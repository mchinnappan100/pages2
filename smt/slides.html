01

Overview — Why Global_Key__c?

✗  The Problem
✗Salesforce record IDs differ between every org — they can never be used cross-org
✗Name is not always unique and may change over time
✗Lookup relationships break on upsert without a stable key
✗No built-in cross-org identifier in standard Salesforce objects
✗Manual ID mapping is error-prone and doesn't scale
✓  The Solution: Global_Key__c
✓Custom External ID field added to every object in the data model
✓Human-readable keys: MFG-001, CUST-002, VEH-004
✓Identical values in source and target org — stable anchor point
✓Used as compositeKey in Load Plan JSON for upsert decisions
✓Enables lookup resolution across parent-child relationships

Load Plan JSON — Structure & Lookup Resolution

load-plan.json — Vehicle__c entry 
{
  "object": "Vehicle__c",

  "compositeKeys": [
    "Global_Key__c"
  ],

  "query": "SELECT
    Global_Key__c, Name, Model__c,
    Manufacturer__r.Global_Key__c,
    VehicleCustomer__r.Global_Key__c
    FROM Vehicle__c",

  "fieldMappings": {
    "Global_Key__c": "Global_Key__c",
    "Name":          "Name",
    "Model__c":      "Model__c",

    "Manufacturer__c": {
      "lookup": {
        "object": "Manufacturer__c",
        "key":    "Global_Key__c",
        "field":  "Manufacturer__r.Global_Key__c"
      }
    },

    "VehicleCustomer__c": {
      "lookup": {
        "object": "VehicleCustomer__c",
        "key":    "Global_Key__c",
        "field":  "VehicleCustomer__r.Global_Key__c"
      }
    }
  }
}
compositeKeys
Defines the External ID field used to determine insert vs. upsert. SMT checks if a record with this key already exists in the target org before deciding the operation.
query (SOQL)
Fetches all required fields from the source org including relationship traversal via dot notation — Manufacturer__r.Global_Key__c — to pull the parent's key.
lookup (nested)
Resolves the foreign key by finding the matching record in the target org using Global_Key__c as the join key. Replaces the raw ID with the correct target ID.
Object order matters
Manufacturer__c and VehicleCustomer__c must be loaded before Vehicle__c. Use sf-load-sequencer-ui to determine the correct array order.

SMT Studio — Migration Workflow

1
Connect Orgs
Add credentials for source org (e.g. af102) and target org (e.g. af200) in SMT Studio's connection manager. Both orgs must have identical schema deployed.
2
Load Load Plan JSON
Import the array of object configurations. SMT reads compositeKeys, SOQL queries, and fieldMappings. Objects are processed in array order.
3
Generate CSV Only
Run "Generate CSV Only" first to preview the extracted data from the source org. Inspect field values and lookups before committing to migration.
4
Start Migration
SMT performs UPSERT on each object in sequence. First run inserts new records into the target org. Lookup fields are resolved using Global_Key__c on each parent.
5
Handle Updates
After modifying records in the source org, re-run SMT. It detects existing Global_Key__c values and issues UPDATE operations on matched records in the target.

Key Takeaways

🔑
Global_Key__c as External ID
A stable, human-readable field (e.g. MFG-001) that uniquely identifies records across any Salesforce org — the foundation of the entire migration strategy.
📋
Declarative Load Plan JSON
A single JSON array defines object order, SOQL queries, field mappings, and nested lookup resolution. Fully version-controllable and repeatable.
🔧
6-Tool Workflow
schema-builder → load-sequencer → plan-generator → sf-utils → smt-studio → sf-data-util covers the complete lifecycle from schema setup to verification.
🔄
Insert & Update Handled
SMT automatically issues INSERT on first run and UPDATE on subsequent runs by detecting existing Global_Key__c values via upsert logic.
✅
Compare to Close the Loop
sf-data-util compare validates source/target parity per object after migration, surfacing any field-level mismatches before sign-off.
🔗
Lookup Resolution
Nested lookup definitions in the Load Plan resolve parent references using Global_Key__c, eliminating the need for manual ID translation.
