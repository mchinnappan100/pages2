Best Git Branching Strategies for Salesforce Development & Deployments

A practical guide to Git branching models that scale with Salesforce metadata, CI/CD pipelines, and sprint-based release management.

Abstract

Salesforce development presents unique challenges for version control due to tightly coupled metadata, org-based environments, and limited rollback options. An effective Git branching strategy is essential to maintain stability, enable parallel development, and support predictable deployments.

This document outlines proven Git branching strategies optimized for Salesforce, aligned with the deployment pipeline: Dev → DevMerge → QA → UAT → PROD.

Why Git Branching Matters in Salesforce

  • Salesforce metadata is shared across teams and features
  • Multiple developers modify the same objects and flows
  • Deployments must map cleanly to environments
  • Hotfixes require fast and safe promotion
  • Auditability and rollback depend on Git history

Common Git Anti-Patterns in Salesforce Projects

❌ Long-lived feature branches merged at the end

❌ Direct commits to main or production branches

❌ One Git branch per org with no promotion flow

❌ Manual cherry-picking between environments

❌ No mapping between Git branches and Salesforce orgs

Recommended Salesforce Git Branching Model

The recommended model uses a small number of long-lived environment branches combined with short-lived feature and hotfix branches. 

flowchart LR
  main[(main / prod)]
  uat[(uat)]
  qa[(qa)]
  devmerge[(devmerge)]

  feature1[feature/*]
  feature2[feature/*]
  hotfix[hotfix/*]

  feature1 --> devmerge
  feature2 --> devmerge

  devmerge --> qa
  qa --> uat
  uat --> main

  hotfix --> main
  hotfix --> qa
  hotfix --> uat

Purpose of Each Branch

feature/*

  • Short-lived branches
  • Mapped to Jira stories
  • Single logical change

devmerge

  • Integration branch
  • Continuous merging
  • CI validation target

qa

  • Sprint-stable branch
  • Regression testing
  • Release candidate

uat

  • Business validation
  • Signed-off builds
  • No direct commits

main (prod)

  • Production source of truth
  • Tagged releases
  • Strict branch protection

Sprint-Based Git Flow

  Dev->>Feature: Create feature branch
  Feature->>DevMerge: Merge via PR
  DevMerge->>QA: Sprint cut
  QA->>UAT: Tested release
  UAT->>Prod: Approved merge

Handling Hotfixes in Salesforce

Hotfixes should be isolated, fast, and safely propagated forward to avoid regression in future releases. 

flowchart LR
  Prod[(main)]
  Hotfix[hotfix/*]
  QA[(qa)]
  UAT[(uat)]

  Prod --> Hotfix
  Hotfix --> Prod
  Hotfix --> QA
  Hotfix --> UAT

Git Best Practices for Salesforce Teams

  • Protect environment branches with PR rules
  • Enforce Apex tests and static analysis in CI
  • Tag every production release
  • Keep feature branches short-lived
  • Align Git branches with Salesforce orgs

Conclusion

Git branching is a foundational decision for Salesforce delivery success. A disciplined, environment-aligned branching strategy enables safe parallel development, predictable deployments, and rapid recovery.

For modern Salesforce teams, Git is not just source control—it is the backbone of release governance.
