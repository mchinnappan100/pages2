⚠ Highest-Impact Security Operation

Changing OWD to Private is one of the highest-impact security changes in Salesforce. It can affect users, integrations, automation, reports, dashboards, Apex code, and Experience Cloud users. A structured approach is mandatory to prevent application outages and data access failures.

01

Perform an Impact Assessment

Identify every object whose OWD will change. For each object, answer:

  • Who currently accesses records they do not own?
  • Which teams rely on visibility through OWD alone?
  • Which integrations query records across ownership boundaries?
  • Which reports and dashboards depend on organization-wide visibility?
  • Which Apex classes use with sharing, without sharing, or inherited sharing?
  • Which Flows, Process Builders, and Approval Processes access records?
  • Which Permission Sets grant View All or Modify All on affected objects?

02

Inventory Current Access Patterns

Use reports and SOQL to understand ownership distribution before making any changes. Baseline data is essential for post-change validation.

Ownership Distribution
-- Understand how records are distributed across owners
SELECT OwnerId, Owner.Name, COUNT(Id) recordCount
FROM Account
GROUP BY OwnerId, Owner.Name
ORDER BY COUNT(Id) DESC
LIMIT 200
Records Owned by Queues
-- Queues do NOT propagate access via role hierarchy
SELECT Id, Name, OwnerId, Owner.Type
FROM Case
WHERE Owner.Type = 'Queue'
Records Owned by Integration / Service Accounts
SELECT OwnerId, Owner.Name, COUNT(Id) cnt
FROM Opportunity
WHERE Owner.Profile.Name LIKE '%Integration%'
   OR Owner.Profile.Name LIKE '%Service%'
GROUP BY OwnerId, Owner.Name
Key Questions
  • Are records owned by queues? Queues do not inherit role hierarchy access.
  • Are records owned by integration users or system service accounts?
  • Are there orphaned records whose owner is inactive or has been deleted?
  • What percentage of records are accessed by users who do not own them?
03

Review & Design Sharing Mechanisms

Before changing OWD, design and implement replacement access. Build sharing first — change OWD last.

Role Hierarchy

Ensures managers inherit access to records owned by their subordinates. Verify the hierarchy covers the intended access paths.

CEO
 ├─ Sales VP
 │   ├─ Sales Manager (East)
 │   │   ├─ Sales Rep 1
 │   │   └─ Sales Rep 2
 │   └─ Sales Manager (West)
 │       ├─ Sales Rep 3
 │       └─ Sales Rep 4
 └─ Support VP
     └─ Support Manager
         └─ Support Agent
Sharing Rules

Criteria-based or ownership-based rules extend access beyond the role hierarchy. Create these before changing OWD to catch gaps during sandbox testing.

  • Share Accounts owned by the Sales East role with Sales Leadership public group
  • Share Cases where Status = 'Escalated' with Support Management
  • Share Opportunities with Amount > 1000000 with Executive Team group

💡 Limit: Up to 300 sharing rules per object. Plan your rule structure to stay within this cap. Criteria-based rules are more flexible but evaluated at record save time.

Account, Opportunity & Case Teams
  • Account Teams — team members gain configurable access (Read, Edit) independent of ownership
  • Opportunity Teams — deal team access for cross-functional collaboration
  • Case Teams — support team visibility for escalated or complex cases
Manual Sharing

Determine whether users rely on the record-level Share button. Manual shares do not automatically re-apply when ownership changes. Audit any workflows that depend on manual share grants.

Apex Managed Sharing

Required for complex access rules that cannot be expressed declaratively. Review all classes that write to __Share objects.

// Grant access via a custom reason — must be defined on the Share object
MyObject__Share share = new MyObject__Share();
share.ParentId      = recordId;
share.UserOrGroupId = userId;
share.AccessLevel   = 'Edit';
share.RowCause      = Schema.MyObject__Share.RowCause.Custom_Reason__c;
insert share;

Audit all trigger/class paths that insert or delete Share records to ensure they still fire correctly after OWD changes. Use Setup → View Sharing on a record to verify applied shares.

04

Review Apex Code

⚠ Most Commonly Missed Area

Apex sharing keywords interact directly with OWD. A change from Public Read Only to Private can silently break SOQL queries that previously returned all records.

Sharing Keyword Behavior Reference
KeywordBehavior with Private OWDAction Required
with sharing Respects OWD — queries return fewer records after change Review all SOQL in these classes carefully
without sharing Bypasses OWD — queries still return all records Verify this is intentional; may now expose too much data
inherited sharing Inherits calling context — behavior depends on call chain Trace all callers; behavior changes based on entry point
(none declared) Defaults to without sharing in most contexts Audit all classes missing an explicit keyword
Example: Query that breaks silently after OWD → Private
// After OWD becomes Private, this query only returns records the
// running user can see — previously it returned ALL accounts.
public with sharing class AccountController {
    @AuraEnabled(cacheable=true)
    public static List<Account> getAllAccounts() {
        return [SELECT Id, Name, OwnerId, Industry FROM Account];
    }
}
Apex Artifacts to Audit
  • Apex controllers backing Visualforce pages and LWC
  • LWC @wire Apex methods and imperative calls
  • Aura controllers (@AuraEnabled methods)
  • Batch Apex — the Database.QueryLocator scope is affected by with sharing
  • Queueable and Schedulable classes that process records
  • Invocable Actions called from Flows
  • Apex test classes — tests may fail if they relied on full visibility

💡 Developer Console Tip: Use System.runAs(targetUser) in Execute Anonymous to verify SOQL result counts from a specific user's perspective before deploying the OWD change.

05

Review Flows & Automation

Flow sharing behavior depends on the run context. This is the second most commonly missed area after Apex code.

Flow Type Context Reference
Flow TypeDefault Run ContextOWD Impact
Record-Triggered Flows System (with record access for trigger record) Get Records for other objects may still be affected
Screen Flows User context Get Records directly affected — user only sees accessible records
Auto-launched Flows (via Process Builder) User context by default Record lookups may return no results post-change
Scheduled Flows System context (automated user) Generally safer, but verify target record access
Validate Every Get Records Element

For every Get Records element querying an object whose OWD changes:

  • Identify the flow's run context (user vs. system mode)
  • Check that fault paths handle zero-record results gracefully
  • Confirm related-record access assumptions (e.g., an Account on an Opportunity)
  • Test using a user persona that does not own the target records

⚠ Flows with Get Records but no fault path will throw null variable errors when the element returns no rows. These failures are often silent until a user hits the affected path in production.

06

Review Reports & Dashboards

Reports and dashboards are data-driven and immediately reflect OWD changes. Establish baselines before the change so discrepancies are clear post-deployment.

Pre-Change Baseline (Required)
  • Export row counts and summary totals from all critical reports
  • Screenshot key dashboards — store in an admin-only folder
  • Document the expected post-change values given the sharing model you designed
High-Risk Report Categories
  • Sales pipeline and quarterly forecast reports
  • Executive dashboards showing portfolio-wide totals
  • Reports scoped to "All Opportunities" or "All Accounts"
  • Cross-object reports joining objects with different OWD settings
  • Reports run by users without View All permissions
Expected Changes After OWD → Private
  • Record counts decrease — users see only owned and shared records
  • Summary totals and rolled-up metrics decrease proportionally
  • Some dashboard components appear empty for users without broad sharing
  • Reports scoped to "My Team" remain unaffected (role hierarchy still works)

💡 Tip: Create a dedicated "OWD Baseline — Pre-Change" report folder shared only with admins. Re-run the same reports post-change and compare the delta against expected sharing model outcomes.

07

Review Integrations

Integration users often rely on implicit full-object visibility. After OWD becomes Private, their result sets may shrink dramatically — often silently breaking data syncs.

The Silent Integration Break
-- Before OWD change: integration user gets ALL accounts
SELECT Id, Name, Type, Industry, OwnerId FROM Account

-- After OWD → Private: returns ONLY accounts owned by the integration user
-- ETL jobs and data warehouses silently sync a fraction of the data
Integration Channels to Audit
  • Connected Apps (OAuth client credentials, JWT flows)
  • Middleware platforms (MuleSoft, Boomi, Informatica, Azure Logic Apps)
  • Scheduled ETL jobs and batch data extracts
  • Agentforce integrations and Data Cloud source connectors
  • Data warehouses and BI tools (Snowflake, Tableau CRM, Power BI)
  • Named Credentials and External Services
Remediation Options for Integration Users
OptionWhen to UseRisk
Permission Set: View All (scoped) Integration needs full read access to specific objects Low if scoped to only required objects
Permission Set: Modify All (scoped) Integration creates or updates records across owners Medium — audit write patterns carefully
Sharing Rules for integration user Integration needs a known subset of records Low — declarative and auditable
Transfer ownership to integration user Integration manages a distinct set of records Medium — impacts reporting and workflows
08

Review Portal / Experience Cloud Access

External users in Experience Cloud sites operate under a different sharing model and are frequently overlooked during internal OWD changes.

External User Types to Review
  • Partner Community Users — portal access based on Account hierarchy; role hierarchy does apply
  • Customer Community Users — typically self-service; limited to records related to their own Contact/Account
  • Customer Community Plus Users — support role hierarchy access within the portal
Sharing Mechanisms for External Users
  • Sharing Sets — grant external users access to records linked to their Account or Contact
  • Share Groups — extend access from an internal group to a portal user group
  • Account Relationships — Partner portal access flows through the Account hierarchy
  • Guest User Sharing Rules — for unauthenticated site pages

⚠ A Private OWD change may remove portal users' access to their own related records if Sharing Sets are not correctly configured. Always test with an actual portal user account in a full sandbox.

10

Test in Full Sandbox

✅ Strongly Recommended: Full Sandbox Only

A Full Sandbox is the only environment that accurately replicates production data volumes, sharing configurations, and integration traffic. Partial or Developer sandboxes may mask issues that only manifest at scale.

Testing Checklist
  • User acceptance testing with all defined personas
  • Integration end-to-end testing with connected external systems
  • Batch Apex jobs — verify Database.QueryLocator scope and record counts
  • Scheduled Apex and Flow jobs — confirm they access the expected records
  • All critical reports and dashboard components
  • Salesforce Mobile app with impacted user accounts
  • Experience Cloud site testing using portal user accounts (not admin)
  • Approval process routing on affected objects
  • List views scoped to team or role-based criteria
Waiting for Recalculation in Sandbox

After changing OWD in sandbox, wait for sharing recalculation to complete before running any access tests. Access results are inconsistent while the recalculation is in progress. Monitor via Setup → Sharing Settings or the Background Jobs queue.

11

Estimate Sharing Recalculation Time

Changing OWD triggers a background sharing recalculation job across all records. This is a critical production planning factor — the org is not fully consistent until recalculation completes.

Factors That Drive Recalculation Time
  • Total record count on affected objects
  • Number and complexity of existing sharing rules
  • Depth of the role hierarchy
  • Number of sharing rules created simultaneously
  • Current platform load and Salesforce infrastructure
Guidance by Data Volume
Record VolumeEstimated DurationRecommendation
< 100K records Minutes Low risk; off-hours preferred but not required
100K – 1M records 30 min – several hours Schedule a maintenance window; notify users
1M – 10M records Hours to a full day Weekend window; user communication required
> 10M records (LDV) Days Engage Salesforce Support; complete LDV review first

⚠ During recalculation, record access is inconsistent. Users may see unexpected "Insufficient Privileges" errors. Plan your maintenance window to allow the recalculation to complete before opening access to users.

Monitoring Recalculation
  • Setup → Sharing Settings — shows pending recalculation status per object
  • Setup → Apex Jobs — background sharing jobs appear in the queue
  • Contact Salesforce Support for LDV guidance before production changes
12

Create a Rollback Plan

Document and rehearse a rollback procedure before touching production. Reverting OWD also triggers a full sharing recalculation — recovery is not instantaneous.

Before the Change — Capture All of These
  • Screenshot of current OWD settings for all affected objects
  • Export of all sharing rules (Setup → Sharing Settings)
  • Current permission set assignments for integration and service users
  • Baseline report snapshots (record counts, totals, dashboard screenshots)
  • List of manual sharing entries on critical records
Rollback Trigger Criteria (Define Before the Change)
  • Critical business process failures within the first hour post-change
  • Integration data sync failures affecting downstream systems or reporting
  • Executive or VP-level users losing access to records required for daily work
  • More than an agreed percentage of active users unable to complete core tasks
Rollback Procedure
1. Navigate to Setup → Sharing Settings
2. Click Edit on the affected object
3. Change OWD back to: Public Read Only
4. Save — sharing recalculation begins immediately
5. Wait for recalculation to complete (monitor Apex Jobs)
6. Validate access with all test personas
7. Document the incident: what broke, when, root cause

💡 Important: Even reverting to Public Read Only requires a full sharing recalculation. Do not expect immediate recovery. Include this recalculation delay in your incident communication plan and SLA expectations.

🔄

Recommended Change Sequence

Follow this sequence to minimize risk. All sharing mechanisms must be in place and validated before the OWD is changed to Private.

flowchart TD A[🔍 Inventory Current Access] --> B[📊 Impact Assessment] B --> C[🏗 Design Sharing Model] C --> D[📋 Create Sharing Rules & Teams] D --> E[🧪 Full Sandbox Testing] E --> F[👥 User Acceptance Testing] F --> G[🔌 Integration Validation] G --> H[📸 Capture Baselines & Rollback Plan] H --> I[⚙ Production OWD Change] I --> J[⏳ Monitor Sharing Recalculation] J --> K[✅ Post-Deployment Validation]

Common Breakages Seen in Real Projects

These are the most frequently observed failures when OWD is changed without thorough preparation:

1 Sales managers suddenly cannot see their team's records — the role hierarchy alone was assumed to be sufficient but was not reviewed.
2 Dashboards show dramatically lower numbers — executive team raises an incident; no pre-change baseline was captured to compare against.
3 Flows fail because Get Records returns no records — null variable errors cascade silently through automated processes until a user hits the affected path.
4 LWC and Aura pages appear empty — Apex controllers annotated with sharing now return empty lists for users who do not own the records.
5 Integration users lose visibility — ETL jobs and middleware pipelines silently sync a fraction of the expected data to downstream systems.
6 Apex tests fail — test data was inserted without Sharing Rules in place; test methods assumed full visibility that no longer exists.
7 Approval processes cannot locate related records — lookup fields on approval step criteria return no value, blocking submissions.
8 Experience Cloud users lose access to their own related records — Sharing Sets were never configured because OWD was Public.
9 Account Team members no longer see related Opportunities — the team members relied on OWD visibility, not the Account Team access level.
10 Custom Apex sharing was never implemented because Public Read masked the requirement — the gap becomes visible only after OWD becomes Private.
🏆

The Golden Rule of OWD Changes

Implement and validate all required sharing mechanisms first, then change OWD to Private as the absolute final step.

This approach ensures access gaps are visible during sandbox testing rather than discovered in production under user pressure. It also makes post-change debugging unambiguous — if access is wrong after the OWD change, the sharing model configuration is the first and only place to investigate.
