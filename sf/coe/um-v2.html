COE Best Practices

Salesforce Persona & Access Governance

A Structured Approach to Scalable Identity and Access Management (IAM)

Executive Summary: This whitepaper outlines a modern, persona-based governance model. By decoupling job functions from technical permissions and leveraging the "Minimum Profile" strategy, organizations can ensure security, simplify auditing, and accelerate user onboarding.

1. Stakeholder Matrix

Stakeholder Core Responsibility
Business Owners Define "Who does what" (Functional requirements and Data Ownership).
Solution Architects Map Personas to technical metadata (Permission Set Groups, Sharing Rules).
Security/IAM Team Manage Identity Lifecycle (AD/Okta), SSO, and MFA enforcement.
DevOps / COE Automate deployment, monitor "Permission Creep," and manage CI/CD.

2. The "Agile" Persona Model

Modern Salesforce security moves away from monolithic Profiles. We implement a three-tier architecture:

  • The Persona (Business Layer): Logical role, e.g., "Field Sales Rep."
  • Permission Set Group (Technical Layer): Bundles all functional permissions required for the persona.
  • Muting Permission Sets (Exception Layer): Used to strip specific permissions from a group without creating unique metadata for sub-roles.

3. Technical Mapping & Security Hierarchy

To align with Salesforce's roadmap for Profile retirement, the following hierarchy is enforced:

  • Minimum Profiles: Used strictly for system settings (Login Hours, IP Ranges, Page Layout assignments).
  • Permission Set Groups (PSG): The primary engine for Object, Field (FLS), and Apex access.
  • Sharing Model: Personas are mapped to Public Groups and Roles to control record-level visibility (OWDs).

4. Enhanced Automated Provisioning Flow

Step 1: User is added to a specific Active Directory Group (e.g., SF_Sales_US).

Step 2: Okta/IdP passes the User.Persona attribute via SAML assertion.

Step 3: Salesforce User Access Policies (Native) automatically detect the attribute and assign:

  • The specific Permission Set Group.
  • Correct Public Groups and Queues.
  • The designated Role in the hierarchy.

Step 4: Just-in-Time (JIT) Provisioning ensures the user is "Ready to Work" upon their first login.

5. Change Management & Governance

When a business process evolves, the following lifecycle is followed:

  1. Requirement: Business Owner submits a request for new capability.
  2. Impact Analysis: Architect determines if it's a "Core Persona" change or a "Modular Add-on."
  3. Development: Permissions updated in Sandbox; Muting sets applied if necessary.
  4. Audit: Quarterly User Access Reviews (UAR) identify and remove "orphan" permissions.

6. Best Practices

  • 80/20 Rule: Aim for 80% of permissions to be standard in the PSG, with 20% handled via modular "Add-on" Permission Sets.
  • Naming Conventions: Standardize across systems: [System]_[Region]_[Function].
  • No Permission on Profiles: Transition all Object/Field permissions to Permission Sets immediately to avoid technical debt.

7. Conclusion

By shifting to a persona-driven model supported by automated User Access Policies, organizations reduce the risk of data breaches and significantly lower the operational overhead of managing complex Salesforce environments.
