Section 00

Overview & Why You Need It

Salesforce replicates your live environment in real-time — which means any record you delete (accidentally or intentionally) is immediately gone. The platform's built-in tools like weekly Data Export are limited, manual, and often insufficient for serious data protection.

⚠️
Critical Stat: According to the 2024 State of Salesforce DevOps Report, 65% of teams experienced data loss last year. The issue isn't just losing data — it's discovering you can't get it back quickly enough.
☁️
Independent Platform
OwnBackup is a separate web application — not embedded in Salesforce. If Salesforce goes down, your backups remain accessible.
🔄
Daily Automated Backups
Choose your exact backup window. Incremental daily backups plus scheduled full-backup days. On-demand backups anytime.
🔍
Precision Repair Tool
Visual diff tool lets you compare data between backup snapshots and restore individual records — not just full-org rollbacks.
🛡️
Metadata Backup
Backs up page layouts, Apex classes, custom objects and configurations — everything that makes your org yours.
📊
Smart Alerts
Proactively monitor data changes. Get notified of unusual deletions, mass updates, or suspicious activity.
🌱
Sandbox Seeding
Clone production data into sandbox environments for development and testing without compromising live data.

Prerequisites & Planning

Before You Start

What You'll Need

✓
System Administrator profile in Salesforce — required for package installation and permission set assignment
✓
OwnBackup subscription — contact your Salesforce Account Executive. Pricing is volume-transparent (not data-volume-based, so no surprise costs)
✓
List of critical objects — identify which standard and custom objects you need protected before setup
✓
Data residency requirements — know your AWS region preference (US, EU, APAC) for storage compliance
✓
Backup window preference — off-peak hours reduce API load on your production org
💡
Planning Tip: Before setup, categorize your objects as High / Medium / Low priority. High = customer records, opportunities, contracts. This shapes your backup policy and recovery time objectives (RTO).
Section 02

Purchase & Install OwnBackup

Step 1

Get OwnBackup from AppExchange

OwnBackup (now branded as "Own") is available on the Salesforce AppExchange. You can start with a free trial.

1
Navigate to appexchange.salesforce.com and search for "OwnBackup" or "Own Data Backup"
2
Click "Get It Now" to start a free trial, or contact your Salesforce AE for a paid subscription quote
3
Select your Salesforce org to install into. Choose "Install for Admins Only" on the Managed Package page
4
When prompted for third-party access, select "Yes, grant access to these third-party web sites", then click Continue
5
Wait for the email confirmation. Package installation typically takes 5–20 minutes
⚠️
Important: If purchasing via Salesforce subscription order, your installation link arrives via email 24–48 hours after your AE processes the order. Use that URL directly rather than AppExchange.
Section 03

Connect Your Salesforce Org

Step 2

Initial Setup & Region Provisioning

After installation, launch OwnBackup and complete the initial org connection setup.

1
In Salesforce, open the App Launcher and search for "OwnBackup" to open the app
2
Click the Settings tab. In the "Provision a Region" section, select your preferred AWS region from the dropdown (consider data residency and GDPR/HIPAA requirements)
3
Alternatively, log in to the OwnBackup web portal at app.ownbackup.com (separate from Salesforce) using your Salesforce credentials via OAuth
4
Click "Add Service" (Services = connections to backup targets). Select Salesforce as the service type
5
Authenticate with OAuth. OwnBackup will request read/write access — required for both backup and restore operations
6
Your org will appear as an active Service in the dashboard. Repeat for additional orgs (production, sandboxes, developer editions)
✅
Multi-Org Support: OwnBackup provides centralized access to backups for multiple orgs from a single dashboard — ideal for organizations with production + multiple sandbox environments.
Section 04

Configure Backup Policies

Step 3

Set Up Backup Schedule & Object Selection

Define what gets backed up, when, and how often. This is the heart of your data protection strategy.

1
In the OwnBackup portal, navigate to your connected service and click Backup Settings
2
Set your daily backup window — choose off-peak hours (e.g., 2:00 AM local time) to minimize API governor limit impact
3
Designate a Full Backup Day (e.g., Sunday). On all other days, OwnBackup performs incremental backups of changed records only
4
Under Object Selection, choose which standard and custom objects to include. Recommended: start with all objects, then exclude low-priority ones
5
Configure Retention Policy — set how many days/versions of backup to retain per service (for compliance: typically 90–365 days)
6
Optionally configure Third-party Export Targets: MySQL, Snowflake, AWS S3, or Azure Blob for additional offsite backup copies
// Recommended Object Priority List — include in your initial backup policy HIGH PRIORITY (back up daily): → Account, Contact, Lead, Opportunity, Case, Contract → Custom financial or PII objects → Junction/relationship objects MEDIUM PRIORITY: → Activity/Task/Event histories → Custom operational objects → ContentDocument / ContentVersion (files & attachments) LOW PRIORITY (weekly or on-demand): → Reference/lookup tables → Archived/read-only records
💡
On-Demand Backups: You can trigger a manual backup at any time — by specific objects, or a full org backup. Useful before major deployments, data migrations, or bulk operations.
Section 05

Set Permissions & User Access

Step 4

Assign Licenses & Permission Sets

Any user who needs to manage or use Backup & Restore requires proper license and permission set assignments.

1
In Salesforce Setup, go to Users → [Select User] → Permission Set License Assignments. Click Edit and assign "Backup and Restore User Permission Set License"
2
On the same user record, go to Permission Set Assignments → Edit Assignments. Find "BackupRestore" in Available Permission Sets, click Add, then Save
3
Navigate to Setup → Permission Sets → BackupRestore → Object Settings. For each object the user needs to back up or restore, click Edit and assign the appropriate Read/Modify permissions
4
Repeat Step 3 for every object in your backup policy. This is required — missing object permissions will silently exclude records from backups
⚠️
Common Mistake: Object permissions must be set individually per object in the BackupRestore permission set. Skipping this for custom objects is one of the most frequent setup errors — the backup will run but silently miss those objects.
Section 06

Run Your First Backup

Step 5

Trigger & Validate Initial Backup

Don't wait for the scheduled backup — run an immediate full backup to validate your setup and establish a baseline.

1
In OwnBackup portal, navigate to your Service (connected Salesforce org)
2
Click "Backup Now" and choose Full Backup for the first run. This captures your entire org as a baseline snapshot
3
Monitor progress in the Activity Log tab — view status, object counts, successes, failures, and error messages per job
4
Once complete, verify record counts in the Backup tab. Compare against your Salesforce object record counts to confirm completeness
5
Check the Compare report to see a snapshot of all backed-up objects and their record counts at this point in time
✅
Validation Step: After your first backup, do a spot-check restore of a non-critical record to confirm the full pipeline works — backup → compare → restore — before you need it in an emergency.
Section 07

Configure Smart Alerts

Step 6

Set Up Proactive Monitoring

Smart Alerts notify you when unusual data activity is detected — mass deletions, bulk updates, or suspicious changes between backups.

1
In the OwnBackup portal, navigate to Alerts → Smart Alerts
2
Click "Create Alert". Select the Service (org) and the Object to monitor (e.g., Account, Contact, Opportunity)
3
Set alert conditions: trigger when record deletions exceed X%, or when a field changes by more than a threshold
4
Configure notification recipients (email addresses) and choose alert frequency
5
Recommended alerts to set up: Account deletions >5%, Opportunity stage mass changes, Contact field blanking
Section 08

Restoring Data

Step 7

Compare & Restore Records

OwnBackup offers two restore paths: full object restore (bulk) and Precision Repair for surgical record-level recovery.

1
Navigate to the Backup tab and select the backup snapshot from the date/time you want to restore from
2
Click "Compare" to generate a comparison report between the selected snapshot and your current org data. This shows exactly what changed, was deleted, or was modified
3
For Precision Repair: click the object name in the Compare report. Use the visual tool to filter records by date range, field value, or change type
4
Apply AND/OR filters to narrow down to only the affected records (e.g., Cases where ContactId was accidentally removed)
5
Select the records to restore and click "Add to Restore Bucket". Review the restore bucket, then click "Preview Restore"
6
OwnBackup will flag any dependency warnings — related objects that also need configuration before data can be accepted back into the org
7
Click "Restore" to complete. Monitor progress in the Activity Log. You'll receive an email confirmation when done
⚠️
Dependency Awareness: OwnBackup will alert you if related objects need to be restored first. For example, restoring Opportunities requires that associated Accounts exist. Always review dependency warnings before confirming.
Section 09

Metadata Backup

Step 8

Protect Your Org's Configuration

Data without metadata is incomplete. Metadata includes page layouts, Apex classes, flows, custom fields, and validation rules — the structure of your org.

1
In OwnBackup portal, navigate to your Service and go to the Metadata tab
2
Click "Configure Metadata Backup". Select metadata component types to include: CustomObjects, ApexClasses, Flows, Layouts, ValidationRules, Profiles, etc.
3
Set a metadata backup schedule (daily recommended) — this is separate from your data backup schedule
4
Use the Metadata Compare view to see exactly which components changed between any two backup points
5
To restore metadata, select the component(s) from a snapshot, review the diff, and click Restore Component
💡
Why Metadata Matters: Native Salesforce Backup (the paid add-on) backs up records, files, and attachments but skips metadata entirely. It's like saving every page from a filing cabinet but losing the folder structure. OwnBackup protects both.
Section 10

Sandbox Seeding

Step 9

Clone Production Data to Sandbox

Use backup snapshots to populate sandbox environments with realistic (but optionally masked) production data for development and testing.

1
In OwnBackup, connect your sandbox org as an additional Service (same process as Step 3, but select sandbox login)
2
Navigate to Seeding tab. Click "New Seed". Choose the production backup snapshot as the Source
3
Select your sandbox org as the Target. Choose objects to include in the seed
4
Optionally configure Data Masking — scramble PII fields (email, phone, name) before seeding into sandbox for compliance
5
Click "Run Seed". Monitor progress in Activity Log. Sandbox will be populated with production-representative data

Section 12

Best Practices & Compliance

🔒 Security Best Practices

→
Keep OwnBackup access credentials separate from Salesforce user accounts where possible
→
Restrict restore permissions to a small group of trusted admins — use separate permission sets
→
Enable MFA on the OwnBackup web portal login
→
Audit the Activity Log monthly for unexpected backup failures or restore events

📋 Compliance & Retention

→
Set retention to 90+ days for most industries; 7 years for regulated data (finance, healthcare)
→
For GDPR: configure data masking before sandbox seeding; document backup scope in your ROPA
→
For HIPAA: choose US-region storage; verify Business Associate Agreement (BAA) with Own
→
Test restore procedures quarterly — document recovery times for your RTO/RPO metrics
