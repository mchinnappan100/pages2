Amazon Aurora Architecture

Overview

Amazon Aurora is AWS's cloud-native relational database, fully compatible with MySQL and PostgreSQL. It delivers up to 5× the throughput of standard MySQL and 3× that of PostgreSQL by decoupling compute from storage and using innovative techniques like redo-log shipping and quorum-based durability.

The core design ships only redo logs (not full data pages) to a distributed, fault-tolerant storage layer. This reduces network I/O, enables fast crash recovery (often <30 seconds), and supports massive scale with shared storage.

Data is divided into 10 GB segments (Protection Groups), each replicated six ways across three AZs (two copies per AZ). Aurora uses redo-log-based writes: only small log records are sent to storage (not full pages), appended to logs, and materialized on-demand. This is the key to high write throughput.

Quorum Rules & Fault Tolerance

  • Writes are durable after 4 out of 6 copies acknowledge (Vw=4, V=6).
  • Reads require 3 out of 6 copies (Vr=3) to guarantee seeing the latest version.
  • Tolerates full AZ loss (2 copies) + 1 additional fault without data loss or write availability impact.
  • Fast repairs: Small segments rebuild in <1 minute on 10 Gbit networks; MTTR is minutes.
  • Self-healing: Continuous scanning detects/replaces bad blocks; membership changes handle permanent failures.

"In Aurora, we use a six-way quorum spread across three AZs, with a write set of four and a read set of three." — AWS Aurora Under the Hood (2017)

Key Features & Innovations

Compute-Storage Separation + Redo Log Shipping

Only redo logs sent to storage → 5× MySQL / 3× PostgreSQL throughput; fast recovery (no replay of data pages).

Quorum Durability & Availability

6 copies / 3 AZs; AZ+1 fault tolerance; 99.99% availability SLA; self-healing storage.

Read Scaling

Up to 15 same-region replicas + 16 per secondary region in Global DB; no replication lag for reads.

Continuous Backup & PITR

Incremental backups to S3; point-in-time recovery to seconds; no performance hit.

I/O-Optimized & Serverless v2

Optimized pricing for high-I/O; auto-scaling from 0 to massive throughput.

Limitless Database

Scale-out sharding for virtually unlimited capacity (preview/GA features as of 2025+).
