Avoid Merge Conflicts on Shared Metadata

Custom Labels, Permission Sets, and shared Apex classes are merge-conflict hotspots. A disciplined ownership model and branching strategy eliminate most conflicts before they happen.

Custom LabelsPermission SetsShared ApexGit Strategy
  • One PR owner per shared file. Designate a metadata steward. Only they merge changes to shared files (CustomLabels.labels, *.permissionset-meta.xml).
  • Prefix Custom Labels with team/feature names (e.g. BILLING_ErrorMsg_001) so they never collide across teams.
  • Decompose Permission Sets. Use Permission Set Groups; each team owns granular PSets assembled — never all editing one giant file.
  • CODEOWNERS file in GitHub/GitLab so shared utility classes require a senior review before merge.
  • Short-lived feature branches. Never let a branch live longer than 2–3 days. Rebase daily against main.
# .github/CODEOWNERS force-app/main/default/labels/ @team-lead @architect force-app/main/default/permissionsets/ @security-lead force-app/main/default/classes/Utils* @platform-team

Run PMD Scans Inside VS Code

Catch Apex anti-patterns, SOQL in loops, and security violations while you type — not at code review. Install the Apex PMD VS Code extension and configure a shared ruleset.

  • Install mohanChinnappan.apex-pmd-code-scanner — runs on every file save.
  • Commit a ruleset.xml to your repo root so every developer uses the same rules.
  • Enforce in CI — run PMD as a pipeline stage; fail the build on critical violations.
  • Top rules: AvoidSoqlInLoops, AvoidDmlStatementsInLoops, ApexCRUDViolation.
// .vscode/settings.json { "apexPMD.runOnFileSave": true, "apexPMD.runOnFileOpen": true, "apexPMD.priorityErrorThreshold": "1" }

Test-Driven Development for Apex

Write test classes alongside — ideally before — writing the production class. TDD reduces regressions, enforces clear method contracts, and makes deployments predictable.

  • One test class per Apex class, named MyClass_Test.cls, committed in the same PR.
  • Aim for 90%+ coverage with meaningful assertions, not just line hits.
  • Use @TestSetup for shared data factories — fast and isolated.
  • Mock callouts with HttpCalloutMock so tests never hit live endpoints.
@isTest private class OrderService_Test { @TestSetup static void setup() { TestFactory.createAccounts(5); } @isTest static void calcDiscount_bulk() { Test.startTest(); OrderService.applyDiscount(orders); Test.stopTest(); System.assertEquals(10, result); } }

Document Every Manual Deployment Step

Deployments that rely on unwritten tribal knowledge are time bombs. Every manual step — pre and post-deploy — must be captured in a versioned runbook before the PR is marked ready.

  • Pre-deploy: sandbox validation, data backups, freeze windows, stakeholder notification.
  • Step-by-step deploy order when metadata has dependencies (fields before layouts).
  • Post-deploy: smoke-test stories, restart scheduled jobs, re-enable paused Flows.
  • Rollback plan with exact steps and named owners — not just "revert the deploy."
  • Store in Git at /docs/runbooks/YYYY-MM-DD-feature.md, versioned with the code.
RunbookPre-DeployPost-DeployRollback

Scratch Orgs for Every Feature

Shared sandboxes breed "works on my machine" bugs and blocked teams. Scratch orgs give every developer a clean, reproducible environment spun up in under 2 minutes.

  • project-scratch-def.json in version control with all features and org preferences.
  • Seed scripts load test data automatically after org creation.
  • One-command setup via Makefile: create, push source, run seeds.
  • Max 7-day expiry — forces fresh environments, prevents config drift.
# Spin up in one command sf org create scratch \ --definition-file config/project-scratch-def.json \ --alias feat --duration-days 7 --set-default sf project deploy start sf apex run --file scripts/apex/seed.apex

Externalise Config with Custom Metadata

Hard-coded URLs, thresholds, and flags in Apex are a deployment and security risk. Use Custom Metadata Types and Named Credentials to make config deployable and environment-aware.

  • Custom Metadata Types for feature flags, batch sizes, timeout values.
  • Named Credentials for all external endpoints — never hard-code URLs or secrets.
  • Environment-specific records (Dev/UAT/Prod) deployed via metadata, not manual setup.
Custom MetadataNamed CredentialsFeature Flags

👀

Lightweight, High-Signal Code Reviews

Slow reviews kill velocity. Establish a PR template and a 24-hour SLA so reviews are fast, focused, and never block the team.

  • PR template enforces: what changed, how to test, ticket link, runbook if needed.
  • Max 400 lines per PR — smaller PRs get reviewed 3x faster with fewer bugs.
  • Automate the boring checks (formatting, PMD, coverage) so reviewers focus on logic.
  • 24h SLA with Slack notifications linked to PR status.
PR Template24h SLA400-line limit

Proactive Org Health Monitoring

Don't discover governor limits or technical debt at deployment time. Continuous org health monitoring surfaces issues weeks before they become blockers.

  • Salesforce Optimizer and Health Check run monthly; results tracked in a team dashboard.
  • Apex Exception Email alerts routed to a shared Slack channel.
  • Track SOQL/DML governor limit usage via debug logs.
  • Flow/Process Builder inventory — know what auto-fires on each object.

Living Architecture Docs & Team Rituals

Velocity compounds when knowledge spreads. A weekly guild sync, an ADR log, and a searchable internal wiki turn individual expertise into team capability.

  • ADRs in /docs/adr/ — why decisions were made, not just what.
  • Weekly 30-min dev guild: one person demos a tip they used that week.
  • Shared VS Code snippet library for boilerplate Apex, SOQL stubs, test factories.
  • Onboarding checklist — new developer first commit within Day 1.
  • Retrospectives after every deployment: what slowed us down?

Installation — 4 Steps

Org-Wide PMD Scanner:
Salesforce Code Analyzer

The official Salesforce CLI plugin — scans your entire org's Apex, LWC, Flows, and Visualforce using PMD, ESLint, RetireJS and the Flow Scanner engine. Required for AppExchange security review. Install from npmjs.com ↗.

🛠️

Install & Run Your First Scan

# 1. Install Salesforce CLI npm install -g @salesforce/cli # 2. Install Code Analyzer plugin (v5) sf plugins install code-analyzer # 3. Verify installation sf code-analyzer --version # 4. Scan entire project (all engines) sf code-analyzer run \ --workspace ./force-app \ --output-file results.html # 5. Security rules only (fast) sf code-analyzer run \ --workspace ./force-app \ --rule-selector Security \ --output-file sec-report.html # 6. SARIF output for GitHub Code Scanning sf code-analyzer run \ --workspace ./force-app \ --output-file results.sarif.json
📊

Scan Engines Included

  • PMD — Apex and Visualforce static analysis. Catches SOQL in loops, empty catch blocks, dead code, naming violations.
  • ESLint — JavaScript / LWC best practices, accessibility and security rules.
  • RetireJS — Detects outdated JS libraries with known CVEs in your static resources.
  • Flow Scanner — Analyses Salesforce Flows for DML in loops, missing null checks, security gaps.
  • CPD (Copy-Paste Detector) — Finds duplicated code blocks across your entire org codebase.
  • Regex Engine — Define custom rules with regular expressions for org-specific standards.

Org Health Monitor:
salesforce-org-health

A CLI utility from npmjs.com ↗ that connects to your org and generates a comprehensive health report covering governor limits, metadata inventory, test coverage, active automations, and failed jobs — from your terminal in seconds.

⚡

Install & Generate a Report

# Install globally npm install -g salesforce-org-health # Authenticate your org first sf org login web --alias my-org # Run full health check sf-org-health --target-org my-org # HTML report (shareable) sf-org-health --target-org my-org \ --format html \ --output-file health.html # JSON output (for dashboards) sf-org-health --target-org my-org \ --format json \ --output-file health.json # Coverage check only sf-org-health --target-org my-org \ --check coverage
npm InstallHTML + JSONCI Friendly
🏥

What the Report Covers

  • Apex Test Coverage — org-wide % and per-class breakdown; flags any class below 75%.
  • Governor Limit Usage — Data Storage, File Storage, Daily API Requests, Concurrent Apex.
  • Metadata Inventory — count of Flows, Triggers, Apex Classes, VF Pages, Custom Objects.
  • Active Flows & Process Builders — lists automations per object; flags objects with 3+ running.
  • Scheduled & Async Jobs — all scheduled Apex, batch, and queueable jobs in the org.
  • Failed Async Jobs — surfaces recently failed batch jobs with error messages.
  • Security Health Check Score — Salesforce Health Check score and failed baselines.
