Salesforce Best Practices

Making effective use of Description and Help Text in Salesforce

Why this matters

Descriptions and Help Text form Salesforce’s built‑in documentation layer. When written well, they reduce user confusion, improve data quality, speed up onboarding, and save admins and developers countless hours.

Description vs Help Text

Description (Admin & Developer‑focused)

Use Description to explain why a field exists and how it is used internally.

  • Business purpose of the field
  • Source of data (manual, formula, integration)
  • Automation, reports, or APIs that depend on it
  • Important constraints or caveats

Help Text (User‑focused)

Help Text should guide users at the moment of data entry and prevent mistakes.

  • Short and actionable (1–2 sentences)
  • Written in plain business language
  • Includes examples or valid ranges when helpful
  • Avoids technical or internal terminology

How to write good Help Text

  • Tell the user exactly what to enter.
  • Call out formats, limits, or validation rules.
  • Explain impact only if it affects the user’s decision.

Example:

Enter a value between 0 and 30. Discounts above 30% require manager approval.

How to write strong Descriptions

Descriptions should help someone understand the field without needing to ask questions.

Purpose: Stores the external order ID from SAP
Source: Populated by nightly SAP integration
Used by: Billing reconciliation reports and order sync jobs
Notes: Must not be edited manually

Golden rules

  1. No custom field should exist without a Description.
  2. All user‑facing fields should include Help Text.
  3. If users ask questions, improve the Help Text.
  4. If admins are confused, improve the Description.
