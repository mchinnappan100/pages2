Release Gate Overview

The Release Gate is a structured set of criteria that must be satisfied before any sprint's work can be promoted from the UAT environment to Production. Everything described here uses JIRA out-of-the-box features — no scripts, plugins, or custom code are required.

02

JIRA Hierarchy & Label Strategy

The solution relies on JIRA's standard hierarchy and the Labels field to track feature ownership and release readiness on Stories.

Issue Hierarchy

Epic and Sprint are orthogonal — not parent/child. An Epic groups work by feature/theme; a Sprint groups work by timebox. A Story belongs to both simultaneously.

WHAT dimension — Epic
Epic → feature / theme
  ├── Story
  │      └── Sub-task
  ├── Task
  │      └── Sub-task
  └── Bug ← P1/P2/P3/P4
WHEN dimension — Sprint
Sprint → timebox
  ├── Story ← same Story as above
  ├── Task
  └── Bug ← same Bug as above

# Bug sits under Epic AND in Sprint
85% Gate uses Sprint dimension

Count Done Stories & Tasks within the Sprint regardless of which Epic they belong to.

Feature % uses Epic dimension

Count Done Stories per Epic using Epic Burndown to check if individual features are release-ready.

Label Taxonomy

Apply consistent labels on Stories and Bugs so JIRA's built-in filters and gadgets can surface them accurately in dashboards.

Label Applied To Purpose
feature-<name> Stories Associates a story with a feature for completion tracking
P1 Bugs Critical — system down, data loss. Hard blocks release.
P2 Bugs Major functionality broken. Hard blocks release.
P3 Bugs Minor impact / workaround available. Max 10 per sprint.
P4 Bugs Cosmetic / enhancement. Max 15 per sprint.
uat-approved Stories UAT sign-off received — story counts as Done for gate
uat-blocked Stories Blocked in UAT — flags for release manager attention
ℹ️

Pro tip: Pre-create all labels in Project Settings → Labels and use a JIRA Field Configuration Scheme to make the Label field mandatory on Bug issue types.

03

Story Completion Gate — 85% Threshold

JIRA's native Burndown Chart, Sprint Report, and Pie Chart gadget give you all the visibility needed to enforce the 85% completion rule — no custom code required.

📊 Story Completion — Sprint 42
PASS
17 of 20 Stories Done 85%

Threshold: ≥85% → Gate Passes  |  3 stories rolled over to next sprint

📊 Story Completion — Sprint 43
BLOCKED
14 of 22 Stories Done 64%

Below threshold — sprint cannot be promoted to PROD

How to Track This in JIRA (No Code)

1
Use the Sprint Report

Navigate to Reports → Sprint Report. JIRA automatically shows completed vs. incomplete stories as story points or issue count. The "Completed Issues" section is your numerator; total sprint issues is your denominator.

2
Add a Pie Chart Gadget to the Dashboard

In your JIRA Dashboard, add the Pie Chart gadget. Set the Statistic Type to Status, filter by Sprint and Issue Type = Story. This gives a live visual of Done vs. In Progress vs. To Do.

3
Epic Burndown for Feature-Level Completion

Use Reports → Epic Burndown to see completion percentage per Epic (Feature). This shows whether a specific feature has met its 85% threshold — useful when releasing only some features.

4
Two Dimensional Filter Gadget

Add the Two Dimensional Filter Statistics gadget to your dashboard. Set X-axis = Status, Y-axis = Label (feature-name). This gives a per-feature completion matrix at a glance.

⚠️

Only count Stories with label uat-approved as truly "Done" for the gate. Stories in "Done" status without UAT sign-off should not count toward the 85%.

JQL Filters to Save in JIRA

Create these as Saved Filters in JIRA (Issues → Manage Filters → Create Filter). They power your dashboard gadgets.

-- P1 Bugs in Active Sprint (should always be 0)
project = MYPROJ AND issuetype = Bug AND labels = P1
AND sprint in openSprints() AND status != Done
-- P2 Bugs in Active Sprint (should always be 0)
project = MYPROJ AND issuetype = Bug AND labels = P2
AND sprint in openSprints() AND status != Done
-- P3 Bugs in Active Sprint (monitor count: allow ≤ 10)
project = MYPROJ AND issuetype = Bug AND labels = P3
AND sprint in openSprints() AND status != Done
-- P4 Bugs in Active Sprint (monitor count: allow ≤ 15)
project = MYPROJ AND issuetype = Bug AND labels = P4
AND sprint in openSprints() AND status != Done
💡

For closed sprints, replace openSprints() with closedSprints() or use sprint = "Sprint 42" to audit a specific sprint retroactively.

06

Release Process & Workflow

The following workflow integrates the gate checks into the sprint lifecycle with clear ownership and ceremony touchpoints.

1
Sprint Planning — Label All Stories

During Sprint Planning, the Scrum Master ensures every Story has a feature-<name> label applied and is linked to an Epic. Bugs created during planning are labelled P1–P4 by the QA lead.

2
During Sprint — Bug Triage Daily

New bugs are labelled at creation. The Release Gate Dashboard is reviewed each daily standup. P1/P2 bugs trigger immediate escalation. P3/P4 counts are tracked against thresholds.

3
UAT Entry — Stories move to UAT column

When stories are ready for UAT, they are moved to the "UAT" status in the board. Testers apply uat-approved label on passing stories and uat-blocked on failing ones.

4
Pre-Release Gate Check (Sprint End - 1 day)

Release Manager opens the Release Gate Dashboard and evaluates all five conditions. The JIRA Sprint Report is used to confirm ≥85% story completion. P1/P2 filters must show zero results. P3/P4 counts are confirmed within tolerance.

5
Go / No-Go Decision

Release Manager records the go/no-go decision as a comment on the Sprint's Release Issue (create a JIRA issue type "Release" for tracking). Screenshots of dashboard gadgets are attached as evidence.

6
Post-Sprint — Retrospective Metric Review

Use the Sprint Report and Velocity Chart to trend completion rates. Bug counts per priority are tracked sprint-over-sprint via saved filter history to identify quality trends.
