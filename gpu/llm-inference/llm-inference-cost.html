Economic Symptoms Mask Physical Laws

Every pricing quirk you see in LLM APIs is a hardware constraint in disguise

The pricing of every major LLM API looks arbitrary on the surface — until you understand the physics. Each pricing tier, discount, and premium is a direct manifestation of a specific hardware constraint. The economics are not business decisions; they are engineering realities expressed in dollars.

Economic Symptom

API providers offer a 6× priced Fast Mode — pay a significant premium for lower latency responses.

Physical Reality

Dictated by Batch Size Amortization — fast mode means smaller batches, so the weight fetch cost is divided by fewer tokens.

Economic Symptom

Providers charge a 50% premium for >200k context lengths — long context costs significantly more per token.

Physical Reality

Dictated by Memory Bandwidth Limits — a larger KV cache must be fetched from HBM on every decode step, consuming precious bandwidth.

Economic Symptom

Massive discounts are offered for 5-minute and 1-hour cache hits — if your context was used recently, you pay far less.

Physical Reality

Dictated by Drain Times of HBM vs. DDR Storage — keeping data in HBM costs a fortune; moving it to DRAM or disk is cheap but slow.

The Core Principle

From rack cable densities to API pricing, the frontier of AI is entirely dictated by the physical cost of moving data. Understanding the physics means you can predict any future pricing decision before it's announced.

02

The Fundamental Equation of Inference

Runtime = max(compute time, memory time) — you are always bottlenecked by one of two things

Every inference request runs on two parallel clocks. The compute clock ticks based on how many multiplications the GPU must perform. The memory clock ticks based on how much data must be moved from HBM into the chip's processing cores. Your response arrives when the slower clock finishes.

Runtime = MAX( T_compute , T_memory ) T_compute = (Batch Size × Active Parameters) / FLOPs ↑ pure hardware arithmetic throughput T_memory = (Total Weights + KV Cache) / Memory Bandwidth ↑ time to physically move data into the chip You are ALWAYS either: Compute-bound → more FLOPs would help Memory-bound → more bandwidth would help ← most LLM inference
🔢

T_compute — Arithmetic Bound

How long it takes the GPU's multiply-accumulate units to process the arithmetic. Scales with batch size and parameter count. Rare for single-user LLM inference.

  • Increases linearly with batch size
  • Increases with active parameter count (MoE helps here)
  • Fixed by peak FLOP/s of the GPU
  • Dominant only at large batch sizes
💾

T_memory — Bandwidth Bound

How long it takes to stream all model weights and the KV cache from HBM into the compute cores. The dominant bottleneck for virtually all real-world LLM inference.

  • Weight fetch: same cost regardless of batch size
  • KV cache: grows linearly with context length
  • Fixed by HBM bandwidth (e.g. 3.35 TB/s on H100)
  • Dominant at small batch sizes and long contexts
Why LLM Inference is Almost Always Memory-Bound

A Llama-3 70B model has 140 GB of weights in FP16. Fetching them from H100 HBM takes ~42ms. For a single user request, the GPU does a trivial amount of arithmetic — far less than 42ms worth. So the bottleneck is the weight fetch, not the computation. You are paying for data movement, not arithmetic.

The Roofline & Cost Parabola

Why there is exactly one optimal batch size — and how it shapes every pricing decision

When you plot cost-per-token against batch size, a characteristic parabolic shape emerges. This is not an accident — it is the mathematical consequence of the fundamental equation. Understanding the shape tells you exactly why "fast mode" costs 6× more.

The Cost Parabola Cost per token ↑ │ Memory-bound region │ (small batch: weight fetch Compute-bound region │ amortised over few tokens) (large batch: arithmetic │╲ time dominates) │ ╲ │ ╲ ◆ Balance Point / │ ╲ (memory time = / │ ╲__compute time)__________________/ │ └────────────────────────────────────────→ Batch Size Left side (low batch): Cost dominated by amortised memory (weight fetch) Balance point: Minimum cost — T_memory = T_compute Right side (high batch): Cost dominated by compute (arithmetic time grows)

Why "Fast Mode" Costs 6× More

Fast mode = small batch. Your request is processed immediately without waiting for other users' requests to accumulate. But the cost of fetching all model weights from HBM is shared by very few tokens — so the per-token cost is far higher. At batch size 1, you pay 100% of the weight fetch cost for your single token. At batch size 300, that same weight fetch is split 300 ways.

Latency vs. Cost — The Fundamental Trade-off

  • Low latency (small batch): high cost per token, immediate response
  • Low cost (large batch): queuing latency, cheaper per token
  • No free lunch: the physics of the memory wall makes both simultaneously impossible
Python · Roofline cost model
import numpy as np

# H100 SXM5 hardware constants
FLOPS_PER_SEC  = 989e12       # 989 TFLOPS FP16 (Tensor Core peak)
MEM_BW_BYTES   = 3.35e12      # 3.35 TB/s HBM3 bandwidth

# Model constants (70B FP16)
PARAMS         = 70e9          # 70 billion parameters
BYTES_PER_PARAM = 2            # FP16 = 2 bytes per parameter
WEIGHT_BYTES   = PARAMS * BYTES_PER_PARAM   # = 140 GB total weight size

# KV cache per token (simplified: 2 * layers * d_model * dtype)
KV_BYTES_PER_TOKEN = 2 * 80 * 8192 * 2  # ≈ 2.6 MB per token in context

def inference_runtime(batch_size, context_len):
    """Estimate runtime using the fundamental roofline equation."""
    # FLOPs per forward pass: 2 * params per token in batch (matmul FLOPs)
    flops          = 2 * PARAMS * batch_size
    t_compute      = flops / FLOPS_PER_SEC              # arithmetic time

    # Memory: fetch all weights + KV cache for each sequence in the batch
    kv_cache_bytes = KV_BYTES_PER_TOKEN * context_len * batch_size
    t_memory       = (WEIGHT_BYTES + kv_cache_bytes) / MEM_BW_BYTES   # bandwidth time

    runtime = max(t_compute, t_memory)                 # bottleneck determines runtime
    bottleneck = "compute" if t_compute > t_memory else "memory"
    return runtime, bottleneck, t_compute, t_memory

# Compare batch sizes at 2k context
print(f"{'Batch':>6} {'Runtime(ms)':>12} {'Bottleneck':>12} {'T_comp(ms)':>12} {'T_mem(ms)':>11}")
for b in [1, 10, 100, 300, 1000, 3000]:
    rt, bn, tc, tm = inference_runtime(b, context_len=2048)
    print(ff"{b:6d} {rt*1000:12.1f} {bn:>12} {tc*1000:12.1f} {tm*1000:11.1f}")

The 20ms Train Schedule

How inference servers batch requests like a train that departs every 20 milliseconds

An optimally-run inference server operates like a commuter train. A new "batch departs" every 20ms — the exact time required to drain High Bandwidth Memory (HBM) of one full set of model weights. If your request arrives just after a batch departs, you wait up to 40ms (two train cycles) for the next one.

The 20ms HBM Drain Train t=0ms Batch 1 Departs ←── 2,000 user sequences loaded │ HBM draining: streaming 140 GB of weights @ 3.35 TB/s t=20ms Batch 2 Departs ←── next 2,000 sequences │ HBM draining... t=40ms Batch 3 Departs t=60ms ... Worst-case queuing latency: just missed t=0ms → wait until t=40ms = 40ms Average queuing latency: ~10ms Optimal batch size formula: Batch Size ≈ 300 × Sparsity For a dense model (sparsity=1): ~300 sequences For MoE with 8 active of 64 experts (sparsity=8): ~2,400 sequences
🚂

Why 20ms?

HBM (High Bandwidth Memory) on an H100 provides 3.35 TB/s. Llama-3 70B weights are 140 GB. Time to stream all weights: 140 GB / 3,350 GB/s = 41.8ms. Each inference server targets one weight fetch per decode step — so batches are scheduled around this physical drain time.

  • H100 HBM3 bandwidth: 3.35 TB/s
  • 70B FP16 model: 140 GB
  • Weight drain time: ~42ms per decode step
  • Optimal scheduling: ~1 batch per drain cycle
👥

The 1,000× Efficiency Gap

A server serving 1 user at a time uses the GPU at roughly 0.1% compute efficiency. Serving 2,000 concurrent users can approach 100% efficiency — the same hardware cost amortised across 1,000× more output tokens. This is why batch pricing exists.

  • Individual requests: inefficient (100% of weight fetch, 1 token benefit)
  • Batched inference: efficient (weight fetch split across thousands of tokens)
  • The 1,000× gap drives 6× "fast mode" premiums
  • Cloud providers maximise GPU revenue by maximising batch fill
The Latency vs. Cost Trade-off

Every millisecond of lower latency costs real money — it means a smaller batch, which means the fixed weight-fetch cost is divided by fewer tokens. A 10ms response requires processing a batch so small that each token effectively pays for its own weight fetch. A 500ms response allows accumulating thousands of concurrent users, amortising that cost to near-zero.

The Context Length Penalty

Why 100k tokens costs 50% more and 1M tokens is prohibitively expensive

Every decode step requires reading the KV cache — the stored key and value vectors for all previous tokens. At 8k tokens, the KV cache is a small fraction of the total memory traffic. At 1M tokens, the KV cache dwarfs the model weights and completely dominates inference cost.

Crucially: compute cost is constant regardless of context length. The matmul arithmetic doesn't care how many tokens came before. Only the memory bandwidth cost — reading the KV cache — scales with context length.

Memory Traffic vs Context Length (70B model, H100) Decode step memory traffic = Weights + KV Cache Context: 8k tokens Weights: ████████████████████████████████████████ 140 GB KV Cache: ██ ~21 MB (negligible) Context: 100k tokens Weights: ████████████████████████████████████████ 140 GB KV Cache: ████████████ ~260 MB (+0.2% overhead) Context: 1M tokens Weights: ████████████████████████████████████████ 140 GB KV Cache: ████████████████████████████████████████████████████████ ~2.6 TB! At 1M tokens: KV cache is 18× larger than the model weights! HBM can't even hold it. Must tier to DRAM → 10–100× slower decode. Conclusion: 100M-token contexts are memory bandwidth limited, not FLOP limited.
Python · KV cache size vs context length
def kv_cache_gb(model_name, context_tokens):
    """Estimate KV cache size for common models."""
    configs = {
        "Llama-3 8B":  {"layers": 32,  "n_kv_heads": 8,  "d_head": 128},
        "Llama-3 70B": {"layers": 80,  "n_kv_heads": 8,  "d_head": 128},
        "GPT-4 (est)": {"layers": 120, "n_kv_heads": 16, "d_head": 128},
    }
    c = configs[model_name]
    # KV cache: 2 (K+V) × layers × n_kv_heads × d_head × tokens × 2 bytes (FP16)
    bytes_total = 2 * c["layers"] * c["n_kv_heads"] * c["d_head"] * context_tokens * 2
    return bytes_total / 1e9   # convert to GB

contexts = [8_000, 32_000, 128_000, 1_000_000]
print(ff"{'Model':15} {'8k':>8} {'32k':>8} {'128k':>8} {'1M':>10}")
for model in ["Llama-3 8B", "Llama-3 70B"]:
    sizes = [kv_cache_gb(model, c) for c in contexts]
    print(ff"{model:15}" + "".join(ff"{s:8.1f}G" for s in sizes))

The Economics of Memory Tiers

HBM, DRAM, and disk — the three tiers that create all cache pricing

Not all memory is created equal. The hierarchy of storage speed and cost directly maps to the cache discount tiers you see in LLM APIs. Understanding why a 5-minute cache discount is so large requires understanding the drain time difference between HBM and DDR.

Memory TierDrain TimeAPI EquivalentCost to HoldCore Trade-off
HBM (High BW Memory)20 millisecondsInstant API ResponseAstronomically expensiveInstant retrieval — but costs a fortune per GB per second
DDR (Main Host RAM)1–10 seconds5-Minute Cache DiscountModerateCheap to hold — but requires copying back to HBM before each use

Spinning Disk / Cloud StorageMinutes–hoursNot offeredCheapestOnly viable for fully cold workloads; retrieval kills latency
Memory Tier Economics — KV Cache Storage Strategy Active inference (user typing): KV cache lives in HBM ← fast decode, expensive real estate User stops for 2 minutes: Evict KV cache to DDR ← "5-minute cache" tier Cost drops ~100× If user returns: reload HBM (1–10 sec penalty → longer TTFT) User stops for 1 hour: Evict KV cache to Flash ← "1-hour cache" tier Cost drops ~10,000× If user returns: reload DDR then HBM (tens of seconds penalty) User gone forever: Delete KV cache entirely ← no storage cost, next request starts fresh
Why Cache Discounts Are So Large

HBM on an H100 costs ~$30/GB/hour when fully utilised. DRAM costs ~$0.03/GB/hour. Flash costs ~$0.003/GB/hour. If your KV cache sits idle in HBM for 5 minutes, you're paying for expensive real estate that could be serving other users. The large cache discounts are an incentive for users to allow the provider to tier their data to cheaper storage — with the trade-off of slightly higher latency on cache reload.

The Physical Rack Limit

Why leaving a single rack causes an 8× bandwidth cliff — and why this shapes model architecture

Inside a single rack of 8 GPUs, NVIDIA's NVLink fabric provides all-to-all communication at full bandwidth. Every GPU can talk to every other GPU at 900 GB/s simultaneously. The moment you need to communicate between racks, you hit InfiniBand — and the effective bandwidth drops by 8× immediately.

This is not a hardware choice — it is a physical constraint. Adding more GPUs to a single NVLink domain requires an exponentially denser tangle of copper cables. Each cable has a minimum bend radius. Each cable has weight. The rack has finite physical volume. You literally cannot fit more high-bandwidth cables in a given physical space.

Network Topology: Inside vs. Outside the Rack Inside the Rack (NVLink 4.0): GPU 0 ←──── 900 GB/s ────→ GPU 7 GPU 0 ←──── 900 GB/s ────→ GPU 3 All-to-all at full bandwidth. Fast. ✓ Leaving the Rack (InfiniBand NDR): GPU 0 ←──── ~112 GB/s ───→ GPU in Rack 2 8× bandwidth drop immediately. ⚠ Why? Physics: NVLink: copper traces on PCB — zero latency, max density InfiniBand: physical cables with min. bend radius, finite weight Cable density: limited by rack volume, cable weight, bend radius Adding GPUs: requires exponentially more cables → physically impossible to scale Practical consequence: Model parallelism inside rack: fast, cheap Model parallelism across racks: expensive, slow Architecture must be designed to minimise cross-rack traffic

Mixture of Experts Is a Hardware Mirror

MoE is not just a model architecture — it is a direct map of rack topology

Mixture of Experts (MoE) models route each token to a small subset of "expert" networks. This routing pattern requires all-to-all communication: any token might need to reach any expert on any GPU. This is exactly the traffic pattern that NVLink handles optimally — and exactly the traffic pattern that cross-rack InfiniBand handles poorly.

This means the expert layout is physically constrained to fit within one rack. The NVLink domain boundary is the MoE boundary. This is why MoE models scale in a way that perfectly reflects rack topology — it is not a design philosophy, it is a hardware necessity.

MoE Token Routing — Bounded by the Rack Router Layer receives token embeddings │ ├──→ Expert GPU 0 (some tokens) ├──→ Expert GPU 1 (some tokens) ├──→ Expert GPU 2 (some tokens) ├──→ Expert GPU 3 (some tokens) ├──→ Expert GPU 4 (some tokens) ├──→ Expert GPU 5 (some tokens) ├──→ Expert GPU 6 (some tokens) └──→ Expert GPU 7 (some tokens) ┌─────────────────────────────────────┐ │ Physical Rack Boundary │ │ All-to-all via NVLink: 900 GB/s │ └─────────────────────────────────────┘ Any GPU might need to consult any expert → true all-to-all pattern This perfectly matches NVLink topology inside the rack Crossing the rack boundary would cause 8× bandwidth collapse
🧠

MoE Economics

A 64-expert MoE model with top-2 routing activates only 2/64 = 3% of parameters per token. This means the effective active parameter count is 3% of total — dramatically reducing both compute and memory bandwidth per token.

  • Active params per token ≈ 2/N_experts × total params
  • Memory bandwidth: only fetch 2 expert weights, not all 64
  • Compute: only execute 2 experts' matmuls per token
  • Trade-off: routing overhead + all-to-all communication cost
🏗️

Expert Parallelism vs. Tensor Parallelism

Expert parallelism distributes expert networks across GPUs inside the rack. Tensor parallelism shards individual weight matrices across GPUs. Both are optimally confined to a single NVLink domain — but for different reasons.

  • Expert parallelism: all-to-all routing → NVLink domain
  • Tensor parallelism: AllReduce per layer → NVLink domain
  • Pipeline parallelism: P2P between stages → spans racks via IB
  • The rack is the natural unit of model parallelism

The Pipeline Parallelism Bubble

How pipelining model layers across racks causes idle GPU time — and why it's still worth it

When a model is too large to fit in a single rack, it is split across multiple racks using pipeline parallelism: Rack 1 holds layers 1–20, Rack 2 holds layers 21–40, etc. Data flows sequentially from rack to rack like an assembly line. The problem: each rack must wait for the previous rack to finish before it can start. This creates idle time — the "pipeline bubble."

The 100× Over-Training Phenomenon

Why labs intentionally train far past the Chinchilla-optimal point

Chinchilla-optimal training (Hoffmann et al., 2022) determined that the compute-efficient ratio is ~20 tokens per parameter. A 70B model should be trained on ~1.4 trillion tokens. Yet Llama-3 was trained on 15 trillion tokens — 10× past optimal. DeepSeek trained on 200 trillion tokens — 100× past Chinchilla optimal.

This is not a mistake. It is a deliberate economic strategy driven by inference costs.

Why 100× Over-Training Makes Economic Sense Chinchilla Optimal: Training compute: C FLOPs Training tokens: ~1.4 trillion (for 70B model) Model capability: baseline 100× Over-trained (Llama-3 / DeepSeek scale): Training compute: 100C FLOPs ← burn it upfront Training tokens: ~200 trillion Model capability: HIGHER (more compressed knowledge) BUT: each inference token is CHEAPER to serve The math: Expected inference volume: hundreds of millions of tokens/day Over N years of deployment: >> 10^18 total inference tokens Cost saved per inference token (smaller active model, denser knowledge) × 10^18 inference tokens = savings >> upfront training over-spend Over-training is front-loading compute to compress parameters, making every future inference token cheaper to serve.
The Balanced Ecosystem

Total ecosystem cost is minimised when compute is perfectly equalised across all phases: Pre-training compute ≈ RL alignment compute ≈ Inference compute. Labs that serve hundreds of millions of users find it economically optimal to over-train the model by the ratio of expected inference volume to training volume — compressing more world knowledge into the same parameter count to reduce per-token inference cost.

Reversible Nets — Cryptography Bypasses the Memory Wall

How invertible cipher logic eliminates the activation memory footprint during training

Standard neural network training must store all intermediate activations in HBM during the forward pass so they can be used during backpropagation. For a large model trained on long sequences, this activation memory can be larger than the model weights themselves.

Reversible Networks (inspired by Feistel ciphers from cryptography) eliminate this requirement. By constructing each layer as an invertible function, the backward pass can recompute all activations purely from the final output — trading extra compute cycles for completely eliminating activation storage.

Standard Network (training) Forward: x₁ → Layer 1 → h₁ ← stored in HBM h₁ → Layer 2 → h₂ ← stored in HBM h₂ → Layer 3 → h₃ ← stored in HBM h₃ → Layer 4 → y Backward: Need h₃ to compute ∂L/∂Layer4 weights Need h₂ to compute ∂L/∂Layer3 weights Need h₁ to compute ∂L/∂Layer2 weights Memory: O(n_layers × activation_size) For GPT-4 scale: hundreds of GB of activations
Reversible Network (RevNet) Forward (Feistel-style): (x₁, x₂) → XOR/ADD → (y₁, y₂) Activations zeroed out! (no HBM storage) Backward: Given (y₁, y₂) at output: Can RECOMPUTE (x₁, x₂) by INVERTING (y₁, y₂) → XOR/ADD inverse → (x₁, x₂) No stored activations needed! Memory: O(1) — constant regardless of depth Trade-off: ~33% more compute for backward pass Worth it when HBM is the bottleneck

The Memory Wall — The Ultimate Constraint

Compute is abundant. Memory bandwidth is the immovable constraint shaping the entire AI industry.

The fundamental asymmetry of modern AI hardware: compute (FLOP/s) has scaled at roughly Moore's Law pace. Memory bandwidth has grown far more slowly. This widening gap — the "memory wall" — is the defining constraint of LLM inference. Every architectural innovation, pricing decision, and infrastructure investment ultimately traces back to this single physical reality.

Hiding the Wall — Batching

Amortise the weight fetch across many concurrent users. The wall doesn't move, but you divide its cost. 2,000 users per batch = each user pays 1/2,000 of the weight fetch cost.

Compressing the Wall — Quantisation

Reduce model weight size via INT4/INT8 quantisation. A 70B INT4 model is 35 GB instead of 140 GB — drains HBM 4× faster. The wall shrinks proportionally to compression ratio.

Bypassing the Wall — MoE

Only fetch the active expert weights (2 of 64 = 3%). The effective bandwidth requirement drops 32× for MoE models. You never fetch the 98% of weights that aren't needed.

Tiering Around the Wall — Cache

Store less-frequently-accessed KV caches in DDR or Flash rather than HBM. Costs 100–10,000× less to hold data there. The 5-minute and 1-hour cache discount tiers are direct expressions of this.

Architectural Wall — Reversible Nets

During training, activation memory can exceed model weight memory. Reversible architectures eliminate activation storage entirely, trading compute for memory bandwidth freedom.

Future: Non-Von Neumann Compute

Processing-in-Memory (PIM) chips compute directly in the DRAM chips. Analog computing eliminates digital data movement. These approaches don't hide the wall — they demolish it architecturally.

The Final Insight

The future of AI architecture isn't just about calculating faster — it is about hiding, compressing, and bypassing the Memory Wall. From rack cable densities to API pricing, from MoE routing to cache tier discounts, every dimension of the LLM ecosystem is ultimately shaped by one immutable physical law: data movement costs energy, time, and money — and there is no free lunch.
