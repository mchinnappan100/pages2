What Is the GIL?

The Global Interpreter Lock — CPython's most famous design choice

The Global Interpreter Lock (GIL) is a mutex inside the CPython interpreter that ensures only one thread executes Python bytecode at a time, even on a multi-core machine. It was introduced in CPython 1.5 (1997) to protect Python's reference-counted garbage collector from concurrent modification without fine-grained locking.

Why Does It Exist?

Python objects are reference-counted. When you write a = b, b's ob_refcnt is incremented. If two threads could increment or decrement ob_refcnt simultaneously without synchronisation, you'd get race conditions leading to premature object deallocation and memory corruption. The GIL is the cheapest fix: one coarse lock around the entire interpreter, rather than a lock per object.

CPython Object Reference Count (ob_refcnt) typedef struct _object { Py_ssize_t ob_refcnt; ← race condition without GIL PyTypeObject *ob_type; } PyObject; Thread 1: ob_refcnt++ (LOAD, ADD, STORE) Thread 2: ob_refcnt++ (LOAD, ADD, STORE) Without GIL (race): With GIL (safe): T1 loads ob_refcnt = 5 T1 holds GIL T2 loads ob_refcnt = 5 T1: 5 → 6, releases GIL T1 stores ob_refcnt = 6 T2 acquires GIL T2 stores ob_refcnt = 6 ← lost T2: 6 → 7, releases GIL Final: 6 (should be 7) GIL is a mutex around the ceval loop (ceval.c: _PyEval_EvalFrameDefault)
Python · Proving the GIL exists
import threading, time

counter = 0

def increment(n):
    global counter
    for _ in range(n):
        counter += 1   # NOT thread-safe: LOAD_GLOBAL, BINARY_ADD, STORE_GLOBAL

# Even with the GIL, += is not atomic because it's 3 bytecode ops.
# The GIL can be released between LOAD and STORE (every sys.getswitchinterval()).
threads = [threading.Thread(target=increment, args=(100_000,)) for _ in range(4)]
[t.start() for t in threads]
[t.join()  for t in threads]
print(counter)   # usually < 400000 — data race despite the GIL!

# Fix: use threading.Lock() or collections.Counter (atomic in C)
import sys
print(sys.getswitchinterval())  # 0.005 s — GIL released every 5 ms
# In Python 3.2+ it's every N "gil_drop_request" checks (default ~100 checks)
Common Misconception

The GIL does not make Python thread-safe for compound operations like counter += 1. It only prevents two threads from running Python bytecode simultaneously at the interpreter loop level — not at the Python expression level. A context switch can still happen between any two bytecodes.

02

Threads & the GIL

When threads help, when they hurt, and the I/O exception

The GIL is released during operations that do not manipulate Python objects — most importantly during blocking I/O (read, write, recv) and most C-extension computations (NumPy, PyTorch CUDA kernels). This is why threads work fine for I/O-bound tasks and why PyTorch's CUDA ops don't block each other.

When Threads Work

  • I/O-bound: network requests, file reads — GIL released during read() syscall, other threads run.
  • C-extension compute: NumPy operations release the GIL via Py_BEGIN_ALLOW_THREADS / Py_END_ALLOW_THREADS. PyTorch CUDA kernels are fully asynchronous and never hold the GIL.
  • GUI callbacks: UI event loop + background I/O thread.

When Threads Don't Work

  • CPU-bound pure Python: sorting, hashing, string processing. Two threads will run at the same speed as one — the GIL serialises them. Use multiprocessing instead.
  • DataLoader preprocessing: decoding JPEG images, tokenisation in Python — all CPU-bound Python bytecode → processes, not threads.
Python · Thread vs Process: CPU-bound benchmark
import threading, multiprocessing, time

def cpu_work():
    return sum(i*i for i in range(10_000_000))

# ── SERIAL ──────────────────────────────────
t0 = time.perf_counter()
cpu_work(); cpu_work(); cpu_work(); cpu_work()
print(ff"Serial:  {time.perf_counter()-t0:.2f}s")   # e.g. 2.4s

# ── 4 THREADS (GIL = no speedup) ────────────
t0 = time.perf_counter()
threads = [threading.Thread(target=cpu_work) for _ in range(4)]
[t.start() for t in threads]; [t.join() for t in threads]
print(ff"Threads: {time.perf_counter()-t0:.2f}s")   # ~2.5s — no speedup!

# ── 4 PROCESSES (true parallelism) ──────────
t0 = time.perf_counter()
ctx = multiprocessing.get_context("spawn")
procs = [ctx.Process(target=cpu_work) for _ in range(4)]
[p.start() for p in procs]; [p.join() for p in procs]
print(ff"Procs:   {time.perf_counter()-t0:.2f}s")   # ~0.7s — 3.5× speedup!

03

multiprocessing Module

Creating processes, sharing state, and the Queue / Pipe primitives

Python's multiprocessing module creates separate OS processes, each with its own Python interpreter and GIL. Communication between processes requires explicit serialisation (via pickle) or shared memory (via mmap). The module provides high-level abstractions: Pool, Queue, Pipe, Value, Array, and SharedMemory.

Python · multiprocessing primitives
import multiprocessing as mp
import numpy as np

# ── Queue: serialise (pickle) items across processes ─────────────
q = mp.Queue(maxsize=10)   # internally backed by a pipe + semaphore

def producer(q):
    for i in range(5):
        q.put({"batch": np.random.randn(32, 512)})  # pickled → pipe → unpickled

def consumer(q):
    while True:
        item = q.get()
        if item is None: break
        process(item["batch"])

# ── Shared numpy array via mmap: ZERO copy ───────────────────────
from multiprocessing import shared_memory

shm = shared_memory.SharedMemory(create=True, size=32*512*4)   # 4 bytes/float32
arr = np.ndarray((32, 512), dtype=np.float32, buffer=shm.buf)

def fill_shared(name, shape):
    shm2 = shared_memory.SharedMemory(name=name)           # attach in child
    buf  = np.ndarray(shape, dtype=np.float32, buffer=shm2.buf)
    buf[:] = np.random.randn(*shape)                       # write directly
    shm2.close()

p = mp.Process(target=fill_shared, args=(shm.name, (32, 512)))
p.start(); p.join()
print(arr[0, :5])   # filled by child process — no copy!
shm.unlink()         # destroy the shared segment
🔌

Queue vs Pipe vs SharedMemory

Three communication primitives with very different performance characteristics. Choose based on message size and frequency.

  • Queue: pickles objects, sends over OS pipe — convenient, ~1 µs + pickle cost
  • Pipe: two-way raw bytes pipe — faster than Queue, still copies
  • SharedMemory: mmap'd region, zero-copy, fastest for large arrays
  • Value/Array: small shared C types (e.g. mp.Value('i', 0) for a counter)
🏊

Pool — Process Pool Executor

mp.Pool manages a fixed set of worker processes. Submit tasks via map, imap, or apply_async. Workers are reused across calls, amortising the fork cost.

  • Pool.map(fn, items): blocks until all done
  • Pool.imap(fn, items): yields results as they arrive
  • Pool.starmap(fn, [(a,b), ...]): multiple args
  • concurrent.futures.ProcessPoolExecutor: modern API, same semantics

Process Start Methods

fork, spawn, and forkserver — how child processes are created

Three Start Methods Compared fork (Linux/macOS default before Python 3.12) Parent ──fork()──→ Child • Copy-on-write clone of entire parent address space • Very fast to start (~1 ms) • Inherits ALL parent state: open files, CUDA context, locks • DANGER: CUDA context is NOT fork-safe → deadlock or corruption! spawn (Windows default; Python 3.12+ default on macOS) Parent ──spawn→ fresh Python ──pickle→ Worker fn • Start a brand-new Python interpreter from scratch • Slow (~50–200 ms per worker, imports whole library) • Clean state: no inherited CUDA context, no dangling locks • Pickle the target function + args to send to child forkserver (PyTorch 2.0+ default on Linux) Parent ──request→ Fork Server ──fork()→ Child • A single clean server process is started at program init • The server forks from its own clean state on demand • Fast as fork (~2 ms), safe as spawn (no CUDA context) • Each child is a fork of the CLEAN server, not the parent
Python · Choosing start methods
import multiprocessing as mp

# ── Method 1: set globally (affects all mp.Process) ──────────────
mp.set_start_method("forkserver")   # call once, before any child process

# ── Method 2: use a context (preferred — isolated scope) ─────────
ctx = mp.get_context("forkserver")

def worker(x):
    return x * x

with ctx.Pool(4) as pool:
    results = pool.map(worker, range(16))
print(results)   # [0, 1, 4, 9, ..., 225]

# ── Why PyTorch uses forkserver ───────────────────────────────────
# If parent has already called torch.cuda.* (e.g. moved a tensor to GPU),
# fork() copies the CUDA context into the child.
# The child then tries to use CUDA, hitting a deadlock or silent corruption
# because CUDA contexts are not designed to be shared across fork().
# forkserver avoids this: child forks from a CLEAN pre-CUDA server process.

# Check which method DataLoader uses internally:
import torch
loader = torch.utils.data.DataLoader(dataset, num_workers=4)
# Internally calls mp.get_context("forkserver") on Linux

Shared Memory Queue

How mmap + semaphores enable zero-copy inter-process tensor transfer

A Python multiprocessing.Queue uses a Pipe (OS pipe, copies bytes) plus a Semaphore for flow control and a background feeder thread for serialisation. For large tensors this is slow: pickle + pipe copy = two full copies of the data. PyTorch's DataLoader avoids this with shared memory.

How Shared Memory Works (mmap + semaphore)

A shared memory segment is an anonymous mmap(MAP_SHARED) region. Both parent and child map the same physical pages into their address spaces. Writing to the region in the worker is immediately visible in the main process — zero bytes copied. A semaphore signals when a new batch is ready.

DataLoader Shared Memory Queue Internals Main Process Worker Process (forkserver) ┌─ Dataset.__getitem__(idx) index_queue ──idx──────────────→ │ (CPU preprocessing) (mp.SimpleQueue/Pipe) └─ tensor = transform(sample) │ tensor.share_memory_() → shm_open() + mmap(MAP_SHARED) → moves backing store to /dev/shm │ data_queue ←──meta────────────── put (storage_handle, shape, dtype) (only handle, not data!) │ Main: rebuild tensor from handle → maps same physical pages → zero copy from worker to main │ pin_memory → cudaMemcpyAsync → GPU
Python · Manual shared-memory tensor pipeline
import torch
import torch.multiprocessing as mp

def worker(index_q, data_q, dataset):
    while True:
        idx = index_q.get()
        if idx is None:
            break
        sample = dataset[idx]                    # preprocessing in worker
        tensor = torch.tensor(sample)

        # Move backing storage to /dev/shm (mmap MAP_SHARED)
        tensor.share_memory_()                   # now backed by shm

        # Send only the metadata handle (not the data bytes!)
        data_q.put(tensor)                       # pickle sends shm handle

ctx = mp.get_context("forkserver")
index_q = ctx.SimpleQueue()
data_q  = ctx.SimpleQueue()

workers = [
    ctx.Process(target=worker, args=(index_q, data_q, dataset))
    for _ in range(4)
]
[w.start() for w in workers]

# Feed indices
for i in range(len(dataset)):
    index_q.put(i)

# Receive tensors — zero copy from worker!
for _ in range(len(dataset)):
    batch = data_q.get()   # deserialized from shm handle, same memory
    train_step(batch.pin_memory().cuda(non_blocking=True))
Semaphore under the hood

PyTorch's SimpleQueue uses a POSIX named semaphore (sem_open) to signal between producer and consumer, and a pipe() or socket to transmit the pickled handle. The actual tensor bytes never pass through the pipe — only the 40-byte storage handle does. This is why DataLoader throughput scales with the number of workers even for large tensors.

Python · DataLoader tuning for maximum throughput
import torch
from torch.utils.data import DataLoader

# Rule of thumb: num_workers = 4 × num_GPUs (or CPU cores - 2)
loader = DataLoader(
    dataset,
    batch_size=64,
    num_workers=8,            # 8 worker processes
    pin_memory=True,          # cudaHostRegister for DMA
    persistent_workers=True,  # don't kill workers between epochs
    prefetch_factor=4,        # each worker preloads 4 batches
    multiprocessing_context="forkserver",  # safe with CUDA
)

Avoid Fork After CUDA Init

Never call torch.cuda.init() or move a tensor to GPU before spawning DataLoader workers. If you must (e.g. you profile before training), use multiprocessing_context="forkserver" explicitly. forkserver forks from a clean server that was started before any CUDA initialisation.

persistent_workers=True

Without this, workers are killed and respawned every epoch — costing hundreds of milliseconds of forkserver overhead. With it, workers stay alive and their memory-mapped shared regions remain valid. Essential for fast short epochs.

Worker-Init Fn for Per-Worker State

Use worker_init_fn to set per-worker random seeds, open per-worker file handles, or set CPU affinity. Without it, all workers share the same random seed (from fork), leading to identical data augmentation across workers.

collate_fn

The default collate stacks a list of tensors into a batched tensor. For variable-length sequences (NLP), write a custom collate_fn that pads to the longest sequence in the batch — much more efficient than padding to max length globally.

CUDA Context & forkserver Safety

Why fork + CUDA is dangerous and how forkserver solves it

When fork() duplicates a process, the child inherits a copy of all file descriptors, memory mappings, and — critically — CUDA's internal state tables. CUDA contexts contain:

  • CUDA stream handles (backed by kernel objects)
  • Module/function handles (pointers to GPU device code)
  • Memory allocations with virtual address mappings registered with the driver
  • Internal mutexes and condition variables

None of these are designed to be shared between OS processes. After fork, both parent and child hold the same CUDA handles but point to kernel resources owned by the parent. Any CUDA call in the child will either deadlock (waiting for a mutex the parent holds), crash (invalid handle), or silently corrupt memory.

Python · Reproducing the fork+CUDA bug
import torch
import multiprocessing as mp

# ── BROKEN: fork after CUDA init ──────────────────────────────────
x = torch.zeros(10).cuda()   # initialises CUDA context in parent

def bad_worker():
    # Child forked AFTER cuda init — inherits broken CUDA context
    y = torch.zeros(10).cuda()   # ← may hang, crash, or corrupt
    print(y)

ctx_fork = mp.get_context("fork")
p = ctx_fork.Process(target=bad_worker)
p.start(); p.join()   # likely deadlock or RuntimeError

# ── FIXED: use forkserver ─────────────────────────────────────────
x = torch.zeros(10).cuda()   # parent has CUDA context

def safe_worker():
    # Child is a fork of the CLEAN forkserver (no CUDA context)
    y = torch.zeros(10).cuda()   # ← creates fresh CUDA context → safe
    print(y)

ctx_fs = mp.get_context("forkserver")
p = ctx_fs.Process(target=safe_worker)
p.start(); p.join()   # works correctly
Production Warning

If you see CUDA errors only when num_workers > 0 in your DataLoader, the cause is almost certainly fork after CUDA initialisation. Fix: add multiprocessing_context="forkserver" to your DataLoader, or ensure you never touch CUDA before creating workers. This is one of the most common hard-to-debug bugs in PyTorch training pipelines.

Free-Threaded CPython (3.13+)

The experimental no-GIL build and what it means for ML

Python 3.13 (PEP 703) ships an experimental free-threaded build (python3.13t) where the GIL can be disabled at interpreter startup with PYTHON_GIL=0. This allows true parallelism for CPU-bound Python threads — finally making threads a viable alternative to processes for pure-Python preprocessing.

What Changed Internally

  • Reference counting: replaced with per-object biased reference counting (fast path: no atomic ops; slow path: atomic CAS for shared objects).
  • Object locking: critical objects (dict, list) get per-object locks instead of relying on the GIL.
  • Thread-local caches: memory allocator uses per-thread arenas to avoid global lock contention.

ML Impact

For PyTorch today, free-threaded CPython has limited benefit — the expensive operations (CUDA kernels, C-extension math) already release the GIL. The gain is in pure-Python preprocessing pipelines (custom tokenisers, Python-level augmentation). Expect production support in Python 3.14–3.15 as third-party extensions (NumPy, PyTorch) gain thread-safety certifications.

Shell · Using free-threaded Python 3.13
# Install free-threaded build (pyenv)
pyenv install 3.13t
pyenv local 3.13t

# Verify GIL is disabled
python -c "import sys; print(sys._is_gil_enabled())"   # False

# Or disable at runtime
PYTHON_GIL=0 python train.py
Python · Thread parallelism without GIL (Python 3.13t)
import threading, time, sys

print(f"GIL enabled: {sys._is_gil_enabled()}")   # False on 3.13t with PYTHON_GIL=0

def cpu_work():
    return sum(i*i for i in range(5_000_000))

# On 3.13t with GIL disabled: true thread parallelism for pure-Python work
t0 = time.perf_counter()
threads = [threading.Thread(target=cpu_work) for _ in range(4)]
[t.start() for t in threads]; [t.join() for t in threads]
elapsed = time.perf_counter() - t0
print(ff"Threaded (no GIL): {elapsed:.2f}s")
# On 3.13t: ~0.6s (4× speedup vs serial)
# On 3.12:  ~2.4s (no speedup — GIL serialises)
