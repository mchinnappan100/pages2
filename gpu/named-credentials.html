Named Credentials &
Secure External Callouts

A comprehensive guide to calling external services from Salesforce without exposing secrets — covering Named Credentials, External Credentials, OAuth 2.0, JWT, mTLS certificates, External Services, and security best practices.

Named Credentials External Credentials OAuth 2.0 JWT Bearer mTLS Certificates External Services

Why Secure Callouts Matter

Salesforce connects to hundreds of external systems — REST APIs, payment gateways, ERP systems, identity providers, AI services. Every connection needs credentials: API keys, passwords, OAuth tokens, certificates. The challenge is keeping those secrets secure while making them usable in Apex code, Flows, and automated processes.

❌ The Insecure Way

  • Hardcoded API keys in Apex
  • Passwords in Custom Settings visible to admins
  • Credentials in Custom Labels (visible in metadata)
  • Tokens stored in text fields on records
  • Credentials committed to Git in metadata

✅ The Secure Way

  • Named Credentials — encrypted, never in code
  • External Credentials — flexible auth per user
  • OAuth 2.0 — token exchange, no password storage
  • JWT Bearer — no human passwords at all
  • Certificates — cryptographic identity
Core Principle

Named Credentials separate where to call (the endpoint URL) from how to authenticate (the credentials). Your Apex code references a logical name — never a URL string, never a password. Salesforce handles the authentication transparently.

CHAPTER ONE
Named Credentials Fundamentals

Named Credentials:
The Foundation of Secure Callouts

Named Credentials are Salesforce metadata records that store endpoint URLs and authentication settings. They are the recommended way to manage credentials for external callouts in Apex, Flow, and External Services — eliminating hardcoded secrets and centralising credential management.

Legacy Named Credentials vs Enhanced Named Credentials

Salesforce introduced Enhanced Named Credentials in Summer '22. They split the old single record into two separate metadata types, giving much more flexibility.

Reuse auth across endpointsNo — one credential per endpointYes — one External Credential, multiple Named Credentials

Recommended?Legacy — being deprecated✅ Use for all new work
Deprecation Notice

Salesforce is deprecating Legacy Named Credentials. All new integrations should use Enhanced Named Credentials with a separate External Credential. Migrate legacy records as part of technical debt reduction.

External Credentials

An External Credential defines the authentication protocol and credentials. It is linked FROM one or more Named Credentials. This separation means you can change the secret in one place and all Named Credentials pointing to it are updated instantly.

Setup Path

Setup → Security → Named Credentials — one screen manages both Named Credentials and their linked External Credentials.

Named Principal vs Per-User Principal

The Principal on an External Credential controls whose identity is used when making calls. This is one of the most important — and most misunderstood — concepts in Named Credentials.

Named Principal

  • A single shared identity for all calls
  • All users share the same token/credentials
  • Org-to-org or server-to-server integrations
  • No user interaction for token refresh
  • Example: A single service account for an ERP integration

Per-User Principal

  • Each Salesforce user has their own token
  • User must authenticate via OAuth flow first
  • Calls are made on behalf of the specific user
  • Audit trail shows individual user actions
  • Example: Google Drive integration per employee
Key Decision

Use Named Principal for system integrations where all calls should use a service account. Use Per-User Principal when the external system needs to know which individual Salesforce user is acting (user-delegated access, per-user audit trails, or when the external system enforces per-user permissions).

Permission Set Required for Per-User

For Per-User Named Credentials, users must be granted access to the External Credential via a Permission Set. Navigate to the Permission Set → External Credential Principal Access → add the External Credential. Users then authenticate individually via their User Settings → Authentication Settings for External Systems.

CHAPTER TWO
Authentication Protocols

Authentication Protocols:
OAuth, JWT, Certificates, and More

The External Credential's authentication protocol determines how Salesforce proves identity to the external service. This chapter covers every supported protocol, when to use each, and how to configure them correctly.

Basic Authentication

Basic Auth sends a Base64-encoded username:password string in the Authorization header of every request.

Configuration

  • External Credential Protocol: Password Authentication
  • Principal stores the username and password (encrypted at rest)
  • Salesforce injects Authorization: Basic <base64> automatically
Security Warning

Basic Auth sends credentials on every request (even over TLS). It should only be used with APIs that don't support OAuth. Always use HTTPS. If the API supports OAuth 2.0 or API keys, prefer those. Never use Basic Auth over unencrypted HTTP.

OAuth 2.0 Flows

OAuth 2.0 is the gold standard for API authentication. Instead of storing a password, Salesforce exchanges credentials for a short-lived access token and uses that for API calls.

Client Credentials Flow (Server-to-Server)

The most common flow for server-to-server integrations. No user interaction required. The org authenticates with a client_id and client_secret to get an access token.

Token Storage and Refresh

Salesforce automatically:

  • Stores access and refresh tokens encrypted per principal
  • Uses the refresh token to get a new access token when the current one expires
  • Retries the callout transparently with the new token
  • Raises an error only if the refresh token itself has expired
Key Concept — Token Refresh is Automatic

Your Apex code doesn't need to handle token expiry. The Named Credential framework detects a 401 response, refreshes the token using the stored refresh token, and retries the original request automatically. This is a major benefit over managing tokens manually.

JWT Bearer Token Flow

JWT (JSON Web Token) Bearer eliminates client secrets entirely. Instead of a password, Salesforce signs a JWT with a private key and sends it to the token endpoint. The external service verifies the signature with the corresponding public key.

Why JWT is Superior to Client Credentials

  • No shared secret — the private key never leaves Salesforce
  • Each token request is cryptographically signed and timestamped
  • Tokens are short-lived and non-replayable (exp + jti claims)
  • Audit trail shows exactly when and by whom tokens were requested

JWT Bearer Flow Setup

  1. Generate a certificate in Salesforce (Setup → Certificate and Key Management → Create Self-Signed Certificate) or upload a CA-signed cert
  2. Register the public key with the external service (they trust signatures from this key)
  3. In the External Credential, set Protocol = JWT, select the certificate, configure issuer/subject/audience
  4. On each callout, Salesforce creates a signed JWT and exchanges it for an access token at the token endpoint
// JWT claims Salesforce constructs automatically
{
  "iss": "your-client-id",         // Issuer — your Connected App consumer key
  "sub": "service@yourorg.com",   // Subject — the user/service account
  "aud": "https://login.salesforce.com", // Audience
  "exp": 1735689600,              // Expiry — 3 minutes from now
  "jti": "a1b2c3d4-unique-id"   // JWT ID — prevents replay attacks
}
// Signed with your private key from Certificate and Key Management
jwt payload

Certificate-Based Authentication (mTLS)

Mutual TLS (mTLS) is the most secure callout method. Both sides present certificates — Salesforce proves its identity to the server AND the server proves its identity to Salesforce. Used for banking, government, and high-compliance integrations.

How mTLS Works

Salesforce
Presents client cert
TLS handshake ↔
External Server
Presents server cert

Both sides verify each other's certificate against a trusted CA. No passwords or tokens needed — identity is proven cryptographically at the transport layer.

Configuring Client Certificates

  1. Setup → Certificate and Key Management → Create or upload your client certificate
  2. Export the public certificate and provide it to the external service's administrators
  3. In the External Credential, set Protocol = Certificate, select your certificate
  4. Salesforce presents this certificate in the TLS ClientHello during each connection

Custom Headers and Parameters

For APIs using API keys, HMAC signatures, or custom auth headers not covered by built-in protocols, use Custom Headers on the External Credential.

// External Credential: Custom Header for API Key
Header Name:  X-API-Key
Header Value: {!$Credential.ExternalCredentialName.api_key}
// The value references a Custom Parameter named 'api_key'
// stored encrypted in the External Credential's principal
named credential config

Custom Parameters

Custom Parameters are named, encrypted values stored on an External Credential principal. They can be referenced in header values using merge field syntax: {!$Credential.ExternalCredentialAPIName.ParameterName}

Best Practice — API Keys

Store API keys as Custom Parameters on the External Credential, then inject them via Custom Headers. Never reference the API key directly in Apex. When the key rotates, update only the External Credential — zero code changes needed.

CHAPTER THREE
Apex Callouts

Making Secure Callouts
from Apex

Apex callouts use the Http, HttpRequest, and HttpResponse classes. Named Credentials eliminate the need to manage URLs, credentials, and token refresh in code — your Apex becomes clean, portable, and credential-free.

HttpRequest Basics

Every Apex HTTP callout uses the same three classes regardless of whether you use Named Credentials or not:

// Without Named Credentials — DO NOT DO THIS
HttpRequest req = new HttpRequest();
req.setEndpoint('https://api.example.com/v1/customers');
req.setMethod('GET');
req.setHeader('Authorization', 'Bearer sk_live_abc123HARDCODED'); // 💀 NEVER DO THIS

Http http = new Http();
HttpResponse res = http.send(req);
System.debug(res.getBody());
apex

Using Named Credentials

With Named Credentials, the endpoint starts with callout: followed by the Named Credential API name. Salesforce automatically appends the base URL, injects authentication headers, and manages token lifecycle.

// With Named Credentials — the RIGHT way
HttpRequest req = new HttpRequest();
req.setEndpoint('callout:Stripe_Payment_API/customers'); // Base URL from Named Credential
req.setMethod('GET');
req.setHeader('Content-Type', 'application/json');
// NO Authorization header — Salesforce injects it automatically

Http http = new Http();
HttpResponse res = http.send(req);

if (res.getStatusCode() == 200) {
    Map<String, Object> body = (Map<String, Object>)
        JSON.deserializeUntyped(res.getBody());
    // process response...
} else {
    throw new CalloutException(
        'HTTP ' + res.getStatusCode() + ': ' + res.getBody()
    );
}
apex

Merge Fields in Callouts

Named Credentials support merge fields in the endpoint and request body when the Named Credential has Allow Merge Fields in URL / Allow Merge Fields in Body enabled. This lets you inject per-user identity or dynamic credential values.

// Merge fields in the endpoint path
req.setEndpoint(
    'callout:My_NC/users/{!$Credential.OAuthToken}/profile'
);

// Available merge field variables
// {!$Credential.OAuthToken}        — the current OAuth access token
// {!$Credential.UserName}           — the username on the principal
// {!$Credential.MyParam}            — a Custom Parameter named 'MyParam'

Asynchronous Callouts

Apex callouts cannot be made from a synchronous context that already performed DML. Use these patterns for async callouts:

Queueable with System.enqueueJob

public class SlackNotifier implements Queueable, Database.AllowsCallouts {

    private String message;

    public SlackNotifier(String msg) { this.message = msg; }

    public void execute(QueueableContext ctx) {
        HttpRequest req = new HttpRequest();
        req.setEndpoint('callout:Slack_Webhook/services/T.../B.../X...');
        req.setMethod('POST');
        req.setHeader('Content-Type', 'application/json');
        req.setBody('{"text":"' + message + '"}');

        HttpResponse res = new Http().send(req);
        if (res.getStatusCode() != 200) {
            System.debug('Slack error: ' + res.getBody());
        }
    }
}

// Call from a trigger or class after DML:
System.enqueueJob(new SlackNotifier('Case #00001 created'));
apex

@Future with Callouts

@future(callout=true)
public static void sendWebhook(String payload) {
    HttpRequest req = new HttpRequest();
    req.setEndpoint('callout:Webhook_NC/events');
    req.setMethod('POST');
    req.setHeader('Content-Type', 'application/json');
    req.setBody(payload);
    new Http().send(req);
}

Governor Limits

Apex callout limits: 100 callouts per transaction, 120 seconds total callout time per transaction, 10MB max response body. Each callout must complete before the next one begins (no true parallelism in Apex). Design for batch processing of large datasets.

Mocking Callouts in Tests

Unit tests cannot make live HTTP callouts. Use HttpCalloutMock to simulate responses — Named Credentials work seamlessly with mocks.

// 1. Implement the mock
@isTest
global class StripeApiMock implements HttpCalloutMock {
    global HttpResponse respond(HttpRequest req) {
        // Assert the Named Credential endpoint is used
        System.assert(req.getEndpoint().startsWith('callout:Stripe_Payment_API'));

        HttpResponse res = new HttpResponse();
        res.setHeader('Content-Type', 'application/json');
        res.setBody('{"id":"cus_123","object":"customer"}');
        res.setStatusCode(200);
        return res;
    }
}

// 2. Use it in the test
@isTest
static void testGetCustomer() {
    Test.setMock(HttpCalloutMock.class, new StripeApiMock());
    // call the production code that uses callout:Stripe_Payment_API
    StripeService.getCustomer('cus_123');
    // assert results...
}
apex

CHAPTER FOUR
Flow & External Services

No-Code Callouts with Flow
and External Services

Not every callout requires Apex. Salesforce Flow can call external HTTP endpoints directly using Named Credentials, and External Services turns an OpenAPI specification into reusable Flow elements and Invocable Actions — no code required.

HTTP Callout Actions in Flow

Flow Builder supports HTTP callout actions natively (generally available since Spring '24). These let you call any REST endpoint directly from a Flow without writing Apex.

Setting Up a Flow HTTP Callout

  1. In Flow Builder, add an Action element
  2. Select HTTP Callout as the action type
  3. Choose your Named Credential from the dropdown
  4. Specify the URL path, HTTP method, request body, and expected response schema
  5. Map response fields to Flow variables for downstream use
Key Benefit

Flow HTTP Callout actions use Named Credentials for authentication — no credentials ever appear in the Flow definition. Admins with zero coding skills can build authenticated integrations while the security team controls the credentials centrally.

// What Salesforce stores in the Flow metadata (simplified)
<actionCalls>
  <actionName>callout__My_Named_Credential</actionName>
  <actionType>externalService</actionType>
  <inputParameters>
    <name>method</name>
    <value><stringValue>GET</stringValue></value>
  </inputParameters>
</actionCalls>
flow xml (metadata)

External Services

External Services turns a REST API specification into a suite of reusable Salesforce actions. Configure it once and use the generated actions everywhere: Flows, Apex as Invocable Methods, and even directly in Agentforce as External Actions.

What External Services Generates

For each operation in your API specification, External Services creates:

  • A named Invocable Action usable in Flow Builder
  • An @InvocableMethod-equivalent callable from Apex
  • Input/output variable schemas with type checking
  • Automatic request/response serialisation

Where External Services Lives

Setup → Integrations → External Services

OpenAPI Registration

External Services supports OpenAPI Specification 2.0 (Swagger) and 3.0. When registering:

  1. Navigate to Setup → External Services → New
  2. Provide the service name and select the Named Credential for authentication
  3. Paste or upload the OpenAPI JSON/YAML specification
  4. Salesforce validates the spec and generates the actions
  5. Each API operation becomes a separate named action
# Minimal OpenAPI 3.0 for External Services registration
openapi: "3.0.0"
info:
  title: "Weather API"
  version: "1.0"
servers:
  - url: "https://api.weatherapi.com/v1"
    # Note: URL here is overridden by the Named Credential in Salesforce
paths:
  /current.json:
    get:
      operationId: getCurrentWeather
      parameters:
        - name: q
          in: query
          required: true
          schema:
            type: string
      responses:
        '200':
          content:
            application/json:
              schema:
                type: object
                properties:
                  temp_c: { type: number }
                  condition: { type: string }
openapi yaml
Best Practice — External Services + Agentforce

External Services actions are automatically available as External Actions in Agentforce. Register your third-party API once in External Services and it immediately appears in Agent Builder's Action catalog — no Apex code needed for third-party API integrations.

CHAPTER FIVE
Security Patterns

Security Boundaries, Certificates,
and What to Avoid

Named Credentials are only one layer of Salesforce's external callout security model. This chapter covers the complete picture — Remote Site Settings, CSP Trusted Sites, certificate management, and the anti-patterns that create vulnerabilities.

Remote Site Settings

Before Apex can call any external URL, the domain must be listed in Remote Site Settings. This is a whitelist that prevents Apex from making callouts to arbitrary internet addresses.

Configuration

Setup → Security → Remote Site Settings → New Remote Site

  • Enter the base URL (protocol + domain + optional port): https://api.stripe.com
  • Subpaths are automatically permitted — you only need the base domain
  • Named Credentials automatically create a Remote Site Setting — no manual step needed
  • You still need to add Remote Site Settings for endpoints NOT covered by Named Credentials
Gotcha

Remote Site Settings apply to Apex callouts only. They do NOT restrict calls made from Lightning Web Components (LWC) running in the browser — that's controlled by CSP Trusted Sites. Two separate whitelists for two separate contexts.

CSP Trusted Sites

Content Security Policy (CSP) Trusted Sites control which external domains browser-side JavaScript (LWC, Aura) can connect to. Required for:

  • LWC components loading external scripts or fonts
  • Client-side API calls from JavaScript in Lightning pages
  • Loading external images or media in LWC
  • Connecting to WebSocket servers from the browser

CSP Directive Types

DirectiveControls
connect-srcAJAX/Fetch calls, WebSocket connections
script-srcLoading external JavaScript files
img-srcLoading images from external domains
font-srcLoading web fonts
frame-srcEmbedding external pages in iframes

Setup → Security → CSP Trusted Sites → New Trusted Site

Certificates and Key Management

Salesforce has two certificate stores serving different purposes:

Salesforce-Generated Self-Signed Certificates

  • Created via Setup → Certificate and Key Management → Create Self-Signed Certificate
  • Private key never leaves Salesforce — most secure option
  • Used for JWT Bearer token signing, mTLS client authentication, SAML signing
  • 2048-bit RSA minimum; 4096-bit recommended for high-value integrations

CA-Signed Certificates (Keystores)

  • Generate a Certificate Signing Request (CSR) in Salesforce, sign with external CA, import the signed cert
  • Or import an existing PKCS#12 (.p12/.pfx) keystore with the full chain
  • Required when the external service demands a certificate signed by a specific CA

Certificate Expiry Monitoring

Critical — Certificate Expiry = Outage

Expired certificates cause immediate, silent outage of all integrations using them. Salesforce does NOT send expiry notifications by default. Set calendar reminders at 90, 60, and 30 days before expiry. Consider implementing the Salesforce Certificate Expiry monitoring via scheduled Apex that queries SetupEntityAccess or monitors certificate dates.

Alternatives for Secret Storage

Sometimes you need to store sensitive values outside Named Credentials. Here are the options ranked by security:

Shield Platform Encryption

Salesforce Shield Platform Encryption can encrypt Custom Settings fields using an org-managed or tenant-managed key. This is the enterprise approach for storing secrets that cannot use Named Credentials (e.g., encryption keys, license keys, webhook signing secrets).

Anti-Patterns to Avoid

❌ Hardcoded Credentials in Apex

// NEVER DO THIS — credentials in source code
private static final String API_KEY = 'sk_live_AbCdEf123456789';
req.setHeader('Authorization', 'Bearer ' + API_KEY);
// Problems: committed to Git, visible to all devs,
//           cannot rotate without deployment, fails code scan
apex — bad example

❌ Credentials in Custom Labels

// NEVER DO THIS — Custom Labels appear in exported metadata
String apiKey = Label.MyService_API_Key;
// Problems: visible in Setup → Custom Labels,
//           exported with change sets, in Git repository,
//           no encryption, visible to all admins
apex — bad example

❌ Credentials in Remote Site Settings URL

// NEVER include auth tokens in Remote Site Settings URLs
// Remote Site Setting: https://api.example.com?apikey=secret123
// The URL appears in Setup and in exported metadata unencrypted
setup — bad example

❌ Callout Endpoints Constructed from User Input

// NEVER construct endpoints from user input — SSRF vulnerability
String userInput = ApexPages.currentPage().getParameters().get('url');
req.setEndpoint(userInput);  // Attacker can call internal services!
// Use Named Credentials — the endpoint is always admin-controlled
apex — ssrf vulnerability
SSRF — Server-Side Request Forgery

If Apex builds callout URLs from user-supplied input, an attacker can make Salesforce call internal systems or metadata services. Named Credentials prevent this because the base URL is always defined by an administrator in Setup — it cannot be overridden by user input. The callout: syntax accepts the path portion only, not the domain.

CHAPTER SIX
Governance & Best Practices

DevOps, Rotation, and
Ongoing Governance

Secure integrations require ongoing governance — not just initial setup. This chapter covers metadata management for CI/CD, credential rotation strategies, and monitoring patterns that keep integrations healthy and auditable.

Metadata and DevOps

What Is and Isn't in Metadata

Named Credentials have a split between configuration (in metadata) and secrets (stored encrypted in Salesforce only):

ComponentIn Metadata (Deployable)Stored Encrypted (Not in Metadata)
Named CredentialName, URL, auth protocol reference—
External CredentialProtocol type, custom header definitionsClient secret, passwords, API keys, tokens
CertificateCertificate definition (name, algorithm)Private key (never exported)
Key Deployment Behaviour

Deploying Named Credentials via change set or Salesforce CLI creates the configuration shell in the target org — but secrets are NOT transferred. After deployment, an admin must manually enter the credentials in the target org. This is by design: secrets should never traverse CI/CD pipelines.

CI/CD Strategy for Named Credentials

  1. Source control: Commit Named Credential and External Credential metadata to Git (they contain no secrets)
  2. Automated deployment: Deploy the shell via Salesforce CLI sf project deploy start
  3. Secret injection: After deployment, use a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager) to inject credentials via the Salesforce Metadata API or UI
  4. Scratch orgs: Use environment variables or a secrets management sidecar to populate credentials in CI scratch orgs
# Deploy Named Credential metadata (no secrets included)
sf project deploy start \
  --source-dir force-app/main/default/namedCredentials \
  --target-org production

# After deploy: inject secrets via Tooling API or Manual UI
# curl to update the External Credential's encrypted field
curl -X PATCH \
  -H "Authorization: Bearer $SF_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"ClientSecret":"'$SECRET_FROM_VAULT'"}' \
  "https://org.salesforce.com/services/data/v62.0/tooling/sobjects/ExternalCredential/Id"
shell

Credential Rotation

Credentials should be rotated on a schedule and immediately upon suspected compromise. The Named Credential architecture makes rotation safe and low-risk.

Rotation Process for API Keys

  1. Generate new API key at the external service (while keeping old key active)
  2. Update the Custom Parameter in the External Credential in Salesforce
  3. Verify callouts work with the new key (run a test callout from Developer Console)
  4. Revoke the old key at the external service
  5. Update the rotation schedule in your secrets management system

Rotation Process for OAuth Client Credentials

  1. Generate new client secret in the external application registration
  2. Update the External Credential with the new secret in Salesforce
  3. Existing cached access tokens continue to work until they expire naturally
  4. Revoke the old client secret after confirming new tokens are being issued
Zero-Downtime Rotation

Most APIs support a brief dual-secret window: generate a new secret while the old one remains valid (usually 24–72 hours). Rotate Salesforce first, confirm traffic flows, then revoke the old secret. This gives a no-downtime rotation window.

Monitoring and Auditing

Event Monitoring for Callouts

Salesforce Shield Event Monitoring logs detailed callout activity including:

  • ApiEvent — every API request including callout endpoints
  • ApexExecutionEvent — which Apex classes executed and callout counts
  • NamedCredentialEvent — credential access and authentication events

Query these via the EventLogFile object in the REST API or use Salesforce's Log Streaming to forward to a SIEM.

Setup Audit Trail

Every change to a Named Credential or External Credential is recorded in Setup → Security → View Setup Audit Trail. This includes:

  • Who created or modified the credential
  • Timestamp of every change
  • What fields were changed (not the values — secrets are never logged)

Certificate Expiry Monitoring

// Scheduled Apex to alert on expiring certificates
global class CertExpiryMonitor implements Schedulable {
    global void execute(SchedulableContext ctx) {
        List<Certificate> certs = [
            SELECT DeveloperName, ExpirationDate
            FROM Certificate
            WHERE ExpirationDate <= :Date.today().addDays(90)
            AND ExpirationDate >= :Date.today()
        ];
        if (!certs.isEmpty()) {
            // Send alert email / publish Platform Event
            String msg = certs.size() + ' certificate(s) expiring within 90 days: ';
            for (Certificate c : certs) {
                msg += c.DeveloperName + ' (' + c.ExpirationDate + ') ';
            }
            Messaging.SingleEmailMessage email = new Messaging.SingleEmailMessage();
            email.setToAddresses(new String[]{ 'security-team@yourcompany.com' });
            email.setSubject('⚠️ Salesforce Certificate Expiry Alert');
            email.setPlainTextBody(msg);
            Messaging.sendEmail(new Messaging.Email[]{ email });
        }
    }
}

The Golden Rules

1. Never hardcode credentials — use Named Credentials or External Credentials. 2. Never put secrets in Custom Labels — they're in source control and visible to all admins. 3. Prefer OAuth or JWT over Basic Auth — no password storage. 4. Secrets are never in metadata — inject post-deployment from a vault. 5. Monitor certificate expiry — no built-in alert; implement it yourself. 6. Named Credential endpoint = SSRF protection — the domain is admin-controlled, not user-controllable.
