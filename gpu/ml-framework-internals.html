Tensor Internals

How a PyTorch tensor is represented in memory

PyTorch Tensor Object Layout Python Tensor (torch.Tensor) └── TensorImpl (C++ object on heap) ├── Storage ─── shared_ptr → raw data buffer (device + allocator) │ └── data_ptr() ← 256-byte aligned, owned by allocator ├── sizes_ [64, 128] ← shape as SmallVector ├── strides_ [128, 1] ← row-major layout (contiguous) ├── storage_offset_ 0 ← offset into storage in elements ├── dtype_ BFloat16 ← ScalarType enum ├── device_ cuda:0 ← Device struct └── autograd_meta_ ← null if no_grad, else AutogradMeta ├── grad_fn_ ← Node* in computation graph ├── grad_ ← accumulated gradient tensor └── requires_grad_← bool
Python · Inspecting tensor internals
import torch

x = torch.randn(3, 4)

# Storage — the raw memory backing the tensor
print(x.storage())             # shows raw float values
print(x.storage_offset())      # 0 for contiguous
print(x.data_ptr())            # address of first element

# Strides reveal memory layout
print(x.stride())              # (4, 1) — row-major
y = x.t()                      # transpose: no copy!
print(y.stride())              # (1, 4) — same storage, different strides
print(y.is_contiguous())       # False
z = y.contiguous()             # now copies, resets strides

# Views share the same storage
a = torch.zeros(12)
b = a.view(3, 4)
b[0, 0] = 99
print(a[0])                   # 99.0 — same data!

# Element access via strides: a[i,j] = storage[offset + i*stride[0] + j*stride[1]]
idx = x.storage_offset() + 1*x.stride(0) + 2*x.stride(1)
print(x[1, 2] == x.storage()[idx])  # True
💾

Memory Allocator

PyTorch uses a caching allocator for CUDA memory. Rather than calling cudaMalloc on every allocation (slow), it maintains a pool of free blocks and reuses them. The allocator splits large blocks and coalesces small ones.

  • torch.cuda.memory_allocated(): live tensors
  • torch.cuda.memory_reserved(): caching allocator pool
  • torch.cuda.empty_cache(): release reserved back to CUDA
  • PYTORCH_NO_CUDA_MEMORY_CACHING=1 to disable
🗜️

Views vs Copies

Operations like reshape, transpose, squeeze, expand return views — they share storage. Operations that require contiguity (e.g. view on a non-contiguous tensor) raise an error. Use reshape which automatically calls contiguous if needed.

  • view(): must be contiguous (raises otherwise)
  • reshape(): safe — may copy or return view
  • as_strided(): arbitrary strides, powerful but dangerous
  • expand(): broadcast without copy (stride=0)

Autograd Engine

Dynamic computation graphs, backward passes, and custom gradient functions

Every tensor with requires_grad=True carries a grad_fn — a pointer to the C++ Node that created it. When you call .backward(), PyTorch traverses the DAG in topological order, calling each node's apply() to accumulate gradients via the chain rule. The engine runs the backward pass in a thread pool, parallelising independent branches.

Computation Graph for z = (x * y).sum() x (leaf) y (leaf) └──────┬──────┘ MulBackward0 ← grad_fn of (x*y) │ SumBackward0 ← grad_fn of z │ z (root) z.backward() traverses upward, calling each Node.apply(grad_output) SumBackward0.apply(ones) → passes ones to MulBackward0 MulBackward0.apply(ones) → x.grad += ones * y, y.grad += ones * x
Python · Custom autograd Function
import torch
from torch.autograd import Function

class SigmoidFunction(Function):
    @staticmethod
    def forward(ctx, x):
        s = 1 / (1 + torch.exp(-x))
        ctx.save_for_backward(s)   # save for reuse in backward
        return s

    @staticmethod
    def backward(ctx, grad_output):
        s, = ctx.saved_tensors
        return grad_output * s * (1 - s)   # chain rule: dL/dx = dL/ds * ds/dx

sigmoid = SigmoidFunction.apply

x = torch.tensor([1.0, 2.0], requires_grad=True)
y = sigmoid(x)
y.sum().backward()
print(x.grad)   # [0.1966, 0.1050]

# Inspect the graph
print(y.grad_fn)              # <SigmoidFunctionBackward>
print(y.grad_fn.next_functions)  # [(AccumulateGrad for x, 0)]

Dispatch & ATen

How torch.add(x, y) finds the right kernel at runtime

PyTorch's operator dispatch is a multi-level key system. When you call torch.add(x, y), the dispatcher looks at the tensor's dispatch keys (a bitset encoding device, dtype, autograd state, etc.) and routes to the highest-priority registered implementation. This layered system allows the same op name to work on CPU, CUDA, XLA, MPS, and fake/meta tensors with zero user-visible branching.

PyTorch Dispatcher Call Stack for torch.add(x_cuda, y_cuda) Python: torch.add(x, y) ↓ Python binding → C++ Dispatcher: compute key set from tensors ↓ key = Autograd | CUDA | BFloat16 Autograd key: record op in graph, call into next ↓ CUDA key: select CUDA kernel (add_cuda_out) ↓ ATen generated code cuBLAS / custom CUDA kernel ↓ returns result Tensor Autograd: attach grad_fn = AddBackward0 ↓ Python: result tensor with requires_grad
Python · Registering a custom operator
import torch
from torch.library import Library, impl

# Declare the op schema
mylib = Library("myops", "DEF")
mylib.define("mish(Tensor x) -> Tensor")

# CPU implementation
@impl(mylib, "mish", "CPU")
def mish_cpu(x):
    return x * torch.tanh(torch.nn.functional.softplus(x))

# CUDA implementation
@impl(mylib, "mish", "CUDA")
def mish_cuda(x):
    return x * torch.tanh(torch.nn.functional.softplus(x))   # or call custom kernel

# Autograd rule
@impl(mylib, "mish", "Autograd")
def mish_autograd(x):
    return torch.ops.myops.mish(x)   # will be differentiated via autograd

# Use it like a native op
x = torch.randn(10, requires_grad=True)
y = torch.ops.myops.mish(x)
y.sum().backward()

torch.compile Internals

TorchDynamo graph capture → TorchInductor code generation

torch.compile Pipeline Python source code (model.forward) ↓ TorchDynamo — Python bytecode interpreter hook FX Graph (torch.fx.GraphModule — pure ATen ops) ↓ AOTAutograd — ahead-of-time backward graph Joint fwd+bwd FX Graph ↓ TorchInductor — code generation backend ├─→ Triton kernels (CUDA / ROCm) └─→ C++ / OpenMP (CPU) ↓ Compiled callable (fast as a native library)
Python · torch.compile usage & modes
import torch

model = MyTransformer().cuda()

# Default: balance of compile time and runtime performance
compiled = torch.compile(model)

# max-autotune: tries many tile sizes, slowest compile, fastest run
compiled = torch.compile(model, mode="max-autotune")

# reduce-overhead: CUDA graphs + minimal overhead  (fixed shapes)
compiled = torch.compile(model, mode="reduce-overhead")

# Inspect the captured FX graph
from torch._dynamo import export
graph_module, guards = export(model)(sample_input)
graph_module.graph.print_tabular()

# Disable compilation for a specific function
@torch.compiler.disable
def my_debug_fn(x):
    return x.cpu().numpy()   # can't compile numpy ops
How Dynamo works

TorchDynamo installs a Python frame evaluation hook (PEP 523). When a compiled function is called, Dynamo intercepts the bytecode execution, traces through it symbolically, and builds an FX graph of the ATen operations. If it hits Python code it can't trace (data-dependent control flow, NumPy, custom C extensions), it "breaks" the graph there and falls back to eager execution — called a graph break. Minimising graph breaks is key to getting full compilation benefit.

DataLoader & Prefetching

Multi-process data loading, memory-mapped datasets, and GPU prefetching

Python · DataLoader internals & tuning
from torch.utils.data import DataLoader, Dataset
import torch

class TokenDataset(Dataset):
    def __init__(self, mmap_file):
        # Memory-mapped file: avoids loading entire dataset into RAM
        self.data = torch.from_file(mmap_file, dtype=torch.int32, shared=True)

    def __len__(self): return len(self.data) // 2048

    def __getitem__(self, idx):
        chunk = self.data[idx*2048:(idx+1)*2048]
        return chunk[:-1], chunk[1:]  # input, target

loader = DataLoader(
    TokenDataset("tokens.bin"),
    batch_size=32,
    num_workers=8,           # spawn 8 worker processes
    pin_memory=True,         # page-lock CPU buffers for fast H2D
    persistent_workers=True, # keep workers alive between epochs
    prefetch_factor=4,       # pre-load 4 batches per worker
)

# Inside DataLoader: worker processes call __getitem__ in parallel,
# fill a shared-memory queue, main process batches + pins + sends to GPU.

Optimiser Internals

Adam, fused kernels, and per-parameter learning rate schedules

Python · Adam update — step-by-step
# What Adam.step() does internally for each parameter group
def adam_step(p, grad, m, v, t, lr, beta1, beta2, eps, wd):
    # Weight decay (decoupled AdamW style)
    p.mul_(1 - lr * wd)
    # First moment (EMA of gradients)
    m.mul_(beta1).add_(grad, alpha=1 - beta1)
    # Second moment (EMA of squared gradients)
    v.mul_(beta2).addcmul_(grad, grad, value=1 - beta2)
    # Bias-corrected estimates
    m_hat = m / (1 - beta1**t)
    v_hat = v / (1 - beta2**t)
    # Parameter update
    p.addcdiv_(m_hat, v_hat.sqrt().add_(eps), value=-lr)

# PyTorch's fused Adam runs ALL of the above in one CUDA kernel,
# avoiding 5 separate memory passes over params + grads + states.
optim = torch.optim.AdamW(
    model.parameters(), lr=3e-4, betas=(0.9, 0.95), fused=True
)
📉

LR Schedulers

Cosine annealing with warmup is the standard for LLM training. PyTorch schedulers call optimizer.param_groups[i]["lr"] = new_lr each step.

  • Warmup: linearly ramp lr for first N steps to stabilise early training
  • Cosine decay: smoothly reduce from peak to min_lr over training horizon
  • OneCycleLR: fast convergence for shorter runs (super-convergence)
  • ReduceLROnPlateau: adaptive, driven by validation metric
🧪

Gradient Accumulation

Simulate large effective batch sizes when GPU memory is limited. Accumulate gradients over N micro-batches before calling optimizer.step(). With DDP, disable gradient synchronisation during accumulation to avoid wasted AllReduce calls.

  • effective_batch = micro_batch × accum_steps × world_size
  • DDP: model.no_sync() context manager avoids premature AllReduce
  • Scale loss by 1/accum_steps before backward to keep gradient magnitudes correct
