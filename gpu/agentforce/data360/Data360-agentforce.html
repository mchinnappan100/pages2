CHAPTER ONE
Foundations

Understanding Data 360

Data 360 is Salesforce's unified data platform — the engine that collects, harmonises, and activates customer data across every touchpoint. Before building intelligent agents, you need to understand what Data 360 is and how its two operational layers unlock different Agentforce capabilities.

From Data Cloud to Data 360

Salesforce rebranded Data Cloud to Data 360 on October 14, 2025. The rebrand reflects the platform's evolution into a complete 360-degree customer data strategy — unifying structured and unstructured data from every source across the Salesforce ecosystem and beyond.

If you encounter the name "Data Cloud" in Salesforce documentation, UI labels, or legacy resources, it refers to the same product now called Data 360. All existing functionality, APIs, and integrations continue to work unchanged.

Layer 1 — Data 360 Enabled

Enabling means you have provisioned and turned on Data 360 in your Salesforce org. This is the minimum requirement for any Agentforce use. Without this layer:

  • The Agentforce Data Library does not function
  • The Einstein Trust Layer does not work
  • Core Agentforce features are unavailable

Layer 2 — Data 360 Implemented

Implementing goes further — you have enabled Data 360 and also:

  • Connected your data sources to Data 360
  • Mapped data to Data 360 data models
  • Optionally harmonised data with identity resolution rulesets
  • Set up transforms, real-time ingestion, zero-copy connections, and other features

Implementation unlocks the full power: diverse unified customer data, powerful data quality tools, and fully customisable RAG (Retrieval-Augmented Generation) for your agents.

Key Rule

Data 360 must be provisioned and enabled for all Agentforce use — not just advanced features. It is a hard prerequisite, not an optional enhancement.

Credits and Digital Wallet

All Data 360 features described in this guide — including those that power Agentforce — consume Data 360 credits. Credits are managed through the Salesforce Digital Wallet, which tracks consumption across features and provides visibility into usage.

Before enabling advanced capabilities, consult the Data 360 Credit Consumption: Quick Look and Salesforce Digital Wallet: Quick Look resources to understand the cost model for your implementation.

CHAPTER THREE
Data Foundation Pillar 1

Unified Customer 360

The unified customer 360 profile is the centrepiece of Data 360. It consolidates every data point about a customer — across every Salesforce cloud and external source — into a single, coherent picture. Agents built on this foundation know your customers inside and out.

What Unified Customer 360 Means

Unified Customer 360 is the result of:

  1. Bringing together data from across Salesforce clouds (Sales, Service, Marketing, Commerce) and external systems
  2. Running identity resolution rulesets to match records belonging to the same customer across sources
  3. Building a unified profile of each customer with full historical context across all data sources

Impact on Agents

Agents built on Data 360's unified profiles can move from generic responses to deeply personalised ones. Instead of asking "what can I help you with today?" with no context, the agent already knows:

  • What the customer recently purchased
  • Which marketing campaigns they engaged with
  • What their browsing history suggests they want next
  • Their support history and open issues

NTO Use Case: Product Recommendation

Northern Trail Outfitters (NTO) built a service agent that handles customer inquiries, including recommending products. Here is how unified customer 360 data transforms the recommendation quality:

NTO Scenario — Unified Customer 360
🏔 "Can you recommend a product for me?"
1
Customer asks NTO's agent for a product recommendation.
2
With unified profiles, the agent sees the customer's recent purchase: shoes from NTO's new line.
3
The agent also sees the customer recently engaged with the marketing campaign for the same line — specifically clicking on the jacket and the shoes.
4
Since the customer already purchased the shoes, the agent recommends the jacket — a relevant, personalised suggestion rather than a generic one.

Without unified data, the agent would either have no purchase history or would see only the CRM record — missing the marketing engagement data that reveals what the customer is interested in.

Identity Resolution

A customer might exist in multiple systems: as a Sales Cloud contact, a Marketing Cloud subscriber, a Commerce Cloud shopper, and a loyalty programme member. Without identity resolution, these are treated as separate people.

Data 360's identity resolution rulesets match records across systems using configurable rules (email address, phone, name + address matching, etc.) and merge them into a single unified individual record. Agents work with this merged record, ensuring they have the complete picture regardless of which channel the customer used.

Agentforce Impact

Unified profiles transform agent responses from generic and disjointed to personalised and contextually aware — considering the customer's full history across every Salesforce cloud and connected system.

CHAPTER FOUR
Data Foundation Pillar 2

Data Transforms

Raw data ingested from multiple sources is rarely clean or consistent. Data transforms allow you to cleanse, standardise, and reshape data before it reaches your agents — preventing subtle data quality issues from causing agent failures in production.

The Data Quality Problem for Agents

Agents rely on data being accurate and consistent to reason and act correctly. Data quality issues that seem minor — inconsistent date formats, varying naming conventions, mismatched units — can cause agents to:

  • Produce incorrect responses (wrong calculations, wrong recommendations)
  • Produce ambiguous responses (uncertain which record to use)
  • Produce no response at all (unable to process the data)

Data 360's transform capabilities let you cleanse and transform data before agents consume it, resolving quality issues at the data layer rather than trying to work around them in agent instructions.

NTO Use Case: Refund Processing

NTO Scenario — Data Transforms
🔄 "I'd like to refund my order."

Before the transform (broken):

1
NTO ingests order records from Agentforce Commerce into Data 360. The data hasn't been cleaned yet.
2
A customer asks the agent to refund their order. The agent looks up the order in Data 360.
3
The order's date is in DD/MM/YY format. NTO's return policy requires orders within 30 days, but the formula only works with MM/DD/YY.
✗
The agent cannot process the refund. The data quality issue blocks a valid customer request.

After the transform (fixed):

✓
NTO runs a data transform to convert all dates to MM/DD/YY. Now the 30-day refund formula works correctly and agents can successfully process refunds.

Batch vs Streaming Transforms

Data 360 supports two types of transforms to cover different data patterns:

TypeWhen It RunsBest For
Batch TransformsOn a schedule (hourly, daily, etc.)Historical data, large-scale cleansing, periodic enrichment
Streaming TransformsIn real time as data arrivesLive data that agents act on immediately
Important

Data transform issues are silent failures at the agent layer — the agent simply gets no result or a wrong result, with no obvious indication that a data quality problem is the cause. Always validate transform correctness in a sandbox before deploying agents to production.

CHAPTER FIVE
Data Foundation Pillar 3

Real-Time Data

The third pillar of Data 360 for Agentforce is real-time data ingestion and activation. Real-time capabilities allow agents to react to what a customer is doing right now — not what they did last week. This enables a fundamentally different class of agent response: contextual, timely, and immediately relevant.

Real-Time Ingestion and Activation

Data 360 can ingest, unify, analyse, and act on data in real time. Events from web sessions, mobile apps, and other real-time streams are immediately available to agents without batch delays.

This capability is what separates a "knows your history" agent from a "knows what you're doing right now" agent — a significant difference in perceived intelligence and relevance.

NTO Use Case: Live Promotions

NTO Scenario — Real-Time Data
⚡ "What promotions are available right now?"
1
As a customer browses NTO's website, they open the agent chat and ask for current promotions.
2
The agent sees the customer's real-time website interactions — and recognises that the customer is currently clicking on backpacks.
3
Instead of listing every active promotion across the entire catalogue, the agent gives the most relevant promotion for what the customer is looking at right now: buy one get one on select backpacks.

Without real-time data, the agent could only respond with generic current promotions, missing the opportunity to connect the promotion directly to the customer's present intent. With real-time data, the agent becomes a dynamic assistant that responds to the moment.

Real-Time Use Cases

Real-time data is especially powerful for: personalised promotions during active browsing, proactive service when errors are detected in real time, product recommendations based on current session behaviour, and fraud prevention alerts during active transactions.

CHAPTER SIX
Data Foundation Pillar 4

Zero Copy

Many enterprises store critical data outside Salesforce — in Databricks, Snowflake, Google BigQuery, or other systems. Zero Copy is Data 360's mechanism for federating access to that data without duplicating it, extending the reach of your agents beyond the Salesforce ecosystem.

What is Zero Copy?

Zero Copy lets you create bidirectional communication between Data 360 and external data systems. Instead of copying data into Salesforce (which is expensive, creates duplication, and creates synchronisation challenges), Zero Copy provides a live connection that reads data in place.

Key Properties of Zero Copy

  • No data duplication — external data stays where it lives; only metadata is managed in Salesforce
  • Bidirectional — Salesforce can read from and write to the external system
  • Unified profiles enriched — external data can enrich the unified customer profile
  • Agent-accessible — agents can query the enriched profile, indirectly accessing external data without knowing it's external

NTO Use Case: Loyalty Programme Enrichment

NTO Scenario — Zero Copy
💎 "What promotions can I get as a member?"
1
NTO stores loyalty programme data in Databricks (not in Salesforce). They use a zero-copy connection to bring the data into Data 360 and enrich unified profiles with loyalty tier and points information.
2
A customer asks the agent for current promotions.
3
The agent looks up the customer's unified profile, which has been enriched with loyalty programme data from Databricks.
4
The agent sees that the customer is a platinum member, which entitles them to an exclusive promotion not available to regular customers.
✓
NTO's agent recommends the exclusive platinum promotion — a response that would have been impossible without zero-copy access to the external loyalty data.
Expanding Agent Reach

Zero Copy is what allows agents to act on data from any system in your enterprise — not just Salesforce. ERP data, loyalty platforms, data warehouses, and custom databases can all enrich agent responses through zero-copy connections.

CHAPTER SEVEN
Next Steps

Retrieval-Augmented Generation
and What Comes Next

With data connected, transformed, and unified in Data 360, you have the foundation needed to configure Retrieval-Augmented Generation (RAG) for your Agentforce agents. RAG is the mechanism that allows agents to retrieve the most relevant data at inference time and inject it into their responses.

What is RAG in Agentforce?

Retrieval-Augmented Generation (RAG) is the process of:

  1. Retrieving relevant data from Data 360 at the time the agent is responding
  2. Augmenting the agent's prompt with that retrieved data as context
  3. Generating a response that is grounded in your actual enterprise data

Without RAG, agents generate responses based only on their training — which knows nothing about your specific customers, products, or business context. With RAG powered by Data 360, agents produce responses grounded in real, current, customer-specific data.

This chapter walks through the end-to-end process of implementing Data 360 for an Agentforce project — from understanding your data requirements to building and deploying a production agent grounded in unified, transformed data. Follow NTO's enterprise architect Pia as she builds a product-issue service agent from scratch.

Understanding Your Data Requirements

Before touching any Salesforce configuration, you need to understand what data your agent needs and where it lives. This planning step prevents costly rework later.

Data Prep Planning Questions

Planning DimensionQuestions to Answer
What data is needed? Do you need CRM records (leads, cases, contacts)?
Do you need unstructured or structured data?
Do you need to upload files (PDFs, docs)?
Where does the data live? If CRM records — which org are they currently in?
Is the data in an external data warehouse? (→ zero-copy connector)
Do you need real-time access?
What processing is needed? Does the data have quality issues? What type? How widespread?
Do you need to unify the data via identity resolution?
Are transforms (batch or streaming) required?
Plan Before You Build

Answering these questions before configuring Data 360 ensures you connect the right sources, apply the right transforms, and create the right data models. Skipping this step leads to partial data, quality issues in production, and rework.

NTO Use Case: Pia Builds a Product-Issue Agent

NTO wants to build an agent that responds to questions about product issues. NTO plans to ground its prompts with advanced Data 360 RAG. Follow NTO's enterprise architect Pia as she configures Data 360 from scratch.

Starting Setup

  1. Enable and provision Data 360 in the org
  2. Set up Data 360 users with appropriate permission sets
  3. Identify the data source: customer case data in Service Cloud
NTO Source Data — Service Cloud Cases
📋 Customer Case Records
1

2
Case 2: Pants style 2000 is too long
3
Case 3: Fitness watch doesn't charge quickly

Step 1 — Ingest: Create a Data Stream

Pia creates a data stream that ingests case data from Service Cloud into Data 360. Case data is stored in the Case Data Lake Object (DLO) — the raw, unharmonised representation of the ingested records.

Step 2 — Transform: Clean the Data

Pia applies a batch data transform to resolve data quality issues — inconsistent names, formats, and other problems. The cleaned output is stored in a new Cleaned Case DLO.

Step 3 — Map: Connect DLO to DMO

Pia adds mappings between the Cleaned Case DLO and the Case Data Model Object (DMO). This step harmonises the raw data into Salesforce's standard data model, making it queryable and joinable with other objects.

Step 4 — Unify: Run Identity Resolution

Pia creates and runs an identity resolution ruleset to link each case to the customer's unified profile.

NTO — Identity Resolution Enriches Context
🔗 Cases Linked to Unified Profiles
1
Case 1 (pants style 1068 too short) → linked to customer Rachel Rodriguez. Rachel's unified profile shows her height: 170 cm.
2
Case 2 (pants style 2000 too long) → linked to a customer whose unified profile shows height: 165 cm.
✓
Now the agent has customer context (height) alongside each case — enabling much more relevant product recommendations.
Why This Matters

Without identity resolution, the agent sees only raw cases — no customer context. With unified profiles, the agent knows the customer's height when a pants case is raised, enabling it to recommend sizes that fit that specific customer rather than giving generic advice.

  1. Create a search index from the Case DMO — indexes all cleaned case records for semantic retrieval
  2. Create a search index from the Unified Individual DMO — indexes customer profiles including enriched attributes like height
  3. Data 360 automatically creates a retriever for each search index — the retriever is the bridge between search indexes and prompt templates
  4. Optionally create an ensemble retriever to search both indexes simultaneously in a single retrieval call
  5. Create a prompt template that calls the retriever(s) — the retriever populates the prompt with the most relevant cases and profile data for each customer question

Building the Agent

With the RAG pipeline ready, Pia creates the agent:

  1. Creates a new agent from an Agentforce Service Agent template
  2. Creates a permission set including access to Prompt Builder, the Case DMO, and the Unified Individual DMO — assigns it to the agent user
  3. Creates an agent action named "Answer Questions with Case" that uses the prompt template
  4. Adds a subagent to the agent
  5. Adds the Answer Questions with Case action to the subagent
  6. Activates and tests the agent
NTO — Agent in Action
👟 Testing the Product-Issue Agent
1
Pia asks: "I have long legs and I'm 170 cm tall. Which pant styles do you recommend and not recommend?"
2
The agent retrieves cases from the Case DMO and the customer's unified profile from the Unified Individual DMO.
✓
Agent answers: "Customers with similar heights report that style 1068 is too short. Customers report that style 2000 is longer. I recommend style 2000, and I don't recommend style 1068."

Post-Deployment Best Practices

After deploying the agent to customer channels, NTO follows these ongoing best practices to ensure continued quality and relevance:

  • Monitor the audit trail and give feedback on agent responses — use thumbs-up/down and reason text
  • Edit prompts to address critical feedback from the audit trail
  • Monitor performance with Agentforce Analytics dashboards (containment rate, acceptance rate, toxicity)
  • Update your RAG implementation to keep your agents' data sources current as products, cases, and customer profiles evolve
