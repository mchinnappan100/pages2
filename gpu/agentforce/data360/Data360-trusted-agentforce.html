CHAPTER ONE
Grounding Agents

Retrieval-Augmented Generation (RAG)

RAG is the mechanism that connects agents to real enterprise data at inference time. Instead of answering from training data alone, a RAG-powered agent retrieves relevant information from a knowledge base and injects it into the prompt — dramatically improving accuracy, relevance, and trustworthiness.

What is RAG?

RAG stands for Retrieval-Augmented Generation — three words that describe a precise pipeline:

  • Retrieval — relevant information is fetched from a knowledge base using a retriever
  • Augmentation — the retrieved information is combined with the original user prompt
  • Generation — the LLM generates a response that is now grounded in real, specific data

Grounding is the broader concept: adding contextual, specific information to prompts to enhance the quality, accuracy, and relevance of LLM-generated output. RAG is the primary mechanism for grounding in Agentforce.

Key RAG Terminology

TermDefinition
Unstructured data Data without a specific, consistent format that can't be easily stored in relational databases (e.g., Knowledge articles, PDFs, web pages). Must be chunked before vectorising.
Chunking Splitting large documents into smaller, searchable pieces before creating embeddings. Chunk size affects retrieval precision.
Vector embeddings A numerical representation of unstructured data that machines can read. Measures semantic similarity between pieces of text, enabling accurate and relevant retrieval.
Vector data store A type of database designed specifically to store and query vector embeddings efficiently.
Search index A data structure that stores chunked, vectorised data that can be searched and retrieved by other applications.
Retriever The link between the prompt and the search index. Searches data sources for relevant information to augment the prompt.
Ensemble retriever A collection of individual retrievers that conducts parallel searches from multiple sources simultaneously, then combines results.

CHAPTER TWO
Quick-Start RAG

Agentforce Data Library (ADL)

The Agentforce Data Library is a pre-configured, quick-start RAG solution that works with Data 360 enabled. It's designed to get agents grounded in data quickly, without requiring full Data 360 implementation — but it comes with specific limitations you need to understand.

What the Agentforce Data Library Does

When you add a Data Library — either from Agentforce Builder or from Setup — Salesforce automatically builds a complete RAG-powered solution using default settings. This includes all three required components:

  • Vector data store — automatically provisioned using Data 360 storage
  • Search index — automatically created and maintained
  • Retriever — automatically configured to connect your agent's prompts to the index

You can customise each component individually after the initial setup, but you don't have to — the defaults work out of the box.

Requirement

The Agentforce Data Library requires only Data 360 enabled (provisioned and turned on). Full Data 360 implementation is not required for ADL.

Supported Data Sources

ADL supports the following unstructured data source types:

  • 📄 Knowledge articles — Salesforce Knowledge base articles
  • 📁 Files — uploaded documents and attachments
  • 🌐 Web search — open web search results
  • 🔧 Custom retriever — a custom retriever you build yourself

ADL Limitations

The trade-off for the quick, easy setup is a constrained set of capabilities. Key limitations:

  • Unstructured data only — no structured CRM data, no records
  • Single source per library — each data library can contain only one data source; no multi-source integration
  • No real-time capabilities — data is not updated in real time
  • No zero-copy capabilities — non-CRM data from external lakes must be physically downloaded as a file and added to the library; there is no live federation
  • No data harmonisation — data is not unified or deduplicated across sources
When to Use ADL

ADL is ideal for: quick prototypes, Knowledge article grounding, low-complexity agents with a single data source, and teams that need to get agents live quickly without full Data 360 implementation. For production agents serving complex, multi-source enterprise data needs, evaluate the advanced Data 360 setup.

CHAPTER THREE
Advanced RAG

Customisable RAG with
Full Data 360 Setup

When the Agentforce Data Library's quick-start approach is insufficient, implementing RAG directly in Data 360 gives you complete control. This path requires more setup effort but unlocks structured data, multi-source integration, real-time grounding, zero-copy federation, and data harmonisation.

Full Custom RAG Setup

Advanced RAG with Data 360 involves building the full pipeline manually:

  1. Ingest data — bring CRM records (including unstructured long-text fields), external files, real-time events, and zero-copy data into Data 360
  2. Chunk the data — split long-text content into searchable pieces
  3. Create a vector data store — set up and configure the embedding storage
  4. Build a search index — create a queryable index over your embeddings
  5. Configure a retriever — link the search index to your agent prompts
  6. Use in agents — reference the retriever from your Agentforce prompts

The result: agents grounded in harmonised, real-time, zero-copy data — both structured and unstructured — providing a comprehensive understanding of your customers and organisation.

Requirement

Full customisable RAG requires Data 360 implemented — provisioned, enabled, with data connected and mapped to data models.

Data Cloud One — Cross-Org Retriever Sync

With Data Cloud One connections, retrievers built in Data 360 are synced from the home org to companion orgs. This allows you to use the same retriever in prompts and flows across multiple Salesforce orgs without rebuilding the pipeline in each one.

Data Graphs for RAG

Data graphs provide a flattened view of related data as a JSON string that can be retrieved quickly. For example, a data graph can model the relationship between a customer profile and their sales order details — providing full relational context to an agent in a single retrieval operation.

Benefits of Data Graphs for RAG

  • Multi-source integration without ensemble retrievers — data graphs natively integrate CRM data and external lake data (via Zero Copy) without requiring a separate ensemble retriever configuration
  • JSON format — data is automatically converted to JSON, which maintains relational structure and is easily understood by LLMs
  • Fast retrieval — the flattened JSON structure is pre-computed, enabling faster retrieval than joining multiple sources at query time

ADL vs Full Data 360 Setup — Full Comparison

Agentforce Data Library

  • Requires Data 360 enabled only
  • ✓ Fast, easy setup of complex pipelines
  • ✗ Limited to Knowledge, files, web search, custom retriever
  • ✗ One data source per library — no multi-source
  • ✗ No real-time capabilities
  • ✗ No zero-copy — external data must be downloaded as file
  • ✗ No data harmonisation or unified profiles

Customisable Data 360 Setup

  • Requires Data 360 implemented
  • ✗ More complex setup (ingestion, modeling, identity resolution)
  • ✓ Ensemble retrievers or data graphs for multi-source
  • ✓ Data Cloud One: retrievers synced to companion orgs
  • ✓ Data transformation, mapping, harmonisation
  • ✓ Harmonised, real-time, zero-copy data
  • ✓ Structured AND unstructured data

CHAPTER FOUR
AI Security

The Einstein Trust Layer

The Einstein Trust Layer is Salesforce's data governance framework for AI. It sits between your agents and the LLM provider, enforcing security, privacy, and safety controls on every prompt and response — powered by Data 360.

What the Einstein Trust Layer Protects

The Einstein Trust Layer protects customer data and ensures responsible AI use across the Salesforce ecosystem through a set of robust security features:

  • Zero Data Retention (ZDR) — the LLM provider does not store, log, or use for training any data sent through the Trust Layer
  • Toxicity Detection — scans LLM responses for harmful, offensive, or inappropriate content before delivery to users
  • Secure data retrieval — ensures that data retrieved for grounding is retrieved securely and only used for the intended purpose
  • Dynamic grounding — enables real-time, contextual grounding of prompts with enterprise data
  • PII masking / Sensitive Data Detection — detects and masks personally identifiable information before prompts reach the LLM
  • Prompt injection detection — identifies adversarial attempts to override system instructions
Requirement

The Einstein Trust Layer requires Data 360 enabled. It will not function without Data 360 provisioned and enabled in your org — regardless of whether you've implemented Data 360 or not.

Trust Layer in the Prompt Pipeline

Every agent interaction passes through the Einstein Trust Layer:

👤 User Prompt Raw user message sent to the agent
↓
🔍 Prompt Injection Detection Scans for adversarial override attempts before processing
↓
🎭 PII / Sensitive Data Masking Detects and masks personal data before it reaches the LLM
↓
📡 Dynamic Grounding (RAG) Retrieves and injects relevant enterprise context into the prompt
↓
🤖 LLM Call (ZDR) Augmented prompt sent to LLM with Zero Data Retention guarantee
↓
🚫 Toxicity Detection Scans LLM response for harmful content before delivering to user
↓
✅ Safe Response to User Grounded, secure, compliant response delivered

CHAPTER FIVE
Accountability

Generative AI Audit &
Feedback Trail

The Audit & Feedback Trail provides the data infrastructure needed to track every AI agent action and output. This data is stored and analysed in Data 360, enabling compliance reviews, quality improvement, and human oversight of AI behaviour.

What Gets Logged

The Generative AI Audit Trail logs detailed information about every agent interaction. This data is stored in Data 360 and available for analysis:

Prompt Data
  • Prompt ID
  • User data (who sent the prompt)
  • Original prompt text
  • PII-masked version of the prompt
Safety Scores
  • Safety score (overall)
  • Toxicity score for the response
  • Trust Layer intervention flags

Feedback API

Beyond passive logging, the Audit Trail supports active human feedback through the Feedback API. This allows users — and reviewers — to rate and correct agent responses, building a continuous improvement loop:

Feedback Reactions
  • 👍 Thumbs-up reaction
  • 👎 Thumbs-down reaction
  • 📝 Reason text explaining the rating
Response Actions
  • Accept (response used as-is)
  • Regenerate (request a new response)
  • Modify (user edits the response)
  • Decline / Ignore (response not used)
  • Modified final response text
Why the Audit Trail Matters

The Audit Trail is not just a compliance tool — it is the foundation for continuous agent improvement. By analysing which responses are accepted, rejected, or modified, you can identify systematic agent weaknesses, refine prompts and grounding, and tune agent behaviour over time. The data is stored in Data 360, making it available for reporting and analysis.

Requirement

The Generative AI Audit & Feedback Trail requires Data 360 enabled. The trail data is stored and processed within Data 360's infrastructure.

CHAPTER SIX
Performance Monitoring

Agentforce Analytics

Once agents are deployed, you need to monitor their performance continuously. Agentforce Analytics provides pre-built insight dashboards powered by Data 360, giving you visibility into agent safety, usage trends, and effectiveness — without building a reporting infrastructure from scratch.

Pre-Built Insight Dashboards

Agentforce Analytics ships with pre-built dashboards that surface the most critical agent performance metrics:

Dashboard TopicWhat It ShowsWhy It Matters
Data Masking How often PII was detected and masked in prompts Validates that the Trust Layer is actively protecting sensitive data
Toxicity in Responses Toxicity scores and flagged response rates over time Identifies safety regressions as agent configuration evolves
User Trends Volume, session length, most common request types Reveals which use cases drive the most agent usage
Acceptance Rates What % of agent responses users accept, regenerate, or modify Measures response quality; low acceptance rates indicate agent improvement areas

Custom Dashboards

Beyond the pre-built dashboards, Agentforce Analytics supports fully custom dashboards using Data 360's reporting and dashboard capabilities. You can build reports tailored to your specific business requirements — combining agent performance data with other Data 360 data sources.

Infrastructure

All Agentforce Analytics data is stored and processed in Data 360. Dashboards and reports are displayed using Data 360 dashboards and reports. Requires Data 360 enabled.

Continuous Improvement Loop

Acceptance rate trends from Analytics → identify low-quality response patterns → improve grounding data in Data 360 → retune prompts → monitor acceptance rate improvement. This is the data-driven agent improvement cycle that Data 360 enables.
