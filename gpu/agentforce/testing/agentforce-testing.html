The Agent Development Lifecycle (ADL)

AI agents and the rise of generative AI are reshaping how Salesforce administrators and developers build solutions. While the familiar Application Lifecycle Management (ALM) stages apply to agents too, generative AI adds unexpected complexity — especially around testing.

ADL vs Traditional ALM

Traditional ALM is mostly sequential — plan everything, then build. The ADL is iterative — you refine, test, and revise while you build. This loop between Configure → Test → Configure is what makes agent development different.

CHAPTER ONE
Foundations

Why Testing Agents Matters

Agent testing is fundamentally different from testing a traditional application. Understanding why — and what makes it uniquely challenging — is the first step to building accurate, reliable agents.

What Agent Testing Validates

Testing validates that your agent's performance aligns with the specifications you designed — the same criteria you defined during planning:

  • Audience — does the agent respond appropriately for your intended users?
  • Scope — does the agent stay within the boundaries you set?
  • Use cases — does it handle the tasks it was designed for?
  • Guardrails — does it respect the limits and restrictions you defined?
  • Tasks — does it call the right subagents, actions, and data?

When an agent gives an inaccurate response, an error message, or a hallucination, the cause might be inside an instruction, an action, data, or a permission set. Testing helps you identify which of these needs refinement.

Probabilistic vs Deterministic Testing

This is the most important conceptual shift for teams moving from traditional development to agent development:

Traditional App Testing (Deterministic)

  • Plan every detail before building
  • Success = predictable, repeatable results
  • Either works as intended, or it doesn't
  • Same input always produces same output
  • Rules-based logic — no ambiguity

Agent Testing (Probabilistic)

  • Plan upfront, then refine while building
  • Results can be varied, unique, surprising
  • Same input can produce many correct responses
  • Or occasionally incorrect ones, or hallucinations
  • Generative AI lacks rules-based logic
The Core Challenge

Because the same input can generate many different yet still correct responses, it is difficult to anticipate all the ways a user might interact with your agent. You need to account for and test a wide variety of scenarios — not just the happy path. This is why batch testing at scale is essential.

CHAPTER TWO
Testing Tools

Agentforce Builder Testing &
Troubleshooting Tools

Agentforce Builder provides three powerful tools for testing and troubleshooting your agent before launch. These tools let you iterate quickly and understand exactly how your agent reaches its responses.

👁️

① Preview Panel

Converse with your agent interactively. Choose between Simulate (mock data) and Live Test (real data) modes.

📋

③ Interaction Summary

Review the steps at a high level: which subagents and reasoning were called to generate the response.

🔍

④ Session Tracing

View every detail of an agent session — reasoning executions, actions, prompts, gateway inputs/outputs, and error messages. Requires Data 360.

🧾

Test Suites (Beta)

Batch test dozens or hundreds of scenarios at once — from Agentforce Studio, not just Builder. The next level after Preview.

Preview Panel — Interactive Testing

The Preview panel is the first place you test your agent. It lets you try out conversations your users might have so you can see if the agent responds the way you intended.

Two Preview Modes

ModeUsesWhen to Use
Simulate Mock data and actions — no real records affected Early testing, safe experimentation without touching production data
Live Test Real Salesforce data and live action execution Validating that the agent correctly reads, processes, and acts on your actual org data

What Preview Output Shows You

  • Whether the agent provides helpful and relevant responses
  • Whether it calls the right actions
  • Whether it references your business processes correctly
  • Whether it stays within the guardrails you set
Example Preview Conversation

User: "I'd like to book a session of the Full Moon Beach Experience."
Agent: "To help you book a session for the Full Moon Beach Experience, I need to verify your details first. Could you please provide your email address and membership number?"
The agent correctly called the verification subagent before proceeding to booking.

Interaction Summary — High-Level Path

The Interaction Summary panel shows you the steps the agent took to reach its response at a high level, including:

  • The input that was received
  • Which subagent was invoked
  • The reasoning steps the agent applied
  • The output evaluation before the final response was delivered

This is your first-level debugging view — useful for understanding the general flow without diving into every technical detail.

Agentforce Session Tracing — Full Detail View

Session Tracing is the deep-dive tool. It exposes every detail of what happened during an agent session, organised under the session ID for each conversation:

  • Reasoning engine executions — step-by-step LLM reasoning
  • Actions called — which actions were invoked and with what inputs
  • Prompt inputs and outputs — the exact prompt sent to the LLM and the response received
  • Gateway inputs/outputs — data flowing through the Einstein Trust Layer
  • Error messages — specific failure details when something goes wrong
  • Final responses — in both text and code view
Requirement

Agentforce Session Tracing requires Data 360 to be provisioned and enabled in your org. All session trace data is stored and processed in Data 360.

Session Tracing After Go-Live

Session Tracing is valuable after your agent is deployed, not just during development. You can review real user conversation exchanges to:

  • Locate and fix unexpected issues in production
  • Adjust the agent to handle input you hadn't anticipated
  • Identify whether additional guardrails are needed
  • Refine instructions or actions for more targeted responses

CHAPTER THREE
Scale Testing

Agentforce Studio
Test Suites (Beta)

After you've iterated on your agent's performance in Agentforce Builder, Test Suites takes your testing to the next level — batch testing dozens or hundreds of scenarios simultaneously, far beyond what's practical in the Preview panel one conversation at a time.

Why Test Suites Is Necessary

The Preview panel is essential for iteration, but there's a fundamental limit: it tests one conversation at a time. Consider all the ways a user might phrase the same request — politely, abruptly, with typos, in different languages, with context, without context. Testing each variation manually would take an enormous amount of time.

Test Suites solves this by enabling batch testing at scale:

  • Test dozens or hundreds of scenarios simultaneously
  • Upload a .csv file of test scenarios written in natural language
  • Or ask Test Suites to use AI to generate test input relevant to the jobs your agent performs

How to Access and Run Test Suites

  1. From the App Launcher, open Agentforce Studio

  2. Click Tests in the Agentforce Studio menu

  3. Create a new test suite for your agent

  4. Add test scenarios — either by uploading a .csv file or using AI to generate them

  5. Run the batch test

  6. Review results for each test input

Reading Test Results

When a batch test completes, the results show you for each test input:

Result FieldWhat It Shows
Input testedThe natural language scenario that was tested
Expected subagents/actionsWhat you expected the agent to call
Actual subagents/actionsWhat the agent actually called
Expected responseThe response you expected
Pass / FailWhether the actual behaviour matched expected

When a Test Fails

If a test input fails, you don't have to guess why. You can:

  1. Copy the failed input from Test Suites

  2. Paste it into the Agentforce Builder Preview panel

  3. Review the path the agent took on the plan canvas to reach the failed response

  4. Identify whether the issue is in instructions, actions, data, or permissions

  5. Refine and re-test

Test Input Sources

Test inputs can come from two sources: (1) Human-written scenarios — you write test cases in natural language and upload as a .csv. (2) AI-generated scenarios — ask Test Suites to generate test input relevant to the jobs your agent performs. The AI-generated option dramatically reduces the time needed to build a comprehensive test suite.

CHAPTER FOUR
Testing Strategy

Testing Considerations &
Refinement Strategies

Agent testing requires ongoing judgment — not just running tools and checking boxes. This chapter covers the key considerations for evaluating your agent's responses and how to address each one, plus the critical sandbox and cost rules every team must follow.

Determining Production Readiness

The probabilistic nature of agent behaviour makes the question "is my agent production-ready?" genuinely difficult to answer. There is no universal pass/fail threshold.

The Human Baseline Approach

A practical starting point: consider how accurately a human would perform the same task (for example, handling reservation questions). Use that human accuracy level as your baseline. Then strive to ensure your agent meets or exceeds that level. This anchors the target in a tangible, business-relevant benchmark rather than an arbitrary percentage.

Every organisation needs to determine its own baseline for pass/fail rates across scenarios. The level of precision varies by industry — a medical information agent requires a higher accuracy bar than a product recommendation agent.

Six Key Evaluation Criteria

As you test, evaluate each response against these six dimensions. For each, there is a clear set of refinement actions when the answer is "no":

Testing QuestionRefinement Actions When Failing
Did the agent follow my instructions? Refine the wording in existing instructions; add instructions to cover different types of input
Is the response accurate, complete, and easy to read? Check that the agent accesses the correct data; break actions or instructions into smaller pieces; refine wording to align with your company's voice
Is the response grounded in my data? Revise instructions to call the desired data; verify agent has permissions to access all required data; ensure data is current and accurate
Is the response aligned with my brand voice? Refine instructions to use your business's terms and phrasing; adjust tone in the language settings tab
How long did the response take? Clarify instructions; break actions or instructions into smaller, more focused pieces
Is there bias or toxicity in the response? Refine or add guardrails to your instructions; review Einstein Trust Layer settings
Is the response reliable every time? Determine if the agent's responses meet or exceed your baseline for accuracy; run Test Suites across many inputs

The Iterative Testing Loop

Getting the responses you want is a process of trial and error. Building an agent is an iterative process — you will go through rounds of:

  • Wordsmithing — precise language in instructions matters enormously
  • Checking permissions — agents can only use data they have access to
  • Validating data — ensure the data the agent accesses is correct and current
  • Adding details or guardrails — more specific instructions produce more predictable responses

The Sandbox Rule — Non-Negotiable

⚠️ Critical Rule

Testing agents — especially in Test Suites (Beta) — can modify your CRM data. Always run Test Suites in a sandbox environment. Never run batch tests in your production environment. A test that accidentally triggers an action could create, update, or delete real records.

Testing Costs

Testing is not free. Running tests in Test Suites (Beta) consumes requests or credits — these are billable generative AI usage metrics that incur costs to your organisation.

  • Each test scenario in a batch run consumes generative AI requests
  • AI-generated test inputs also consume usage
  • Consult the Generative AI Billable Usage Types documentation for specifics
  • Speak with your account executive to understand your organisation's budget
Cost Efficiency Tip

Before running a large batch test, refine your test inputs on a small sample first. Getting your test scenarios right before running hundreds of them reduces wasted spend on tests that would fail for the wrong reasons (poorly written test cases vs actual agent issues).
