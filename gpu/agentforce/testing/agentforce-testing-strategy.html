🔁 Agentforce · Five-Step Testing Strategy

Refine Your Agents with a
Five-Step Strategy

The AI Agent Testing Loop is a structured, repeatable strategy for systematically fine-tuning your Agentforce agents — from writing test scenarios and setting evaluation parameters, to running batch tests, validating with humans, and iterating until you reach your target accuracy.

Test Scenarios Evaluation Params Batch Testing Human Validation Iterate

Learning Objectives

After completing this guide, you will be able to:

  • Explain why it's important to have a testing strategy before you begin testing your agent.
  • Describe each of the five steps in the AI Agent Testing Loop.
  • Apply the strategy to real-world agent scenarios using Test Suites (Beta).

Why a Testing Strategy Matters

Agent testing is the foundation for releasing agents that are reliable and trustworthy. Without a structured approach, you risk testing the same scenarios repeatedly, missing important edge cases, or spending time on refinements that don't address the real cause of failures.

With so many variables at play — instructions, actions, data, permissions, user phrasing — a strategy gives you a disciplined path through complexity, ensuring you build toward your desired level of accuracy systematically rather than by trial and chance.

When to Use This Strategy

This strategy picks up after you've manually tested your agent in the Preview panel and iterated on its responses. At that point, you're ready to move to batch testing in Test Suites (Beta). The five-step loop guides that phase of your testing work.

The AI Agent Testing Loop is a cycle — you move through all five steps, use the results to refine, then loop back to run again with updated tests. Each iteration brings your agent closer to production-ready accuracy.

STEP ONE
Foundation

Identify Test Scenarios
& Create Test Data

Good batch testing starts with good test scenarios. This step focuses on identifying what to test, where scenarios come from, and the three attributes that make a test suite comprehensive.

Two Ways to Create Test Scenarios

Before batch testing in Test Suites (Beta), you need a set of scenarios that represent the full range of user inputs your agent needs to handle.

MethodHowBest When
Write your own Write scenarios in natural language, save as a .csv file, and upload to Test Suites You have clear knowledge of expected user inputs; precise control over test coverage
AI-generated Click Create test suite in Test Suites (Beta) — AI generates test cases using your agent's metadata and data it has permission to access Faster start; discovering scenarios you hadn't thought of; expanding an existing set

Where to Look for Scenario Ideas

Review these elements in Agentforce Builder to build comprehensive scenario ideas:

  • Subagent Classification Description — what does each subagent claim to handle?
  • Subagent Scope fields — what are the parameters the subagent should operate within?
  • Instructions — how is the agent directed to perform each task?
Coral Cloud Resorts — Example Scenarios
Experience Management Subagent Test Inputs

The Experience Management subagent handles bookings, modifications, and queries about experiences. Test scenarios derived from its scope include:

  • "Tell me about the [experience name] experience." — tests information retrieval
  • "Do you have any bookings for the [experience name] in July?" — tests record lookup with date filter
  • "I need to change my booking." — tests modification intent without a specific booking provided
  • "I'd like to confirm my booking." — tests confirmation intent
  • "Can I bring my dog to the Beach Bonfire experience?" — tests an out-of-scope or edge-case query

Three Attributes of a Good Test Suite

📦
Volume
A sufficient number of test cases to ensure coverage of different scenarios and edge cases. Not too few to miss patterns, not so many you can't review them.
🌐
Diversity
A wide range of inputs, contexts, and variations — including inputs outside scope and those that challenge guardrails. Real users will surprise you.
✅
Quality
Well-defined, accurate, and relevant test cases aligned with the agent's objectives. Poorly written test cases produce misleading results.

CSV Files & Recommended Starting Size

Test Suites (Beta) uses .csv files to hold test scenarios. Whether you write scenarios yourself or AI-generate them, the workflow results in a .csv that you can use as-is or edit.

Recommended Starting Point

Start with 10–20 AI-generated test scenarios in Test Suites (Beta). Download the .csv and review the scenarios against your agent's parameters. This saves time — you can revise the generated scenarios and add your own to align with expected user input. As your agent's performance improves, generate larger batches.

STEP TWO
Configuration

Set Evaluation Parameters

Before running tests, you configure how Test Suites (Beta) should evaluate your agent's performance. The New Test workflow guides you through four screens of configuration that shape how each test scenario is scored.

The New Test Workflow — Four Screens

When you create a new test suite, you're guided through four configuration screens:

  1. Basic information — test name and the agent you're testing
  2. Context variables — optionally simulate information about the user or conversation context (e.g., logged-in user's account tier, region)
  3. Evaluation criteria — select which performance and quality criteria Test Suites uses to evaluate each response
  4. Test case generation / upload — generate via AI or upload your .csv file
Best Practice

Test all of the evaluation criteria options to ensure your agent is reliable and performing well across every dimension — not just the ones you're most concerned about. Gaps in evaluation criteria are a common source of production surprises.

Test Case Anatomy

After completing the New Test workflow, each test case in your list contains four key fields that are evaluated when the test runs:

Test Case Fields
Utterance
The input query sent to the agent — the natural language test input. This is what the simulated user "says".
Expected Subagent
The relevant subagent the agent should route to when processing this utterance.
Expected Actions
The related actions the agent should execute for this input — which flows, Apex methods, or prompt templates should be called.
Expected Response
The desired result described in plain language — what a correct response should contain or accomplish.

STEP THREE
Execution

Run Tests & Evaluate Results

With scenarios defined and evaluation parameters configured, you run the batch test and review how each test case performed against its expected behaviour.

Running the Test Suite

After completing the New Test workflow, click Run Test Suite. Test Suites (Beta) executes every test case in the batch, sending each utterance to the agent and recording its response.

What the Results Show

For each test case in the completed batch, you can review:

  • The actual agent response — what the agent said
  • Results for each evaluation criterion you selected in the New Test workflow
  • A clear Pass / Fail indicator per test case
What to Focus On

Pay particular attention to failed tests — but also look for patterns in passed tests. Consistent passes on a certain type of query can confirm your agent handles that scenario well. Clusters of failures on similar inputs point to a systemic issue in a specific instruction or action.

STEP FOUR
Human Review

Validate Results with
Human Review

Automated tests tell you pass/fail — but they can't catch everything. Human review of test results is a critical step that catches tone mismatches, context-specific inaccuracies, and subtle issues that automated scoring misses.

Why Human Validation Is Essential

Generated tests handle many scenarios efficiently, but automated evaluation has limits. A human reviewer brings judgment that no automated metric can fully replace:

  • Tone mismatches — the response may be factually correct but sound wrong for your brand
  • Context-specific inaccuracies — subtle errors that only someone familiar with your business would catch
  • Toxic or unwanted content — responses that technically passed automated safety checks but would be inappropriate in context
  • Missed subtleties — the agent answered the literal question but missed what the user actually needed
Human Review Is Not Optional

Skipping human validation is one of the most common mistakes in agent deployments. Automated tests validate structure and factual accuracy, but only a human reviewer can assess whether responses feel right for your users, align with your brand voice, and handle nuanced situations appropriately.

What Human Validators Should Check

Review DimensionWhat to Look For
User alignmentDoes the response address what the user actually asked?
Brand voiceDoes the tone, vocabulary, and formality match your company's communication style?
SafetyIs the response free of harmful, biased, or toxic content?
CompletenessDid the agent give the full answer, or did it stop short?
Context accuracyAre details like names, dates, and product specifics correct in context?

STEP FIVE
Refinement

Review Results & Iterate

The final step transforms test results into actionable improvements. Testing is inherently iterative — you refine based on what you learned, then loop back to run tests again with your updated agent.

What to Refine Based on Test Results

Test results point you toward the specific components that need work:

  • Subagent instructions — refine wording, add specificity, or cover missing scenarios
  • Actions — fix or replace actions that aren't being called correctly
  • Data — test results may reveal outdated data your agent has access to
  • Permissions — failed data access may indicate permissions that need adjusting

The Iterative Loop

After refining, you don't just call it done — you loop back through the five steps:

  1. Update or expand your test scenarios to cover the refined behaviours
  2. Adjust evaluation parameters if the refinements changed expected behaviour
  3. Run the test suite again
  4. Human-validate the new results
  5. Assess whether you've reached your acceptable accuracy level
When to Stop Iterating

The loop continues until your agent meets or exceeds your organisation's defined baseline for accuracy. There is no universal stopping point — it depends on your use case, industry, and risk tolerance. Use the human performance baseline from your planning phase as the target.

ONGOING
Continuous Quality

Retesting Over Time

Agents evolve, and so does your business. A well-tested agent at launch can drift in performance as the world around it changes. Retesting for continued accuracy and trust is an ongoing responsibility, not a one-time event.

What Triggers the Need to Retest

Any of the following changes can impact your agent's performance and should trigger a new round of testing:

🗂
Data Changes
Updates to the data your agent accesses — new records, schema changes, outdated information entering the knowledge base.
🔐
Permission Changes
Changes to what the agent user can see or do — new permission sets, profile changes, or restricted access.
⚙️
Agent Updates
Updates to subagents, actions, prompts, or instructions — even small wording changes can shift behaviour.
📦
Product Updates
Changes to Agentforce features, capabilities, or related platform components that your agent relies on.
🏢
Business Changes
New products, revised policies, changed business processes, or updated brand guidelines that affect expected responses.
🔁
Scheduled Cadence
Even without specific changes, run periodic retests to catch drift and ensure the agent continues meeting your accuracy baseline.
Continuous Trust

Continued testing helps your agent evolve to stay relevant as your business goals change. The investment in maintaining a good test suite pays dividends every time a change is made — you can quickly verify that your agent still behaves as intended.
