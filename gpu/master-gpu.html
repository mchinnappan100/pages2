Streaming Multiprocessor (SM)

The fundamental execution unit of every NVIDIA GPU. Each SM contains CUDA cores, Tensor Cores, schedulers, shared memory, and register files — thousands of threads run in parallel.

  • CUDA Cores: scalar ALUs for FP32/INT32 ops
  • Tensor Cores: matrix-multiply accelerators (GEMM)
  • Warp = 32 threads executing SIMT
  • Shared memory: fast scratchpad per SM
  • L1 cache unified with shared memory
H100 SXM SM breakdown: 128 FP32 CUDA Cores per SM 64 INT32 CUDA Cores per SM 4 Tensor Core Units per SM (4th gen) 4 Warp Schedulers per SM 256 KB shared memory (configurable) Total SMs: 132 → 16,896 CUDA Cores
🧮

Tensor Cores

Introduced with Volta (2017), Tensor Cores perform fused mixed-precision matrix multiplications in a single clock — the secret sauce behind LLM training speed.

  • 4th-gen Tensor Cores in H100: FP8/FP16/BF16/TF32/INT8
  • FP8 doubles throughput over FP16
  • Structured sparsity: 2× speedup via 2:4 sparse math
  • Each TC: 4×4 matrix ops per clock cycle
Tensor Core generations: Volta V100 FP16 → 125 TFLOPS Turing T4 INT8 → 130 TOPS Ampere A100 BF16 → 312 TFLOPS (sparsity: 624) Hopper H100 FP8 → 1979 TFLOPS (sparsity: 3958) Blackwell B200 FP4 → 4500 TFLOPS (est.)
💾

Memory Hierarchy

GPU memory is a multi-level hierarchy: registers → L1/shared → L2 → HBM/GDDR. Bandwidth is the primary bottleneck for most AI workloads.

  • HBM3: 3.9 TB/s (H100) vs GDDR6X: ~1 TB/s
  • L2 cache: 50 MB on H100
  • Shared memory: up to 228 KB per SM
  • Unified Memory: CPU/GPU address-space sharing
  • NVLink 4: 900 GB/s chip-to-chip bandwidth
Memory bandwidth comparison: Level Size Bandwidth Latency Registers ~256KB ~50 TB/s 1 cycle L1/Shared 228KB/SM ~20 TB/s ~28 cyc L2 Cache 50 MB ~10 TB/s ~200 cyc HBM3 80 GB 3.9 TB/s ~600 cyc NVLink — 900 GB/s μs
🔄

Execution Model: SIMT

GPUs execute threads in lock-step groups of 32 called warps using Single Instruction Multiple Thread (SIMT). This is what enables thousands of parallel ops.

  • Thread → Warp (32) → Block → Grid
  • Hardware warp scheduling hides memory latency
  • Divergent branches reduce utilization (warp divergence)
  • Occupancy: active warps / max possible warps
// Kernel launch hierarchy dim3 grid(1024); // blocks in grid dim3 block(256); // threads per block myKernel<<<grid, block>>>(data); // → 262,144 total threads // → 8,192 warps of 32 threads each // Scheduled across SMs automatically
🌐

Interconnects: NVLink & NVSwitch

Multi-GPU systems use NVLink for direct GPU-to-GPU communication, avoiding PCIe bottlenecks. NVSwitch creates full-mesh all-to-all connectivity in DGX systems.

  • NVLink 4.0: 900 GB/s bidirectional per GPU
  • NVSwitch 3rd gen: 57.6 TB/s aggregate fabric
  • DGX H100: 8 GPUs, 72 NVLink connections
  • Enables model parallelism across GPUs seamlessly
DGX H100 Node topology: GPU0 ─── GPU1 ─── GPU2 ─── GPU3 │ ╲ │ ╱ │ ╲ │ │ NVSwitch NVSwitch │ │ ╱ │ ╲ │ ╱ │ GPU4 ─── GPU5 ─── GPU6 ─── GPU7 All-to-all: 900 GB/s per GPU pair Total fabric: 57.6 TB/s aggregate

Precision Formats

Modern GPUs support a spectrum of numeric precisions. Choosing the right format trades accuracy for speed and memory — crucial for LLM training vs inference.

  • FP64: double precision — scientific computing
  • FP32: full precision — baseline training
  • TF32: 10-bit mantissa, full FP32 range (Ampere+)
  • BF16 / FP16: mixed-precision training (AMP)
  • INT8 / FP8: quantized inference, 2–4× speedup

How GPUs Changed LLM Training & Inference

The computational revolution enabling the AI era

🏋️ Training at Scale

GPT-3 (175B params) required ~3,640 petaflop/s-days. A100 clusters made this feasible — it would take 355 years on a single RTX 3080. GPUs enable parallelism that collapses this to weeks.

GPT-3 compute3.14×10²³ FLOPs
H100 cluster (1k GPUs)~2 weeks
Key enablerTensor Cores + NVLink
⚡ Mixed Precision Training

Automatic Mixed Precision (AMP) stores weights in FP32 but performs forward/backward passes in FP16/BF16. Tensor Cores accelerate this natively, giving 2–8× speedup with no accuracy loss.

Speedup vs FP322–8×
Memory savings~50%
Framework supportPyTorch AMP, JAX
🔀 Parallelism Strategies

Modern LLMs don't fit on one GPU. Megatron-LM and DeepSpeed combine Data, Tensor, and Pipeline parallelism across thousands of GPUs, enabled by NVLink and InfiniBand.

Data parallelismSame model, diff data shards
Tensor parallelismSplit layers across GPUs
Pipeline parallelismDifferent layers per GPU
🚀 Inference Optimization

Techniques like KV caching, FlashAttention, continuous batching, and quantization (GPTQ, AWQ, FP8) dramatically reduce serving cost — from $0.06/1k tokens to under $0.001.

FlashAttention speedup2–4× attention ops
INT4 quantization4× VRAM reduction
Continuous batching3–10× throughput gain
🧠

FlashAttention

A memory-efficient attention algorithm that tiles computations to avoid materializing the full N×N attention matrix — enabling long context windows without OOM errors.

  • IO-aware: minimizes HBM reads/writes via tiling
  • FlashAttention-2: 2× speedup over v1
  • FlashAttention-3: Hopper-specific, FP8 support
  • Context length: 1M+ tokens now feasible
# PyTorch SDPA (uses FlashAttention under hood) from torch.nn.functional import scaled_dot_product_attention with torch.backends.cuda.sdp_kernel( enable_flash=True, enable_math=False ): attn_output = scaled_dot_product_attention( Q, K, V, is_causal=True )
📊

KV Cache & Paged Attention

During autoregressive generation, the Key/Value tensors for past tokens are cached so they aren't recomputed each step. PagedAttention (vLLM) manages this like virtual memory.

  • KV cache size: 2 × layers × heads × d_head × seq_len
  • Llama 3 70B: ~1.5 GB KV cache per 4k context
  • PagedAttention: near-zero waste, high concurrency
  • Speculative decoding: draft model accelerates KV reuse
vLLM serving example: from vllm import LLM, SamplingParams llm = LLM(model="meta-llama/Llama-3-70b", tensor_parallel_size=4) outputs = llm.generate( prompts, SamplingParams(temperature=0.8) ) # PagedAttention handles KV cache automatically # Achieves near 100% GPU memory utilization
⚖️

Quantization

Reducing weight precision from FP16 to INT8/INT4 shrinks model size 2–4× and boosts throughput, enabling larger models on fewer GPUs with minimal quality loss.

  • GPTQ: post-training quantization to INT4/INT3
  • AWQ: activation-aware, better accuracy than GPTQ
  • GGUF (llama.cpp): CPU+GPU mixed-precision
  • FP8 native: H100 hardware support, near-lossless
Llama 3 70B VRAM requirements: FP32 280 GB # impractical FP16 140 GB # 2× H100 80GB INT8 70 GB # 1× H100 80GB INT4 35 GB # 2× RTX 4090 24GB Q2_K 18 GB # 1× RTX 4090 ✓

The Future of GPU & AI Compute

Emerging architectures, interconnects, and paradigms

🔮

Optical Interconnects

Copper NVLink hits physical limits around 2026. Silicon photonics will replace electrical interconnects, enabling terabit-per-second chip-to-chip links across racks.

  • Co-Packaged Optics (CPO): laser integrated on die
  • 10–100× bandwidth density over copper at scale
  • NVIDIA Rubin: expected to debut photonic links
  • Reduces power per bit by 5–10×
🧬

3D Chiplet Stacking & HBM4

Chiplet designs (AMD MI300X pioneered this) separate compute and memory dies. HBM4 (2025+) promises 8+ TB/s bandwidth, solving the memory wall for ever-larger models.

  • HBM4: 8 TB/s per stack target, 64 GB per stack
  • 3D-IC: SRAM cache stacked directly on compute die
  • Hybrid bonding replaces solder bumps for density
  • TSMC CoWoS-S: current advanced packaging
🌱

Energy & Sustainability

A single H100 cluster training a large model can consume megawatt-hours. Future GPUs must improve FLOPS/watt dramatically. Liquid cooling and power-efficient architectures are critical.

  • GB200 NVL72: ~120 kW per rack — liquid cooling mandatory
  • Analog in-memory computing: 1000× efficiency potential
  • Sparse computation: only activate needed neurons
  • TSMC N2 / Intel 18A: next-node efficiency gains
🤯

Neuromorphic & Analog Computing

Beyond von Neumann: Intel Loihi 2, IBM NorthPole, and startups like Rain AI are building brain-inspired chips that compute at the memory location, eliminating the memory bottleneck.

  • IBM NorthPole: 22 nm, no off-chip memory access
  • 25× energy efficiency over GPU inference
  • Still limited to inference; training remains GPU domain
  • 5–10 year timeline to production deployment
🔢

FP4 & Ultra-Low Precision

Blackwell introduced FP4 Tensor Cores. Research shows models can run at 4-bit with minimal accuracy loss with the right calibration, doubling throughput over FP8.

  • FP4 = 2× throughput vs FP8 with structured sparsity
  • NVIDIA B200: 9 PFLOPS FP4 peak (dense)
  • Quantization-Aware Training (QAT) closes accuracy gap
  • Binary / ternary networks: research frontier
🌐

Distributed Inference at Edge

Models will increasingly be sharded across networks of edge devices. PetalsML-style collaborative inference, on-device NPUs, and model distillation bring LLMs to every form factor.

  • Apple Neural Engine: 38 TOPS, 0.5W — on every iPhone
  • Qualcomm Hexagon: Snapdragon X Elite, 45 TOPS
  • NVIDIA Jetson AGX Orin: 275 TOPS, 15W edge
  • Collaborative inference: P2P GPU sharing networks

Best Practices for GPU-Accelerated AI

What high-performance teams do differently

Profile Before You Optimize

Always use NVIDIA Nsight or PyTorch Profiler before tuning. Most bottlenecks are in data loading or memory allocation, not the GPU kernels themselves.

python -m torch.profiler ... # or: nsys profile -t cuda python train.py # Look for: SM utilization, memory BW util

Use BF16 / AMP for Training

Automatic Mixed Precision cuts memory and speeds training 2–4× with no code changes needed. BF16 is preferred over FP16 for stability (wider dynamic range).

from torch.cuda.amp import autocast with autocast(dtype=torch.bfloat16): loss = model(batch) scaler.scale(loss).backward()

Maximize GPU Utilization

Target ≥ 80% SM utilization. Use large batch sizes, reduce CPU↔GPU data transfers, and overlap computation with async data prefetching via DataLoader num_workers.

DataLoader(dataset, batch_size=512, # maximize num_workers=8, # overlap pin_memory=True, # faster transfer prefetch_factor=4)

Use FlashAttention for Transformers

Always enable FlashAttention when running attention-heavy models. It's now default in most frameworks and delivers 2–4× speedup with significantly lower memory usage.

model = AutoModel.from_pretrained( "meta-llama/Llama-3-8B", attn_implementation="flash_attention_2", torch_dtype=torch.bfloat16 )

Quantize for Inference Deployment

Never deploy FP16 inference on production if INT8 or FP8 quality is acceptable. AWQ or FP8 quantization gives 2–4× throughput with <1% quality loss on most benchmarks.

# AWQ quantization (4-bit) from awq import AutoAWQForCausalLM model = AutoAWQForCausalLM.from_pretrained(...) model.quantize(tokenizer, quant_config={ "w_bit": 4, "q_group_size": 128 })

Right-Size GPU Memory for the Job

Match GPU VRAM to your model + batch size + optimizer state. Training needs ~6× model params in bytes (FP16 + optimizer). Underestimating causes OOM; overestimating wastes cost.

Rule of thumb (FP16 training): Params (B) × 6 = VRAM (GB needed) 7B model → ~42 GB minimum 13B model → ~78 GB (2× H100) 70B model → ~420 GB (6× H100)

Use Gradient Checkpointing for Large Models

Gradient checkpointing trades compute for memory — recomputing activations on the backward pass instead of storing them. Reduces activation memory by ~80% at a 20–30% compute cost.

model.gradient_checkpointing_enable() # Reduces activation memory ~80% # ~25% slower training — worth it for # fitting 70B models on 4× A100

Monitor Thermal & Power in Production

GPUs throttle when hitting thermal limits. In production inference servers, monitor GPU temp, power draw, and SM clock rates. Target < 80°C for sustained peak performance.

nvidia-smi dmon -s pucvmet # Watch: sm%, mem%, temp, power # Alert if: temp > 83°C # power > TDP × 0.95 # sm_util < 40%

Pin CUDA Kernels to the Right Stream

CUDA streams allow overlapping kernel execution with memory transfers. Use multiple streams to keep the GPU busy during memory copies and avoid idle cycles.

stream1 = torch.cuda.Stream() stream2 = torch.cuda.Stream() with torch.cuda.stream(stream1): output1 = model_part1(data1) with torch.cuda.stream(stream2): data2 = fetch_next_batch()

Choose Parallelism Strategy by Model Size

Data parallelism scales well to ~8 GPUs. Beyond that, use tensor + pipeline parallelism (Megatron-LM) or fully sharded data parallelism (FSDP / DeepSpeed ZeRO).

≤ 8 GPUs → DDP (Data Parallel) 8–64 GPUs → FSDP / ZeRO-3 64+ GPUs → Megatron 3D Parallel (TP × PP × DP)

Compile Models with torch.compile()

PyTorch 2.0's torch.compile() fuses kernels and optimizes the computation graph, delivering 20–50% throughput gains with a single line change.

model = torch.compile( model, mode="max-autotune", # or "reduce-overhead" fullgraph=True ) # One-time compile cost; then 20–50%+ faster

Use Continuous Batching for Serving

Traditional static batching wastes GPU cycles waiting for the slowest sequence. Continuous batching (vLLM, TGI) inserts new requests mid-batch, achieving 3–10× higher throughput.

docker run --gpus all \ -p 8000:8000 \ vllm/vllm-openai:latest \ --model meta-llama/Llama-3-70B-Instruct \ --tensor-parallel-size 4 \ --max-num-seqs 256
