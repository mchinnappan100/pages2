What Is RLHF?

Teaching models human preferences — not just human text

RLHF (Reinforcement Learning from Human Feedback) is the technique that made ChatGPT, Claude, and Gemini behave like helpful assistants rather than raw text predictors. The core insight: pretraining teaches a model what language looks like, but RLHF teaches it how to be helpful, honest, and harmless.

Instead of humans writing the ideal answer (supervised fine-tuning), humans compare multiple answers and say which is better. This preference data trains a reward model that can score any response. The language model is then optimised with RL to produce high-scoring responses.

Why RLHF — the key difference Supervised Fine-Tuning (SFT) only: Human writes: "Paris is the capital of France." Model learns to imitate this style. Problem: imitation is limited by what humans bothered to write. RLHF adds preference learning: Human sees 3 responses to "Explain quantum computing." Human ranks: Response A > Response C > Response B Model learns the underlying preference: clear, accurate, concise Model can now produce responses better than any single example RLHF captures preferences, not just demonstrations.

The Three-Stage RLHF Pipeline

SFT → Reward Model → PPO fine-tuning

Full RLHF Pipeline Stage 1: Supervised Fine-Tuning (SFT) Pretrained LLM │ Human-written demonstrations (prompt → ideal response) │ Fine-tune with cross-entropy loss │ SFT Model ← starting point for RL Stage 2: Reward Model Training Prompt + Response A, Response B (from SFT Model) │ Human annotators choose preferred response (A > B) │ Train reward model: RM(prompt, response) → scalar score │ Reward Model (RM) ← captures human preferences Stage 3: RL Fine-Tuning (PPO) SFT Model (copy → Policy π_θ) SFT Model (frozen → Reference π_ref) RM (frozen) │ For each prompt: Generate response with π_θ Score with RM: r = RM(prompt, response) KL penalty: r_adj = r − β·KL(π_θ || π_ref) Update π_θ with PPO to maximise r_adj │ RLHF-aligned Model

Stage 1 — Supervised Fine-Tuning

Teaching the conversational format before RL begins

Before RL can improve a model, the model needs to be able to follow instructions at all. SFT fine-tunes the pretrained model on a curated dataset of (prompt, ideal response) pairs — human-written demonstrations of helpful, correct behaviour.

SFT is standard supervised learning: minimise cross-entropy loss between the model's output and the human-written response. The resulting SFT model forms the starting point for RL and also the frozen reference model that prevents RL from drifting too far.

Python · SFT training loop sketch
import torch, torch.nn as nn

# SFT dataset: (prompt, ideal_response) pairs
# e.g. {"prompt": "Explain DNA", "response": "DNA is a molecule..."}

for batch in sft_dataloader:
    prompt_ids    = batch["prompt_ids"]      # (B, T_prompt)
    response_ids  = batch["response_ids"]    # (B, T_response)

    # Concatenate prompt + response, mask loss on prompt tokens
    input_ids = torch.cat([prompt_ids, response_ids], dim=1)
    labels    = input_ids.clone()
    labels[:, :prompt_ids.size(1)] = -100   # ignore prompt tokens in loss

    # Standard cross-entropy: model learns to predict response given prompt
    logits = model(input_ids).logits
    loss   = nn.functional.cross_entropy(
        logits[:, :-1].reshape(-1, vocab_size),
        labels[:, 1:].reshape(-1)
    )
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()

Stage 2 — Reward Model Training

Learning to predict human preferences from pairwise comparisons

The reward model (RM) is a neural network trained to assign a scalar score to (prompt, response) pairs that correlates with human preference. It is typically initialised from the SFT model with the final LM head replaced by a linear layer that outputs a single scalar.

Training data: human annotators are shown the same prompt with two (or more) responses generated by the SFT model. They choose which response is better. This creates a pairwise preference dataset: (prompt, chosen_response, rejected_response).

Reward Model Training Loss (Bradley-Terry preference model): L_RM = -E[(prompt, chosen, rejected)] log σ(RM(prompt, chosen) − RM(prompt, rejected)) Where: σ(x) = sigmoid(x) = 1 / (1 + e^{-x}) RM(prompt, response) = scalar reward score Intuition: RM(chosen) − RM(rejected) should be LARGE and POSITIVE If the margin is high → sigmoid → close to 1 → log → close to 0 (low loss) If the margin is negative (model wrong) → sigmoid → <0.5 → large loss
Python · Reward model preference loss
import torch
import torch.nn.functional as F

def reward_model_loss(reward_chosen, reward_rejected):
    """
    reward_chosen:   scalar RM score for the preferred response
    reward_rejected: scalar RM score for the less preferred response
    Bradley-Terry model: P(chosen > rejected) = σ(r_chosen - r_rejected)
    """
    margin = reward_chosen - reward_rejected
    # Maximise margin → minimise negative log sigmoid of margin
    loss = -F.logsigmoid(margin).mean()
    return loss

# Example
r_chosen   = torch.tensor([2.1, 1.8, 3.0])   # RM scores for preferred responses
r_rejected = torch.tensor([0.5, 1.2, 1.1])   # RM scores for rejected responses

loss = reward_model_loss(r_chosen, r_rejected)
print(ff"RM loss: {loss.item():.4f}")   # lower = RM correctly ranks chosen > rejected

# Accuracy: what % does the RM correctly rank?
accuracy = (r_chosen > r_rejected).float().mean()
print(ff"Ranking accuracy: {accuracy.item():.2%}")

Stage 3 — PPO Fine-Tuning

Optimising the policy to maximise human-preferred rewards

With a trained reward model, we now optimise the language model (policy) to generate responses that score highly. But we must be careful: unconstrained RL optimisation can exploit the reward model — finding adversarial responses that get high scores but are incoherent or harmful. This is called reward hacking.

The KL divergence penalty against the original SFT model prevents this: it penalises the policy for drifting too far from the SFT baseline, keeping the model coherent and safe.

RLHF PPO Objective: r_adjusted(prompt, response) = RM(prompt, response) − β · KL(π_θ(response|prompt) || π_ref(response|prompt)) L_PPO = E[min(rₜ·Aₜ, clip(rₜ, 1-ε, 1+ε)·Aₜ)] Where: π_θ = policy being trained (initialised from SFT model) π_ref = frozen SFT model (reference; prevents reward hacking) β = KL penalty coefficient (typically 0.02 – 0.2) RM = reward model (frozen)
🎭

Reward Hacking

When the policy exploits flaws in the reward model — finding responses that score high but are actually bad. Classic example: generating very long, verbose responses that the RM was not trained to penalise.

  • KL penalty prevents excessive divergence from SFT
  • Diverse evaluation prompts during RM training helps
  • Constitutional AI (Anthropic) adds explicit safety rules
  • Monitoring RM scores over training detects hacking
⚖️

The KL Penalty Trade-off

β controls the balance between reward maximisation and staying close to the SFT model. Small β: more reward, risk of hacking. Large β: safer, but less improvement. Most production systems anneal β or use adaptive schedules.

  • β too small: model degenerates, reward hacking
  • β too large: barely any RL improvement over SFT
  • Typical range: 0.02 – 0.2 for language models
  • KL measured per token then summed over sequence

DPO — Direct Preference Optimisation

RLHF without the explicit reward model or RL training loop

DPO (Rafailov et al., 2023) observes that the optimal RLHF policy has a closed-form relationship to the preference data. Instead of training a separate reward model and running PPO, DPO directly optimises the language model on preference pairs using a classification-style loss. The result: same alignment quality, 3× simpler pipeline.

DPO Loss: L_DPO = -E[(prompt, chosen, rejected)] log σ(β · [log π_θ(chosen|prompt)/π_ref(chosen|prompt) − log π_θ(rejected|prompt)/π_ref(rejected|prompt)]) Intuition: Increase log P(chosen) and decrease log P(rejected) — relative to reference β controls how strongly to push toward chosen vs staying near reference No separate reward model required! No PPO training loop! SimPO (2024): removes reference model entirely — even simpler.

Challenges & Pitfalls

What can go wrong in RLHF — and how to mitigate it

👤

Annotator Disagreement

Different annotators have different preferences. The reward model averages over these disagreements, potentially learning a muddled preference signal. Careful annotator selection, guidelines, and consensus scoring help.

💸

Human Annotation Cost

Collecting high-quality pairwise preferences at scale is expensive and slow. This is the primary motivation for RLAIF — replacing human annotators with AI models for the majority of comparisons.

📊

Distribution Shift

The RM was trained on responses from the SFT model. As the policy evolves, responses drift out-of-distribution for the RM. The RM's scores become unreliable, causing reward hacking.

🔄

Sycophancy

Models trained with RLHF can learn to be sycophantic — agreeing with the user rather than being accurate, because agreement often gets higher human ratings. Requires careful evaluation and anti-sycophancy reward shaping.

Key Takeaway

RLHF is the technique that turns a knowledgeable but socially-unaware language model into a genuinely helpful assistant. The reward model captures human values; PPO optimises toward them; the KL penalty keeps the model grounded. Every major commercial AI assistant — GPT-4, Claude, Gemini — uses some form of this pipeline.
