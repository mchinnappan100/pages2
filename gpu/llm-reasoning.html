Overview & Taxonomy

Four broad categories of LLM reasoning and execution strategies

Modern LLMs use a variety of reasoning, planning, retrieval, and execution strategies. Some are prompting techniques applied at inference time; others are full agent architectures. They are not mutually exclusive — production systems routinely combine several. A coding agent might use ReAct to decide when to search documentation, RAG to retrieve API references, tool use to run tests, Reflexion to fix failures, and test-time scaling to explore multiple repair strategies before returning the final code.

Chain-of-Thought (CoT)

Generate intermediate reasoning steps before the final answer

Chain-of-Thought prompting encourages the model to produce a reasoning trace — a sequence of intermediate steps — before committing to a final answer. This dramatically improves performance on tasks that require multi-step logic, arithmetic, and symbolic reasoning, because it forces the model to decompose the problem rather than guess directly.

There are three main variants:

  • Standard CoT — provide worked examples with reasoning steps in the prompt.
  • Zero-shot CoT — simply append "Let's think step by step." No examples needed. Surprisingly effective (Kojima et al., 2022).
  • Few-shot CoT — provide several solved examples with full reasoning chains. Works best on complex multi-step problems.
Standard CoT — Flow Question │ ▼ Reasoning Step 1 ▼ Reasoning Step 2 ▼ Reasoning Step 3 ▼ Final Answer Each step reduces the problem. The model cannot skip ahead.
Few-shot CoT — Prompt Structure Example 1: Q: Roger has 5 apples. Buys 3 more. Gives away 2. How many? A: 5 + 3 = 8. 8 − 2 = 6. Answer: 6 Example 2: Q: ... A: reasoning... answer: ... Now solve: Q: [new problem] A: [model reasons step by step]
Key Papers

Chain-of-Thought Prompting Elicits Reasoning in Large Language Models — Wei et al., Google, 2022.  |  Large Language Models are Zero-Shot Reasoners — Kojima et al., 2022 (zero-shot CoT: "Let's think step by step").

✅

Strengths

CoT consistently improves performance on tasks that humans solve by working through steps — arithmetic, logic puzzles, commonsense reasoning, multi-step word problems.

  • Better mathematical reasoning
  • Better multi-step logical deduction
  • Better planning and decomposition
  • Reasoning trace is auditable by humans
  • Zero-shot variant requires no example engineering
⚠️

Limitations

CoT reasoning is not guaranteed to be faithful — the model may produce a plausible-looking reasoning chain that does not reflect its actual internal computation. It also increases token usage.

  • Reasoning trace may be post-hoc rationalisation
  • More tokens → higher cost and latency
  • Small models (<7B) often cannot follow CoT reliably
  • Does not guarantee factual accuracy

Self-Consistency

Sample multiple reasoning paths and take the majority answer

Instead of relying on a single reasoning chain, Self-Consistency samples k diverse reasoning paths from the model (by using temperature > 0) and returns the most frequent final answer. The intuition: if several independent chains of thought arrive at the same answer, that answer is more likely to be correct — even if individual chains contain errors.

Self-Consistency substantially improves accuracy on math benchmarks over greedy CoT (e.g. +17.9% on GSM8K with PaLM 540B), and requires no extra training — only multiple inference calls.

Self-Consistency — Sample k Reasoning Paths Question │ ┌──┴─────────────────────────┐ ▼ ▼ ▼ Reasoning A Reasoning B Reasoning C ← sample with T>0 ▼ ▼ ▼ Answer: 6 Answer: 6 Answer: 4 ← different paths Majority vote → Answer: 6 ✓ k=10–40 samples typical. Cost: k × single-chain cost. No model fine-tuning needed.

Tree of Thoughts & Graph of Thoughts

Search over branching reasoning paths for deliberate problem solving

Tree of Thoughts (ToT) extends CoT to a search problem. Instead of a single linear chain, the model generates multiple candidate thoughts at each step, evaluates them (by self-assessment or a separate evaluator), and searches the tree using BFS or DFS to find the most promising path. Useful for problems where a wrong early step leads to an irrecoverable dead-end.

Graph of Thoughts (GoT) generalises ToT further: the reasoning structure is a directed graph rather than a tree. This allows ideas to be merged (combining two partial solutions) and revisited (looping back to refine an earlier idea), capturing reasoning patterns that a tree cannot express.

Tree of Thoughts Start / | \ A B C ← generate candidates / \ | A1 A2 B1 ← expand promising | Evaluate ← prune weak branches | Best path Search: BFS, DFS, or beam search Evaluation: LLM self-score or verifier
Graph of Thoughts Idea A ───── Idea B │ │ ▼ ▼ Idea C ───── Idea D ← merge two paths │ ▼ Final Answer Allows merging partial solutions. Allows revisiting earlier ideas. Generalises tree to arbitrary graph.
Key Paper

Tree of Thoughts: Deliberate Problem Solving with Large Language Models — Yao et al., 2023. Best for: planning, puzzles, creative writing, game-playing, coding tasks where early choices matter.

Least-to-Most Prompting

Decompose a hard problem into progressively simpler sub-problems

Least-to-Most prompting addresses a key failure mode of standard CoT: when a problem is significantly harder than anything in the prompt examples, the model cannot generalise. The solution is to explicitly decompose the problem into a sequence of sub-problems from simplest to most complex, solving each in turn and building on prior answers.

This two-stage process — first decompose, then solve sequentially — excels at symbolic reasoning, compositional generalisation, and tasks requiring length generalisation beyond training distribution.

Least-to-Most — Two Stages Stage 1: Decompose Stage 2: Sequential Solve Hard Problem Each answer feeds into the next │ │ ▼ Easy Sub-problem 1 → Answer 1 What sub-problems must be │ solved first? Easy Sub-problem 2 (uses A1) → A2 ▼ │ Sub-problem 1 Easy Sub-problem 3 (uses A2) → A3 Sub-problem 2 │ Sub-problem 3 Hard Problem Answer ✓

PAL & Program of Thoughts

Offload computation to a Python interpreter — don't calculate in your head

Program-Aided Language Models (PAL) address the fundamental unreliability of LLM arithmetic. Instead of computing numerically, the model generates a Python program that computes the answer, then executes it. This decouples language understanding (what the model is good at) from arithmetic execution (what Python is good at).

Program of Thoughts (PoT) is a hybrid: the model interleaves natural-language reasoning with code generation, allowing it to reason about what to compute before writing the code that computes it. Particularly powerful for data analysis and symbolic math.

PAL — Generate then Execute Question (math / data) │ ▼ LLM: Generate Python │ ▼ Python Interpreter │ ▼ Answer (exact, no hallucination)
Program of Thoughts — Hybrid Question │ ▼ Reason (what do I need to compute?) │ ▼ Write Program │ ▼ Execute │ ▼ Reason (interpret result) │ ▼ Answer
Python · PAL example — word problem to code
# Prompt: "Roger has 5 tennis balls. He buys 2 cans, each with 3 balls.
# He gives away 4 balls. How many does he have?"

# LLM generates:
tennis_balls = 5
cans_bought   = 2
balls_per_can = 3
balls_given   = 4

answer = tennis_balls + (cans_bought * balls_per_can) - balls_given
print(answer)   # → 7  (exact, no arithmetic errors)

ReAct — Reason + Act

The most influential agent framework: interleave thinking with taking actions

ReAct is the foundation of most modern AI agent frameworks including Agentforce, LangChain agents, and AutoGPT. The model alternates between producing Thought traces (internal reasoning) and Action calls (external tool use), observing the results, and iterating until the goal is reached. This grounds the model's reasoning in real-world information rather than pure internal generation.

ReAct Loop Question │ Thought: I need to find the current weather in Boston. │ Action: Search("Boston weather today") │ Observation: 75°F, partly cloudy │ Thought: 75°F is warm. No umbrella needed. Should I check tomorrow too? │ Action: Search("Boston weather tomorrow") │ Observation: Rain expected, 55°F │ Thought: Tomorrow it rains. I have enough info. │ Answer: Today: no umbrella (75°F). Tomorrow: bring an umbrella (rain).
Key Paper

ReAct: Synergizing Reasoning and Acting in Language Models — Yao et al., 2022. Used by: LangChain agents, Agentforce Atlas, OpenAI Assistants API, Amazon Bedrock Agents, AutoGPT.

💡

Why ReAct Works

The Thought-Action-Observation cycle grounds the model's reasoning in external reality. The model cannot hallucinate a search result — it must observe the actual output. This dramatically reduces confabulation for factual questions.

  • Thoughts provide interpretable reasoning traces
  • Actions retrieve real-world information
  • Observations correct the model's beliefs
  • Loop terminates naturally when goal is achieved
  • Generalises to any tool: search, code, APIs, databases
🏗️

ReAct in Production

Production systems add guardrails around the raw ReAct loop: maximum action limits, action allow-lists, human-in-the-loop approval for destructive actions, and fallback escalation when the loop exceeds budget.

  • Set max_iterations (typically 10–25 per session)
  • Maintain an action allow-list to prevent unsafe tool calls
  • Log every Thought-Action-Observation for audit
  • Escalate to a human when max steps reached without resolution

Plan-and-Execute

Separate planning from execution for structured multi-step workflows

Plan-and-Execute separates the agent into two distinct roles. A Planner (typically a large, capable model) generates a full multi-step plan upfront. An Executor (potentially smaller, cheaper) carries out each step sequentially, feeding results back. This is more efficient for well-defined workflows where the plan is unlikely to change mid-execution, and easier to debug since the plan is visible before execution begins.

Plan-and-Execute Architecture Goal │ ▼ ┌─── Planner (large model) ───┐ │ Generates full plan │ │ ├── Step 1: Research X │ │ ├── Step 2: Analyse Y │ │ └── Step 3: Write report │ └─────────────────────────────┘ │ plan is visible & auditable ▼ ┌─── Executor (can be smaller) ─┐ │ Executes Step 1 → result 1 │ │ Executes Step 2 → result 2 │ │ Executes Step 3 → result 3 │ └───────────────────────────────┘ │ ▼ Final Answer Used in: LangChain Plan-and-Solve, BabyAGI, OpenAgents

Reflexion & Self-Refine

Learn from mistakes and iteratively improve outputs

🔍

Reflexion

The model attempts a task, receives an evaluation signal (from a test suite, environment feedback, or self-evaluation), generates a verbal reflection on what went wrong, stores it in memory, and retries. Similar to how a programmer debugs iteratively.

  • Attempt → evaluate → reflect → improved attempt
  • Reflection stored in episodic memory buffer
  • Each retry benefits from all prior reflections
  • Particularly effective for coding and planning tasks
  • Paper: Reflexion (Shinn et al., 2023)
✍️

Self-Refine

A simpler iterative loop: the model generates a draft, critiques it (using the same or a separate model), then rewrites based on the critique. The loop repeats until the output meets a quality threshold or a maximum iteration count is reached.

  • Draft → critique → rewrite → repeat
  • No external feedback signal needed
  • Useful for writing, code generation, summarisation
  • Quality improves each iteration — up to a point
  • Paper: Self-Refine (Madaan et al., 2023)
Reflexion vs Self-Refine Reflexion Self-Refine Attempt Draft │ │ External Evaluation Critique (same model) (test suite / env feedback) │ │ Rewrite Reflection (verbal, in memory) │ │ Critique again Improved Attempt │ │ Final output loop

MCTS & Test-Time Scaling

Spend more compute at inference to explore and verify before committing

Monte Carlo Tree Search (MCTS) is borrowed from game AI (AlphaGo, AlphaZero). Applied to LLMs, it explores many candidate solution paths by simulating rollouts, scoring them with a value function, and iteratively focusing search on the most promising branches — before selecting the final answer. Increasingly used for code generation and mathematical theorem proving.

Test-Time Scaling / Deliberative Reasoning is the generalisation: rather than always generating a single response immediately, the model spends additional inference-time compute on difficult problems. This can involve generating multiple CoT paths and verifying them, using tools, revising drafts, or running MCTS. Many recent frontier models (OpenAI o1/o3, Google Gemini Thinking, DeepSeek-R1) employ adaptive test-time reasoning — spending more tokens on hard problems and fewer on easy ones.

MCTS for LLM Reasoning Root / | \ Try Try Try ← expansion: generate candidates | | Sim Sim ← simulation: rollout to end | Score ← evaluation: reward / verifier │ Backpropagate ← update node values │ Best path selected ← after N simulations Used in AlphaCode 2, rStar-Math, OpenAI o1-like systems
⏱️

Test-Time Scaling Laws

Research shows that for hard reasoning tasks, spending more tokens on deliberation at inference time can match or exceed the gains from scaling model parameters. The key: the model must have a way to verify whether a candidate solution is correct.

  • More inference compute → better accuracy on hard problems
  • Requires a process reward model (PRM) or verifier
  • Optimal for math, code, science, logic
  • Adaptive: hard problems get more compute automatically
  • Examples: OpenAI o1/o3, DeepSeek-R1, Gemini 2.0 Flash Thinking
🌳

Process Reward Models (PRM)

A PRM evaluates the quality of intermediate reasoning steps, not just the final answer. This is the key enabling technology for MCTS and test-time scaling — without a reliable step-level evaluator, you cannot prune bad branches or verify partial solutions.

  • Outcome reward model (ORM): scores final answer only
  • Process reward model (PRM): scores each reasoning step
  • PRM enables beam search over reasoning chains
  • Training PRMs requires step-level human annotations

RAG & Agentic RAG

Ground the model in external knowledge to eliminate hallucination

Retrieval-Augmented Generation (RAG) replaces reliance on parametric (baked-in) model knowledge with real-time retrieval from an external knowledge base. A retriever finds the most relevant documents for the query; these are injected into the LLM context before generation. Essential for enterprise AI where accuracy and currency of information are paramount.

Agentic RAG gives the model autonomy over the retrieval process itself — it decides what to search, when to search (including mid-reasoning), and whether the retrieved information is sufficient or another search is needed. This enables multi-hop reasoning over distributed knowledge bases.

RAG Variants Basic RAG Agentic RAG ────────────────── ────────────────────────────────── Query Question │ │ Retriever Plan ← what do I need to find? │ │ Top-K Documents Search 1 ← semantic search │ │ LLM (with context) Read & Assess │ │ Answer Need more? ← decide │ Search 2 ← multi-hop │ Synthesise Answer RAG variants: Graph RAG · Hybrid RAG · Multi-hop RAG · Corrective RAG
🕸️

Graph RAG

Constructs a knowledge graph from source documents. Retrieval traverses graph edges rather than doing flat vector search, enabling reasoning over entity relationships and multi-hop paths that dense retrieval misses.

  • Entities and relations extracted at indexing time
  • Community summaries enable global reasoning
  • Better for "how are A and B related?" queries
  • Microsoft GraphRAG: open source implementation
🔀

Hybrid RAG

Combines dense vector retrieval (semantic similarity) with sparse keyword retrieval (BM25/TF-IDF) and re-ranks results. Captures both exact keyword matches and semantic relevance — neither method alone handles all query types.

  • Dense: finds semantically similar passages
  • Sparse: finds exact keyword matches
  • Re-ranker: cross-encoder scores final top-K candidates
  • Reciprocal Rank Fusion (RRF) for merging rankings

Tool Use & Multi-Agent Systems

Extend LLMs with external capabilities and collaborative agent networks

Tool Use / Function Calling allows LLMs to invoke external tools in a structured way. The model generates a JSON-formatted tool call; the framework executes it and returns the result. Modern APIs from OpenAI, Anthropic, and Google all support structured function calling, allowing models to reliably call calculators, databases, search engines, code interpreters, and custom APIs.

Multi-Agent Systems compose several specialised agents into a collaborative network. A manager agent decomposes the goal and routes sub-tasks to specialist agents (researcher, coder, reviewer, planner). This scales to complex automation tasks that exceed any single agent's context or capability.

Tool Use — Structured Function Calling User: "What's 347 × 892, and who wrote Hamlet?" │ LLM decides to call two tools: │ Tool Call 1: { "name": "calculator", "args": {"expr": "347*892"} } Result 1: 309524 │ Tool Call 2: { "name": "search", "args": {"query": "who wrote Hamlet"} } Result 2: William Shakespeare │ LLM synthesises: "347 × 892 = 309,524. Hamlet was written by William Shakespeare." Common tools: Calculator · Python · Web Search · Database · Email · Calendar · APIs
Multi-Agent System User Goal │ ▼ Manager Agent (orchestrator — decomposes goal, routes tasks) │ ┌──────┬───────────┬──────────────┬──────────┐ ▼ ▼ ▼ ▼ ▼ Researcher Coder Planner Reviewer Writer │ │ │ │ search run create critique docs tests schedule output │ ▼ Final Answer Frameworks: Microsoft AutoGen · CrewAI · LangGraph · Amazon Multi-Agent Bedrock

Mixture of Experts (MoE)

A model architecture — sparse expert activation for efficient scale

Mixture of Experts is a model architecture, not a prompting strategy. A MoE model replaces the dense feed-forward layers in a transformer with a set of expert networks plus a router. For each token, the router selects a small subset of experts (typically 2 out of 8–128) to activate. The rest remain idle. This allows the model to scale total parameter count dramatically (more parameters = more knowledge) while keeping the compute cost per token constant.

Mixture of Experts — Per-Token Routing Token embedding │ ▼ Router (learned gating function) │ ┌──────────────────────────────────────┐ │ Scores all N experts for this token │ │ Selects top-K (typically K=2) │ └──────────────────────────────────────┘ │ Expert 3 (activated) Expert 17 (activated) Expert 1 (idle) Expert 7 (idle) ... N-2 others idle │ Weighted sum of active expert outputs │ Output token Examples: Mixtral 8×7B (8 experts, top-2) · GPT-4 (rumoured MoE) Gemini 1.5 (MoE) · DeepSeek-MoE · Switch Transformer
⚡

Why MoE is Efficient

A dense 70B model activates all 70B parameters for every token. A MoE model with 141B total parameters but 2 of 8 experts active uses only ~35B active parameters per token — same compute, double the knowledge capacity.

  • Active parameters ≈ dense model of much smaller size
  • Total parameters (knowledge) ≈ much larger model
  • Training FLOPs competitive with dense models
  • Inference: fast if experts fit in GPU memory
⚠️

MoE Challenges

MoE models have higher memory requirements (all experts must be loaded), can suffer from load imbalance (some experts overloaded, others starved), and require careful training to prevent router collapse.

  • Memory: all N experts loaded even if only K active
  • Load imbalance: auxiliary loss needed to spread tokens evenly
  • Expert collapse: router may ignore most experts
  • Serving complexity vs dense model

Constitutional AI & Debate

Alignment strategies: principled self-revision and adversarial argumentation

📜

Constitutional AI (CAI)

Developed by Anthropic, Constitutional AI trains models to be helpful, harmless, and honest by having the model evaluate its own outputs against a written set of principles (a "constitution"). Instead of relying solely on human preference labels, the model critiques and revises its responses against these principles — scalable AI feedback.

  • Model generates a response, then critiques it against each principle
  • Model revises based on critique — iteratively
  • Revised responses used to fine-tune with RLHF or RLAIF
  • Scales alignment supervision without per-response human labels
  • Foundation of Claude's training methodology
⚔️

Debate

Two (or more) models argue opposing positions on a question; a judge (another model or human) evaluates the arguments and selects the winner. The intuition: it is easier for a judge to spot flaws in an argument than to generate the correct answer directly — especially for complex or deceptive questions.

  • Proposed for AI safety: harder for a deceptive model to win debates
  • Judge need not understand the full problem — just spot bad arguments
  • Works for factual QA, ethics, policy decisions
  • Paper: AI Safety via Debate (Irving et al., OpenAI, 2018)
Constitutional AI — Self-Critique Loop Original Response │ ▼ Check principle 1: "Is this response harmful?" Critique: "Step 3 could facilitate harm because..." │ Revise response │ Check principle 2: "Is this honest?" Critique: "The uncertainty is understated..." │ Revise response │ ▼ Final revised response (passes all principles)

The Four Categories in Depth

  • Reasoning Strategies — Chain-of-Thought, Self-Consistency, Tree/Graph of Thoughts, Least-to-Most, PAL, PoT. These techniques shape how the model thinks — producing step-by-step traces, exploring multiple paths, or offloading computation to external interpreters.
  • Planning and Execution Strategies — ReAct, Plan-and-Execute, Reflexion, Self-Refine, MCTS, Test-Time Scaling. These govern how the model acts in the world — interleaving reasoning with external actions, planning ahead, recovering from failures, and allocating inference compute intelligently.
  • Knowledge Augmentation Strategies — RAG, Agentic RAG, Tool Use, Function Calling, Multi-Agent. These determine what the model knows at inference time — grounding responses in external databases, APIs, and the outputs of other specialised agents.
  • Architectural Strategies — Mixture of Experts, Constitutional AI, Debate. These affect how the model is built or trained — structuring the model's internal computation or training process rather than how it is prompted at inference time.
Combining Strategies

Modern production AI systems routinely combine many of these strategies. A coding agent might use ReAct to decide when to search documentation, RAG to retrieve API references, Tool Use to run tests, Reflexion to fix failures, and Test-Time Scaling to explore multiple repair strategies before returning the final code — all in a single session.

Choosing a Reasoning Strategy

For math and logic: start with zero-shot CoT ("Let's think step by step"). If still failing, try few-shot CoT, then Self-Consistency (k=10–20), then PAL for arithmetic-heavy problems. Reserve Tree of Thoughts for planning tasks where intermediate steps can be evaluated.

Choosing a Planning Strategy

For open-ended agent tasks: ReAct is the best default — flexible and interpretable. For structured workflows with known steps: Plan-and-Execute. For tasks with a clear correctness signal (code tests, unit tests): add Reflexion on top of ReAct for iterative improvement.

Choosing a Knowledge Strategy

For enterprise QA over a fixed document corpus: standard RAG. If queries involve entity relationships: Graph RAG. If accuracy on exact keywords matters: Hybrid RAG. If the model needs to decide what to search mid-reasoning: Agentic RAG. For computation, databases, and APIs: Tool Use.

Cost vs Accuracy Trade-offs

Self-Consistency costs k× inference tokens but improves accuracy with no fine-tuning. Test-Time Scaling is expensive but can reach frontier accuracy on hard problems. For budget-constrained deployments: zero-shot CoT is the best free lunch — often 10–15% accuracy gain for free.
