Scale Fundamentals

Why large-scale training is a distributed systems problem first

Modern foundation models contain hundreds of billions to trillions of parameters. A single A100 80 GB GPU holds at most ~20B parameters in BF16 — making multi-node, multi-GPU training a hard requirement, not an optimization. The system designer must simultaneously maximize compute utilisation, minimise communication overhead, and tolerate hardware failures at scale.

The Roofline Model

Every kernel is either compute-bound (arithmetic intensity exceeds the hardware's ops-per-byte ratio) or memory-bandwidth-bound. Knowing which regime a workload lives in determines the right optimization strategy.

📐

Model FLOPs Utilisation (MFU)

The single most important training efficiency metric. MFU = (observed throughput) / (theoretical peak FLOP/s). World-class runs achieve 35–55% MFU on H100 clusters.

  • Chinchilla (2022): ~30% MFU on TPUv4
  • PaLM (2022): 46% MFU on TPUv4 pods
  • GPT-NeoX: ~49% on A100 DGX
  • MFU drops under communication, memory stalls, load imbalance
🔢

Parameter & Memory Arithmetic

For a model with P parameters: mixed-precision training requires ~16× P bytes (params + grads + optimizer states). A 70B model needs ~1.1 TB just for Adam states.

  • BF16 params: 2 bytes/param
  • FP32 master copy: 4 bytes/param
  • FP32 Adam m + v: 8 bytes/param
  • Activations: batch × seq × hidden × layers
  • Gradient checkpointing trades compute for memory
📡

Bandwidth Hierarchy

Communication cost depends on where the data lives. Designing systems means matching data placement to the fastest available path.

  • HBM3 (H100): 3.35 TB/s per GPU
  • NVLink 4.0: 900 GB/s bidirectional (8-GPU)
  • PCIe 5.0 x16: 128 GB/s
  • InfiniBand HDR: 200 Gbps (25 GB/s)
  • Ethernet 400G: 50 GB/s

Parallelism Strategies

Data, tensor, pipeline, and sequence parallelism — and how to combine them

Data Parallelism (DP)

Each worker holds a full copy of the model. The global batch is sharded across workers; each computes local gradients independently. An AllReduce synchronises gradients before the optimizer step. This is the easiest form of parallelism but requires each worker to hold the full model in memory.

Python · PyTorch DDP sketch
# Distributed Data Parallel — each rank gets a full model replica
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

dist.init_process_group(backend="nccl")           # NCCL for GPU
rank  = dist.get_rank()
world = dist.get_world_size()

model = MyModel().to(rank)
model = DDP(model, device_ids=[rank])              # wraps & hooks gradients

for batch in dataloader:                           # each rank sees 1/world of data
    loss = model(batch)
    loss.backward()                                # DDP overlaps AllReduce with backward
    optimizer.step()
    optimizer.zero_grad()

Tensor Parallelism (TP)

A single weight matrix is split across devices. For a linear layer Y = XW, column-parallel splits W column-wise so each device computes a partial output; row-parallel splits row-wise and requires an AllReduce to sum partials. Megatron-LM pioneered this for transformer blocks.

Python · Column-parallel linear (Megatron-style)
# Weight W shape [H, H] split into W_local [H, H/tp_size] per rank
class ColumnParallelLinear(torch.nn.Module):
    def __init__(self, in_f, out_f, tp_group):
        super().__init__()
        tp = dist.get_world_size(tp_group)
        self.weight = torch.nn.Parameter(          # local shard
            torch.empty(out_f // tp, in_f))
        self.tp_group = tp_group

    def forward(self, x):
        out = torch.nn.functional.linear(x, self.weight)
        # AllGather across TP ranks to reconstruct full output
        full = [torch.empty_like(out) for _ in range(dist.get_world_size(self.tp_group))]
        dist.all_gather(full, out, group=self.tp_group)
        return torch.cat(full, dim=-1)

Pipeline Parallelism (PP)

Model layers are partitioned into stages, each on a different device. Micro-batches stream through the pipeline. GPipe uses synchronous bubbles; PipeDream and 1F1B (one-forward-one-backward) schedules interleave forward and backward passes to reduce the pipeline bubble from O(p) to O(1/m) where p=stages, m=micro-batches.

1F1B Pipeline Schedule (4 stages, 8 micro-batches) Stage 0: F0 F1 F2 F3 F4 F5 F6 F7 B3 B2 B1 B0 Stage 1: F0 F1 F2 F3 F4 F5 F6 F7 B3 B2 B1 Stage 2: F0 F1 F2 F3 F4 F5 F6 F7 B3 B2 Stage 3: F0 F1 F2 F3 F4 F5 F6 F7 B3 F = forward micro-batch B = backward micro-batch Bubble fraction ≈ (p-1)/(m+p-1) → small when m >> p

3D Parallelism

Real large-scale runs combine DP × TP × PP. The standard layout: TP within a node (NVLink), PP across nodes in a rack (IB), DP across racks. Megatron-Turing NLG 530B used TP=8, PP=35, DP=560 across 280 DGX A100 nodes.

Communication Primitives

Collective operations underpinning every distributed training framework

🔄

AllReduce

Every rank contributes a tensor; the element-wise sum (or mean) is available on all ranks. Ring-AllReduce achieves near-optimal bandwidth: each rank sends/receives exactly 2(N-1)/N of the data volume.

  • Used by DDP for gradient synchronisation
  • Ring algorithm: O(2×(N-1)/N × size) bytes
  • Tree algorithm: lower latency for small messages
  • NCCL selects algorithm automatically
✂️

ReduceScatter + AllGather (ZeRO)

ZeRO (Zero Redundancy Optimizer) replaces AllReduce with ReduceScatter (grads) + AllGather (params). Each rank owns a shard of parameters, gradients, and optimizer states, enabling linear memory scaling.

  • ZeRO-1: partition optimizer states only
  • ZeRO-2: + partition gradients
  • ZeRO-3 / FSDP: + partition parameters
  • Communication volume same as AllReduce but memory ÷ world_size
↕️

Point-to-Point (P2P)

Used in pipeline parallelism to send activations from stage i to stage i+1. NCCL's send/recv uses RDMA for zero-copy transfers over InfiniBand.

  • torch.distributed.send / recv
  • isend/irecv for non-blocking overlap
  • Batched P2P in PyTorch 2.x for pipeline schedules
Python · ZeRO-3 via FSDP
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy, MixedPrecision
import torch

mp_policy = MixedPrecision(
    param_dtype=torch.bfloat16,
    reduce_dtype=torch.float32,
    buffer_dtype=torch.bfloat16,
)

model = FSDP(
    MyLargeModel(),
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3
    mixed_precision=mp_policy,
    device_id=torch.cuda.current_device(),
)
# Parameters are sharded across all ranks.
# AllGather happens just-in-time before each forward/backward pass.
# ReduceScatter replaces AllReduce for gradients.

Memory Management

Fitting trillion-parameter models and their activations into finite GPU memory

Gradient Checkpointing (Activation Recomputation)

Instead of storing all activations during the forward pass, discard them and recompute on-the-fly during backward. Reduces activation memory from O(L) to O(√L) layers at the cost of ~30% extra compute.

Python · Activation checkpointing
from torch.utils.checkpoint import checkpoint

class CheckpointedTransformerLayer(torch.nn.Module):
    def forward(self, x):
        # Recompute this layer's activations during backward pass
        return checkpoint(self._forward_impl, x, use_reentrant=False)

    def _forward_impl(self, x):
        x = self.attn(x)
        x = self.ffn(x)
        return x

# Full model with selective checkpointing every k layers
for i, layer in enumerate(model.layers):
    if i % 2 == 0:
        layer = CheckpointedTransformerLayer(layer)

Mixed Precision Training

Store parameters as BF16 (2B), keep FP32 master copy + Adam states (12B/param). BF16 has wider exponent range than FP16, avoiding overflow in LLM training. Loss scaling not required with BF16.

Flash Attention

Rewrite attention to tile across SRAM, avoid materialising the N×N attention matrix in HBM. Reduces attention memory from O(N²) to O(N), enabling 100K+ context lengths on a single GPU.

Paged Attention (vLLM)

Manage KV-cache like OS virtual memory — allocate KV cache in non-contiguous pages, eliminating fragmentation during inference serving. Enables 24× higher throughput vs naive serving.

CPU Offloading

Offload optimizer states and/or parameters to CPU DRAM between steps (DeepSpeed ZeRO-Infinity, ZeRO-Offload). Trades PCIe bandwidth for near-unlimited "GPU" memory. Best when CPU↔GPU PCIe is not the bottleneck.

05

Job Scheduling

Cluster managers, gang scheduling, and preemption for ML workloads

Training a large model occupies thousands of GPUs for weeks. Cluster schedulers must provide gang scheduling (all workers start simultaneously), topology-aware placement (put TP ranks on NVLink-connected GPUs, PP across IB), and checkpoint-based preemption.

Common Schedulers

  • Slurm — dominant in HPC; used by most academic GPU clusters. MPI-native.
  • Kubernetes + Volcano / KubeFlow — cloud-native; supports heterogeneous workloads but gang scheduling is more complex.
  • Ray Train — Python-native distributed training with automatic fault recovery and elastic resizing.
  • AWS SageMaker / GCP Vertex — managed services handling provisioning, checkpointing, and spot-instance recovery.

Topology-Aware Placement

For 3D parallelism the mapping of ranks to hardware is critical. Best practice: TP group = one NVLink domain (8 GPUs on DGX), PP group = one IB rail, DP group = across rails. Megatron-LM's --nproc-per-node / --nnodes flags automate this.

Best Practice

Always save distributed checkpoints atomically — write to a temp path then rename. Use torch.distributed.checkpoint (PyTorch 2.1+) which shards the checkpoint across ranks and saves in parallel, reducing checkpoint time from minutes to seconds for large models.

06

Fault Tolerance

Handling GPU failures in multi-week training runs without losing progress

💾

Checkpoint Strategy

Save model + optimizer + RNG state + data loader position every N steps. Keep last K checkpoints. The cost of checkpointing must be amortised over MTBF (mean time between failures).

  • Async checkpointing: snapshot state dict to CPU, write in background thread
  • Distributed checkpointing: each rank writes its own shard in parallel
  • Recommended frequency: every 30–60 min for large runs
  • MTBF of 10K A100 cluster ≈ 3–6 hours
🔁

Elastic Training

Change the number of workers mid-run without restarting. PyTorch Elastic (torchrun) uses rendezvous to re-form process groups after node failures and rebalances data shards.

  • torchrun --nnodes=MIN:MAX for elastic node count
  • C10d rendezvous: etcd or file-based coordination
  • Membership changes trigger a new process group
  • State must be reloaded from last checkpoint

Benchmarks & Profiling

Measuring and diagnosing performance in large-scale ML systems

Python · PyTorch Profiler
import torch
from torch.profiler import profile, record_function, ProfilerActivity

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=torch.profiler.schedule(wait=1, warmup=2, active=5),
    on_trace_ready=torch.profiler.tensorboard_trace_handler("./log"),
    record_shapes=True,
    with_stack=True,
) as prof:
    for step, batch in enumerate(dataloader):
        with record_function("forward"):
            loss = model(batch)
        with record_function("backward"):
            loss.backward()
        optimizer.step()
        prof.step()

# Identify top kernels by CUDA time
print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=20))
