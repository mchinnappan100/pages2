The Kernel's Unified View

Linux has no separate "thread" concept — everything is a task

Most operating systems distinguish processes (heavy-weight, isolated) from threads (light-weight, shared address space) at the kernel level. Linux does not. The kernel has a single abstraction: the task, represented by struct task_struct. Both processes and threads are tasks; what differs is which resources they share.

When you call pthread_create() in C, or threading.Thread() in Python, the C library (glibc / NPTL) ultimately calls the clone() syscall with flags that tell the kernel to share the parent's address space, file descriptor table, and signal handlers. The kernel creates a new task_struct pointing to the same mm_struct (memory map) — that is a thread. A traditional fork() creates a new task_struct with a copy-on-write clone of the parent's mm_struct — that is a process.

Linux: "Processes" vs "Threads" from the kernel's perspective Process A (fork'd) Thread B (clone'd with CLONE_VM) task_struct A task_struct B pid = 1001 pid = 1002 tgid = 1001 ← own thread group tgid = 1001 ← same thread group as A mm → mm_struct_A (CoW copy) mm → mm_struct_A (SAME pointer!) files→ files_struct_A (own) files→ files_struct_A (SHARED) signal→ signal_struct_A (own) signal→ signal_struct_A (SHARED) getpid() in A → 1001 getpid() in B → 1001 (tgid, POSIX process id) gettid() in A → 1001 gettid() in B → 1002 (Linux kernel tid) getppid() in B → parent of the thread group leader
Key Insight

getpid() in POSIX returns the thread group ID (tgid) — so all threads in a process report the same PID. The unique per-task identifier is the Linux TID, returned by gettid() (or syscall(SYS_gettid)). The kernel scheduler operates on TIDs, not PIDs.

Process Lifecycle: fork / exec / exit / wait

How a new process is born, transformed, and reaped by the kernel

Full Process Lifecycle fork() kernel: copy_process() alloc new task_struct (dup_task_struct) copy mm_struct (copy_on_write page tables) copy files_struct (dup file descriptors) copy signal_struct add to process tree (parent.children list) set state = TASK_RUNNING add to scheduler run queue returns: child pid to parent, 0 to child exec() (e.g. execve("/bin/ls", …)) kernel: do_execve() load ELF binary (find PT_LOAD segments) replace mm_struct (flush old VMA list, map new segments) reset signal handlers (inherited dispositions cleared) set new entry point (PC ← ELF e_entry) NOTE: PID unchanged, same task_struct — only mm replaced exit() (or return from main → __libc_start_main → exit()) kernel: do_exit() set state = TASK_DEAD (briefly TASK_ZOMBIE) release mm, files, fs, signals notify parent via SIGCHLD reparent children to init (pid 1) if parent dies first remain as ZOMBIE until parent calls wait() waitpid() kernel: do_wait() reap zombie: free task_struct, return exit status Zombie without waitpid() = resource leak (PID slot held)
C · fork / exec / wait pattern 
#include <unistd.h>
#include <sys/wait.h>
#include <stdio.h>

int main() {
    pid_t child = fork();          // kernel: copy_process()

    if (child == 0) {
        // ── CHILD ──────────────────────────────────────────────────
        printf("child pid=%d ppid=%d\n", getpid(), getppid());
        // Replace child image with /bin/ls (exec family)
        execl("/bin/ls", "ls", "-la", (char*)NULL);  // do_execve()
        // execl only returns on error
        perror("execl"); return 1;
    } else {
        // ── PARENT ─────────────────────────────────────────────────
        int status;
        waitpid(child, &status, 0);      // reap zombie, get exit code
        if (WIFEXITED(status))
            printf("child exited: %d\n", WEXITSTATUS(status));
    }
}
👻

Zombie Processes

A process that has called exit() but whose parent has not yet called wait(). The kernel keeps the task_struct alive (in TASK_ZOMBIE state) to preserve the exit status. Zombies consume a PID slot but no memory.

  • Visible as "Z" in ps aux STATE column
  • Fix: parent must call waitpid() or install SIGCHLD handler
  • Signal SIGCHLD SIG_IGN: kernel auto-reaps (Linux extension)
  • Orphan: parent dies → reparented to PID 1 (systemd/init)
🧟

Orphan Processes

A process whose parent has exited. Linux automatically reparents orphans to PID 1 (systemd / init), which periodically calls wait() to reap them. Used intentionally for daemonisation.

  • Double-fork daemon pattern: fork → setsid → fork → exit grandparent
  • Result: grandchild owned by init, detached from terminal
  • Modern approach: use systemd units instead
  • PyTorch DataLoader uses daemon=True workers (auto-reaped)

clone() — The Universal Primitive

How fork(), pthread_create(), vfork(), and containers all use one syscall

clone() is the Linux syscall that creates both threads and processes. It accepts a flags bitmask that controls exactly which kernel resources the new task shares with its creator. fork() and vfork() are just wrappers around clone() with fixed flag sets.

clone() flags — what gets shared vs copied Flag Shared (set) Copied (unset) CLONE_VM same address space CoW copy of mm_struct ← thread vs process CLONE_FILES same fd table dup fd table CLONE_FS same cwd/root copy fs_struct CLONE_SIGHAND same signal handlers copy sighand_struct CLONE_THREAD same thread group new thread group (new TGID) CLONE_NEWPID — new PID namespace ← containers CLONE_NEWNET — new network namespace CLONE_NEWNS — new mount namespace CLONE_SYSVSEM same SysV semaphores — pthread_create() uses: CLONE_VM | CLONE_FILES | CLONE_FS | CLONE_SIGHAND | CLONE_THREAD fork() uses: SIGCHLD (all resources copied with CoW) vfork() uses: CLONE_VFORK | CLONE_VM (parent blocks until child exec/exit) Container uses: CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS | CLONE_NEWUTS | ...
C · Direct clone() call (educational) 
#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/wait.h>

#define STACK_SIZE (1024*1024)

int thread_fn(void *arg) {
    printf("thread tid=%d  arg=%s\n", gettid(), (char*)arg);
    return 0;
}

int main() {
    char *stack = malloc(STACK_SIZE);

    // Create a thread: CLONE_VM | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD
    pid_t tid = clone(
        thread_fn,
        stack + STACK_SIZE,   // stack grows downward
        CLONE_VM | CLONE_FILES | CLONE_FS | CLONE_SIGHAND | CLONE_THREAD,
        "hello from clone"
    );
    printf("parent pid=%d  new tid=%d\n", getpid(), tid);
    waitpid(tid, NULL, 0);

    // Create a process (fork equivalent): no CLONE_VM
    pid_t child = clone(
        thread_fn,
        stack + STACK_SIZE,
        SIGCHLD,              // no sharing flags → separate process
        "hello from process"
    );
    waitpid(child, NULL, 0);
    free(stack);
}
CoW Memory on fork()

After fork(), both parent and child share the same physical pages. The kernel marks all writable pages as copy-on-write: the first write to a shared page triggers a page fault, at which point the kernel allocates a new physical frame and copies the content. This makes fork() fast — even a 10 GB process forks in milliseconds — but the first-write cost is paid lazily.

POSIX Threads (pthreads / NPTL)

The userspace threading library built on top of clone()

Linux's POSIX thread implementation is NPTL (Native POSIX Thread Library), merged into glibc. Each pthread maps 1:1 to a kernel task (no green threads, no M:N mapping). The kernel scheduler sees and schedules every thread individually.

Thread Stack Layout

Each thread gets its own stack, allocated by pthread_create() via mmap(MAP_PRIVATE|MAP_ANONYMOUS). Default stack size is 8 MB (adjustable via ulimit -s or pthread_attr_setstacksize()). A guard page (non-readable) sits below each stack; overflow triggers a segfault rather than silent corruption.

Process address space with 3 threads High address [ Thread 3 stack 8 MB ] ← mmap'd by pthread_create [ guard page 4 KB ] [ Thread 2 stack 8 MB ] [ guard page 4 KB ] [ Main thread stack ] ← set up by kernel at exec time [ mmap region ] shared libs, anonymous mmap [ heap ↑ ] ALL threads share the same heap [ bss / data / text ] shared (same mm_struct) Low address (0x400000) TLS (Thread-Local Storage): __thread variables — each thread has its own copy stored in a per-thread block pointed to by fs: register (x86-64)
C · pthreads: create, join, mutex, condition variable 
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>

// ── Shared state ──────────────────────────────────────────────────
static pthread_mutex_t  mtx  = PTHREAD_MUTEX_INITIALIZER;
static pthread_cond_t   cond = PTHREAD_COND_INITIALIZER;
static int              ready = 0;

void* producer(void *arg) {
    pthread_mutex_lock(&mtx);
    ready = 1;
    pthread_cond_signal(&cond);     // wake one waiting consumer
    pthread_mutex_unlock(&mtx);
    return NULL;
}

void* consumer(void *arg) {
    pthread_mutex_lock(&mtx);
    while (!ready)                  // always loop — spurious wakeups
        pthread_cond_wait(&cond, &mtx); // atomically releases mutex + sleeps
    printf("consumer woke, ready=%d\n", ready);
    pthread_mutex_unlock(&mtx);
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t2, NULL, consumer, NULL);
    pthread_create(&t1, NULL, producer, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
}
// Compile: gcc -lpthread -o demo demo.c

Signals

Asynchronous notifications between kernel and processes

Signals are a form of limited IPC — the kernel can interrupt a process and invoke a registered handler. Each signal has a default action: terminate, ignore, stop, or continue. sigaction() overrides the default. In multi-threaded programs, signal delivery is complex: a signal sent to the process can be handled by any thread that hasn't masked it; a signal sent to a specific TID (via tgkill()) goes only to that thread.

Key Signals (ML-relevant subset) SIGKILL (9) Cannot be caught or ignored. Kernel kills task unconditionally. Use: kill runaway training job. Leaves GPU memory allocated! SIGTERM (15) Default terminate; CAN be caught. Allows graceful shutdown. Use: spot-instance preemption signal — save checkpoint on receipt. SIGINT (2) Ctrl-C from terminal. Python raises KeyboardInterrupt. SIGCHLD (17) Child process state change (exit, stop, continue). DataLoader: parent catches SIGCHLD to reap dead workers. SIGSEGV (11) Invalid memory access (page fault with no mapping). Common cause: stack overflow in worker thread, C extension bug. SIGBUS (7) Bus error — misaligned access or mmap'd file truncated under read. SIGUSR1/2 User-defined. PyTorch Lightning uses SIGUSR1 for profiler toggle. SIGALRM (14) Timer expiry (from alarm()). Used for watchdog timeouts.
Python · Signal handling for graceful training shutdown 
import signal, sys, torch

checkpoint_path = "checkpoint_latest.pt"
should_stop = False

def handle_sigterm(signum, frame):
    global should_stop
    print("\nSIGTERM received — will checkpoint and exit after this step")
    should_stop = True

def handle_sigusr1(signum, frame):
    # Toggle profiler (torch.profiler) without stopping training
    print("SIGUSR1: toggling profiler")

signal.signal(signal.SIGTERM, handle_sigterm)
signal.signal(signal.SIGUSR1, handle_sigusr1)

for step, batch in enumerate(dataloader):
    loss = model(batch)
    loss.backward()
    optimizer.step()

    if step % 500 == 0:
        torch.save({"step": step, "model": model.state_dict()}, checkpoint_path)

    if should_stop:
        torch.save({"step": step, "model": model.state_dict()}, checkpoint_path)
        print(f"Saved checkpoint at step {step}. Exiting.")
        sys.exit(0)

# Send from shell:  kill -SIGTERM $(pidof python3)
#                   kill -SIGUSR1 $(pidof python3)

# Python: read own cgroup memory limit
import pathlib

def get_memory_limit_bytes():
    # Works inside Docker/K8s pods (cgroups v2)
    p = pathlib.Path("/sys/fs/cgroup/memory.max")
    if p.exists():
        val = p.read_text().strip()
        return None if val == "max" else int(val)
    return None

def get_cpu_quota():
    # Returns float CPUs available, or None if unlimited
    p = pathlib.Path("/sys/fs/cgroup/cpu.max")
    if p.exists():
        quota, period = p.read_text().split()
        if quota == "max": return None
        return int(quota) / int(period)
    return None

cpus = get_cpu_quota()
mem  = get_memory_limit_bytes()
print(ff"CPUs: {cpus}  Memory: {mem//1e9:.0f} GB" if mem else "No limits")
ML in Containers: Common Pitfall

Inside a Docker container or Kubernetes pod, os.cpu_count() and multiprocessing.cpu_count() return the host CPU count, ignoring cgroup CPU limits. A pod limited to 4 CPUs on a 96-core host will report 96 — causing DataLoader to spawn 96 workers and immediately exceed the cgroup pids.max or thrash the scheduler. Always read the cgroup quota and cap num_workers accordingly.

Python · /proc inspector for ML training diagnostics 
import os, pathlib, re

class ProcInspector:
    def __init__(self, pid=None):
        self.pid = pid or os.getpid()
        self.base = pathlib.Path(f"/proc/{self.pid}")

    def threads(self):
        # List all TIDs in this process's thread group
        return [int(t.name) for t in (self.base / "task").iterdir()]

    def rss_mb(self):
        # Resident Set Size in MB
        m = re.search(r"VmRSS:\s+(\d+)", (self.base/"status").read_text())
        return int(m.group(1)) / 1024

    def cpu_time_s(self):
        # CPU time in seconds from /proc/pid/schedstat
        ns = int((self.base/"schedstat").read_text().split()[0])
        return ns / 1e9

    def open_fds(self):
        return len(list((self.base/"fd").iterdir()))

    def numa_pages(self):
        # Per-NUMA-node page counts from /proc/pid/numa_maps
        counts = {}
        for line in (self.base/"numa_maps").read_text().splitlines():
            for m in re.finditer(r"N(\d+)=(\d+)", line):
                n = int(m.group(1))
                counts[n] = counts.get(n, 0) + int(m.group(2))
        return counts

p = ProcInspector()
print(ff"Threads: {len(p.threads())} | RSS: {p.rss_mb():.0f} MB | FDs: {p.open_fds()}")
print(ff"CPU time: {p.cpu_time_s():.2f}s | NUMA pages: {p.numa_pages()}")

Kernel Synchronisation Primitives

futex, spinlock, RCU — how the kernel protects shared data structures

futex — Fast Userspace Mutex

A futex (fast userspace mutex) is a 32-bit integer in shared memory. In the uncontended case, locking and unlocking use only atomic CPU instructions — no syscall, no kernel involvement, ~5 ns. Only when the lock is contended does the locker call futex(FUTEX_WAIT) to sleep in the kernel. The unlock path calls futex(FUTEX_WAKE) only if there are waiters. All pthreads mutexes, condition variables, and semaphores are built on futex.

futex fast path (uncontended lock) Lock (uncontended): CAS(*futex_addr, 0, 1) ← atomic compare-and-swap in user space success → locked, NO syscall Lock (contended): CAS fails (futex_addr == 1) syscall futex(FUTEX_WAIT, addr, 1, NULL) ← sleep in kernel kernel: add thread to wait queue, switch out Unlock: *futex_addr = 0 ← atomic store if (waiters) syscall futex(FUTEX_WAKE, addr, 1) ← wake one waiter Performance rule: Uncontended lock/unlock: ~5 ns (2 atomic ops, no syscall) Contended: ~1 µs (2 syscalls + context switch)
C · Raw futex usage (educational — use pthreads in production) 
#include <linux/futex.h>
#include <sys/syscall.h>
#include <stdatomic.h>
#include <unistd.h>

static _Atomic int futex_var = 0;

static inline void futex_wait(_Atomic int *addr, int val) {
    syscall(SYS_futex, addr, FUTEX_WAIT_PRIVATE, val, NULL, NULL, 0);
}
static inline void futex_wake(_Atomic int *addr) {
    syscall(SYS_futex, addr, FUTEX_WAKE_PRIVATE, 1, NULL, NULL, 0);
}

void lock() {
    int expected = 0;
    while (!atomic_compare_exchange_weak(&futex_var, &expected, 1)) {
        futex_wait(&futex_var, 1);   // sleep if still locked
        expected = 0;
    }
}
void unlock() {
    atomic_store(&futex_var, 0);
    futex_wake(&futex_var);          // wake one waiter (if any)
}
🔄

RCU (Read-Copy-Update)

The kernel's mechanism for very-high-read, low-write data structures (routing tables, process list). Readers take no locks at all — they run in an RCU read-side critical section (rcu_read_lock()). Writers make a copy, update it, then atomically swap the pointer and wait for existing readers to finish (a grace period) before freeing the old copy.

  • Read side: ~1 ns (disable preemption + memory barrier)
  • Write side: wait for grace period (~ms) then free old copy
  • Used for: task list, network routing tables, file system dentries
  • User-space equivalent: liburcu (used in LTTng, DPDK)
🌀

Spinlocks

Used in kernel interrupt context where sleeping is forbidden. A spinlock busy-waits (spins on a CAS loop) until the lock is available. Cheap when held for nanoseconds; catastrophic if the lock holder is preempted (spins waste CPU). Inappropriate in user space — use futex-based mutexes.

  • kernel: spin_lock() / spin_unlock() — disables preemption
  • spin_lock_irqsave(): also disables interrupts (for IRQ handlers)
  • User space equivalent: pthread_spin_lock() — only for <50 ns critical sections
  • Never hold a spinlock across code that may sleep

ML Workload Relevance

Applying Linux process/thread knowledge directly to PyTorch training

DataLoader Worker Count

Each DataLoader worker is a separate Linux task (clone'd process). Set num_workers to match available CPU cores on the NUMA node hosting your GPU. Read the actual core count from the cgroup CPU quota (/sys/fs/cgroup/cpu.max), not os.cpu_count() which ignores container limits.

Avoid Zombie Workers

When a DataLoader worker crashes (SIGSEGV, OOM), it becomes a zombie until the main process calls waitpid(). PyTorch's _MultiProcessingDataLoaderIter installs a SIGCHLD handler that sets a flag; the main loop then checks and reaps. If you write custom workers, always handle SIGCHLD or use Process.join(timeout=).

SIGTERM for Spot Preemption

Cloud spot/preemptible instances send SIGTERM 30–120 seconds before reclaim. Install a handler that sets a flag checked at the end of each training step. Save a checkpoint on receipt, then exit cleanly with code 0 so the job scheduler can reschedule.

Thread Affinity for NCCL

NCCL spawns internal communication threads. For best performance, pin NCCL threads to CPU cores on the same NUMA node as the GPU. Use NCCL_THREAD_THRESHOLDS and NCCL_SOCKET_IFNAME env vars; or set pthread_setaffinity_np() on NCCL's thread before initialising the communicator.

OOM Killer & Training Jobs

The Linux OOM killer selects a process to kill when physical memory is exhausted, based on /proc/$PID/oom_score. Large training processes naturally have high OOM scores. Protect critical processes with echo -1000 > /proc/$PID/oom_score_adj (root required), or better — set tight cgroup memory limits to prevent OOM situations entirely.

futex Contention in PyTorch

High futex syscall counts in strace -c output indicate lock contention — often from many DataLoader workers competing on the shared memory queue semaphore. Reduce contention by increasing prefetch_factor (workers stay ahead) or using a lock-free ring buffer (torch's SimpleQueue uses one internally since 2.1).

strace -c -p $PID -e trace=futex,read,write,mmap sleep 10
Summary: Processes vs Threads for ML

Use threads for I/O overlap, CUDA stream management, and background checkpointing — the GIL is not held during CUDA ops or I/O syscalls, so they run in parallel. Use processes (via forkserver) for CPU-bound preprocessing in DataLoader workers, or any code that runs pure Python. For cross-node collective communication, NCCL manages its own kernel threads internally — you don't create them manually.
