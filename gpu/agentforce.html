What is Agentforce?

Autonomous AI agents that take action — not just answer questions

Agentforce is Salesforce's platform for building and deploying autonomous AI agents that can reason, plan, and take multi-step actions on behalf of users and businesses. Unlike chatbots that respond to explicit prompts, Agentforce agents independently assess a situation, decide which actions to take, execute those actions against real Salesforce data and external systems, and iterate until the goal is achieved.

Agentforce is built on three pillars: the Atlas Reasoning Engine (the agent's brain), the Einstein Trust Layer (enterprise-grade safety and privacy), and Salesforce Data Cloud (grounded, real-time customer data). It ships as part of the Einstein 1 Platform and is available across Sales, Service, Marketing, Commerce, and custom-built use cases.

Agents vs Copilots vs Chatbots

  • Chatbot — rule-based decision tree; answers within scripted flows; no autonomy.
  • Copilot — AI assistant that responds to user prompts; one-step, human-directed.
  • Agent — autonomous; receives a high-level goal; plans multi-step; executes actions; monitors outcome; iterates without per-step human instructions.
🧠

Atlas Reasoning Engine

The orchestration brain of every Agentforce agent. Atlas uses a ReAct (Reasoning + Acting) loop — it analyses the user's goal, routes to the right Subagent via the Agent Router, executes actions, evaluates the result, and decides the next step.

  • ReAct: interleaves reasoning traces with action calls
  • Selects from available Subagents and Actions via the Agent Router
  • Handles multi-turn, multi-step autonomous workflows
  • Stops when goal is met or escalates to a human
🛡️

Einstein Trust Layer

A security and privacy layer that sits between every agent and the underlying LLM. It ensures zero data retention by the LLM provider, prevents prompt injection, masks PII, and audits every interaction.

  • Zero data retention: LLM provider cannot store prompts/responses
  • Dynamic grounding: injects CRM context without exposing raw data
  • Toxicity detection and content moderation
  • Full audit trail in Salesforce org (Einstein Activity Audit)
🗄️

Data Cloud Grounding

Agents are grounded in unified customer data from Salesforce Data Cloud. Instead of hallucinating, the agent retrieves real CRM records, order histories, case notes, and knowledge articles at runtime.

  • Real-time data retrieval from Salesforce objects
  • Semantic search over Data Cloud vector embeddings
  • Knowledge base grounding (Salesforce Knowledge articles)
  • No fine-tuning needed — grounding via RAG at inference time

Key Architectural Principles

  • Metadata-driven — agents, subagents, and actions are Salesforce metadata records, not code deployments. Configure in Agentforce Builder, deploy with a package.
  • CRM-native — runs inside the Salesforce org; has direct access to all objects, permissions, sharing rules, and flows without API calls.
  • LLM-agnostic — the LLM Gateway supports multiple providers (OpenAI, Azure OpenAI, Anthropic, Amazon Bedrock) behind a single abstraction. Swap providers without changing agent logic.
  • Human-in-the-loop — any Subagent can include an escalation rule that hands off to a live agent via Salesforce Omni-Channel with full conversation context transferred.

Einstein Trust Layer

Enterprise-grade security, privacy, and compliance for every LLM call

🛡️ Core Guarantee

The Einstein Trust Layer ensures that no customer data sent to an LLM is ever retained, trained on, or logged by the LLM provider. Every prompt is secured with contractual zero-data-retention agreements with all supported providers. This is the foundational enterprise commitment that differentiates Agentforce from consumer AI tools.

🔒

Zero Data Retention (ZDR)

Salesforce has contractual agreements with every supported LLM provider: prompts and completions are never logged, stored, or used for model training by the provider. This is enforced at the infrastructure level — not just a policy.

  • Applies to all providers: OpenAI, Azure OAI, Anthropic, Bedrock
  • Enforced by dedicated Salesforce API endpoints with ZDR headers
  • Verified through third-party audits (SOC 2 Type II, ISO 27001)
  • Customer data never crosses into LLM provider training pipelines
🎭

PII Masking & De-masking

Before a prompt reaches the LLM, the Trust Layer scans for sensitive data (SSNs, credit card numbers, phone numbers, email addresses, account numbers) and replaces them with placeholder tokens. The response is de-masked before delivery to the user.

  • Detection engine covers GDPR, HIPAA, PCI-DSS data categories
  • Configurable masking policies per org and data category
  • Reversible: placeholder map maintained in Trust Layer session
  • Audit log records which fields were masked in each interaction
💉

Prompt Injection Defence

Prompt injection attacks embed adversarial instructions in user input to override the agent's system prompt, change its persona, or exfiltrate data. The Trust Layer classifies input and blocks suspected injection attempts before they reach the LLM.

  • Pattern-based + ML classifier for injection detection
  • Attempts to override system prompt are blocked and logged
  • Jailbreak patterns (DAN, role-play overrides) filtered
  • Blocked requests trigger Einstein Activity Audit event
📋

Einstein Activity Audit

Every LLM interaction — input prompt, retrieved grounding data, LLM response, trust layer decisions — is written as an AiInteractionLog record in the Salesforce org. Admins can query, export, and review every interaction.

  • SOQL-queryable: SELECT * FROM AiInteractionLog
  • Stores: user, agent, topic, action, prompt (masked), response
  • Retention configurable per org data policies
  • Feeds into Salesforce Shield Event Monitoring for SIEM integration
SOQL · Querying Einstein Activity Audit logs
-- Query all AI interactions for audit review
SELECT
    Id,
    CreatedDate,
    CreatedById,
    AiApplicationName,
    AiInteractionStatus,
    InputTokenCount,
    OutputTokenCount,
    IsContentFiltered,
    IsPiiDetected
FROM AiInteractionLog
WHERE CreatedDate = LAST_N_DAYS:7
  AND IsPiiDetected = true
ORDER BY CreatedDate DESC
LIMIT 200

-- Find blocked prompt injection attempts
SELECT Id, CreatedDate, AiApplicationName, AiInteractionStatus
FROM AiInteractionLog
WHERE AiInteractionStatus = 'Blocked'
  AND CreatedDate = THIS_MONTH

Subagents & Actions

The building blocks that define what an agent knows and can do

Subagents (formerly Topics)

A Subagent defines a domain of expertise for the agent. It contains: a natural-language description of what it covers, instructions for how to handle it, and the list of Actions the agent can invoke within that subagent. The Agent Router classifies each user request to a subagent at runtime.

Actions

An Action is a callable tool — a specific operation the agent can execute. Actions are typed and declared with input/output schemas so the LLM knows how to call them correctly. There are four action types:

  • Flow Action — invokes a Salesforce Flow (auto-launched). Most common; no code required.
  • Apex Action — calls an @InvocableMethod Apex class for complex logic.
  • MuleSoft API Action — calls an external REST API via MuleSoft Anypoint.
  • Prompt Template Action — generates LLM-powered text (summarisation, draft email, etc.) and returns it as a string for further use.
Apex · InvocableMethod for an Agentforce Action
// Apex class exposed as an Agentforce Action
public class OrderStatusAction {

    public class Request {
        @InvocableVariable(label='Order ID' required=true)
        public String orderId;
    }

    public class Result {
        @InvocableVariable(label='Order Status')
        public String status;
        @InvocableVariable(label='Estimated Delivery')
        public String estimatedDelivery;
    }

    @InvocableMethod(
        label='Get Order Status'
        description='Returns the current status and estimated delivery for an order.'
    )
    public static List<Result> getOrderStatus(List<Request> requests) {
        List<Result> results = new List<Result>();
        for (Request req : requests) {
            Order__c order = [SELECT Status__c, EstimatedDelivery__c
                              FROM Order__c
                              WHERE Id = :req.orderId LIMIT 1];
            Result r = new Result();
            r.status            = order.Status__c;
            r.estimatedDelivery = String.valueOf(order.EstimatedDelivery__c);
            results.add(r);
        }
        return results;
    }
}

ReAct Reasoning Engine (Atlas)

How the agent thinks, plans, and acts autonomously

Atlas implements the ReAct (Reasoning + Acting) pattern. Instead of answering in one shot, the agent produces a thought (reasoning trace), decides on an action, observes the result, and loops until it achieves the goal or determines it cannot proceed without human input.

ReAct Loop — Atlas Reasoning Engine User: "My order #12345 hasn't arrived. It's been 6 days." Thought 1: The user is asking about a late order. I should look up the order status first. Agent Router → Subagent: Order Management Action 1: GetOrderStatus(orderId="12345") Observation: status="In Transit", estimatedDelivery="2025-03-10" (3 days late) Thought 2: Order is 3 days late (less than 5). My instructions say offer refund only if > 5 days. I should apologise and provide tracking info. Let me get the tracking URL. Action 2: GetTrackingURL(orderId="12345") Observation: trackingUrl="https://carrier.com/track/ABC123" Thought 3: I have tracking info. I can now respond to the user with an apology and tracking link. Goal achieved — no further actions needed. Response: "I'm sorry for the delay on order #12345. It is currently in transit and estimated to arrive on March 10. You can track it here: [link]. Let me know if it doesn't arrive by then and I'll escalate for you."
🔁

Multi-Step Autonomy

Atlas can chain up to a configurable maximum number of actions (default 25) in a single session before requiring human confirmation. Each step refines the agent's understanding of the goal.

  • Max action steps configurable per agent (default 25)
  • Each thought-action-observation cycle is a single LLM call
  • Agent can re-plan if an action fails or returns unexpected data
  • Loops terminate on goal achievement, failure, or max steps
🎯

Agent Router (formerly Topic Selector)

At the start of each turn, the Agent Router classifies the user's message to the most relevant Subagent using the LLM. If confidence is low or no Subagent matches, the agent asks for clarification or escalates to a human.

  • Uses LLM zero-shot classification over Subagent descriptions
  • Falls back to a Default Subagent if no match exceeds threshold
  • Context from prior turns influences routing decisions
  • Mid-conversation Subagent switching is supported
🚨

Escalation & Handoff

When the agent cannot resolve a request, hits a guardrail, or detects high customer frustration, it seamlessly hands off to a human agent via Salesforce Omni-Channel with the full conversation transcript and context attached.

  • EscalateToAgent built-in action available in every Subagent
  • Context packet: conversation history, CRM data retrieved, actions taken
  • Omni-Channel routing rules apply — right human, right queue
  • CSAT surveys triggered post-interaction for both AI and human

Data Cloud & RAG Grounding

How agents access real customer data without hallucinating

Agentforce agents are grounded using Retrieval-Augmented Generation (RAG) powered by Salesforce Data Cloud. When a user asks a question, the Trust Layer retrieves the most relevant CRM data — account records, case history, knowledge articles, order data — and injects it into the LLM prompt as grounding context. The agent answers from fact, not hallucination.

Vector Search in Data Cloud

Data Cloud stores vector embeddings of Salesforce records (Knowledge articles, case summaries, product documentation) using the Einstein vectorisation pipeline. At query time, semantic search finds the most contextually relevant records using cosine similarity, regardless of exact keyword matches.

RAG Pipeline — Data Cloud + Agentforce INDEXING TIME (asynchronous) Salesforce Knowledge / Cases / Products ↓ Einstein Vectorisation (batch + streaming) Data Cloud Vector Store Embedding model: text-embedding-ada-002 or equivalent Stored as float32 vectors with Salesforce record IDs QUERY TIME (synchronous, per user message) User message ↓ Embed query with same model Semantic Search (cosine similarity, top-K results) ↓ Returns K most relevant records Trust Layer ↓ Apply OLS/FLS — user can only see records they have access to ↓ Inject retrieved context into LLM prompt LLM (via ZDR Gateway) ↓ Generate grounded, factual response Agent Response — cites specific records, no hallucination
Apex · Querying Data Cloud Vector Search from an Action
// Use ConnectApi to perform semantic search against Data Cloud
public class KnowledgeSearchAction {

    public class Request {
        @InvocableVariable(required=true)
        public String searchQuery;
    }

    public class Result {
        @InvocableVariable
        public String articles; // JSON array of top-K articles
    }

    @InvocableMethod(
        label='Search Knowledge Base'
        description='Performs semantic search over Knowledge articles and returns the top matches.'
    )
    public static List<Result> search(List<Request> requests) {
        List<Result> results = new List<Result>();
        for (Request req : requests) {
            // Einstein Search: respects user's data access permissions
            List<Knowledge__kav> articles = [
                SELECT Title, Summary, UrlName
                FROM Knowledge__kav
                WHERE PublishStatus = 'Online'
                  AND Language = 'en_US'
                  AND Title LIKE :'%' + req.searchQuery + '%'
                LIMIT 3
            ];
            Result r = new Result();
            r.articles = JSON.serialize(articles);
            results.add(r);
        }
        return results;
    }
}

New Agentforce Builder

Hybrid reasoning, Canvas View, Agent Script, and mock-data previews — Feb 2026

What Changed — Feb 2026

Salesforce launched a redesigned Agentforce Builder powered by an upgraded Atlas Reasoning Engine. The core shift: from purely prompt-driven behaviour to a hybrid reasoning model — a graph-based engine that enables defined, controlled, and previewable agent flows. Located in Agentforce Studio (App Launcher → "Agentforce Studio"), not in Setup. The classic builder has no sunset date and runs alongside the new one.

Hybrid Reasoning — What Changed

Previously: Atlas focused on retrieving and interpreting prompts, then determined action flow entirely from prompt context — behaviour was emergent and hard to predict.

Now: A graph-based engine enables defined, controlled, and previewable flows. Admins can specify deterministic logic paths using Agent Script while still leveraging natural language for flexible handling. This is called hybrid reasoning.

Agent Script

Agent Script is a high-level scripting language introduced with the new builder. It supports both deterministic logic (if/then, conditionals, variable checks) and natural language prompts in the same script, giving admins fine-grained control over agent behaviour without requiring code.

Agent Script · Conditional order-late apology example
// Old way: vague instruction (unpredictable)
// instructions: "Apologise if the order is late"

// New way: Agent Script (deterministic + grounded)

subagent OrderManagement {
  // Step 1: Check whether the Is Late variable already has a value
  if @IsLate == null {
    // Step 2: Run action to determine lateness
    run action CheckIfLate(orderId=@OrderId) → @IsLate
  }

  // Step 3: Conditionally trigger apology only when confirmed late
  if @IsLate == true {
    "I sincerely apologise that your order has been delayed.
     Let me look into what we can do to make this right."
    run action OfferCompensation(orderId=@OrderId)
  } else {
    "Your order is on track! Here is the latest status:"
    run action GetOrderStatus(orderId=@OrderId)
  }
}

Best Practices

Design, security, testing, and deployment guidelines for production agents

Enable All Trust Layer Controls

Always enable PII masking, toxicity filtering, and prompt injection detection — even for internal agents. Turn on Einstein Activity Audit from day one. Retroactively adding audit logging means you lose historical compliance evidence.

Use ZDR-Certified LLM Providers

Only select LLM providers listed in the Einstein LLM Gateway with verified Zero Data Retention agreements. Do not route agent traffic to LLMs via external integrations outside the Trust Layer — this bypasses PII masking and audit logging.

Write Precise Subagent Descriptions

Agent Router classification is zero-shot: the LLM reads your Subagent description at runtime to decide which one matches. Vague descriptions cause misrouting. Describe what the Subagent handles AND what it does not handle. Avoid overlap between Subagents.

Narrow Action Permissions

Each Action runs with the permissions of the Running User (usually the agent's Connected App user). Grant only the minimum object and field permissions needed. Use a dedicated Integration User for agent actions with a restrictive Permission Set.

Always Provide an Escalation Action

Every Subagent should include an EscalateToAgent action. Define clear escalation triggers in the Subagent instructions or Agent Script (e.g. "escalate if the customer expresses frustration more than twice" or "escalate if the issue is not resolved in 3 actions"). Never leave a user stranded.

Test Adversarial Inputs

Before going live, run prompt injection tests: try to override the system prompt, exfiltrate data via the agent, or make the agent act outside its defined Subagents. Use the new Agent Preview (mock data) mode in Agentforce Studio to test without org impact. Verify the Trust Layer blocks adversarial inputs and creates audit records. Document results for compliance review.

Limit Action Chain Depth

Set a conservative max-actions limit (10–15 for customer-facing agents). Long chains increase cost, latency, and failure risk. Design Subagents so that most goals are achievable in 3–5 actions. Use Agent Script conditionals to short-circuit chains early, and escalation as the fallback for complex cases.

Version Control Agent Metadata

Use Salesforce DX (SFDX) to track GenAiAgent, GenAiAgentTopic (Subagent), and GenAiAgentAction metadata in Git. Deploy via CI/CD pipelines with sandbox → UAT → production promotion. Use Agentforce Studio for new agents; avoid making production changes directly in either builder.
