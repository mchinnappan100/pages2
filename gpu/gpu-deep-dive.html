Why GPUs for Machine Learning?

The fundamental reason a GPU outperforms a CPU for neural network math

A CPU is optimised for serial latency — completing a single complex task as fast as possible. It has a small number of powerful cores (8–128), large caches, branch prediction, and out-of-order execution. A GPU is optimised for parallel throughput — completing thousands of simple tasks simultaneously. It has thousands of simple cores, minimal control logic, and minimal cache per core.

Neural network training and inference are dominated by matrix multiplications. A matrix multiply of shape (4096 × 4096) involves 4096³ ≈ 68 billion multiply-add operations. These operations are completely independent of each other — perfectly parallelisable. GPUs are purpose-built for exactly this pattern.

Key Insight

The performance gap is not about clock speed — a modern GPU core runs at ~1.8 GHz while a CPU core runs at ~5 GHz. The advantage is parallelism: 16,896 GPU cores working simultaneously vs 8–16 CPU cores. For matrix multiplication, this delivers 100–1000× the throughput.

Streaming Multiprocessor (SM)

The SM is the fundamental execution unit. Each SM can run up to 2048 threads simultaneously (64 warps × 32 threads). The warp schedulers pick 4 warps per cycle to issue, hiding memory latency by switching to a different warp whenever one is waiting for a memory operation.

Warp — the SIMT unit

A warp is 32 threads that execute the same instruction simultaneously (Single Instruction, Multiple Threads). If threads within a warp take different branches, the GPU must serialise the divergent paths — called warp divergence. Avoiding divergence in performance-critical kernels is essential.

🧵

Thread & Block Hierarchy

CUDA programs are organised as a grid of thread blocks. Each block runs on one SM. Threads within a block share fast shared memory and can synchronise.

  • Thread: one execution lane (1 of 32 in a warp)
  • Warp: 32 threads executing SIMT
  • Block: up to 1024 threads, shares SM resources
  • Grid: many blocks, covers the full problem
  • Occupancy: active warps / max warps per SM
📦

Register File

Each thread has private registers — the fastest storage on the GPU (~1 cycle latency). The register file is large (64 KB per SM) but limited per thread. Using too many registers reduces occupancy.

  • 255 registers available per thread (Ampere+)
  • All registers for all warps reside on-chip simultaneously
  • Register pressure: more registers = fewer active warps
  • Compiler chooses register count; override with --maxrregcount

Why HBM instead of GDDR?

HBM (High Bandwidth Memory) stacks DRAM dies vertically on an interposer next to the GPU die, connected by thousands of short wires. This gives massive bandwidth (3.35 TB/s for H100) compared to GDDR6 (~600 GB/s for gaming GPUs). For ML inference, bandwidth is the bottleneck — loading 70B FP16 weights from HBM at 3.35 TB/s still limits decode to ~24 tokens/sec at batch size 1.

Shared Memory as a Programmer-Controlled Cache

Shared memory is the most important memory level for kernel performance. Unlike L1/L2 caches (managed by hardware), shared memory is explicitly managed by the programmer. GEMM (matrix multiply) kernels load tiles of A and B into shared memory, do the computation from fast SRAM, then write results to HBM — avoiding repeated slow HBM reads.

Tensor Cores

The dedicated matrix-multiply engines that power modern AI

Introduced in the Volta architecture (2017), Tensor Cores are specialised hardware units that perform a fused multiply-accumulate operation on small matrices in a single clock cycle. A 4th-generation Tensor Core (H100) computes a 16×8×16 matrix multiplication in one clock — far faster than using CUDA cores for the same operation.

Tensor Core Operation (WMMA — Warp-level Matrix Multiply Accumulate) D = A × B + C A: (16 × 16) FP16/BF16/FP8 input matrix fragment B: (16 × 16) FP16/BF16/FP8 weight matrix fragment C: (16 × 16) FP32 accumulator (for precision) D: (16 × 16) FP32 output One Tensor Core executes this in 1 clock cycle. 4th-gen H100 Tensor Cores support: FP8, FP16, BF16, TF32, INT8 Throughput comparison (H100 SXM5): CUDA Cores (FP32): 67 TFLOPS Tensor Cores (BF16): 989 TFLOPS ← 14.8× faster Tensor Cores (FP8): 1979 TFLOPS ← 29.5× faster
🔢

Precision Formats

Each generation of Tensor Cores adds support for lower-precision formats. Lower precision = 2× throughput and 2× less memory, with some accuracy trade-off managed by careful quantisation.

  • TF32 (19-bit): FP32 range + FP16 precision. Drop-in for FP32 training.
  • BF16: same exponent as FP32, 7-bit mantissa. Stable for LLM training.
  • FP16: narrower range than BF16. Needs loss scaling.
  • FP8 (E4M3/E5M2): 2× BF16 throughput. Used for inference and fine-tuning.
  • INT8: integer matmul for quantised inference.
📐

Flash Attention & Tensor Cores

Flash Attention 3 (H100-specific) uses wgmma (warp-group matrix multiply accumulate) instructions that pipeline Tensor Core execution with asynchronous memory loads, achieving near-peak Tensor Core utilisation for attention computation.

  • wgmma instruction: warp-group level (128 threads)
  • Async copy (cp.async): hide HBM latency during Tensor Core execution
  • Pingpong pipeline: two tiles in flight simultaneously
  • Achieves ~70–80% of theoretical Tensor Core peak for attention

Transformer Engine (H100)

The Transformer Engine is a hardware + software feature introduced in H100 that automatically manages mixed-precision transformer computation. It dynamically selects FP8 or BF16 for each layer based on the scale of activations, maximising Tensor Core throughput while maintaining numerical stability — without manual precision management by the programmer.

Multi-GPU Scaling

🔗

NVLink 4.0

NVIDIA's proprietary GPU-to-GPU interconnect. 900 GB/s bidirectional in an 8-GPU DGX system. All 8 GPUs can exchange data simultaneously at full bandwidth via NVSwitch fabric — critical for tensor parallelism where every layer needs AllReduce across heads.

  • H100: 900 GB/s bi-directional total (18 links × 50 GB/s)
  • NVSwitch 3.0: non-blocking all-to-all at full bandwidth
  • Used for: tensor parallelism, pipeline P2P activations
  • Latency: ~1–2 µs for small messages
🌐

InfiniBand HDR/NDR

High-speed networking between nodes in a GPU cluster. RDMA (Remote Direct Memory Access) allows one GPU to read/write another node's GPU memory without CPU involvement, enabling efficient collective communication (AllReduce, AllGather).

  • HDR: 200 Gbps per port (25 GB/s)
  • NDR: 400 Gbps per port (50 GB/s)
  • RDMA: DMA directly from GPU HBM to remote HBM
  • NCCL: NVIDIA collective communication library over IB

GPU Programming Model

CUDA, PTX, and the software stack from Python to silicon

Software Stack — from Python to Hardware Python (PyTorch) ↓ operator dispatch (ATen) CUDA Runtime API (cudaMalloc, cudaMemcpy, kernel launch) ↓ CUDA Driver API (cuLaunchKernel, cuMemAlloc) ↓ CUDA C++ Kernels (written in .cu files, compiled by nvcc) ↓ PTX (Parallel Thread Execution — GPU assembly language) ↓ SASS (Shader Assembly — GPU binary, architecture-specific) ↓ GPU Hardware Execution cuBLAS: optimised GEMM kernels (used by PyTorch linear layers) NCCL: collective communication (AllReduce, AllGather) cuDNN: optimised deep learning ops (conv, attention) Triton: Python-level GPU kernel authoring → PTX
Python · Key GPU operations in PyTorch
import torch

# Device selection and info
device = torch.device("cuda:0")
print(torch.cuda.get_device_name(0))              # "NVIDIA H100 SXM5 80GB"
props = torch.cuda.get_device_properties(0)
print(ff"SMs: {props.multi_processor_count}")        # 132
print(ff"VRAM: {props.total_memory/1e9:.0f} GB")     # 80 GB

# Tensor operations — dispatched to cuBLAS Tensor Cores
A = torch.randn(4096, 4096, device=device, dtype=torch.bfloat16)
B = torch.randn(4096, 4096, device=device, dtype=torch.bfloat16)
C = A @ B   # → cuBLAS GEMM → Tensor Cores → ~3 ms on H100

# Asynchronous execution — GPU runs independently of CPU
stream = torch.cuda.Stream()
with torch.cuda.stream(stream):
    result = (A @ B).relu()   # non-blocking: CPU continues immediately
stream.synchronize()          # wait for GPU to finish

# Mixed precision (Tensor Core optimal)
with torch.autocast(device_type="cuda", dtype=torch.bfloat16):
    output = model(input)     # matrix ops run in BF16 on Tensor Cores

# Memory profiling
print(ff"Allocated: {torch.cuda.memory_allocated()/1e9:.2f} GB")
print(ff"Reserved:  {torch.cuda.memory_reserved()/1e9:.2f} GB")

Chapter Summary

GPUs win for ML because neural networks are embarrassingly parallel matrix operations — exactly what thousands of CUDA cores and Tensor Cores are built for. The memory hierarchy (registers → shared memory → L2 → HBM) determines performance: keeping data as close to the compute units as possible (Flash Attention, kernel fusion) is the core optimisation strategy. NVLink and InfiniBand scale this to multi-GPU and multi-node clusters.
