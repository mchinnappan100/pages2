Processes & Threads

task_struct, fork/exec, pthreads, and Python's GIL

Linux task_struct (process descriptor) struct task_struct { pid_t pid; ← process ID (unique per thread) pid_t tgid; ← thread group ID (== PID of main thread) mm_struct *mm; ← virtual memory descriptor (shared in threads) files_struct *files; ← open file descriptors fs_struct *fs; ← filesystem root / cwd signal_struct *signal; ← signal handlers sched_entity se; ← CFS scheduling entity cpumask_t cpus_mask; ← CPU affinity state; flags; priority; comm[16]; } fork() → copy task_struct, copy-on-write page tables clone() → share mm/files/etc (creates a thread)

Python GIL and ML DataLoaders

CPython has a Global Interpreter Lock (GIL) — only one thread runs Python bytecode at a time. This is why PyTorch's DataLoader uses processes (multiprocessing) not threads for workers. Workers are forked processes with separate address spaces; they communicate via a shared-memory queue (using mmap + semaphores) to avoid serialisation overhead.

PyTorch 2.0+ uses multiprocessing.get_context("forkserver") on Linux by default to avoid fork-induced CUDA context corruption in child processes.

Python · Process vs thread for parallel work
import multiprocessing as mp
import threading, os

# Threads: share address space, subject to GIL for Python, not for C extensions
def cpu_bound_python():
    # GIL prevents true parallelism — use processes
    result = sum(range(10**7))

# Processes: separate address space, true parallelism, higher fork cost
ctx = mp.get_context("forkserver")   # safe with CUDA

def worker_fn(rank, shared_array):
    os.sched_setaffinity(0, {rank})   # pin to CPU core
    shared_array[rank] = rank ** 2

shared = mp.Array("i", 8)
procs  = [ctx.Process(target=worker_fn, args=(i, shared)) for i in range(8)]
[p.start() for p in procs]
[p.join()  for p in procs]
print(list(shared))  # [0, 1, 4, 9, 16, 25, 36, 49]

Python · mmap for zero-copy dataset loading
import mmap, os, struct
import torch
import numpy as np

# Memory-map a binary file — kernel maps pages lazily on access
fd = os.open("tokens.bin", os.O_RDONLY)
size = os.fstat(fd).st_size
mm = mmap.mmap(fd, size, mmap.MAP_PRIVATE, mmap.PROT_READ)
os.close(fd)

# Wrap as numpy array — zero copy! Physical pages loaded on demand (demand paging)
data = np.frombuffer(mm, dtype=np.int32)

# For huge page support (2 MB pages = fewer TLB misses)
# echo always > /sys/kernel/mm/transparent_hugepage/enabled
# madvise(MADV_HUGEPAGE) on the mapping

mm.madvise(mmap.MADV_SEQUENTIAL)  # hint: prefetch pages sequentially

# Explicit page prefault (avoid first-access latency during training)
mm.madvise(mmap.MADV_WILLNEED)    # kernel reads-ahead the entire file
📄

Huge Pages (THP)

Standard pages are 4 KB; TLB (Translation Lookaside Buffer) entries are precious (~1500 on modern CPUs). Large model weight tensors accessed repeatedly during training benefit hugely from 2 MB transparent huge pages — fewer TLB misses → better throughput.

  • Enable: echo always > /sys/kernel/mm/transparent_hugepage/enabled
  • PyTorch allocator: PYTORCH_NO_CUDA_MEMORY_CACHING=0 + THPAllocator
  • madvise(MADV_HUGEPAGE) for specific mappings
  • hugetlbfs: pre-allocate 1 GB pages for maximum effect
📌

Pinned (Page-locked) Memory

CUDA DMA engines can only transfer from pinned host memory (pages locked from being swapped out). torch.zeros(n, pin_memory=True) calls cudaMallocHost internally, enabling PCIe DMA without CPU involvement.

  • malloc → mmap anonymous → pin = three-step process
  • Pin cost is ~1 ms; saves ~1 ms per batch copy on repeated use
  • Don't over-pin: excessive pinning causes system memory pressure
  • Async copy: cudaMemcpyAsync on a non-default stream

System Calls & Context Switches

The cost of kernel transitions and how to minimise them

A syscall transitions the CPU from user mode (ring 3) to kernel mode (ring 0). Modern CPUs use SYSCALL/SYSRET instructions; each round-trip costs ~100–400 ns due to register save/restore, stack switching, and Spectre mitigations (KPTI page table flush on older kernels).

Relevant Syscalls for ML Workloads

  • mmap / munmap — allocate/free large memory regions (model weights)
  • madvise — hint kernel about access patterns (MADV_SEQUENTIAL, MADV_HUGEPAGE)
  • read / pread — data loading from NVMe/NFS
  • io_uring — async I/O ring buffer, zero-syscall-per-op for batched I/O
  • futex — fast userspace mutex (used by PyTorch's inter-process semaphores)
  • sched_setaffinity — bind process to specific CPU cores
  • perf_event_open — hardware performance counters
Python · strace to diagnose syscall overhead
# Profile syscalls of a training process
# $ strace -c -p $(pidof python) -e trace=mmap,read,write,futex
# % time     seconds  usecs/call     calls  syscall
# 62.14    0.482130        5831        82  mmap
# 25.01    0.194211          18     10724  futex
#  8.33    0.064726          18      3457  read

# Diagnose with perf stat (hardware counters)
# $ perf stat -e cache-misses,cache-references,instructions,cycles \
#       python train.py 2>&1 | tail -20

import ctypes, os

# madvise from Python via ctypes
MADV_HUGEPAGE  = 14
MADV_DONTFORK  = 10   # don't copy-on-write in child (save fork cost)
libc = ctypes.CDLL("libc.so.6", use_errno=True)

def madvise(addr, length, advice):
    ret = libc.madvise(ctypes.c_void_p(addr), ctypes.c_size_t(length), ctypes.c_int(advice))
    if ret != 0:
        raise OSError(ctypes.get_errno(), os.strerror(ctypes.get_errno()))

I/O & Storage for ML

NVMe, RAID, io_uring, and high-throughput dataset loading

Linux Storage I/O Stack Application (Python / PyTorch DataLoader) ↓ read() / io_uring / mmap VFS (Virtual File System) ↓ file_operations.read Page Cache ← 4 KB pages cached in RAM; hit avoids disk I/O ↓ cache miss Block Layer ├─ I/O Scheduler (mq-deadline / none for NVMe) └─ Block Device Driver ↓ NVMe Controller (PCIe 4.0 ×4 = 8 GB/s, ~20 µs latency) ↓ NAND Flash / QLC / TLC io_uring: submit 128 reads in one syscall, poll for completions — zero syscall per I/O
⚡

io_uring for Dataset Loading

io_uring (Linux 5.1+) uses two ring buffers in shared memory between kernel and user space. Submit batch I/O requests with no syscalls; poll the completion ring without blocking. Ideal for loading many files concurrently in DataLoader workers.

  • liburing: C library for io_uring ergonomics
  • Throughput: up to 2× vs traditional pread for small random reads
  • SQPOLL mode: kernel thread polls submission ring (zero syscalls)
  • Python: use liburing via cffi or pread threads for simplicity
🗄️

Direct I/O & Page Cache

By default, reads go through the page cache. For training where each sample is accessed once (streaming), O_DIRECT bypasses the cache, avoids polluting RAM, and avoids copy from kernel buffer to userspace.

  • O_DIRECT: requires 512-byte aligned buffers
  • posix_fadvise(POSIX_FADV_SEQUENTIAL): prefetch pages ahead
  • posix_fadvise(POSIX_FADV_DONTNEED): drop pages after reading
  • GPUDirect Storage: DMA directly NVMe → GPU, bypasses CPU/RAM

Linux CPU Scheduler

CFS, real-time scheduling, and priorities for ML worker processes

The Completely Fair Scheduler (CFS) uses a red-black tree ordered by virtual runtime. The task with the smallest vruntime is selected next. CFS aims to give each process exactly its fair share of CPU time weighted by nice priority.

Tuning for ML

  • GPU-bound training: CPU scheduler barely matters — GPU is the bottleneck. But DataLoader workers and communication threads benefit from lower niceness (higher priority).
  • NUMA-aware scheduling: Keep a process's memory and its CPU core on the same NUMA node. Crossing NUMA nodes doubles memory latency.
  • CPU isolation (isolcpus): Reserve cores solely for ML process, preventing OS jitter from preempting compute threads. Critical for latency-sensitive inference.
Shell · Scheduler tuning commands
# Set realtime scheduling for inference server (FIFO, priority 50)
chrt -f -p 50 $PID

# Pin process to specific CPU cores (avoid NUMA crossing)
taskset -cp 0-7 $PID

# Set nice value (lower = higher priority; only root can go negative)
renice -n -5 -p $PID

# View per-CPU stats and NUMA topology
numactl --hardware
numastat -p $PID

# Run process on NUMA node 0 with memory allocated locally
numactl --cpunodebind=0 --membind=0 python train.py

# Check scheduler statistics
cat /proc/$PID/schedstat   # run-time, wait-time, context switches
perf sched latency -p $PID

NUMA & CPU Affinity

Non-uniform memory access topology and its impact on multi-GPU servers

Dual-socket NUMA topology (common on DGX A100) NUMA Node 0 NUMA Node 1 ┌────────────────────┐ ┌────────────────────┐ │ CPU 0-23 (24 cores)│ QPI/UPI │ CPU 24-47 (24 cores)│ │ RAM 0: 512 GB DRAM │━━━━━━━━━│ RAM 1: 512 GB DRAM │ │ │ │ │ │ PCIe→ GPU 0-3 │ │ PCIe→ GPU 4-7 │ └────────────────────┘ └────────────────────┘ Local access: ~80 ns Remote access (cross-QPI): ~140 ns Pin DataLoader workers for GPU 0-3 to NUMA 0 CPUs workers for GPU 4-7 to NUMA 1 CPUs
ML Impact

On a multi-GPU server, PCIe-attached GPUs are physically connected to one CPU socket. If a DataLoader worker on NUMA node 1 reads data and sends it to a GPU on NUMA node 0, every transfer traverses the QPI link — doubling effective memory latency and halving bandwidth. Always bind workers to the same NUMA node as their target GPU. Check GPU → NUMA mapping with cat /sys/bus/pci/devices/<bdf>/numa_node.

IPC & Shared Memory

How PyTorch DataLoader workers share tensors without copying

Python · Shared memory tensor (PyTorch)
import torch
import torch.multiprocessing as mp

# Share a tensor across processes via POSIX shared memory (shm_open + mmap)
tensor = torch.zeros(1000, 1000)
tensor.share_memory_()    # moves backing store to /dev/shm

def worker(t):
    t[0, 0] = 42         # writes directly into shared region

p = mp.Process(target=worker, args=(tensor,))
p.start(); p.join()
print(tensor[0, 0])    # 42.0 — no serialisation!

# CUDA IPC: share GPU tensor between processes
gpu_t = torch.zeros(100, device="cuda")
handle = gpu_t.storage().share_cuda_()    # returns IPC handle
# In child process: torch.Tensor._new_shared_cuda(handle)

# Python 3.8+ multiprocessing.shared_memory for raw shared buffers
from multiprocessing import shared_memory
shm = shared_memory.SharedMemory(create=True, size=1024*1024)
buf = shm.buf     # memoryview — zero-copy access from any process with the name
