What Is Prompt Engineering?

A prompt is any input you give a large language model (LLM) — a question, instruction, code snippet, document, or combination of all of these. Prompt engineering is the discipline of designing those inputs to reliably produce the outputs you need.

While LLMs respond naturally to human language, there is a meaningful difference between casual interaction and engineered prompts. Prompt engineering is less about grammar and more about understanding the underlying mechanics: how models interpret context, why they hallucinate, how to anchor them to evidence, and how to structure instructions so they are unambiguous.

Core Insight

Think of prompting like managing a brilliant but literal intern. They will do exactly what you ask — and nothing more. The skill is in the asking. Vague instructions produce vague results; precise, well-structured instructions produce precise, reliable results.

Why It Matters at Scale

A single well-crafted prompt can be reused thousands of times — in automated pipelines, customer-facing products, CI/CD workflows, and data extraction jobs. A poorly crafted prompt at the same scale produces thousands of bad outputs. The leverage is enormous in both directions.

CHAPTER ONE
Core Technique

Zero-Shot Prompting

The simplest form of prompting — a direct instruction with no examples or demonstrations. Zero-shot prompts rely entirely on the model's pre-trained knowledge and your instruction clarity.

What It Is

A zero-shot prompt gives the model a task with no prior examples. The model must understand and execute the instruction from its training alone. "Zero-shot" refers to the number of demonstrations provided — zero.

✦ Zero-Shot Prompt
Summarize this meeting transcript and create a list of action steps for each participant.
◎ Why It Works
Summarization and list creation are common tasks the model has seen extensively in training. The instruction is clear, the scope is bounded, and there is no ambiguity about format expectations.

When to Use Zero-Shot

  • Simple, well-defined tasks: Summarizing, translating, classifying with obvious categories
  • Creative brainstorming: Generating ideas where any reasonable output is useful
  • Broad knowledge questions: Explanations, definitions, general analysis
  • Rapid prototyping: First-pass exploration before refining with examples

Limitations

Zero-shot prompting often falls short when:

  • The output format is specific and non-standard (use few-shot instead)
  • The task requires domain-specific terminology or style
  • Accuracy on multi-step reasoning is critical (use chain-of-thought)
  • The task is highly ambiguous — the model will make assumptions you may not agree with

❌ Weak Zero-Shot

  • "Write marketing copy for a resort."
  • "Analyze this code."
  • "Help me with this email."

✅ Strong Zero-Shot

  • "Write marketing copy in a humorous tone for a relaxed, family-friendly beach resort in California targeting parents with children under 12."
  • "Identify security vulnerabilities in this Python function. List each with its CWE category."
  • "Rewrite this email to be 30% shorter while preserving all action items."
Best Practice

Specificity is the most powerful lever in zero-shot prompting. Add: tone, audience, format, length, and constraints to any instruction. Each addition narrows the output space toward what you actually want.

Few-Shot Prompting

Few-shot prompting gives the model examples of input-output pairs before presenting the actual task. The model learns the pattern from your examples and applies it. It is one of the most reliable techniques for consistent, formatted output.

When to Use Few-Shot

  • You need specific output formatting (tables, JSON, bullet structures)
  • You want the AI to match a tone or style (your company's brand voice)
  • The task is domain-specific with terminology not common in general training data
  • Zero-shot produces inconsistent results — examples anchor the output
  • You need to demonstrate edge case handling (what to do when data is missing)

Writing Effective Examples

The quality of your examples determines the quality of few-shot results. Follow these principles:

PrincipleWhat It MeansExample
2–4 examplesEnough to establish pattern; too many wastes tokens2 for simple formats, 4 for complex classification
Diverse coverageExamples should cover different cases, not repeat the same scenarioOne easy case + one edge case
Match real inputsExamples should resemble the actual inputs you will sendIf inputs are messy, show messy input examples
Show the reasoningFor complex tasks, include WHY the output is correct"Because X is true, the label is Y"
Consistent structureUse identical formatting for all examplesAlways: Input: / Output: labels

Example: Case Summary (Support Agent)

✦ Few-Shot Prompt
You're a Salesforce support agent. Summarize support cases in this format: Case: [issue description] Resolution: [what was done] Next Steps: [follow-up required] Example 1: Case: Customer reports dashboard not loading on mobile devices. Resolution: Cleared browser cache and verified dashboard settings for mobile compatibility. Next Steps: Follow up in 2 days to confirm functionality. Example 2: Case: User cannot reset password using the self-service portal. Resolution: Reset the user's password manually and updated portal configuration settings. Next Steps: Monitor for similar issues across other users. Now summarize this case: Case: Client unable to upload large files (>20MB) through the Experience Cloud site.
◎ AI Output
Case: Client unable to upload large files (>20MB) through the Experience Cloud site. Resolution: Adjusted the file upload size limits in site configuration and enabled large file support. Next Steps: Notify client of the update and verify successful upload by end of week.

Example: Lead Qualification with Few-Shot

✦ Few-Shot Prompt (Classification)
Classify leads as Hot, Warm, or Cold based on these signals. Example 1: Input: Source: Referral | Industry: Tech | Budget: $80,000 | Timeline: 1 month Output: Hot Example 2: Input: Source: Web | Industry: Nonprofit | Budget: $5,000 | Timeline: 6 months Output: Cold Example 3: Input: Source: Trade Show | Industry: Finance | Budget: $40,000 | Timeline: 3 months Output: Warm Now classify: Input: Source: Event | Industry: Healthcare | Budget: $25,000 | Timeline: 2 months
◎ AI Output
Warm
Key Insight — Why Examples Beat Instructions

You can try to describe a format in words: "use a structured three-part response with bolded headers and bullet sub-points." Or you can show it once. Showing is almost always more reliable. The model learns from demonstration faster and more accurately than from description alone.

Chain-of-Thought Prompting

Chain-of-Thought (CoT) prompting guides the model to reason step by step before giving a final answer. By externalising intermediate reasoning, CoT dramatically improves accuracy on complex, multi-step, or logic-heavy tasks.

Why CoT Works

When a model jumps directly to an answer, it compresses all reasoning into a single prediction step. For simple tasks this is fine, but for multi-step problems, skipping intermediate steps introduces compounding errors. CoT forces the model to "show its work" — each step is a checkpoint where errors can be caught and corrected.

User Input
→
Step 1
Gather facts
→
Step 2
Apply logic
→
Step 3
Draw conclusion
→
Final Answer

Zero-Shot CoT

The simplest form of CoT: just add "Think step by step" or "Let's reason through this" to your prompt. This single phrase activates multi-step reasoning without any examples.

✦ Zero-Shot CoT Prompt
A sales rep closed 3 deals in January worth $12,000, $8,500, and $21,000. Their monthly quota is $35,000 and they receive a 5% bonus on everything above quota. Think step by step: Did they hit quota, and if so, how much bonus do they earn?
◎ AI Reasoning Output
Step 1: Calculate total revenue $12,000 + $8,500 + $21,000 = $41,500 Step 2: Compare to quota $41,500 > $35,000 → Quota achieved ✓ Step 3: Calculate amount above quota $41,500 − $35,000 = $6,500 above quota Step 4: Calculate bonus $6,500 × 5% = $325 Answer: Yes, the rep hit quota. They earn a $325 bonus.

Multi-Step CoT with Examples

For more complex reasoning, provide CoT examples that demonstrate both the reasoning process AND the final answer. This is called few-shot CoT.

✦ Few-Shot CoT Prompt (Lead Qualification)
Qualify leads step by step before assigning Hot, Warm, or Cold. Example: Input: Source: Referral | Industry: Tech | Budget: $80,000 | Timeline: 1 month Reasoning: Referral source indicates strong intent. Tech industry has quick budgets. High budget and short timeline align with immediate purchase readiness. Qualification: Hot Now qualify: Input: Source: Event | Industry: Healthcare | Budget: $25,000 | Timeline: 2 months
◎ AI Output with Reasoning
Reasoning: Event leads indicate genuine interest. Healthcare is a stable industry with structured procurement. The budget is moderate and the 2-month timeline is actionable without urgency. Qualification: Warm

When to Use CoT

  • Math and calculation: Revenue forecasts, pricing logic, quota calculations
  • Multi-condition logic: "If X and Y are true, but Z is not, then…"
  • Debugging: Trace an error's root cause step by step
  • Decision-making: Case routing, escalation decisions, lead scoring
  • Code review: Walk through logic to find the bug
Research Finding

Chain-of-thought prompting dramatically improves accuracy on reasoning-heavy tasks. The key mechanism: each intermediate step is committed to text, which constrains future steps. The model cannot "change its mind" about a prior step — the visible reasoning acts as a logical anchor.

Core Technique

Role-Based Prompting

Role-based prompting assigns a specific persona to the AI before asking it to complete a task. The model aligns its vocabulary, reasoning style, tone, and decision-making to the assigned role — producing outputs that sound like they came from that person.

How It Works

LLMs are trained on text written by many different types of people. When you assign a role, you activate the subset of the model's knowledge associated with that role's typical writing patterns, vocabulary, and decision logic. A "senior software engineer" and a "customer success manager" will answer the same question very differently.

✦ Role-Based Prompt — Sales Follow-Up
Act as a Salesforce account executive. You've just had a discovery call with a prospective customer. Draft a follow-up email that: - Recaps the key pain points discussed - Highlights how Salesforce addresses them - Includes a clear CTA for the next meeting Use a tone that is professional, confident, and focused on business value. Call details: - Customer struggles with disconnected data across sales and service teams - Currently using spreadsheets and legacy tools - Interested in improving cross-team collaboration and real-time visibility
◎ AI Output
Subject: Next Steps: Connecting Your Sales & Service Teams Hi [Name], Great connecting today. I appreciated hearing about the challenges your teams face — particularly around data silos and the reliance on spreadsheets for cross-team coordination. As we discussed, Salesforce Customer 360 brings your sales and service data onto a single platform, giving every team member real-time visibility into the customer journey — no more hunting across tools. I'd love to show you what this looks like in practice. Are you available for a 30-minute demo early next week? Best, [Your Name] Account Executive, Salesforce

Combining Roles with Merge Fields

Role-based prompts become even more powerful when combined with dynamic data. Use merge field syntax to personalise prompts at scale:

{{Account.Name}} has submitted a support case:

Case subject: {{Case.Subject}}

Customer tier: {{Account.Customer_Tier__c}}

Draft a personalised response acknowledging their issue and
outlining immediate next steps. Tone: empathetic, action-oriented.

Role Stacking

You can assign multiple role constraints in one prompt to get highly specific personas:

✦ Role-Stacking Prompt
Act as a senior Salesforce solution architect with 10 years of enterprise CRM experience, currently advising a Fortune 500 retail client. Review the following integration design and identify risks. Write in a consulting report style — direct, technical, and executive-facing.

When to Use Role-Based Prompting

  • Producing domain-specific language (legal, medical, technical, financial)
  • Matching brand voice and tone for customer-facing content
  • Coaching simulations — AI acts as a sales coach, manager, or reviewer
  • Perspective-taking — "Act as a sceptical customer" to stress-test a pitch
  • Building AI assistants with a consistent persona across multi-turn conversations
Pro Pattern — Adversarial Role

Assign an adversarial role to stress-test your work: "Act as a security auditor and find vulnerabilities in this code," or "Act as a sceptical investor and challenge every assumption in this business case." Adversarial roles surface weaknesses that neutral prompts miss.

CHAPTER FIVE
Core Technique

Conversational Prompting

Conversational prompts create dialogue-like interactions where the AI adapts its responses based on evolving user input. Unlike single-turn prompts, conversational prompting shapes behaviour across multiple exchanges.

Why Conversational Prompting Works Differently

In a conversation, context accumulates. Each exchange adds information the model can use to give more relevant, personalised responses. Conversational prompting is less about a single perfect prompt and more about designing the arc of the interaction.

✦ Conversational System Prompt
You are a helpful Salesforce onboarding assistant. Guide new administrators through org setup step by step. Ask clarifying questions before giving instructions. Offer options when there are multiple valid approaches. Make the experience feel like a live conversation with an expert — not a manual. When the user completes a step, acknowledge it and move naturally to the next.
◎ Sample Conversation Flow
AI: Hi! Welcome to Salesforce. I'm here to help you set up your org. First question: are you planning to use Salesforce primarily for Sales, Service, or both? User: Sales. AI: Great! Let's start with your lead and opportunity pipeline. Do you already have a defined sales process you want to mirror in Salesforce, or would you prefer to start with the default pipeline and customise from there? User: I have a custom process — 6 stages. AI: Perfect. Let's map those out. What is the name of your first stage, and what does "won" look like at the end of the pipeline?

Designing Effective Conversational Flows

  • State management: Design the system prompt to instruct the model to track what has been discussed and refer back to it
  • Progressive disclosure: Don't ask all questions at once — sequence them naturally
  • Acknowledgement patterns: Instruct the model to acknowledge user input before proceeding
  • Graceful recovery: Define what the model should do when the user gives an unexpected answer
  • Exit conditions: Specify when the conversation is "complete" to avoid infinite loops
Use Cases

Conversational prompting excels in: onboarding flows, customer support bots, guided configuration wizards, coaching dialogues, interview simulations, and any scenario where user input shapes the next step. It is the foundation of Agentforce and Claude's interactive capabilities.

Counterfactual Prompting

Counterfactual prompts ask the AI to reason about alternate realities — what would have happened if things were different. This technique is invaluable for coaching, root cause analysis, scenario planning, and strategic decision support.

The Structure of a Counterfactual Prompt

Every counterfactual prompt has three parts:

  1. What actually happened — the factual scenario
  2. Why it happened — root cause analysis
  3. What could have changed it — the counterfactual insight
✦ Counterfactual Prompt — Lost Deal Analysis
You are a sales coach analysing lost opportunities. Given the opportunity data below, explain why the deal was lost and then offer a counterfactual: what specific changes could have led to a different outcome? Opportunity data: - Outcome: Lost - Deal size: $45,000 - Close date: Missed by 2 weeks - Decision maker: Not involved - Competitor: BigDigital - Primary objection: Platform felt too complex - Rep notes: Demo focused heavily on features, not business outcomes
◎ AI Output
Root cause: The deal was lost primarily because the decision maker was not engaged during the buying process, and the demo emphasised technical features rather than business outcomes — making the platform appear complex and hard to justify. Counterfactual: If the decision maker had been identified and involved 2–3 weeks earlier, and if the demo had opened with a business impact narrative (time saved, revenue recovered, team visibility), the customer would likely have perceived the platform as a solution rather than a burden. An outcome-led presentation — showing ROI before features — could have materially changed the result.

Other Counterfactual Use Cases

ScenarioCounterfactual Question
Failed marketing campaign"What would the conversion rate have been if the headline had been benefit-focused?"
Missed SLA"What process change would have prevented this escalation?"
Data model decision"If we had chosen a different data architecture, what technical debt would we have avoided?"
Employee attrition"What early signals, if acted upon at 90 days, might have changed this outcome?"

Creative Prompting

Creative prompts unlock the generative strengths of LLMs — producing original names, narratives, campaigns, slogans, and concepts that would take humans much longer to brainstorm. The key is giving the AI just enough creative direction without over-constraining its output.

The Creative Direction Balance

Creative prompts need enough direction to be on-brand, but enough freedom to surprise you. Too constrained → generic outputs. Too open → irrelevant outputs. Aim for: describe the feeling and constraints, not the exact content.

✦ Creative Prompt — Campaign Names
Act as a creative marketer. Generate 5 campaign name ideas for a customer loyalty campaign emphasising connection, trust, and long-term partnership. The names should be: - Engaging and memorable - Suitable for digital and event marketing - Brand-aligned with Salesforce's tone: inspiring, human, forward-looking - Avoiding jargon or overly technical language For each name, include a one-sentence tagline.
◎ AI Output
1. Forever Forward — Because the best partnerships are built to last. 2. Trailblazer Together — Every trailblazer needs a trusted guide. 3. Built to Last — Strong relationships. Stronger results. 4. Trusted by Design — Where every interaction earns confidence. 5. Momentum Matters — Your growth, powered by the people who believe in it.

Techniques for Better Creative Output

  • Give a tone, not a script: "Witty and warm" beats "Use humour but not sarcasm"
  • Use contrast: "Like Apple meets the warmth of Patagonia"
  • Specify the audience: "For C-suite buyers who value trust over features"
  • Ask for multiple variants: "Give me 5 options ranging from formal to playful"
  • Iterate with feedback: "Keep #2, make the other 4 more energetic"
CHAPTER EIGHT
Advanced Pattern

Structured Output & JSON Schemas

Structured output prompting coerces the model into returning machine-parseable data — JSON, YAML, CSV, or custom formats — rather than prose. It is the foundation of production AI pipelines where downstream systems need to consume model output programmatically.

Why Structure Matters in Production

In production, AI output doesn't just go to a human — it flows into databases, APIs, dashboards, and notification systems. A single inconsistently formatted response can break an entire pipeline. Structured output prompting makes AI output reliable by design.

Technique 1: Schema in the Prompt

Define the output structure in the prompt itself, with a JSON schema or example:

✦ Structured Output Prompt
Extract the following information from the support email below. Return ONLY valid JSON — no markdown, no explanation. Schema: { "customer_name": string, "issue_type": "billing" | "technical" | "feature_request" | "other", "urgency": "high" | "medium" | "low", "product_mentioned": string | null, "action_required": string } Email: "Hi, I've been a customer for 3 years and my invoice this month is double what it should be. I need this fixed urgently before month-end close. This is for our Sales Cloud subscription."
◎ AI Output (JSON)
{ "customer_name": null, "issue_type": "billing", "urgency": "high", "product_mentioned": "Sales Cloud", "action_required": "Investigate and correct double-charged invoice before month-end" }

Technique 2: Tool Use for Guaranteed Schema Compliance

For Claude and GPT-4 APIs, using tool calling with a JSON schema is the most reliable structured output method. The model is forced to call the tool with schema-compliant arguments — syntax errors become impossible.

# Define extraction as a tool
tools = [{
    "name": "extract_case_data",
    "description": "Extract structured data from a support case email",
    "input_schema": {
        "type": "object",
        "properties": {
            "issue_type": {
                "type": "string",
                "enum": ["billing", "technical", "feature_request", "other"]
            },
            "urgency": {"type": "string", "enum": ["high", "medium", "low"]},
            "product": {"type": ["string", "null"]},  # nullable!
            "action_required": {"type": "string"}
        },
        "required": ["issue_type", "urgency", "action_required"]
    }
}]

# Force the tool call
response = client.messages.create(
    model="claude-opus-4-5",
    tools=tools,
    tool_choice={"type": "tool", "name": "extract_case_data"},
    messages=[{"role": "user", "content": email_text}]
)

result = response.content[0].input  # Guaranteed schema-compliant
python

Key Schema Design Principles

  • Use nullable fields for information that may be absent — prevents hallucination to fill required fields
  • Use enums for categorical fields — prevents freeform values that break downstream logic
  • Add "other" + a detail field for extensible categories: {"type": "other", "detail": "..."}
  • Separate semantic validation from schema validation — tool use eliminates syntax errors but not logical errors (values that don't add up)
Important Distinction

Schema validation eliminates syntax errors (malformed JSON). It does NOT prevent semantic errors (line items that don't sum to total, values placed in wrong fields). Always add a semantic validation step after extraction for business-critical data.

System Prompts & XML Structuring

The system prompt is your primary configuration lever. It runs before every user message and establishes the model's persona, constraints, output format, and scope. XML tags are a powerful structural tool for organising complex multi-section prompts.

System Prompt Architecture

A well-structured system prompt has five components:

## Component 1: Role Definition
You are a senior customer success manager at Acme Corp.

## Component 2: Capability Scope
You help customers resolve billing disputes, understand their subscription,
and escalate technical issues. You do NOT provide legal advice.

## Component 3: Output Format
Always structure responses with:
1. A brief acknowledgement of the customer's issue
2. Your recommended action or explanation
3. Clear next steps with timeframes

## Component 4: Constraints
- Never disclose internal pricing structures
- Never make refund commitments above $500 without escalation
- If asked about competitors, politely redirect to our value

## Component 5: Background Context
The current date is {{today}}. The customer's account tier is {{account_tier}}.
system prompt

Using XML Tags for Structure

Claude and many modern LLMs parse XML tags naturally. Use them to clearly separate distinct sections of a complex prompt — the model treats each section independently and accurately.

✦ XML-Structured Prompt
<context> <customer_tier>Enterprise</customer_tier> <product>Sales Cloud</product> <account_age_years>4</account_age_years> </context> <task> Draft a renewal proposal email for this customer. Emphasise the ROI they've achieved and introduce the new AI features available in their tier. </task> <constraints> - Max 200 words - Professional but warm tone - Include one specific metric placeholder: {{key_metric}} - End with a soft CTA for a renewal call, not a hard close </constraints>

XML Tag Best Practices

  • Use tags to separate document content from instructions: wrap the document in <document>, keep instructions outside
  • Use descriptive tag names that match the semantic content: <customer_complaint> not <text>
  • Tags are especially useful for multi-part inputs: multiple documents, data + instructions, context + task
  • Use <thinking> tags to instruct extended reasoning before the final answer

Retrieval-Augmented Generation (RAG)

RAG combines a retrieval step (searching a knowledge base) with a generation step (the LLM). The retrieved content is injected into the prompt as grounding context — enabling the model to answer questions about your specific data without fine-tuning.

The RAG Prompt Pattern

User Question
→
Vector Search
Retrieve relevant docs
→
Inject into Prompt
Add as <context>
→
LLM Generates
Grounded answer
✦ RAG Prompt Template
Use the following retrieved documents to answer the user's question. If the answer is not contained in the documents, say "I don't have that information" — do NOT invent an answer. <retrieved_documents> <doc id="1" source="knowledge_base" relevance="0.92"> {{retrieved_content_1}} </doc> <doc id="2" source="product_faq" relevance="0.87"> {{retrieved_content_2}} </doc> </retrieved_documents> User question: {{user_question}} When answering: - Cite which document supports each claim (e.g., "According to doc 1...") - Be direct and concise - If documents conflict, acknowledge the conflict and present both views

Grounding Anti-Hallucination

The key instruction in RAG prompts: "Do not use information outside the provided documents." Variations:

  • "If the answer is not in the provided context, say 'I don't have that information.'"
  • "Only answer based on the documents below. Do not draw on prior knowledge."
  • "If you are unsure, express uncertainty rather than guessing."
CHAPTER ELEVEN
Advanced Pattern

Self-Correction & Validation Loops

Self-correction loops ask the model to review and improve its own output. Combined with programmatic validation, they form retry pipelines that dramatically improve output quality without manual review.

The Retry-with-Error Pattern

When an extraction fails schema validation, don't just retry the same prompt — tell the model exactly what went wrong:

def extract_with_retry(document, schema, max_retries=3):
    for attempt in range(max_retries):
        result = call_claude(document, schema)
        errors = validate(result, schema)

        if not errors:
            return result  # success

        # Retry WITH the specific validation error
        document = f"""
Original document: {document}

Your previous extraction attempt:
{result}

Validation errors found:
{errors}

Please correct your extraction to fix these specific errors.
"""
    raise ExtractionFailed(f"Failed after {max_retries} attempts")
python

Self-Evaluation Prompt

Ask the model to rate its own confidence and flag uncertain fields — route low-confidence outputs to human review:

✦ Self-Evaluation Prompt
Extract the data from the document below. For each field, also provide a confidence score (0.0–1.0) and a brief reason if confidence is below 0.8. Return JSON: { "fields": {...}, "confidence": { "field_name": {"score": 0.95, "reason": null}, "another_field": {"score": 0.6, "reason": "Document mentions two possible values"} } }
When Retries Won't Help

Retrying is effective for format errors (the model can produce the right structure if corrected) and structural extraction errors (values in wrong fields). It is NOT effective when the information is simply absent from the source document. Always check whether the failure is fixable before retrying.

CHAPTER TWELVE
Advanced Pattern

Multi-Agent Prompting

Multi-agent systems use multiple specialised AI instances — each with its own prompt, role, and tools — coordinated by an orchestrator. Prompt engineering for multi-agent systems requires thinking about how context flows between agents, not just within a single prompt.

Coordinator Prompt Design

The coordinator's prompt should specify goals, not step-by-step procedures. Over-specification creates rigid pipelines that fail on edge cases:

❌ Procedural (fragile)

  • "First call web_search, then call document_analysis, then call synthesis."
  • "Always run all three agents in sequence."
  • "If web_search returns 5 results, call document_analysis."

✅ Goal-Based (flexible)

  • "Research this topic comprehensively. Use available agents as needed based on what you find."
  • "Goal: produce a cited, multi-source report. Delegate to specialists appropriately."
  • "If an agent returns insufficient results, adapt your approach."

Context Passing Between Agents

Subagents do NOT automatically inherit context from the coordinator. You must explicitly include relevant prior findings in each subagent's prompt:

synthesis_prompt = f"""You are a synthesis specialist. Produce a comprehensive report
from the research below.

Web search findings:
{web_search_results}

Document analysis findings:
{document_analysis_results}

For each claim in your report:
- Cite the specific source (URL or document name)
- Note the date of the source
- Flag any contradictions between sources

Do not add information beyond what is provided above.
"""

Error Propagation in Multi-Agent Prompts

Design subagent prompts to return structured errors — not silent failures:

✦ Subagent Error Reporting Prompt
If you cannot complete your assigned task, do NOT return empty results. Instead, return a structured error report: { "status": "error", "error_type": "timeout" | "access_denied" | "no_results" | "partial", "attempted": "description of what was tried", "partial_results": [...], // whatever was retrieved before failure "suggested_recovery": "what the coordinator could try instead" } A timeout and an empty result set are different situations — report them differently.

The Prompt Lifecycle

A prompt that delivers value in production is not written once and forgotten. It passes through a lifecycle: development, testing, monitoring, and iteration. Understanding this lifecycle is what separates a one-time experiment from a reliable, scalable AI feature.

The Four Stages

1. Develop
Design & build
→
2. Test
Evaluate output
→
3. Monitor
Measure in production
→
4. Iterate
Improve continuously

Stage 1: Develop

Use tools like Anthropic's Workbench, Salesforce Prompt Builder, or LangSmith to author and version prompts. Capture:

  • The business objective this prompt serves
  • The intended input types (expected range of inputs)
  • The success criteria (what does a correct output look like?)

Stage 2: Test

Build an evaluation set — a collection of representative inputs with expected outputs. Run the prompt against this set and measure:

  • Accuracy: Does the output match the expected output?
  • Format compliance: Does it match the required structure?
  • Edge case coverage: Does it handle missing data, unusual formats, adversarial inputs?
  • False positive rate (for classification/review prompts)

Stage 3: Monitor

Once deployed, track:

  • Output quality over time — models update; prompts that worked yesterday may behave differently tomorrow
  • User feedback signals — dismissals, edits, escalations after AI responses
  • Distribution of outputs — if all results fall into one category, something is wrong
  • Latency and cost — especially for high-volume prompts

Stage 4: Iterate

Use monitoring data to identify prompt improvements:

  • High false positive rate → add exclusion examples or explicit criteria
  • Format inconsistency → add few-shot examples anchoring the format
  • Reasoning errors → add chain-of-thought guidance
  • Scope creep → tighten the constraint section of the system prompt
Version Control for Prompts

Treat prompts like code: version them, document changes, and maintain a changelog. A prompt that changed without documentation is a prompt you cannot debug. Use Prompt Templates in Salesforce Prompt Builder, or store prompts in your source control system alongside the code that uses them.

Anti-Patterns to Avoid

These are the most common prompt engineering mistakes that lead to unreliable, expensive, or unsafe AI outputs.

The Most Dangerous Anti-Patterns

❌ Anti-Pattern 1 — Vague Confidence Instructions

"Only report high-confidence findings." Models cannot reliably self-calibrate confidence. This instruction suppresses output volume but not false positive rate — the model remains overconfident on the same errors it was wrong about before. Replace with explicit categorical criteria: "Flag X only when Y is true."

❌ Anti-Pattern 2 — Prompt-Based Enforcement for Critical Business Rules

"Always verify identity before processing a refund." Prompt instructions have a non-zero failure rate — even excellent prompts fail on some inputs. For rules with financial or safety consequences, use programmatic enforcement (code gates) not prompt guidance.

❌ Anti-Pattern 3 — Required Fields for Optional Information

Making every field required in a JSON schema forces the model to fabricate values for fields not present in the source document. Always mark fields as nullable when the source may not contain them.

❌ Anti-Pattern 4 — Self-Review

"Review your previous answer for errors." A model reviewing its own output retains the reasoning context that produced the error — it is unlikely to catch mistakes it did not notice the first time. Use an independent review instance with no knowledge of the original generation.

❌ Anti-Pattern 5 — Infinite Retry Loops

Retrying the same prompt indefinitely for missing information that is simply not in the source document. Retries help with format errors; they cannot create information that does not exist. Implement maximum retry limits and distinguish "fixable" from "unfixable" failures.

❌ Anti-Pattern 6 — Silent Error Suppression

Having agents return empty results instead of errors when they fail. Empty results look like "no data found" to the coordinator, when the truth is "the tool crashed." Always return structured error context — failure type, what was attempted, partial results.
