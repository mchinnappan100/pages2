A complete practitioner guide to building Autolaunched Flows that power Agentforce agent actions — from variable design and error handling to modular architecture, permissions, and live worked examples.

Autolaunched Flows Agent Actions Input / Output Variables Error Handling Permissions Record Variables

Why Flows Are the Engine of Agentforce

Agentforce agents are remarkably powerful at understanding language, retrieving context, and reasoning about what to do next. But without flows, an agent can only read data — it cannot change anything. Flows are the only low-code mechanism for an agent to create, update, or delete Salesforce records.

Flows also solve a precision problem. When agents query data freely, they choose what they think is relevant — and they can be wrong. A flow constrains the query: it retrieves exactly the fields you specify, with exactly the filters you define, and returns only what the agent actually needs. This dramatically improves accuracy.

❌ Without Flows

  • Agent can only read data, not change it
  • Agent decides which fields to use
  • Risk of irrelevant or sensitive data reaching the agent
  • No guaranteed query filters
  • No structured error feedback to the agent

✅ With Flows

  • Agent can create, update, and delete records
  • Flow controls exactly which fields are returned
  • Only relevant, approved data reaches the agent
  • Precise SOQL-equivalent filters guaranteed
  • Fault paths provide meaningful error messages
Core Mental Model

Think of flows as typed function calls for your agent. The agent supplies typed inputs, the flow executes with guaranteed precision, and the agent receives typed outputs. The agent reasons and communicates; the flow acts and retrieves.

CHAPTER ONE
Flow Fundamentals for Agents

What Makes a Flow
Agent-Ready

Not every flow can be an agent action. This chapter covers the specific requirements — flow type, descriptive naming, mandatory variables, and record variable design — that make a flow ready to be assigned to an Agentforce agent action.

The Only Supported Flow Type

Agent actions support only one specific flow type: Autolaunched Flow (No Trigger). This is a critical constraint that trips up many builders.

Flow TypeSupported as Agent Action?Notes
Autolaunched Flow (No Trigger)✅ Yes — requiredThe only supported type
Record-Triggered Flow❌ NoEven though it's technically autolaunched
Schedule-Triggered Flow❌ No—
Screen Flow❌ NoAgents can't interact mid-flow
Platform Event–Triggered Flow❌ No—
Existing Flows

If you want to use an existing flow as an agent action and it's not of the Autolaunched Flow (No Trigger) type, you must re-create it using the correct type. There is no conversion path — a new flow must be built.

The reason Screen Flows are excluded is fundamental: agents cannot pause mid-execution to wait for user input inside a flow. If interaction is needed, the agent handles it conversationally before running the flow — passing the user's answers as input variables.

Naming and Descriptions — The Most Important Non-Technical Requirement

AI works on language. The agent reads your flow's name, description, and variable names to understand what the flow does and how to use it. Poor naming causes the agent to misuse flows, skip them entirely, or use the wrong data.

Flow Description = Agent Instruction

This is the most important pattern to internalize: the flow's description automatically becomes the agent action's instructions when you add it to an agent. Write it as a directive, not a label.

ContextLess EffectiveMore Effective
Flow descriptionUpdates a phone numberUpdate the user's phone number associated with their contact record. If no matching contact exists, create a new contact record.
Output variable nameOrderCollSorted_Cupcake_Orders_Collection
Output variable descriptionA collection of cupcake ordersA collection of the customer's cupcake orders, matching the email address given by the customer, sorted by most recent.
Input variable nameidContact_ID
Get Records descriptionGets bookingsRetrieves all upcoming bookings assigned to the found contact that have not been canceled. The bookings are stored in the Contact_Bookings variable.
Best Practice — Write Descriptions as Instructions

The flow description becomes the agent action instruction. Write it in imperative form: "Find all the Booking records related to the guest's contact record. Only retrieve Bookings whose date hasn't occurred yet." — not "A flow that retrieves bookings."

Variable Name Rules

  • Use underscores between words — the API Name field doesn't allow spaces. Write Account_ID_Number not AccountIDNumber
  • Be descriptive: Contact_ID tells the agent far more than cid
  • Use complete words, not abbreviations: Error_Message_Output not ErrMsg

Input and Output Variables — Always Required

Every flow used as an agent action must have at least one input variable and at least one output variable. This is both a technical requirement and a best practice — flows that accept and return nothing give the agent no context and no result to reason about.

Input Variables

Input variables are how the agent passes data into the flow. Common examples:

  • A Contact_ID (Text) — the ID of the customer the agent is working with
  • A Booking_ID (Text) — the ID of the record to update or retrieve
  • A New_Phone_Number (Text) — a value the agent received from the user

Always set Available for input on variables the flow receives from the agent. Set Available for output on variables the flow returns to the agent. These are separate checkboxes — a variable can be both, neither, or either.

The Always-Include Output Variable: Error Message

Even if a flow produces no meaningful data to return, always include an error message output variable. Without it, when the flow fails, the agent has nothing to report — it may apologise generically, present unrelated data, or behave unpredictably.

// Always include this output variable pattern in every agent-ready flow
API Name:    Error_Message_Output
Description: The error message to show to the customer when this flow
             doesn't finish successfully.
Data Type:   Text
Input:       Not selected
Output:      Selected ✓
flow variable spec

Record Variables — More Powerful Than You Think

Record variables are not just for internal flow operations. Agents can receive them as output! Instead of returning four separate text fields from a Case record, return one record variable containing all four fields. The agent receives the structured record and presents it in a readable format automatically.

Record vs Record Collection

Variable TypeReturnsUse When
Record VariableOne record with selected fieldsThe flow retrieves a single matching record
Record Collection VariableMultiple records with selected fieldsThe flow retrieves many matching records (e.g., all bookings)
Key Insight

When an agent receives a record variable or collection, it automatically organises the data into a readable format before presenting it to the customer. You don't need to format the output yourself — just return the record and let the agent handle presentation.

CHAPTER TWO
Designing Agent-Ready Flows

Architecture, Data Control,
and Permissions

Good flow design for agents goes beyond making it work once. This chapter covers modular architecture, controlling exactly what data reaches the agent, fault path error handling patterns, and managing the dedicated agent user's permissions.

Modular Flow Design — One Flow, One Task

Agents work best with many small, focused actions — not one large flow trying to do everything. Each flow should complete exactly one logical task. This modularity enables flows to be reused across multiple agents and combined in different sequences.

Finding the Right Boundaries

The key question when drawing the boundary of a flow is: what is one complete task the agent might be asked to do?

Interaction Boundaries

Agents cannot ask the customer for information mid-flow. If you need user input partway through a process, that's a natural flow boundary. The agent gathers the input conversationally, then runs the next flow with that input as a variable. Design your flows around this constraint.

Example: Opportunity Management Agent

Consider an agent that manages opportunities. Instead of one flow handling everything, break it into tasks:

Agent asks:
which opportunity?
→
Flow 1:
Get Opportunity Data
→
Agent asks:
what action?
→
Flow 2a:
Update Stage
  • Flow 1: Retrieve the opportunity (takes name/ID input, returns opportunity data)
  • Flow 2a: Update the Stage field (takes opp ID + new stage, returns updated record)
  • Flow 2b: Update the Close Date (takes opp ID + new date, returns updated record)
  • Flow 2c: Update both Stage and Close Date simultaneously (one request = one flow is fine)

Notice that Flow 2a and 2b are separate because they represent separate user requests. But updating both at once doesn't need a third separate flow — it's one combined action the user explicitly asked for.

Controlling What Data Reaches the Agent

Agents are generous with data — when they receive it, they tend to use it, present it, or reason about it. Never send an agent data you don't want it to use. This applies especially to sensitive fields and fields that might confuse the agent's reasoning.

The Get Records Data Setting

The default Get Records configuration stores all fields automatically. For agent-facing flows, always change this:

❌ Default (Avoid)

  • How to Store Record Data: Automatically store all fields
  • Returns every field on the object — 40+ fields for a typical record
  • Sensitive fields (SSN, salary, internal notes) reach the agent
  • Agent may present or reason about fields unexpectedly

✅ Agent-Safe (Use This)

  • How to Store Record Data: Choose fields and assign variables (advanced)
  • You explicitly select only the fields needed
  • Sensitive fields never enter the agent's context
  • Agent works with a clean, minimal data set
// Get Records element — agent-safe configuration
How Many Records to Store:   All records (or Only the first record)
How to Store Record Data:    Choose fields and assign variables (advanced) ✓
Record Collection:           [Your output variable name]

// Then select ONLY the fields the agent needs:
Field 1: Experience_Name__c
Field 2: Date__c
Field 3: Start_Time__c
Field 4: End_Time__c
// NOT: Id, OwnerId, CreatedById, LastModifiedDate, SystemModstamp...

Error Handling — Fault Paths Are Mandatory

A flow that can fail without reporting the error leaves your agent helpless. Use fault paths with Assignment elements to capture failure information and return it to the agent via your error output variable.

How Fault Paths Work

Every data element (Get Records, Create Records, Update Records, Delete Records) supports a fault path. The fault path runs only when that specific element fails. An Assignment element on the fault path sets your error message variable to a customer-friendly message.

Writing Customer-Friendly Error Messages

The error message set on the fault path is what the agent will share with the customer. Write it conversationally, matching your agent's persona:

// Good error message — friendly, specific, offers next step
"I'm sorry, I'm having a problem finding your booked activities.
 Would you like to speak to our support team?"

// Good error message — contextual to the operation that failed
"I'm sorry, I'm having a problem canceling that activity booking.
 Would you like to speak to our support team?"

// Bad — too technical, exposes internals
"Flow element Get_Contact_s_Upcoming_Bookings failed: SOQL error"
error message examples
Important — Error Messages Don't Auto-Escalate

The text "Would you like to speak to our support team?" in an error message does not automatically trigger escalation. It's a conversational prompt. To enable actual escalation, you must create a separate agent action that performs the escalation logic. The error message only informs the agent what to say.

Agent Permissions and the Agent User

Different agent types use permissions differently, and getting this wrong is a security risk:

Agent TypeWhose Permissions Are UsedKey Consideration
Employee-facing agentsThe individual Salesforce user interacting with the agentNatural data access scoping per employee
External-facing agents (e.g., customer service on website)A single dedicated Agent UserThis shared user must have minimal, carefully scoped permissions

Finding and Configuring the Agent User

  1. Open Agentforce Builder for your agent

  2. Navigate to the Agent Details page

  3. Find the Agent User field — this shows the dedicated user (e.g., EinsteinServiceAgent User)

  4. Navigate to that user in Setup → Users

  5. Assign only the permission sets, profiles, and roles needed to run the agent's flows and fulfill its tasks — nothing more

Security Risk — Over-Permissioned Agent User

The Agent User for an external-facing agent can access all Salesforce data it has permission to read. Giving it System Administrator or overly broad permissions means any customer conversation could potentially expose sensitive internal data. Apply the principle of least privilege strictly.

CHAPTER FIVE
Connecting Flows to Agents

From Flow to Agent Action:
The Full Connection

A completed, activated flow doesn't automatically become available to an agent. This chapter walks through the Subagent → Agent Action → Flow connection, explains how to sequence multiple flows, and covers writing effective action instructions.

Subagents and Agent Actions

Flows are never assigned directly to an agent. The hierarchy is: Agent → Subagent → Agent Action → Flow.

What Is a Subagent?

A Subagent is a logical grouping of related agent actions. Think of it as a department: a "Bookings Management" Subagent might contain actions for viewing bookings, cancelling bookings, and rescheduling bookings. A separate "Account Management" Subagent might handle profile updates and payment methods.

Modular subagents can be reused across multiple agents. An agent for a customer support portal might share the same Bookings Management Subagent as an agent for a mobile app.

Sequencing Multiple Flows

Many agent interactions require running multiple flows in sequence — the output of one becomes the input of the next. The agent handles this coordination using Conversation Variables.

The Authentication → Task Pattern

A very common pattern in external-facing agents:

  1. Customer makes a request (e.g., "cancel my booking")

  2. Agent asks for verification details (email + member number)

  3. Agent runs Flow 1: Get Customer Details → receives Contact_ID

  4. Agent stores Contact_ID in a Conversation Variable

  5. Agent asks which booking to cancel

  6. Agent runs Flow 2: Get Bookings passing Contact_ID → receives booking list

  7. Agent presents list, customer selects → agent has Booking_ID

  8. Agent runs Flow 3: Cancel Booking passing Booking_ID

Subagents Don't Share Flows Automatically

If the Get Customer Details flow is on the Bookings Subagent, the agent can pass the Contact_ID output to the next flow on the same Subagent. The agent acts as the coordinator — you give it clear instructions about what data to pass between action calls.

Writing Effective Agent Action Instructions

When you create an agent action for a flow, the flow's description is automatically used as the action's instruction. This is why we write the flow description as a directive.

What Makes Good Instructions

  • Tell the agent WHEN to run the flow — not just what it does
  • Specify what inputs to pass — e.g., "pass the Contact_ID retrieved from the Get Customer Details action"
  • Specify how to handle the output — e.g., "present the returned booking records to the customer"
  • Handle the error output — e.g., "if Error_Message_Output is not empty, share that message with the customer"
// Example agent action instruction (written as flow description)
"Find all the Booking records related to the guest's contact record.
 Only retrieve Bookings whose date hasn't occurred yet.
 Pass the Contact_ID from the Get Customer Details action as input.
 If Error_Message_Output is populated, share that message with the
 customer instead of the bookings."
agent action instruction

The Flow → Agent Action Lifecycle

Understanding when Salesforce reads your flow's metadata is critical for maintaining accuracy over time.

Critical — Static Snapshot at Action Creation

When you create an agent action from a flow, Salesforce takes a one-time snapshot of the flow's input and output variables. This snapshot is static — it does NOT automatically update when you change the flow's variables later.

If you add, rename, or remove a variable in the flow, you must: (1) delete the agent action, then (2) create a new agent action pointing to the updated flow.

Change to the FlowAgent Action ImpactRequired Action
Change element logic (filter conditions, field values)None — logic changes apply immediatelyNo action needed
Add a new version of the flowAction continues using the active versionNo action needed if variable signatures unchanged
Add, rename, or remove a variableAction uses stale variable snapshotDelete action → create new action
Change variable data type or availabilityAction uses stale snapshotDelete action → create new action

CHAPTER SIX
Pitfalls, Limits & Best Practices

What to Avoid and
How to Ship Reliably

A summary of the most common mistakes, hard technical limits, and a pre-launch checklist to validate your flows are truly agent-ready before connecting them to a live agent.

Anti-Patterns to Avoid

❌ Using "Automatically Store All Fields"

The biggest data risk in agent flows. Default Get Records stores everything — including sensitive fields the agent should never see. Always use "Choose fields and assign variables" and explicitly select only needed fields.

❌ Skipping the Error Message Output Variable

Without an error output variable, when a flow fails the agent has no structured feedback. It may apologise generically, present unrelated data, or behave unpredictably. Include Error_Message_Output on every agent flow — it is as essential as the functional output variables.

❌ Long Text Strings in Text Variables

Text variables hold a maximum of 255 characters. Text exceeding this limit is silently truncated — no error is raised. If you need to pass more than 255 characters to a flow, work with a developer to create an Apex-defined variable instead.

❌ Unsupported Output Data Types

The following data types cannot be passed back to the agent via output variables:

Currency Picklist Multi-Select Picklist Apex-Defined

If a flow attempts to pass any of these types as an output variable, the agent can no longer handle any output variables from that flow — not just the incompatible one.

❌ Not Activating the Flow Before Creating the Agent Action

A flow must be in Active status to be selected as an agent action. Saving a flow in Draft status will make it invisible in the agent action flow selection list.

❌ One Giant Flow for Everything

Monolithic flows that attempt to handle many tasks are harder to debug, impossible to reuse, and confusing for the agent. Each flow should complete one clear task. If you need to handle multiple things, chain multiple focused flows.

❌ Changing Flow Variables Without Rebuilding the Action

Because agent actions take a one-time snapshot of variables at creation, adding or changing variables in the flow without deleting and recreating the agent action leaves the action with an outdated schema — leading to silent failures or missing data.

Pre-Launch Checklist

Before connecting a flow to an agent action and deploying to users, verify every item:

  1. Flow type is correct: Autolaunched Flow (No Trigger) — not record-triggered, not screen flow

  2. Flow is Active: Draft flows cannot be assigned as agent actions

  3. At least one input variable exists with Available for input checked and a clear, descriptive name and description

  4. At least one output variable exists with Available for output checked and a descriptive name and description

  5. Error_Message_Output variable exists on every flow as a Text output variable

  6. Every data element has a fault path (Get Records, Update Records, Create Records, Delete Records)

  7. Get Records uses "Choose fields" mode — not "Automatically store all fields"

  8. Only necessary fields are selected in Get Records — no sensitive or irrelevant fields

  9. No unsupported output types: no Currency, Picklist, Multi-Select Picklist, or Apex-Defined output variables

  10. Flow description is written as an agent instruction — directive, specific, actionable

  11. All variable descriptions are thorough — explains what the data is, where it came from, how it should be used

  12. Agent User permissions reviewed — minimal access, only what's needed to run this flow's operations

SubagentA logical grouping of related agent actions within an agent. Handles a domain of related tasks (formerly called a Topic).
Agent UserThe dedicated Salesforce user whose permissions are applied when an external-facing Agentforce agent runs flows. Found on the Agent Details page.
Input VariableA flow variable with "Available for input" selected. Receives data from the agent when the flow is invoked.
Output VariableA flow variable with "Available for output" selected. Returns data from the flow back to the agent after execution.
Record VariableA flow variable of type Record that holds a single Salesforce record with selected fields. Can be passed to the agent as output.
Record Collection VariableA flow variable of type Record with "Allow multiple values (collection)" enabled. Holds multiple Salesforce records.
Error_Message_OutputThe standard name for a Text output variable that holds a customer-facing error message, set by an Assignment element on a fault path.
Fault PathA special path on a data element (Get Records, Update Records, etc.) that runs only when that element fails. Used to handle errors gracefully.
Assignment ElementA flow element that sets variables to specified values. Used on fault paths to set error messages before flow failure is reported to the agent.
Get RecordsA flow element that queries Salesforce records using filter conditions. In agent flows, always use "Choose fields and assign variables" mode.
Static Variable SnapshotThe one-time copy of a flow's input/output variable schema taken when an agent action is created. Does not update automatically when flow variables change.
Choose fields and assign variablesThe Get Records data storage mode that requires explicitly selecting which fields to retrieve and assigning them to a variable. Mandatory for agent flows.
Conversation VariableA typed data slot in an agent session. Actions write to them; the agent can pass them as inputs to subsequent flow actions.
Modular Flow DesignThe practice of creating many small, single-purpose flows (one task each) rather than one large flow. Enables reuse and simplifies agent action configuration.
