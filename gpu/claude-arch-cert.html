CHAPTER ONE
Agentic Architecture & Orchestration

Building Autonomous
Agent Systems

Agents are Claude-powered systems that perceive input, reason, take actions via tools, and loop until a goal is achieved. This chapter covers the agentic loop, orchestration patterns, multi-agent coordination, and how to design safe, reliable autonomous systems.

27% of exam — highest weight

The Agentic Loop

The agentic loop is the fundamental execution pattern of an autonomous Claude agent. The model receives a task, reasons about it, optionally calls tools, processes tool results, then continues reasoning until a final answer is produced or the loop is terminated.

User Task
→
Claude Reasons
→
Tool Call
→
Tool Result
→
Reasons Again
→
Final Answer

stop_reason Values

The stop_reason field in the API response is how your code detects what to do next:

stop_reasonMeaningAction
end_turnClaude finished respondingReturn response to user
tool_useClaude wants to call one or more toolsExecute tools, append results, call API again
max_tokensHit the token limitHandle truncation or extend
stop_sequenceHit a custom stop sequenceProcess partial output
# Minimal agentic loop in Python
import anthropic

client = anthropic.Anthropic()
messages = [{"role": "user", "content": "What is the weather in Tokyo?"}]

while True:
    response = client.messages.create(
        model="claude-opus-4-5",
        max_tokens=4096,
        tools=tools,
        messages=messages
    )

    # Append assistant turn to message history
    messages.append({"role": "assistant", "content": response.content})

    if response.stop_reason == "end_turn":
        break  # Done — return final text to user

    if response.stop_reason == "tool_use":
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })
        messages.append({"role": "user", "content": tool_results})
python
Exam Focus

The message history must alternate between user and assistant roles. Tool results are sent back as a user turn containing a list of tool_result blocks — NOT as a new assistant turn. This role alternation is a frequently tested concept.

Orchestration Patterns

Single-Agent Loop

One Claude instance with a set of tools. Simple, low-latency, best for tasks where all sub-tasks can be completed by one context. The loop runs until end_turn or a maximum step limit is reached.

Orchestrator + Subagent Pattern

An orchestrator Claude receives the high-level task and breaks it into sub-tasks. It calls subagent tools that each launch their own Claude API calls. Subagents are specialised, focused, and cheaper to run in parallel.

Orchestrator Claude
→
dispatch_task tool
→
Subagent A
Orchestrator Claude
→
dispatch_task tool
→
Subagent B

Pipeline Pattern

Sequential agents where the output of Agent A becomes the input of Agent B. Each agent is specialised — a planner agent, a researcher agent, a writer agent. The final agent produces the polished output.

Parallel Fan-Out

The orchestrator dispatches identical or different sub-tasks to multiple agents simultaneously, then collects and synthesises their results. Useful for parallelising independent research tasks.

Key Concept — When to Use Each Pattern

Single agent: task fits in one context, low latency needed. Orchestrator: tasks decompose cleanly, subagents can specialise. Pipeline: sequential enrichment required. Parallel: independent subtasks with speed requirement.

Guardrails and Loop Safety

Autonomous agents running in loops without safety limits can cause runaway behaviour. Every production agent must implement:

  • Maximum iterations: Hard limit on the number of agentic loop iterations (typically 10–50 depending on task complexity).
  • Token budget: Track cumulative token usage; abort if budget exceeded.
  • Human-in-the-loop (HITL) checkpoints: For high-stakes actions (irreversible writes, sending emails, executing code in production), pause and require human confirmation.
  • Tool permission scoping: Restrict tools to only those needed for the current task. Don't give an agent database write access if only read access is required.
  • Idempotency: Design tools to be safely retried — the same tool call twice should not cause double-execution of side effects.
  • Timeout handling: Tool calls must have timeouts; surface errors cleanly to Claude rather than hanging indefinitely.
# Agentic loop with safety controls
MAX_ITERATIONS = 20
iteration = 0
total_tokens = 0
TOKEN_BUDGET = 100_000

while iteration < MAX_ITERATIONS:
    iteration += 1
    response = client.messages.create(...)

    total_tokens += response.usage.input_tokens + response.usage.output_tokens
    if total_tokens > TOKEN_BUDGET:
        raise AgentBudgetExceeded(f"Spent {total_tokens} tokens")

    if response.stop_reason == "end_turn":
        break

    # Check for irreversible tools — require human approval
    for block in response.content:
        if block.type == "tool_use" and block.name in HITL_TOOLS:
            approved = request_human_approval(block.name, block.input)
            if not approved:
                raise ActionDenied(block.name)

Multi-Agent Architecture Patterns

The Subagent as a Tool

In multi-agent systems, a subagent is exposed to the orchestrator as a tool. The orchestrator calls run_research_agent with a task description; the tool implementation launches a complete Claude API loop and returns the final answer.

{
  "name": "run_research_agent",
  "description": "Delegate a research task to a specialised subagent. Returns a detailed report.",
  "input_schema": {
    "type": "object",
    "properties": {
      "task": {"type": "string", "description": "The research question to investigate"},
      "max_steps": {"type": "integer", "description": "Maximum reasoning steps"}
    },
    "required": ["task"]
  }
}
json

Shared Memory and State

Agents in a multi-agent system need to share state. Common patterns:

  • Message-passing: Results returned through tool call outputs — stateless, clean, auditable.
  • Shared external store: A database or key-value store both agents read/write. Requires careful locking.
  • Conversation thread injection: Prepend prior agent outputs as context to the next agent's system prompt.
Common Pitfall

Never pass sensitive credentials (API keys, tokens) through tool outputs that go back into the conversation — these become part of the model's context and may appear in logs or future completions. Use a secrets vault and inject credentials only in the execution environment.

CHAPTER TWO
Tool Design & MCP Integration

Extending Claude with
Tools and MCP Servers

Tools give Claude the ability to take actions in the real world — search the web, query databases, write files, call APIs. The Model Context Protocol (MCP) is the open standard that enables Claude to connect to any external service through a consistent interface.

18% of exam weight

Tool Schema Design

Tools are defined using JSON Schema. The quality of a tool definition directly impacts how accurately Claude decides when and how to call it. Claude reads your schema like documentation.

The Three Pillars of a Good Tool

  • Name: A clear verb-noun pairing (search_web, create_file, get_weather). Claude uses the name first to decide relevance.
  • Description: The most important field. Explain WHAT the tool does, WHEN to use it (vs alternatives), and what the output represents. Be explicit about limitations.
  • Input Schema: Use required for all non-optional parameters. Use enum to restrict choices. Add description to each property — these guide Claude's parameter selection.
{
  "name": "search_knowledge_base",
  "description": "Search the internal product knowledge base for documentation, FAQs, and troubleshooting guides. Use this when the user asks about product features, setup, or errors. NOT for general web search.",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "The search query. Be specific and use product terminology."
      },
      "category": {
        "type": "string",
        "enum": ["docs", "faq", "troubleshooting", "api"],
        "description": "Filter by content category. Use 'troubleshooting' for error messages."
      },
      "max_results": {
        "type": "integer",
        "default": 5,
        "minimum": 1,
        "maximum": 20
      }
    },
    "required": ["query"]
  }
}

Tool Call Lifecycle

Understanding the exact sequence of API calls and message formats is critical for the exam:

  1. Send tools in the API request: Include the tools array in every API call where tools may be needed.
  2. Claude returns a tool_use block: Contains id, name, and input (the JSON arguments Claude selected).
  3. Your code executes the tool with the provided inputs.
  4. Return a tool_result block: A user message containing the result, referenced by the same tool_use_id.
  5. Call the API again — Claude reads the result and continues.
# Step 4: Returning tool results correctly
messages.append({
    "role": "user",
    "content": [
        {
            "type": "tool_result",
            "tool_use_id": "toolu_01A09q90qw90lq917835lq9",  # Must match the tool_use block id
            "content": "Tokyo: 22°C, Partly Cloudy, Humidity 68%"
        }
    ]
})

When Claude calls multiple tools in a single turn, ALL tool results must be returned in one user message as an array of tool_result blocks before the next API call. You cannot return them one at a time in separate API calls.

Model Context Protocol (MCP)

MCP is an open standard that defines how Claude (or any AI model) connects to external tools and resources. It separates the tool runtime from the model — you build an MCP server once and any MCP-compatible client can use it.

MCP Architecture

MCP Client
(Claude Code/API)
⇄ stdio/HTTP+SSE
MCP Server
(Your code)
→
External Service
(DB, API, File)

MCP Primitives

  • Tools: Actions the model can invoke (execute code, search, write). Server-side functions exposed to the client.
  • Resources: Data the model can read (files, database rows, API responses). Identified by URIs.
  • Prompts: Reusable prompt templates the user or client can invoke by name with arguments.

MCP Transport Mechanisms

  • stdio: MCP server runs as a subprocess; client communicates via stdin/stdout. Default for local/CLI tools. Simple, no network needed.
  • HTTP + SSE: Server runs as an HTTP service; client uses Server-Sent Events for streaming responses. Used for remote servers and multi-client scenarios.
# Minimal MCP server with FastMCP (Python)
from fastmcp import FastMCP

mcp = FastMCP("My Tools Server")

@mcp.tool()
def get_stock_price(ticker: str) -> dict:
    """Get the current stock price for a given ticker symbol."""
    # ... implementation ...
    return {"ticker": ticker, "price": 142.50, "currency": "USD"}

@mcp.resource("config://app")
def get_config() -> str:
    """Return application configuration."""
    return '{"env": "prod", "version": "2.1"}'

if __name__ == "__main__":
    mcp.run()  # Defaults to stdio transport

External Service Integration

When designing tool integrations with external services, follow these principles:

  • Use Named Credentials: Never hardcode API keys in tool definitions. Pass them via environment variables or secret managers.
  • Handle errors gracefully: Return informative error strings in tool_result — Claude uses them to self-correct or report to the user.
  • Scope permissions narrowly: A tool that only needs to read a database should not have write permissions.
  • Return structured data: JSON objects are easier for Claude to reason about than free-form text.

Hook Exit Codes

Hook scripts communicate back to Claude Code through exit codes and stdout:

  • Exit 0: Allow the action to proceed
  • Exit 1 (PreToolUse): Block the tool use — Claude receives an error message
  • stdout JSON: A hook can return {"decision": "block", "reason": "..."} to provide Claude with context about why the action was blocked
Exam Focus

PreToolUse hooks can BLOCK tool execution by exiting with a non-zero code. PostToolUse hooks cannot reverse an action — they run AFTER the tool has already executed. The hook receives tool input via environment variables and stdin.

Permissions Model

Claude Code operates under a least-privilege permission model. You specify exactly what tools and file paths Claude is allowed to use.

Permission Rule Syntax

// Permission patterns
"Bash(npm run *)"        // Allow npm run commands
"Bash(git log *)"        // Allow git log but not git push
"Read(**)"               // Allow reading any file
"Write(src/**)"          // Allow writing only under src/
"Write(*.md)"            // Allow writing Markdown files only
"Bash(rm *)"             // Allow rm (be careful!)
json

Three-Tier Trust

Claude Code distinguishes between three trust levels for MCP servers:

  • User-approved (full trust): Servers in ~/.claude/settings.json — can use all tools
  • Project-approved: Servers in .claude/settings.json — team-shared tools
  • Prompt-injected (distrusted): Servers added via prompts — Claude displays a warning and requires explicit user approval

CHAPTER FOUR
Prompt Engineering & Structured Output

Crafting Precise Prompts and
Reliable Structured Output

Prompt engineering is the art of giving Claude clear, unambiguous instructions. Structured output enables production reliability — parsing Claude's responses programmatically without fragile string matching.

20% of exam weight

System Prompts

The system prompt is your primary lever for shaping Claude's behaviour. It is processed before any human turn and establishes persona, capabilities, constraints, and output format expectations.

System Prompt Anatomy

  • Role definition: Tell Claude who it is and its specific domain of expertise.
  • Capability scope: What Claude should and should not do in this context.
  • Output format instructions: How Claude should format its responses — always specify this for production systems.
  • Constraints: Hard rules — topics to avoid, required disclaimers, safety behaviours.
  • Context: Background information Claude needs but users don't need to repeat each turn.
system_prompt = """You are a senior software engineer assistant at Acme Corp.

## Your Role
Help engineers write, review, and debug Python and TypeScript code. You have
access to the codebase via the provided tools.

## Behaviour Rules
- Always explain your reasoning before providing code
- When reviewing code, output structured feedback using the format below
- NEVER suggest using deprecated libraries (check versions first)
- If you are uncertain, say so — don't hallucinate function names

## Output Format for Code Reviews
Respond in this exact JSON structure:
{"severity": "critical|major|minor|info", "issues": [...], "suggestions": [...]}
"""
python

Effective Constraint Writing

Best Practice

Tell Claude what TO DO rather than what NOT to do where possible. Instead of: "Don't be verbose." Write: "Be concise — maximum 3 sentences per answer." Positive instructions are more reliable than negative restrictions.

XML Structuring in Prompts

Claude is trained to parse XML tags naturally. Use them to organise complex prompts with multiple sections — they improve Claude's ability to locate and reference different parts of the input.

# Using XML tags to structure complex prompts
user_message = """

  <user_role>Data Scientist</user_role>

  <libraries_available>pandas, numpy, scikit-learn, plotly</libraries_available>
</context>

<task>
  Analyse the attached CSV data and create a model to predict customer churn.
  Return the code along with a brief explanation of the approach.
</task>

<constraints>
  - Use only the libraries listed above
  - Target F1 score of 0.80+
  - Model must be serialisable with joblib
</constraints>

<data>
  {csv_data}
</data>
"""

When to Use XML Tags

  • Separating document content from instructions (<document> vs task)
  • Multi-part inputs with different types (code, data, instructions)
  • Specifying output format sections Claude should fill
  • Long prompts where you need Claude to reference specific sections

Structured Output Patterns

Pattern 1: JSON in the System Prompt

The most common pattern — instruct Claude to output JSON in the system prompt and validate programmatically. Optionally pre-fill the assistant turn with {" to constrain the start of the response.

        {"role": "assistant", "content": '{"entities":'}  # Pre-fill constrains output

Pattern 2: Tool Use for Structured Output

Define a tool with your desired schema and force Claude to call it. The input field of the tool_use block is always valid JSON matching your schema — making this the most reliable structured output method.

extraction_tool = {
    "name": "extract_entities",
    "description": "Extract named entities from the text.",
    "input_schema": {
        "type": "object",
        "properties": {

            "locations": {"type": "array", "items": {"type": "string"}},
            "dates": {"type": "array", "items": {"type": "string"}}

    max_tokens=1024,

result = response.content[0].input  # Guaranteed to match schema

Advanced Prompt Engineering Techniques

Chain-of-Thought (CoT)

For complex reasoning tasks, instruct Claude to show its work before giving a final answer. Use a <thinking> tag pattern or simply say "Think step by step before answering." CoT significantly improves accuracy on multi-step reasoning, math, and coding.

Few-Shot Examples

Provide 2–5 input/output examples before the actual task. Examples are more effective than abstract descriptions for format compliance. Use consistent XML structure for the examples.

system = """Classify support tickets as: billing, technical, feature_request, or other.

Examples:
<example>
  <ticket>My invoice shows double charge from last month</ticket>
  <label>billing</label>
</example>
<example>
  <ticket>The app crashes when I upload files over 10MB</ticket>
  <label>technical</label>
</example>

Output only the label string, nothing else."""
python

Extended Thinking

Claude's extended thinking mode enables deeper reasoning for complex problems. Enable it via the thinking parameter with a budget_tokens limit. Thinking blocks appear in the response but are not charged to your output tokens for most models.

response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000
    },
    messages=[{"role": "user", "content": "Design the optimal database schema for..."}]
)
# Response contains thinking blocks + text blocks
thinking_block = [b for b in response.content if b.type == "thinking"][0]
answer_block   = [b for b in response.content if b.type == "text"][0]
python
Exam Focus

When using extended thinking + tool use together, thinking blocks from prior turns MUST be included in the message history when continuing the conversation. Stripping them breaks the expected conversation flow. The budget_tokens value is a maximum, not a guarantee — Claude may use fewer tokens.

CHAPTER FIVE
Context Management & Reliability

Context Windows, Caching,
and Production Reliability

Production AI applications must handle limited context windows, manage conversation history efficiently, and operate reliably at scale. This chapter covers prompt caching, token management strategies, long-conversation patterns, and reliability best practices.

15% of exam weight

Prompt Caching

Prompt caching allows you to mark stable content (large system prompts, documents, tool definitions) so Claude can reuse the KV cache from previous requests. This reduces latency by up to 85% and saves up to 90% of input token costs on cached content.

Enabling Prompt Caching

Add {"type": "ephemeral"} cache control to the last message block of the content you want cached. The cache lasts for a minimum of 5 minutes and refreshes on each use.

response = client.messages.create(
    model="claude-opus-4-5",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": large_system_prompt  # Thousands of tokens
        },
        {
            "type": "text",
            "text": company_knowledge_base,  # Static document
            "cache_control": {"type": "ephemeral"}  # Cache everything up to this point
        }
    ],
    messages=conversation_history  # Dynamic part — not cached
)

# Check cache usage in response
print(response.usage.cache_creation_input_tokens)   # Tokens written to cache
print(response.usage.cache_read_input_tokens)        # Tokens read from cache (cheap!)
python

Cache Placement Strategy

The cache breakpoint is the position in the content array where the cache boundary is placed. Everything before the breakpoint is cached. For maximum savings:

  • Place the cache breakpoint after the LAST piece of static content
  • The dynamic/changing parts (user messages, recent context) go AFTER the breakpoint
  • Cache up to 4 breakpoints per request
Exam Focus

Cache creation incurs a one-time cost (~25% more than normal). Cache reads are ~90% cheaper. The minimum cacheable content is 1,024 tokens for Claude Sonnet and 2,048 tokens for Claude Haiku. Requesting cache below this threshold is silently ignored.

Long Conversation Management

As conversations grow, they eventually approach or exceed the context window limit. Several strategies handle this gracefully:

Strategy 1: Sliding Window (Keep Recent + Summary)

When the conversation history exceeds a threshold, summarise early turns and replace them with the summary. Keep the most recent N turns verbatim for continuity.

def trim_conversation(messages: list, max_tokens: int = 50_000) -> list:
    total = count_tokens(messages)
    if total <= max_tokens:
        return messages

    # Keep the last 6 turns verbatim
    recent = messages[-6:]
    older  = messages[:-6]

    # Summarise older turns with a separate API call
    summary = summarise_conversation(older)
    summary_msg = {
        "role": "user",
        "content": f"[Earlier conversation summary: {summary}]"
    }
    return [summary_msg] + recent

Error Handling and Retries

The Claude API returns standard HTTP status codes. Build retry logic for transient errors:

Status CodeTypeAction
400Bad requestFix the request — do not retry
401Auth errorCheck API key — do not retry
429Rate limitedExponential backoff + jitter
500Server errorExponential backoff, up to 3 retries
529OverloadedExponential backoff
import time, random

def call_with_retry(fn, max_retries=3):
    for attempt in range(max_retries):
        try:
            return fn()
        except anthropic.RateLimitError:
            wait = (2 ** attempt) + random.uniform(0, 1)
            time.sleep(wait)
        except anthropic.APIStatusError as e:
            if e.status_code >= 500:
                time.sleep(2 ** attempt)
            else:
                raise  # 4xx — do not retry
    raise MaxRetriesExceeded()

Input Validation and Safety

  • Validate tool inputs before execution — never trust Claude's arguments as safe. SQL injection, path traversal, and command injection can come through tool inputs.
  • Sanitise user input before inserting into system prompts — prompt injection attacks can override your instructions.
  • Use the API's safety system — the anthropic_dangerous_direct_browser_access system prompt flag can be used for specific access patterns, but test your application's behaviour under adversarial input.
