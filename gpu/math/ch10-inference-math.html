Autoregressive Decoding

One token at a time — the math of sequential generation

LLMs generate text one token at a time. Each token generation is a full forward pass through the model. The output probability distribution P(next | context) is sampled to get the next token, which is appended to the context, and the process repeats.

This sequential dependency is why inference latency scales linearly with output length — and why parallelising generation is non-trivial.

Generation of a T-token response: Step 1: context = [t₁, t₂, ..., t_prompt] logits = model(context) ← full forward pass over prompt t_new = sample(softmax(logits[-1])) ← sample last token's distribution Step 2: context = [t₁, ..., t_prompt, t_new] logits = model(context) ← forward pass over all tokens again! t_new2 = sample(softmax(logits[-1])) ... Step T: context = [t₁, ..., t_prompt, t_new, ..., t_newT] Without KV cache: each step recomputes all previous keys and values. Compute cost: O(T²) total ← quadratic in response length! With KV cache: store keys and values from previous steps. Compute cost: O(T) total ← linear in response length!

KV Cache Math

What exactly is cached — and how much memory it costs

In the attention mechanism, for each layer and each head, we compute K and V from the token embeddings. During autoregressive generation, the new token contributes new K and V vectors, but all previous K and V vectors are unchanged. The KV cache stores all previous K and V vectors, avoiding recomputation.

At each new token step, we only need to compute K and V for the new token, then concatenate with the cached K and V, and run attention with the new token's query attending to all cached keys.

KV cache memory per token: = 2 × n_layers × n_kv_heads × d_head × bytes_per_element For LLaMA-3 8B (FP16): n_layers = 32 n_kv_heads = 8 (GQA: 8 key-value heads for 32 query heads) d_head = 128 bytes = 2 (FP16) Per token = 2 × 32 × 8 × 128 × 2 = 131,072 bytes ≈ 0.125 MB/token For a 4096-token context: 0.125 × 4096 ≈ 512 MB For a 128K-token context: 0.125 × 131072 ≈ 16 GB ← needs a lot of VRAM!
Python · KV cache and incremental attention
import numpy as np

class KVCache:
    def __init__(self, max_len, n_layers, n_heads, d_head):
        self.cache = np.zeros((2, n_layers, max_len, n_heads, d_head))
        # cache[0] = keys, cache[1] = values
        self.pos = 0

    def update(self, layer, k_new, v_new):
        """Append new K, V for position self.pos at given layer."""
        self.cache[0, layer, self.pos] = k_new   # shape (n_heads, d_head)
        self.cache[1, layer, self.pos] = v_new
        return (self.cache[0, layer, :self.pos+1],   # all keys so far
                self.cache[1, layer, :self.pos+1])   # all values so far

def incremental_attention(q_new, k_all, v_all, d_head):
    """
    q_new: (n_heads, 1,   d_head)  — query for the new token only
    k_all: (n_heads, T+1, d_head)  — all keys including new
    v_all: (n_heads, T+1, d_head)  — all values including new
    """
    # Score: new query against ALL keys (including cached)
    scores = (q_new @ k_all.transpose(0, 2, 1)) / np.sqrt(d_head)
    # shape: (n_heads, 1, T+1)

    # No masking needed — new token only attends forward (already autoregressive)
    e = np.exp(scores - scores.max(axis=-1, keepdims=True))
    attn = e / e.sum(axis=-1, keepdims=True)

    out = attn @ v_all   # (n_heads, 1, d_head)
    return out

# Example: KV cache for 1 layer, 8 heads, d_head=64, max 100 tokens
cache = KVCache(max_len=100, n_layers=1, n_heads=8, d_head=64)
print(ff"KV cache memory: {cache.cache.nbytes/1e6:.2f} MB (1 layer)")

Quantization

Reducing model size by using fewer bits — with controlled precision loss

Quantization reduces the number of bits used to represent weights (and sometimes activations). A FP32 weight uses 32 bits; an INT8 weight uses 8 bits; INT4 uses 4 bits. The savings are dramatic — a 70B FP16 model is 140 GB; at INT4 it is ~35 GB.

The key mathematical operation is rounding: map a floating-point value to the nearest representable integer value, then store a scale factor to recover the approximate original value.

Linear (absmax) quantization: scale = max(|W|) / (2^(b-1) - 1) ← b = number of bits W_quant = round(W / scale) ← integer in range [-127, 127] for INT8 W_dequant = W_quant × scale ← approximate reconstruction Quantization error: ||W - W_dequant||₂ ≈ scale × √(n/12) (rounding error ≤ scale/2 per element) INT8: scale = max(|W|)/127 → max error per element = max(|W|)/254 INT4: scale = max(|W|)/7 → max error per element = max(|W|)/14 → INT4 has 18× larger error per element than INT8! This is why INT4 LLMs work: the model is overdetermined (billions of weights) and the average prediction is robust to per-weight noise.
Python · INT8 quantization from scratch
import numpy as np

def quantize_int8(W):
    """Absmax quantization to INT8."""
    scale = np.max(np.abs(W)) / 127.0
    W_int8 = np.round(W / scale).astype(np.int8)
    return W_int8, scale

def dequantize(W_int8, scale):
    return W_int8.astype(np.float32) * scale

# Original weight matrix (FP32)
W = np.random.randn(768, 768).astype(np.float32)

W_int8, scale = quantize_int8(W)
W_reconstructed = dequantize(W_int8, scale)

error = np.linalg.norm(W - W_reconstructed, 'fro') / np.linalg.norm(W, 'fro')
print(ff"Relative quantization error: {error:.4f}  ({error*100:.2f}%)")
print(ff"Memory: FP32={W.nbytes/1e6:.2f} MB, INT8={W_int8.nbytes/1e6:.2f} MB")
# INT8 is 4× smaller than FP32, ~1% relative error

# INT4 simulation (NF4 used in QLoRA)
scale4 = np.max(np.abs(W)) / 7.0
W_int4 = np.round(np.clip(W / scale4, -7, 7)).astype(np.int8)
W_rec4 = W_int4.astype(np.float32) * scale4
error4 = np.linalg.norm(W - W_rec4, 'fro') / np.linalg.norm(W, 'fro')
print(ff"INT4 relative error: {error4:.4f}  ({error4*100:.2f}%)")

Speculative Decoding

Use a small model to propose tokens; verify with the large model in parallel

Speculative decoding exploits the fact that for many tokens (common words, obvious continuations), a small draft model can correctly predict what the large model would predict. The algorithm:

  1. Run a small draft model (e.g. 7B) to propose K candidate tokens
  2. Run the large model (e.g. 70B) on all K+1 tokens in parallel (one forward pass)
  3. Accept each draft token if the large model agrees (with probability-ratio criterion)
  4. If a token is rejected, sample from the corrected distribution and discard remaining drafts

This achieves 2-3× speedup because the large model's bottleneck is memory bandwidth (loading weights), not compute — and loading weights for 1 forward pass vs K+1 is the same cost.

Acceptance criterion (token-level): For draft token t with P_draft(t|ctx) and P_target(t|ctx): Accept with probability: min(1, P_target(t) / P_draft(t)) If rejected: sample from adjusted distribution: P_adjusted(x) = max(0, P_target(x) - P_draft(x)) / Z This guarantees the final output distribution is identical to the large model's distribution — no quality loss!

Throughput, Latency & Memory Bandwidth

Why inference is memory-bandwidth-bound — not compute-bound

For batch_size=1 autoregressive decoding (typical for interactive use), the bottleneck is memory bandwidth, not arithmetic throughput. For each new token, every weight in the model must be loaded from GPU HBM memory. With 70B parameters at FP16: 140 GB of memory reads per token.

Memory bandwidth bound (batch_size=1): Time per token ≥ model_size_bytes / memory_bandwidth LLaMA-3 70B FP16 on H100: model_size = 70 × 10⁹ × 2 bytes = 140 GB H100 BW = 3.35 TB/s min_time = 140 GB / 3350 GB/s = 0.042 s = 42 ms per token max_tokens/s ≈ 24 This is the mathematical ceiling — no algorithmic trick can beat it without reducing the number of weight bytes loaded per token. Techniques that reduce memory-bound bottleneck: Quantization (INT4): 140 GB → 35 GB → 4× faster → ~100 tokens/sec Batching: serve B=32 users simultaneously → same weight load, 32× more output tokens → throughput ≈ 32 × 24 = 768 tokens/sec total KV cache: moves decode from O(T²) compute to O(T) — but still BW bound Speculative decoding: amortise BW cost over K≈4 accepted tokens
Python · Estimating LLM inference throughput
def estimate_inference_throughput(
    n_params, dtype_bytes=2,   # FP16 = 2 bytes
    memory_bw_gb_s=3350,       # H100 HBM3
    batch_size=1,
    speculation_factor=1,      # >1 with speculative decoding
):
    """Estimate decode throughput in tokens/second."""
    model_bytes  = n_params * dtype_bytes
    memory_bw    = memory_bw_gb_s * 1e9   # bytes/sec

    # Time to load all weights once (one token for batch=1)
    time_per_step = model_bytes / memory_bw  # seconds

    # Tokens per second = (batch_size × speculation_factor) / time
    tokens_per_sec = (batch_size * speculation_factor) / time_per_step
    return tokens_per_sec

# LLaMA-3 8B on H100
tps_8b_fp16 = estimate_inference_throughput(8e9,   dtype_bytes=2)
tps_8b_int4 = estimate_inference_throughput(8e9,   dtype_bytes=0.5)  # 4-bit
tps_70b_fp16 = estimate_inference_throughput(70e9, dtype_bytes=2)
tps_70b_spec = estimate_inference_throughput(70e9, dtype_bytes=2, speculation_factor=3)

print(ff"LLaMA-3  8B FP16  (batch=1):       {tps_8b_fp16:.0f} tok/s")
print(ff"LLaMA-3  8B INT4  (batch=1):       {tps_8b_int4:.0f} tok/s")
print(ff"LLaMA-3 70B FP16  (batch=1):       {tps_70b_fp16:.0f} tok/s")
print(ff"LLaMA-3 70B FP16 + speculative×3: {tps_70b_spec:.0f} tok/s")

The One Equation

The entire LLM forward pass is: logits = Embedding(tokens) → N × (LayerNorm + Attention + LayerNorm + FFN) → Linear → Softmax. Every operation is a matrix multiply or an element-wise function. That's it.

The One Loss

Training minimises L = -(1/N) Σ log P(wᵢ|context). This is cross-entropy = negative log-likelihood = MLE. The gradient of this loss with respect to every weight is computed by backpropagation via the chain rule.

The One Bottleneck

At inference, the bottleneck is memory bandwidth — loading 140 GB of weights from GPU HBM. Quantization (fewer bits), batching (more users share the load), and speculative decoding (more tokens per weight load) are all mathematical solutions to this bandwidth limit.
