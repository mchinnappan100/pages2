History & Timeline

From simple RNNs to modern LLMs — the 30-year arc

1986

Backpropagation Through Time (BPTT)

Rumelhart, Hinton & Williams show how to train recurrent networks by unrolling them through time. Simple RNNs work in theory — but vanishing gradients make them fail in practice on long sequences.

RNN Foundations
1997

Long Short-Term Memory (LSTM)

Hochreiter & Schmidhuber introduce the LSTM cell — gated memory that can selectively remember or forget information across hundreds of time steps. Solves the vanishing gradient problem. Becomes the dominant sequence model for 20 years.

LSTM Born
2014

Sequence-to-Sequence with Attention

Bahdanau et al. add an attention mechanism on top of LSTMs for machine translation. Instead of compressing the entire input into a single context vector, the decoder attends to all encoder hidden states. Attention is born — but still uses LSTMs underneath.

LSTM + Attention
2017

"Attention Is All You Need" — Transformer

Vaswani et al. (Google) remove the RNN entirely. Pure self-attention, fully parallelisable, processes all positions simultaneously. Trains 10× faster on GPUs. Immediately achieves state-of-the-art on translation. The RNN era begins to end.

Transformer Born
2018–2019

BERT, GPT, GPT-2

Pre-trained transformers dominate every NLP benchmark. BERT (Google) uses the encoder for understanding; GPT (OpenAI) uses the decoder for generation. LSTMs are relegated to niche applications.

Transformer Dominates
2020–2025

GPT-3 → GPT-4 → Gemini → Claude → LLaMA

Scaling transformers to hundreds of billions of parameters produces emergent capabilities. Every frontier LLM is a transformer. LSTM-based models are no longer competitive at scale. New linear-attention architectures (Mamba, RWKV) revisit recurrent ideas with transformer-era techniques.

The Vanishing Gradient Problem

Why simple RNNs fail on long sequences — the problem LSTM was invented to solve

A simple RNN processes a sequence by updating a hidden state at each step: h_t = tanh(W_h · h_{t-1} + W_x · x_t). To train it, we back-propagate the gradient from the loss through time. But at each step, the gradient is multiplied by the Jacobian of the recurrence — and if those values are less than 1, the gradient decays exponentially as we go back in time.

For a sequence of length 100, the gradient from the last token must pass through 100 multiplications. With typical weight magnitudes, this gradient is effectively zero by the time it reaches early tokens. The model cannot learn long-range dependencies.

Vanishing Gradient in Simple RNN (sequence length 6) Forward: x₁ → [h₁] → [h₂] → [h₃] → [h₄] → [h₅] → [h₆] → Loss Backward (gradient flow): ∂L/∂h₁ = ∂L/∂h₆ × W^5 ← gradient multiplied by W 5 times If largest eigenvalue of W = 0.9: After 5 steps: 0.9⁵ = 0.59 After 20 steps: 0.9²⁰ = 0.12 After 100 steps: 0.9¹⁰⁰ ≈ 0.00003 ← essentially zero Problem: model cannot learn dependencies longer than ~5–10 steps LSTM solution: constant error carousel — gradient flows through the cell state unchanged (additive update, not multiplicative)

The Mathematics of Vanishing Gradients

During Backpropagation Through Time (BPTT), the gradient of the loss at the final step must flow back through every time step. At each step, the gradient is multiplied by the Jacobian of the hidden state update. For a vanilla RNN with h_t = tanh(W·h_{t-1} + U·x_t), the chain rule gives:

BPTT Gradient — exact mathematics RNN hidden state: h_t = tanh(W · h_{t-1} + U · x_t + b) Gradient of loss at step T w.r.t. h_k (k < T): ∂L/∂h_k = (∂L/∂h_T) × Π_{t=k+1}^{T} (∂h_t/∂h_{t-1}) where ∂h_t/∂h_{t-1} = diag(tanh'(z_t)) · W z_t = W · h_{t-1} + U · x_t + b So: ∂L/∂h_k = (∂L/∂h_T) × (diag(f'₁)W)(diag(f'₂)W)···(diag(f'_{T-k})W) = (∂L/∂h_T) × W^{T-k} with per-step scaling by tanh derivatives Key bound (Hochreiter 1991): ‖∂h_t/∂h_{t-1}‖ ≤ ‖W‖ · max|tanh'(z)| ≤ ‖W‖ · 1.0 (tanh derivative ≤ 1 everywhere) ≤ ‖W‖ · 0.25 (if z large, tanh' → 0 fast) If ‖W‖ < 1 → ‖∂L/∂h_k‖ ≤ (‖W‖·0.25)^{T-k} → 0 exponentially If ‖W‖ > 1 → ‖∂L/∂h_k‖ → ∞ exponentially (exploding) For T-k = 100 and ‖W‖·0.25 = 0.9: 0.9^100 ≈ 2.7 × 10^{-5} ← essentially zero

Why tanh Makes It Worse

The derivative of tanh(x) = 1 − tanh²(x). At x=0 it equals 1 (best case), but for |x| > 2 it drops below 0.07. Hidden states in trained RNNs are rarely near zero — so in practice the per-step Jacobian norms are well below 1, making the gradient decay even faster than the pure eigenvalue bound suggests.

Why ReLU Doesn't Fully Fix It

Replacing tanh with ReLU gives a derivative of exactly 1 in the positive region (no shrinkage) — but the gradient can still vanish if neurons die (output zero, zero gradient). And if all activations are positive, it can explode. ReLU helps somewhat but does not solve the fundamental problem for very long sequences.

How LSTM Solves It — The Constant Error Carousel

Hochreiter & Schmidhuber's key insight: make the gradient path through time additive instead of multiplicative. The cell state update is:

c_t = f_t ⊙ c_{t-1} + i_t ⊙ ĉ_t

The gradient through the cell state: ∂c_t/∂c_{t-1} = f_t — element-wise scaling by the forget gate, not matrix multiplication by W. If the forget gate learns to be near 1.0 for a dimension, that gradient flows back unchanged through arbitrarily many steps. This is the "constant error carousel" — a highway for gradients.

Gradient highway in LSTM vs vanilla RNN Vanilla RNN — multiplicative bottleneck: ∂L/∂h_{t-100} = ∂L/∂h_T × W × W × W × ... × W (100 matrix multiplies) → exponential decay or explosion LSTM — additive highway through cell state: ∂L/∂c_{t-100} = ∂L/∂c_T × f_T × f_{T-1} × ... × f_{t-99} (element-wise scaling by forget gates, NOT matrix multiply) If f = 1.0: gradient passes through unchanged If f = 0.0: gradient is zeroed (model chose to forget) The forget gate IS the gradient gate. Model learns: when f_t ≈ 1 → "keep remembering → also keep gradient flowing" when f_t ≈ 0 → "forget this → also block gradient (intentional)"
Python · Demonstrating vanishing gradients — RNN vs LSTM
import torch
import torch.nn as nn
import numpy as np

def measure_gradient_norms(model_type, seq_len=100, hidden=64):
    """Measure how gradient magnitude changes with distance from the loss."""
    torch.manual_seed(42)

    if model_type == "rnn":
        model = nn.RNN(input_size=16, hidden_size=hidden, batch_first=True)
    else:
        model = nn.LSTM(input_size=16, hidden_size=hidden, batch_first=True)

    x = torch.randn(1, seq_len, 16, requires_grad=True)
    output, _ = model(x)

    # Gradient from the LAST token's output w.r.t. every input position
    loss = output[0, -1, :].sum()  # loss at final step
    loss.backward()

    # Gradient norm at each input position (how much each step influenced loss)
    grad_norms = x.grad[0].norm(dim=1).detach().numpy()  # (seq_len,)
    return grad_norms

# Compare gradient flow at key positions
seq_len = 50
rnn_grads  = measure_gradient_norms("rnn",  seq_len)
lstm_grads = measure_gradient_norms("lstm", seq_len)

positions = [0, 10, 25, 40, 49]
print("Position  RNN Grad Norm     LSTM Grad Norm")
print("-" * 50)
for p in positions:
    dist = seq_len - 1 - p   # distance from last step
    print(ff"pos={p:3d} (dist={dist:2d})   {rnn_grads[p]:.2e}         {lstm_grads[p]:.2e}")

# Expected output (approximately):
# pos= 0 (dist=49)   ~1e-15   (near zero)    ~1e-3   (much larger)
# pos=49 (dist= 0)   ~1e-1    (large)        ~1e-1   (large — both same near loss)
# → RNN gradient collapses to zero after ~10-20 steps
# → LSTM gradient remains orders of magnitude larger at distant positions
📉

Vanishing Gradients

Gradient magnitude shrinks to zero during backprop through many time steps. Caused by repeated multiplication by W in BPTT — eigenvalues <1 compound to near-zero. Early tokens receive zero gradient and cannot contribute to learning.

  • Mathematical cause: ‖∂h_t/∂h_{t-1}‖ < 1 per step
  • Compounded over T steps: exponential decay
  • tanh derivative ≤ 0.25 for large activations — worsens the problem
  • LSTM fix: additive cell state, not multiplicative recurrence
  • Transformer fix: residual connections create direct gradient paths
📈

Exploding Gradients

The opposite: when eigenvalues of W exceed 1, gradients grow exponentially. Weights become NaN. Common in deep RNNs and also in transformers before gradient clipping became standard.

  • Mathematical cause: ‖∂h_t/∂h_{t-1}‖ > 1 per step
  • Fix: gradient clipping — scale down if ‖∇‖ > threshold
  • LSTM partial fix: forget gate bounded in (0,1) limits growth
  • Transformer fix: LayerNorm + Adam's adaptive lr + warmup
  • PyTorch: torch.nn.utils.clip_grad_norm_(params, 1.0)
🛤️

Residual Connections (Transformer Fix)

Transformers solve the vanishing gradient problem differently: every block has a residual (skip) connection x = x + f(x). The gradient always has a direct path: ∂L/∂x_input = ∂L/∂x_output · (1 + ∂f/∂x_input). The "+1" guarantees unattenuated gradient flow to every layer.

  • Introduced by ResNet (He et al. 2015) for vision
  • Transformers use residuals around attention AND FFN
  • Gradient highway: ∂L/∂x includes 1 — no shrinkage
  • Enables training 100+ layer transformers stably
  • Combined with LayerNorm: near-ideal gradient conditioning

LSTM Architecture

Three gates, a cell state, and the "constant error carousel"

An LSTM cell maintains two hidden states: the hidden state h_t (short-term memory, sent to the next layer) and the cell state c_t (long-term memory, a "highway" that flows through time with only additive updates). Three learnable gates control information flow.

LSTM Cell — Gates and State Update Inputs: x_t (current token), h_{t-1} (previous hidden), c_{t-1} (cell state) Forget Gate: f_t = σ(W_f · [h_{t-1}, x_t] + b_f) "How much of c_{t-1} should I forget?" f_t ∈ (0,1) Input Gate: i_t = σ(W_i · [h_{t-1}, x_t] + b_i) ĉ_t = tanh(W_c · [h_{t-1}, x_t] + b_c) "What new information to write into the cell state?" Cell State Update: c_t = f_t ⊙ c_{t-1} + i_t ⊙ ĉ_t ← ADDITIVE update: gradient flows unchanged (no vanishing!) Output Gate: o_t = σ(W_o · [h_{t-1}, x_t] + b_o) h_t = o_t ⊙ tanh(c_t) Gradient highway through c_t: ∂c_t/∂c_{t-1} = f_t (not multiplied by W) → gradient flows back via the cell state with only element-wise scaling → no exponential decay for long sequences!
🚪

Forget Gate f_t

Decides what fraction of each dimension of the cell state to keep. A value of 0 = completely forget; 1 = keep perfectly. The model learns when to reset memory (e.g. at sentence boundaries).

  • Output: vector in (0,1)^d — one value per cell dimension
  • Learns to forget irrelevant context (e.g. prior sentence topic)
  • Initialise bias to 1: start by forgetting little (common practice)
✏️

Input Gate i_t & Candidate ĉ_t

Two-part write: i_t decides how much to write at each dimension, ĉ_t is the candidate content to write. Together they control: what new information enters the long-term memory.

  • i_t ∈ (0,1): write magnitude gate
  • ĉ_t ∈ (-1,1): content (tanh bounded)
  • Combination: selective write — write some content, some dimensions
📤

Output Gate o_t

Controls which parts of the cell state to expose as the hidden state h_t (the output sent to the next layer and next time step). The cell can hold information privately, revealing it only when relevant.

  • o_t ∈ (0,1): how much of cell state to expose
  • tanh(c_t): squashes cell state to (-1,1) before exposing
  • Enables "private memory" — cell holds info not yet needed
04

Transformer Architecture

No recurrence, no memory state — just attention over all positions at once

The Transformer discards recurrence entirely. Instead of processing tokens one at a time and maintaining a hidden state, it processes all tokens simultaneously. Each token attends to every other token via scaled dot-product attention — directly computing pairwise relationships without any sequential bottleneck.

This design has three critical consequences: it is fully parallelisable (no sequential dependency during training), it has direct access to all positions (no information bottleneck from compressing everything into a fixed hidden state), and it scales naturally on GPUs/TPUs where massively parallel computation is the strength.

Transformer Block — Decoder Only (GPT-style) Token Embeddings + Positional Encoding [B, T, d_model] │ ┌──────────────────────────────────────────────────────┐ │ RMSNorm │ │ Causal Self-Attention │ │ Q = xW_Q, K = xW_K, V = xW_V │ │ scores = Q@K^T / √d_k (all pairs at once)│ │ scores[i,j] = -∞ if j > i (causal mask) │ │ attn = softmax(scores) │ │ out = attn @ V │ │ Residual: x = x + out │ │ RMSNorm │ │ SwiGLU FFN: SiLU(xW₁) ⊙ xW₂ → W₃ │ │ Residual: x = x + ffn(x) │ └──────────────────────────────────────────────────────┘ │ × N layers RMSNorm → Linear → logits [B, T, vocab_size] Key: ALL positions processed IN PARALLEL during training (unlike LSTM which must process step-by-step)

Parallelism & Training Speed

Why transformers train 10–100× faster than LSTMs on modern hardware

The single biggest advantage of transformers over LSTMs is training speed on modern GPUs. The LSTM's sequential dependency is a fundamental obstacle: to compute h_t, you must have already computed h_{t-1}. For a sequence of length 2048, this means 2048 sequential operations — you simply cannot parallelise across the time dimension.

A transformer processes all 2048 tokens in parallel. The attention computation is a single large matrix multiply (O(T² × d)), which is exactly what Tensor Cores are built for. GPUs achieve near-peak utilisation; LSTMs are typically at 5–10% utilisation on the same hardware.

The numbers

  • LSTM training throughput: ~500–2000 tokens/sec on a single A100 (sequential bottleneck)
  • Transformer training throughput: ~50,000–200,000 tokens/sec on a single A100 (fully parallel)
  • This 100× gap means that training a GPT-3 scale model with LSTMs would take ~100 years instead of ~1 month
The BPTT Bottleneck

Even beyond forward-pass parallelism, LSTM training suffers from a second bottleneck during backpropagation: Backpropagation Through Time (BPTT) must unroll the full sequence and propagate gradients step by step. With a sequence length of 2048, this means 2048 sequential backward passes. Transformers backpropagate through a fixed-depth computation graph regardless of sequence length — massively faster.

Memory & Long-Range Context

How each architecture retrieves information from far back in the sequence

LSTM: The Information Bottleneck

At every time step, all information from the past must fit into a fixed-size hidden state h_t (typically 256–1024 dimensions). The cell state c_t provides some relief, but it too has a fixed capacity. As the sequence grows, old information gets overwritten or diluted. The model must implicitly decide what to compress into this bottleneck — and it often makes poor choices for rare but important long-range dependencies.

Transformer: Direct Access via Attention

A transformer token at position 500 can attend directly to any token at positions 1–499 with equal ease. There is no bottleneck. The attention score is computed as a dot product between query and key vectors — O(1) path length between any two positions. This is why transformers are dramatically better at tasks that require connecting information from widely separated parts of a document.

The O(1) path length advantage

Graph-theoretically, the "information path length" between two positions measures how many operations are needed to connect them:

  • LSTM: O(T) — information must travel through T intermediate hidden states
  • Transformer: O(1) — attention connects any two positions in one operation
Information path length comparison LSTM: connecting token 1 to token 500 t₁ → h₁ → h₂ → h₃ → h₄ → ... → h₄₉₉ → h₅₀₀ 499 intermediate steps. Information about t₁ has been overwritten 499 times. Probability of surviving = very low. Transformer: connecting token 1 to token 500 t₁ ←──── direct attention ────→ t₅₀₀ 1 operation. score(q₅₀₀, k₁) is directly computed. No intermediate bottlenecks. No forgetting. Practical implication: Task: "In the following 2000-word document, what colour was the car mentioned in paragraph 1?" LSTM: paragraph 1 info must survive 1999 hidden state updates → likely forgotten Transformer: paragraph 1 is always in the KV cache → directly attended to

PyTorch Implementations

Both architectures from scratch — side by side

Python · LSTM from scratch
import torch
import torch.nn as nn

class LSTMCell(nn.Module):
    """Single LSTM cell — one time step."""
    def __init__(self, input_size, hidden_size):
        super().__init__()
        # All 4 gates in one fused linear layer (efficient)
        self.W = nn.Linear(input_size + hidden_size, 4 * hidden_size)

    def forward(self, x, h_prev, c_prev):
        # x: (B, input_size)  h_prev,c_prev: (B, hidden_size)
        gates = self.W(torch.cat([x, h_prev], dim=1))  # (B, 4H)
        i, f, g, o = gates.chunk(4, dim=1)              # split 4 gates

        i = torch.sigmoid(i)   # input gate:  what to write
        f = torch.sigmoid(f)   # forget gate: what to forget
        g = torch.tanh(g)      # candidate:   new content
        o = torch.sigmoid(o)   # output gate: what to expose

        # Cell state: additive update (constant error carousel)
        c_next = f * c_prev + i * g     # ⊙ = element-wise
        h_next = o * torch.tanh(c_next)
        return h_next, c_next


class LSTMModel(nn.Module):
    """Multi-layer LSTM for language modelling."""
    def __init__(self, vocab_size, embed_dim, hidden_size, n_layers):
        super().__init__()
        self.embedding = nn.Embedding(vocab_size, embed_dim)
        self.cells = nn.ModuleList([
            LSTMCell(embed_dim if i == 0 else hidden_size, hidden_size)
            for i in range(n_layers)
        ])
        self.fc = nn.Linear(hidden_size, vocab_size)

    def forward(self, tokens):
        B, T = tokens.shape
        x = self.embedding(tokens)   # (B, T, embed_dim)
        H = self.cells[0].W.out_features // 4

        # Initialise hidden + cell states to zero
        states = [(torch.zeros(B, H), torch.zeros(B, H))
                  for _ in self.cells]

        outputs = []
        for t in range(T):          # ← sequential loop: the bottleneck
            inp = x[:, t, :]
            for l, cell in enumerate(self.cells):
                h, c = states[l]
                h, c = cell(inp, h, c)
                states[l] = (h, c)
                inp = h               # feed to next layer
            outputs.append(h)

        out = torch.stack(outputs, dim=1)  # (B, T, H)
        return self.fc(out)                   # (B, T, vocab)

# Shape check
lstm = LSTMModel(vocab_size=1000, embed_dim=128, hidden_size=256, n_layers=2)
tokens = torch.randint(0, 1000, (4, 64))   # batch=4, seq_len=64
logits = lstm(tokens)
print(ff"LSTM logits: {logits.shape}")    # (4, 64, 1000)
Python · Transformer (GPT-style decoder) from scratch
import torch
import torch.nn as nn
import torch.nn.functional as F

class CausalSelfAttention(nn.Module):
    def __init__(self, d_model, n_heads):
        super().__init__()
        self.n_heads = n_heads
        self.d_head  = d_model // n_heads
        self.qkv     = nn.Linear(d_model, 3 * d_model, bias=False)
        self.proj    = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x):
        B, T, C = x.shape
        q, k, v = self.qkv(x).split(C, dim=2)
        q = q.view(B,T,self.n_heads,self.d_head).transpose(1,2)
        k = k.view(B,T,self.n_heads,self.d_head).transpose(1,2)
        v = v.view(B,T,self.n_heads,self.d_head).transpose(1,2)
        # Flash Attention (PyTorch 2.0+) — fused, O(N) memory
        out = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        return self.proj(out.transpose(1,2).reshape(B, T, C))


class TransformerBlock(nn.Module):
    def __init__(self, d_model, n_heads, d_ff):
        super().__init__()
        self.norm1 = nn.RMSNorm(d_model)
        self.attn  = CausalSelfAttention(d_model, n_heads)
        self.norm2 = nn.RMSNorm(d_model)
        # SwiGLU FFN: SiLU(xW1) ⊙ xW2 → xW3
        self.w1 = nn.Linear(d_model, d_ff, bias=False)
        self.w2 = nn.Linear(d_model, d_ff, bias=False)
        self.w3 = nn.Linear(d_ff, d_model, bias=False)

    def forward(self, x):
        x = x + self.attn(self.norm1(x))
        x = x + self.w3(F.silu(self.w1(self.norm2(x))) * self.w2(self.norm2(x)))
        return x


class GPTModel(nn.Module):
    def __init__(self, vocab_size, d_model, n_heads, d_ff, n_layers, max_seq_len):
        super().__init__()
        self.embed   = nn.Embedding(vocab_size, d_model)
        self.blocks  = nn.Sequential(*[
            TransformerBlock(d_model, n_heads, d_ff) for _ in range(n_layers)
        ])
        self.norm    = nn.RMSNorm(d_model)
        self.head    = nn.Linear(d_model, vocab_size, bias=False)

    def forward(self, tokens):
        x = self.embed(tokens)     # (B, T, d_model)
        x = self.blocks(x)         # ← ALL positions in PARALLEL (no loop!)
        return self.head(self.norm(x))  # (B, T, vocab)

               d_ff=1024, n_layers=4, max_seq_len=512)
tokens = torch.randint(0, 1000, (4, 64))
logits = gpt(tokens)
print(ff"Transformer logits: {logits.shape}")   # (4, 64, 1000)
# Note: no for-loop over time steps — fully parallel!

When to Use Each

Practical decision guide — including the new hybrid alternatives

Use LSTM When

Streaming, low-latency real-time inference on edge devices. The O(1) hidden state update is perfect for microcontrollers or IoT. Also good for very short sequences (<50 steps) where parallel advantage is minimal.

Use Transformer When

Virtually all NLP tasks at any scale. If you have GPU/TPU compute, transformers are faster to train and achieve better quality. Text generation, classification, translation, summarisation — transformers dominate.

LSTM Still Useful For

Time series forecasting on edge hardware (smartwatches, sensors), speech processing on embedded systems, and legacy production systems. Also as a component inside larger architectures for recurrent state tracking.

Consider Mamba / RWKV Instead

New architectures (Mamba, RWKV, RetNet) combine LSTM-like O(N) inference cost with transformer-like training parallelism. They use structured state space models or linear attention to achieve the best of both worlds. Watch this space.

Linear attention, Mamba, and the return of recurrent ideas

The transformer's O(T²) memory bottleneck limits it to ~128K–1M token contexts. Researchers are revisiting recurrent ideas with modern techniques — aiming for O(T) training AND O(1) inference, which neither pure LSTM nor pure Transformer achieves simultaneously.

The Core Insight

LSTM and Transformer represent two points in a design space: LSTM optimises for efficient inference (constant memory, O(1) updates) at the cost of training speed and representational power. Transformer optimises for training speed and long-range dependencies at the cost of quadratic memory and linear inference cost. The race in 2024–2025 is to find architectures that achieve all four properties simultaneously. Mamba, RWKV, and hybrid models are the leading candidates.
