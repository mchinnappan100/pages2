Chapter 1 — Virtual Memory Architecture

Paging and Page Tables

Every virtual address a process uses is translated to a physical address through a multi-level page table walk. On x86-64 with 4-level paging, this is a 4-level tree lookup: PGD → PUD → PMD → PTE → physical frame.

/* The 4-level page table walk — what the MMU does on every memory access
 * that misses the TLB. Understanding this explains why hugepages matter. */

/*
 * Virtual address breakdown (48-bit, 4-level paging):
 *
 * Bit 47..39 = PGD index (9 bits, 512 entries)
 * Bit 38..30 = PUD index (9 bits, 512 entries)
 * Bit 29..21 = PMD index (9 bits, 512 entries)
 * Bit 20..12 = PTE index (9 bits, 512 entries)
 * Bit 11..0  = Page offset (12 bits, 4096 byte page)
 *
 * Translation:
 * 1. CR3 register → PGD base physical address
 * 2. PGD[vaddr[47:39]] → PUD physical address
 * 3. PUD[vaddr[38:30]] → PMD physical address
 * 4. PMD[vaddr[29:21]] → PTE physical address
 * 5. PTE[vaddr[20:12]] → Page frame physical address
 * 6. + vaddr[11:0]     → final physical byte
 *
 * For a 4KB page: 4 memory accesses for EVERY virtual→physical translation!
 * TLB caches these translations to avoid repeating the walk.
 * TLB miss on modern CPU: ~50-100 cycles penalty.
 */

/* Walk the page table for the current process from user space.
 * Uses /proc/self/pagemap to find physical addresses. */
#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>
#include <unistd.h>
#include <fcntl.h>

uint64_t virt_to_phys(void *vaddr) {
    int      pagemap_fd = open("/proc/self/pagemap", O_RDONLY);
    uint64_t vpage      = (uint64_t)vaddr / getpagesize();
    uint64_t entry;

    pread(pagemap_fd, &entry, sizeof(entry), vpage * sizeof(uint64_t));
    close(pagemap_fd);

    if (!(entry & (1ULL << 63))) {
        printf("Page not present (swapped out or not faulted)\n");
        return 0;
    }
    uint64_t pfn   = entry & ((1ULL << 55) - 1);   /* page frame number */
    uint64_t paddr = pfn * getpagesize() + (uint64_t)vaddr % getpagesize();
    return paddr;
}

int main() {
    volatile int x = 42;    /* force the page to be faulted in */
    uint64_t vaddr = (uint64_t)&x;
    uint64_t paddr = virt_to_phys((void*)vaddr);
    printf("Virtual:  0x%016llx\n", vaddr);
    printf("Physical: 0x%016llx\n", paddr);
    printf("Page offset: %lld\n", vaddr % getpagesize());
    return 0;
}

TLB and Hugepages — The LLM Performance Secret

The TLB (Translation Lookaside Buffer) is a hardware cache of recent virtual-to-physical translations. A modern CPU has only 64–2048 TLB entries. When loading a 140 GB model with 4KB pages, you have 140 GB / 4 KB = 35 million pages — vastly more than the TLB can hold.

Hugepage Impact on LLM Inference

With 2MB hugepages: 140 GB / 2 MB = 70,000 pages — potentially fitting in the TLB. On a 96-core H100 server, switching to hugepages for model weight storage reduces LLM inference latency by 8–15% with zero code changes. This is free performance.

Chapter 2 — Memory Allocators

The Buddy Allocator

The Linux buddy allocator manages physical pages in power-of-two sized blocks (1, 2, 4, 8, ... 1024 pages). When you allocate memory, it finds the smallest power-of-two block that fits and splits larger blocks if needed.

/* Demonstrating buddy allocator behaviour by allocating order-9 (2MB) pages.
 * In practice: use mmap with MAP_HUGETLB for huge page allocation. */
#include <sys/mman.h>
#include <stdio.h>
#include <string.h>

#define HUGE_PAGE_SIZE (2UL * 1024 * 1024)  /* 2 MB */

void* alloc_huge_pages(size_t size) {
    /* Round up to 2MB boundary */
    size = (size + HUGE_PAGE_SIZE - 1) & ~(HUGE_PAGE_SIZE - 1);

    void* ptr = mmap(NULL, size,
                     PROT_READ | PROT_WRITE,
                     MAP_PRIVATE | MAP_ANONYMOUS | MAP_HUGETLB,
                     -1, 0);
    if (ptr == MAP_FAILED) {
        perror("mmap MAP_HUGETLB failed — check /proc/sys/vm/nr_hugepages");
        /* Fallback to regular pages */
        ptr = mmap(NULL, size, PROT_READ | PROT_WRITE,
                   MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
    }
    printf("Allocated %.1f GB at %p "
           "(page size: %s)\n",
           (double)size / (1<<30), ptr,
           ptr != MAP_FAILED ? "2MB hugepages" : "4KB regular");
    return ptr;
}

int main() {
    /* Allocate 4 GB — typical for loading a 7B model layer */
    void* weights = alloc_huge_pages(4UL * 1024 * 1024 * 1024);
    if (weights == MAP_FAILED) return 1;
    memset(weights, 0, 4096);  /* fault in first page */
    munmap(weights, 4UL * 1024 * 1024 * 1024);
    return 0;
}

The Slab Allocator

The slab allocator handles frequent kernel object allocations (task_struct, file, inode) efficiently. For ML engineers, the most important slab caches are those backing network socket buffers (for NCCL) and file objects (for data loading). You can inspect them with:

# View slab allocator statistics
cat /proc/slabinfo | head -20
# key fields: name, active_objs, num_objs, objsize, objperslab

# Watch for slab cache growth during training (potential memory leak indicator)

Chapter 3 — NUMA Architecture

GPU Server NUMA Topology

On a DGX H100 (8-GPU server), there are 2 CPU sockets = 2 NUMA nodes. GPUs 0–3 connect via PCIe to NUMA node 0; GPUs 4–7 to NUMA node 1. Memory access latency from NUMA node 0 to node 1's RAM is 2–3× higher than local access.

# Explore your NUMA topology
numactl --hardware
# Output shows: nodes, CPUs per node, distances between nodes

# Show which GPUs are on which NUMA node
for gpu in $(nvidia-smi --query-gpu=index --format=csv,noheader); do
    bus=$(nvidia-smi --query-gpu=pci.bus_id --format=csv,noheader -i $gpu)
    numa=$(cat /sys/bus/pci/devices/${bus,,}/numa_node 2>/dev/null || echo unknown)
    echo "GPU $gpu → NUMA node $numa"
done

NUMA-Aware Allocation in C

#include <numa.h>
#include <stdio.h>
#include <stdlib.h>

/* Allocate GPU activation buffers on the correct NUMA node.
 * Rule: allocate host buffers on the NUMA node local to the GPU
 *       you're going to DMA transfer to/from. */
void* alloc_local_to_gpu(int gpu_id, size_t size) {
    /* Find the NUMA node for this GPU via sysfs */
    char path[256];
    FILE* f;
    int   numa_node = 0;

    /* Get PCI bus ID from nvidia-smi and look up numa_node */
    snprintf(path, sizeof(path),
             "/sys/bus/pci/devices/0000:00:%02x.0/numa_node", gpu_id * 2);
    f = fopen(path, "r");
    if (f) { fscanf(f, "%d", &numa_node); fclose(f); }

    printf("GPU %d is on NUMA node %d — allocating %.1f GB there\n",
           gpu_id, numa_node, (double)size / (1<<30));

    /* Allocate on the specific NUMA node */
    void* ptr = numa_alloc_onnode(size, numa_node);
    if (!ptr) {
        fprintf(stderr, "NUMA allocation failed\n");
        return NULL;
    }
    return ptr;
}

/* Python equivalent (using PyTorch):
 *   tensor = torch.empty(size, device='cpu', pin_memory=True)
 *   # PyTorch respects numactl affinity when pin_memory=True */

Transparent Hugepages Configuration

#!/bin/bash
# Configure Transparent Hugepages (THP) for LLM workloads

# Option 1: always — OS automatically promotes 4KB pages to 2MB
# Best for training servers with dedicated workloads
echo always > /sys/kernel/mm/transparent_hugepage/enabled
echo always > /sys/kernel/mm/transparent_hugepage/defrag

# Option 2: madvise — only promote pages that explicitly request it
# Best for shared servers (more controlled)
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled

# In Python: request hugepages for tensor storage
python3 - <<'EOF'
import ctypes, os, mmap

# Use MADV_HUGEPAGE to request THP for a specific allocation
libc = ctypes.CDLL('libc.so.6')
MADV_HUGEPAGE = 14

import torch
# Create a large tensor and advise the kernel to use hugepages
t = torch.zeros(1024 * 1024 * 256, dtype=torch.float32)  # 1 GB
data_ptr = t.data_ptr()
size = t.numel() * t.element_size()
libc.madvise(ctypes.c_void_p(data_ptr), ctypes.c_size_t(size), ctypes.c_int(MADV_HUGEPAGE))
print(f"Requested THP for {size/1e9:.1f} GB tensor at 0x{data_ptr:x}")
EOF

Chapter 5 — OOM Killer & Memory Limits

/* The Linux OOM killer selects which process to kill when memory is exhausted.
 * Its selection is based on 'oom_score' — higher score = killed first.
 *
 * For AI training: protect your training process from OOM kill. */

#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>

void protect_from_oom(void) {
    int  fd;
    char val[] = "-1000\n";   /* -1000 = never kill this process */

    fd = open("/proc/self/oom_score_adj", O_WRONLY);
    if (fd < 0) {
        perror("open oom_score_adj (needs root or CAP_SYS_RESOURCE)");
        return;
    }
    write(fd, val, sizeof(val) - 1);
    close(fd);
    printf("Process protected from OOM kill "
           "(oom_score_adj = -1000)\n");
}

/* Python equivalent for protecting a training process:
 *   import subprocess
 *   pid = os.getpid()
 *   subprocess.run(['sudo', 'bash', '-c',
 *                   f'echo -1000 > /proc/{pid}/oom_score_adj']) */

Chapter 6 — mmap for Tensor Storage

"""
mmap_tensors.py — Memory-map large model weight files.
This is how llama.cpp and safetensors load models faster than torch.load().
With mmap: OS loads pages on demand, NUMA-aware, supports lazy loading.
"""
import mmap, struct, numpy as np, torch
from pathlib import Path

class MmapTensorStore:
    """Store tensors in a flat binary file and mmap them on load.
    OS brings pages into RAM only when accessed — zero startup latency."""

    def __init__(self, path: str):
        self.path = Path(path)
        self.tensors: dict = {}

    def save(self, tensors: dict):
        # Format: [n_tensors][name_len][name][dtype_code][ndim][shape][data]...
        with open(self.path, 'wb') as f:
            f.write(struct.pack('I', len(tensors)))
            for name, tensor in tensors.items():
                arr  = tensor.numpy()
                bname = name.encode()
                f.write(struct.pack('I', len(bname))); f.write(bname)
                f.write(struct.pack('B', arr.dtype.itemsize))
                f.write(struct.pack('I', arr.ndim))
                for d in arr.shape: f.write(struct.pack('Q', d))
                f.write(arr.tobytes())
        print(f"Saved {len(tensors)} tensors to {self.path} "
              f"({self.path.stat().st_size / 1e9:.2f} GB)")

    def load_mmap(self) -> dict:
        """Load tensors using mmap — O(1) load time regardless of file size."""
        with open(self.path, 'rb') as f:
            # mmap the entire file — OS will page-in on access
            mm = mmap.mmap(f.fileno(), 0, access=mmap.ACCESS_READ)

        pos = 0
        n   = struct.unpack_from('I', mm, pos)[0]; pos += 4
        tensors = {}

        for _ in range(n):
            nlen  = struct.unpack_from('I', mm, pos)[0]; pos += 4
            name  = mm[pos:pos+nlen].decode(); pos += nlen
            isz   = struct.unpack_from('B', mm, pos)[0]; pos += 1
            ndim  = struct.unpack_from('I', mm, pos)[0]; pos += 4
            shape = struct.unpack_from(f'{ndim}Q', mm, pos); pos += 8*ndim
            dtype = {2: np.float16, 4: np.float32}[isz]
            nbytes = np.prod(shape) * isz
            # Key: np.frombuffer on mmap slice — zero copy, pages loaded on demand
            arr   = np.frombuffer(mm[pos:pos+nbytes], dtype=dtype).reshape(shape)
            tensors[name] = torch.from_numpy(arr.copy())
            pos += nbytes

        print(f"Loaded {n} tensors via mmap in O(1) time")
        return tensors


if __name__ == '__main__':
    store = MmapTensorStore('/tmp/weights.bin')

    # Save some fake LLM weights
    weights = {
        'transformer.wte.weight':     torch.randn(32000, 4096, dtype=torch.float16),
        'transformer.h.0.attn.c_attn': torch.randn(4096, 12288, dtype=torch.float16),
    }
    store.save(weights)

    import time
    t0 = time.perf_counter()
    loaded = store.load_mmap()
    t1 = time.perf_counter()
    print(f"mmap load time: {(t1-t0)*1000:.1f} ms "
          f"(vs torch.load which reads entire file)")
