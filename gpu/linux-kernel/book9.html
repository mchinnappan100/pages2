Linux Security,
Namespaces & Containers for AI

Linux namespaces, cgroups v2, capabilities, seccomp syscall filtering, the NVIDIA Container Toolkit, and Kubernetes GPU scheduling — the complete guide to running AI workloads securely on shared multi-tenant infrastructure.

Why Security Matters at Scale

A single LLM training job can access terabytes of proprietary training data. In a multi-tenant GPU cluster, one compromised container can exfiltrate model weights, corrupt shared checkpoints, or hijack GPU resources from other tenants. Linux's layered security model prevents all of these.

Chapter 1 — Linux Namespaces

The Seven Namespace Types

Linux namespaces are the foundation of container isolation. Each namespace type wraps a specific global resource, making processes within a namespace see only their namespace's view of that resource.

NamespaceFlagIsolatesAI Relevance
PIDCLONE_NEWPIDProcess IDsTraining workers can't see host PIDs
NetworkCLONE_NEWNETNetwork interfaces, routesContainer gets its own network stack
MountCLONE_NEWNSFilesystem mount pointsDataset and model weight storage isolation
UserCLONE_NEWUSERUser/group IDsRun as root inside container, not on host
UTSCLONE_NEWUTSHostname, domain nameEach training job sees its own hostname
IPCCLONE_NEWIPCSysV IPC, POSIX message queuesShared memory for NCCL isolated per container
cgroupCLONE_NEWCGROUPcgroup root directoryContainer sees only its cgroup subtree

Building a Minimal Container from Scratch

#define _GNU_SOURCE
#include <sched.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>
#include <sys/mount.h>

#define STACK_SIZE (1024 * 1024)

/* The process that runs inside our minimal container */
int container_init(void *arg) {
    /* Inside new namespaces — set up the container environment */

    /* Mount /proc so ps/top work inside the container */
    mount("proc", "/proc", "proc", 0, NULL);

    printf("[Container] PID=%d (isolated from host)\n", getpid());
    printf("[Container] Hostname: "); fflush(stdout);
    system("hostname");

    /* In a real container: exec the actual workload here */
    system("/bin/sh");

    umount("/proc");
    return 0;
}

int main() {
    char *stack = malloc(STACK_SIZE);

    /* Create a new process with isolated namespaces:
     * CLONE_NEWPID:  new PID namespace  (container PIDs start at 1)
     * CLONE_NEWNS:   new mount namespace (independent filesystem view)
     * CLONE_NEWNET:  new network namespace (isolated networking)
     * CLONE_NEWUTS:  new UTS namespace (independent hostname)
     * CLONE_NEWIPC:  new IPC namespace (isolated shared memory)
     * SIGCHLD:       signal parent on exit */
    pid_t pid = clone(container_init,
                      stack + STACK_SIZE,
                      CLONE_NEWPID | CLONE_NEWNS |
                      CLONE_NEWNET | CLONE_NEWUTS | CLONE_NEWIPC | SIGCHLD,
                      NULL);

    if (pid < 0) { perror("clone"); free(stack); return 1; }
    printf("[Host] Container started with PID %d\n", pid);

    /* Set container's hostname — visible only inside its UTS namespace */
    char hostname_cmd[64];
    snprintf(hostname_cmd, sizeof(hostname_cmd),
             "nsenter --target %d --uts hostname training-job-001", pid);
    system(hostname_cmd);

    waitpid(pid, NULL, 0);
    printf("[Host] Container exited\n");
    free(stack);
    return 0;
}

Chapter 2 — cgroups v2

The Unified cgroup v2 Hierarchy

cgroups v2 uses a single unified hierarchy (unlike v1's multiple hierarchies). Every process belongs to exactly one cgroup in the hierarchy. Resources are controlled by controller files in /sys/fs/cgroup/.

#!/bin/bash
# Create a complete cgroup v2 hierarchy for a multi-GPU training job

CGROUP_ROOT="/sys/fs/cgroup"
JOB_ID="training_job_42"
JOB_PATH="$CGROUP_ROOT/$JOB_ID"

echo "=== Setting up cgroup v2 for training job $JOB_ID ==="

# Create the job's cgroup
mkdir -p "$JOB_PATH"

# Enable controllers for this cgroup
echo "+cpu +memory +io" > "$JOB_PATH/cgroup.subtree_control"

# --- MEMORY LIMITS ---
# Hard limit: 480 GB (leave 32 GB for OS)
echo $((480 * 1024 * 1024 * 1024)) > "$JOB_PATH/memory.max"
# Swap limit: 0 (no swap for training)
echo 0 > "$JOB_PATH/memory.swap.max"
# OOM kill: yes (don't let it silently starve)
echo 1 > "$JOB_PATH/memory.oom.group"

# --- CPU LIMITS ---
# CPU weight: 800 (out of 1000 default — high priority)
echo 800 > "$JOB_PATH/cpu.weight"
# CPU burst: allow short bursts above quota
echo 100000 > "$JOB_PATH/cpu.max.burst"

# --- I/O LIMITS for data loading ---
# Find NVMe device major:minor
NVME_MAJOR=$(ls -la /dev/nvme0n1 | awk '{print $5}' | tr -d ',')
NVME_MINOR=$(ls -la /dev/nvme0n1 | awk '{print $6}')
# Limit reads: 4 GB/s (leave headroom for other I/O)
echo "$NVME_MAJOR:$NVME_MINOR rbps=$((4*1024*1024*1024))" > "$JOB_PATH/io.max"

# --- MOVE TRAINING PROCESSES INTO CGROUP ---
# torchrun will have spawned workers; add all of them
for pid in $(pgrep -f "train.py"); do
    echo "$pid" > "$JOB_PATH/cgroup.procs"
    echo "  Added PID $pid to cgroup"
done

echo "=== Cgroup created: $JOB_PATH ==="
echo "Memory limit: 480 GB"
echo "CPU weight:   800"
echo "Swap:         disabled"
echo "I/O limit:    4 GB/s read"

Chapter 3 — Linux Capabilities

Traditional Unix permissions are binary: root or not root. Linux capabilities divide root's power into 37 fine-grained privileges. AI containers need only a handful:

#include <sys/capability.h>   /* link with -lcap */
#include <stdio.h>

void print_capabilities(void) {
    cap_t caps = cap_get_proc();
    char *text = cap_to_text(caps, NULL);
    printf("Current capabilities: %s\n", text);
    cap_free(text);
    cap_free(caps);
}

/* Minimum capabilities for an AI training container:
 *
 * CAP_SYS_NICE:     setpriority() for training process (nice=-10)
 * CAP_IPC_LOCK:     mlock() for pinned memory (DataLoader)
 * CAP_NET_ADMIN:    needed by some NCCL transports
 * CAP_SYS_PTRACE:   Nsight and perf profiling
 *
 * MUST DROP (dangerous):
 * CAP_SYS_ADMIN:    mount, kernel module load — too powerful
 * CAP_NET_RAW:      raw socket access — security risk
 * CAP_SYS_RAWIO:    direct hardware access
 */
c — capabilities.c (link: -lcap)
# Docker run with minimal capabilities for AI training
docker run \
    --rm \
    --gpus all \
    --cap-drop ALL \
    --cap-add SYS_NICE \      # setpriority for training
    --cap-add IPC_LOCK \      # pinned memory for DataLoader
    --cap-add NET_ADMIN \     # NCCL network setup
    --security-opt seccomp=ai_seccomp.json \
    --shm-size=64g \          # shared memory for NCCL
    --ulimit memlock=-1 \     # unlimited locked memory
    nvcr.io/nvidia/pytorch:24.01-py3 \
    torchrun --nproc_per_node=8 train.py

Chapter 5 — AppArmor & LSM

# AppArmor profile for AI training container
# File: /etc/apparmor.d/ai-training

profile ai-training flags=(attach_disconnected, mediate_deleted) {
  # Allow reading model weights from NFS
  /models/** r,
  /datasets/** r,

  # Allow writing checkpoints
  /checkpoints/** rw,
  /tmp/** rw,

  # Allow CUDA/NCCL shared libraries
  /usr/local/cuda/** mr,
  /usr/lib/x86_64-linux-gnu/libcuda** mr,

  # Allow GPU device files (managed by NVIDIA CTK)
  /dev/nvidia* rw,
  /dev/nvidiactl rw,
  /dev/nvidia-uvm rw,

  # Allow proc/sys reads needed by PyTorch
  /proc/sys/kernel/numa_balancing r,
  /proc/sys/vm/overcommit_memory r,
  /proc/meminfo r,
  /sys/fs/cgroup/** r,

  # Deny access to sensitive host paths
  deny /etc/shadow r,
  deny /etc/sudoers r,
  deny /proc/sysrq-trigger w,
  deny /sys/kernel/debug/** rw,
}

Chapter 6 — NVIDIA Container Toolkit

How the NVIDIA Container Toolkit Works

The NVIDIA Container Toolkit (NCT) is an OCI runtime hook that intercepts container creation and injects GPU access into the container. It solves a fundamental problem: containers use namespaces, but GPU devices live outside namespaces by default.

/* The NVIDIA Container Runtime hook sequence:
 *
 * 1. Docker/Kubernetes requests container creation
 * 2. nvidia-container-runtime intercepts the OCI spec
 * 3. Hook reads NVIDIA_VISIBLE_DEVICES env var (e.g., "0,1,2,3")
 * 4. Hook binds /dev/nvidia{0,1,2,3} and /dev/nvidiactl into the container
 * 5. Hook bind-mounts CUDA libraries from host into container
 *    (so container doesn't need to include large CUDA stack in its image)
 * 6. Hook sets up nvidia-smi process group for the container
 *
 * Why bind-mount CUDA from host?
 *   - CUDA driver version must match kernel module version
 *   - Container image doesn't need to include drivers (saves 5-10 GB)
 *   - Driver updates on host are immediately available to all containers
 */

Multi-Tenant GPU Isolation

#!/bin/bash
# Run two AI jobs on the same 8-GPU server with strict GPU isolation

# Job 1: Gets GPUs 0-3, 240 GB RAM, 40 CPU cores
docker run -d \
    --name job1 \
    --gpus '"device=0,1,2,3"' \
    --cpuset-cpus "0-39" \
    --memory 240g \
    --memory-swap 240g \
    -e CUDA_VISIBLE_DEVICES=0,1,2,3 \
    -e NCCL_P2P_LEVEL=NVL \
    nvcr.io/nvidia/pytorch:24.01-py3 \
    torchrun --nproc_per_node=4 train_job1.py

# Job 2: Gets GPUs 4-7, 240 GB RAM, 40 CPU cores
docker run -d \
    --name job2 \
    --gpus '"device=4,5,6,7"' \
    --cpuset-cpus "40-79" \
    --memory 240g \
    --memory-swap 240g \
    -e CUDA_VISIBLE_DEVICES=4,5,6,7 \
    -e NCCL_P2P_LEVEL=NVL \
    nvcr.io/nvidia/pytorch:24.01-py3 \
    torchrun --nproc_per_node=4 train_job2.py

# Verify isolation — each job sees only its 4 GPUs
docker exec job1 nvidia-smi --list-gpus
docker exec job2 nvidia-smi --list-gpus

Chapter 7 — Kubernetes GPU Scheduling

# PyTorch distributed training Job on Kubernetes with GPU scheduling
# kubernetes/llm-training-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: llm-training
  namespace: ml-workloads
spec:
  parallelism: 4        # 4 nodes × 8 GPUs = 32 GPUs total
  completions: 4
  template:
    spec:
      restartPolicy: OnFailure
      hostIPC: true     # needed for NCCL shared memory
      volumes:
      - name: shm
        emptyDir:
          medium: Memory
          sizeLimit: 64Gi
      - name: datasets
        persistentVolumeClaim:
          claimName: imagenet-pvc
      - name: checkpoints
        persistentVolumeClaim:
          claimName: checkpoints-pvc
      containers:
      - name: trainer
        image: nvcr.io/nvidia/pytorch:24.01-py3
        resources:
          requests:
            nvidia.com/gpu: 8     # request 8 GPUs per pod
            cpu: "80"
            memory: 480Gi
          limits:
            nvidia.com/gpu: 8
            cpu: "80"
            memory: 480Gi
        volumeMounts:
        - name: shm
          mountPath: /dev/shm
        - name: datasets
          mountPath: /data
        - name: checkpoints
          mountPath: /checkpoints
        env:
        - name: NCCL_IB_DISABLE
          value: "0"
        - name: NCCL_DEBUG
          value: "WARN"
        - name: MASTER_ADDR
          valueFrom:
            fieldRef:
              fieldPath: status.hostIP
        command:
        - torchrun
        - --nnodes=4
        - --nproc_per_node=8
        - --node_rank=$(JOB_COMPLETION_INDEX)
        - /workspace/train.py

Kubernetes GPU Scheduling Tips

① Use nvidia.com/gpu resource requests — the k8s-device-plugin handles allocation. ② Set hostIPC: true for NCCL shared memory segments. ③ Use emptyDir with Memory medium for /dev/shm to get fast inter-process communication. ④ Set topology constraints to ensure pods land on the same physical rack (minimise IB hop count). ⑤ Use PodDisruptionBudget to prevent eviction during multi-day training runs.
