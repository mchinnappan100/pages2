Profiling for AI

perf, eBPF, ftrace, NVIDIA Nsight Systems, Nsight Compute, PyTorch Profiler, and the Roofline model — every tool you need to find and fix every nanosecond of wasted compute in your AI training and inference workloads.

Profile First, Optimise Second

The most common mistake in AI systems optimisation is guessing at bottlenecks. A developer assumes their training is GPU-bound and spends two weeks optimising the attention kernel — only to discover the actual bottleneck was NVMe I/O in the data loader. Profiling first takes two hours and points you directly at the real problem.

The Profiling Hierarchy

Start coarse (is the GPU idle? is CPU pegged?), then drill down (which kernel is slow?), then go fine-grained (is it memory-bound or compute-bound?). This book follows that hierarchy — start with system-level tools, then GPU profilers, then kernel-level analysis.

Chapter 1 — perf: Linux Performance Tool

perf stat — Hardware Performance Counters

perf stat reads hardware performance counters — the PMU (Performance Monitoring Unit) built into every modern CPU. It gives you cache misses, branch mispredictions, IPC (instructions per cycle), and more in a single command.

# Basic perf stat — measure a training step
perf stat -e \
    cpu-cycles,instructions,\
    cache-misses,cache-references,\
    branch-misses,branch-instructions,\
    dTLB-load-misses,dTLB-loads,\
    LLC-loads,LLC-load-misses \
    python3 -c "
import torch
x = torch.randn(4096, 4096, device='cpu')
for _ in range(100):
    y = x @ x
"

# Sample output interpretation:
# Instructions per cycle (IPC) = instructions / cpu-cycles
#   IPC < 1.0: memory-bound (waiting for DRAM)
#   IPC 2-4:   compute-bound (good utilisation)
# LLC-load-misses / LLC-loads = LLC miss rate
#   > 5%: significant DRAM pressure
# dTLB-load-misses / dTLB-loads = TLB miss rate
#   > 1%: consider hugepages (see Book 2)

# Profile specific CPU events for ML workloads
perf stat -e \
    fp_arith_inst_retired.scalar_single,\   # scalar FP ops
    fp_arith_inst_retired.256b_packed_single,\ # AVX2 FP ops
    avx512_fp_packed.op_count \             # AVX-512 ops
    python3 train.py --steps=10

perf record & FlameGraph

# Record a CPU profile of training (10-second sample)
perf record \
    --freq=99 \                  # 99 Hz sampling (avoid lockstep with 100Hz timers)
    --call-graph=dwarf \         # capture full stack traces
    --output=training.perf.data \
    -- python3 train.py --steps=1000

# Generate FlameGraph (install: https://github.com/brendangregg/FlameGraph)
perf script -i training.perf.data | \
    stackcollapse-perf.pl | \
    flamegraph.pl \
    --title="Training CPU Flamegraph" \
    --width=1400 \
    > training_flamegraph.svg

# Open in browser: open training_flamegraph.svg
# Wide towers = functions consuming most CPU
# Tall stacks  = deep call chains

# Focus on Python: filter to Python frames only
perf script -i training.perf.data | \
    grep -E 'python|torch|_C\._' | \
    stackcollapse-perf.pl | \
    flamegraph.pl > python_flamegraph.svg

Chapter 2 — eBPF for Live Tracing

bpftrace Programs

eBPF (extended Berkeley Packet Filter) lets you run sandboxed programs in the kernel that trace any function, syscall, or hardware event — in production, with zero recompilation and minimal overhead.

#!/usr/bin/env bpftrace
# trace_cuda_malloc.bt — Trace every CUDA memory allocation from Python
# Run: sudo bpftrace trace_cuda_malloc.bt

BEGIN {
    printf("Tracing CUDA allocations... Ctrl+C to stop.\n");
    printf("%-10s %-8s %-16s %-10s\n", "TIME(ms)", "PID", "FUNC", "BYTES");
}

# Trace cudaMalloc calls
uprobe:/usr/local/cuda/lib64/libcudart.so:cudaMalloc {
    @start[tid] = nsecs;
    @size[tid]  = *((uint64_t*)arg1);   /* size is second argument */
}

uretprobe:/usr/local/cuda/lib64/libcudart.so:cudaMalloc /@start[tid]/ {
    $elapsed = (nsecs - @start[tid]) / 1000000;
    printf("%-10llu %-8d cudaMalloc    %-10llu bytes (%llu ms)\n",
           elapsed, pid, @size[tid], $elapsed);
    delete(@start[tid]);
    delete(@size[tid]);
}
