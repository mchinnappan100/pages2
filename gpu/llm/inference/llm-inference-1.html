Chapter 1

Transformer Architecture

The neural network that changed everything — from machine translation to trillion-parameter language models.

The Transformer, introduced in "Attention Is All You Need" (Vaswani et al., 2017), replaced recurrent networks with a purely attention-based architecture. Unlike RNNs that process tokens sequentially, Transformers process all tokens in parallel, making them dramatically faster to train on modern hardware.

📐 High-Level Architecture

A decoder-only Transformer (the kind used in GPT, Llama, Claude) stacks N identical blocks. Each block contains two sub-layers: multi-head self-attention and a position-wise feed-forward network.

📏

d_model

  • Residual stream width
  • Llama-2 7B: 4096
  • Llama-3 70B: 8192
  • GPT-4 (est.): 12288
🔀

n_heads

  • Parallel attention heads
  • head_dim = d_model / n_heads
  • Llama-2 7B: 32 heads
  • Each head: 128 dims
🧱

n_layers

  • Depth of the model
  • Llama-2 7B: 32 layers
  • Llama-3 70B: 80 layers
  • More layers = more capacity
⚡

d_ffn

  • FFN inner dimension
  • Typically 4× d_model
  • SwiGLU: ~8/3 × d_model
  • Most params live here

🔗 Residual Connections

Every sub-layer output is added back to its input: x = x + SubLayer(LN(x)). This is "pre-norm" style (used by Llama, GPT-2 variants). Residual connections allow gradients to flow directly through the network during backpropagation, enabling training of very deep models without vanishing gradients.

📐 Layer Normalization

RMSNorm (Root Mean Square Normalization) normalizes activations at each layer to prevent internal covariate shift. Llama uses RMSNorm instead of LayerNorm — it drops the mean-subtraction step, making it ~10% faster with negligible quality difference.

RMSNorm(x) = x / RMS(x) × γ where RMS(x) = sqrt( (1/d) × Σᵢ xᵢ² )
Key Insight: The Transformer's power comes from two things working together: (1) attention allows every token to gather information from every other token in O(n²) time, and (2) the feed-forward network processes each token independently but with rich non-linear transformations. Together they create an extremely expressive function approximator.
Chapter 2

Tokenization & Embeddings

How raw text becomes the numerical representations that neural networks can process.

🔤 Byte Pair Encoding (BPE)

Modern LLMs use subword tokenization via BPE. The algorithm starts with individual characters and iteratively merges the most frequent adjacent pairs into new tokens, building a vocabulary of typically 32,000–128,000 tokens. This balances character-level flexibility with word-level efficiency.

1

Initialize with characters

"hello" → ['h', 'e', 'l', 'l', 'o']

2

Count pair frequencies across corpus

Find the most frequent adjacent pair, e.g. ('l', 'l')

3

Merge and add to vocabulary

('l', 'l') → 'll' ; repeat until vocab_size reached

4

Encode new text using merge rules

"hello world" → [9906, 1917] (Llama 3 tokenizer)

📦 Token Embeddings

The embedding table has shape (vocab_size, d_model). Each token ID indexes into this table to produce a dense vector. For Llama-3 8B: (128256, 4096) — that's ~500M parameters just in the embedding layer, often weight-tied with the final LM head.

📍 Positional Encoding: RoPE

Transformers have no built-in notion of order. Rotary Position Embeddings (RoPE) solve this by rotating the query and key vectors by an angle proportional to their position. This encodes relative positions — a token at position 5 "knows" it's 3 steps from position 2.

RoPE rotates Q and K vectors by angle θ·m: Q_rotated[i] = Q[i] · cos(m·θᵢ) + Q'[i] · sin(m·θᵢ) The attention score then depends on relative position (m - n): score(q_m, k_n) = f(m - n)
Why RoPE wins: Unlike learned absolute position embeddings (BERT, GPT-2), RoPE generalizes to sequences longer than those seen during training. It's used in Llama, Mistral, Falcon, GPT-NeoX, and most modern open-source models.
Chapter 3

The Attention Mechanism

The core operation that lets every token attend to every other token — the engine of contextual understanding.

⚡ Scaled Dot-Product Attention

Given a sequence of tokens, attention produces output vectors by computing, for each query, a weighted sum of value vectors — where weights are determined by similarity between the query and all keys.

Attention(Q, K, V) = softmax( QKᵀ / √d_k ) · V Q = input × W_Q (queries: what am I looking for?) K = input × W_K (keys: what do I contain?) V = input × W_V (values: what do I return if matched?) d_k = head dimension (e.g., 128 for Llama-2 7B) √d_k scaling: prevents dot products from growing too large in high dimensions

🔀 Multi-Head Attention (MHA)

Instead of a single attention computation, MHA runs h independent attention heads in parallel, each with its own W_Q, W_K, W_V projections. Outputs are concatenated and projected back.

pythonMulti-Head Attention
import torch
import torch.nn as nn
import torch.nn.functional as F
import math

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model: int, n_heads: int):
        super().__init__()
        self.n_heads  = n_heads
        self.head_dim = d_model // n_heads

        self.W_q = nn.Linear(d_model, d_model, bias=False)
        self.W_k = nn.Linear(d_model, d_model, bias=False)
        self.W_v = nn.Linear(d_model, d_model, bias=False)
        self.W_o = nn.Linear(d_model, d_model, bias=False)

    def forward(self, x: torch.Tensor, mask=None):
        B, T, C = x.shape
        H, D    = self.n_heads, self.head_dim

        # Project and reshape to (B, n_heads, T, head_dim)
        q = self.W_q(x).view(B, T, H, D).transpose(1, 2)
        k = self.W_k(x).view(B, T, H, D).transpose(1, 2)
        v = self.W_v(x).view(B, T, H, D).transpose(1, 2)

        # Attention scores
        scale  = math.sqrt(D)
        scores = (q @ k.transpose(-2, -1)) / scale   # (B,H,T,T)

        # Causal mask: prevent attending to future tokens
        if mask is not None:
            scores = scores.masked_fill(mask == 0, float('-inf'))

        attn   = F.softmax(scores, dim=-1)          # (B,H,T,T)
        output = attn @ v                             # (B,H,T,D)

        # Concatenate heads and project
        output = output.transpose(1, 2).contiguous().view(B, T, C)
        return self.W_o(output)

🏎️ Grouped Query Attention (GQA)

MHA requires one K/V pair per head per token — expensive to cache. GQA reduces this by sharing K and V across groups of Q heads. Llama-2 70B uses 8 KV heads with 64 Q heads (8:1 ratio), reducing KV cache size by 8×.

🔢 Feed-Forward Network (FFN)

The FFN applies two linear transformations with a non-linearity in between, independently to each token position. Most modern LLMs use SwiGLU (Swish-Gated Linear Unit) which uses a gating mechanism.

pythonSwiGLU FFN (Llama-style)
class SwiGLUFFN(nn.Module):
    def __init__(self, d_model: int, d_ffn: int):
        super().__init__()
        # SwiGLU uses 3 projections instead of 2
        self.w_gate = nn.Linear(d_model, d_ffn, bias=False)
        self.w_up   = nn.Linear(d_model, d_ffn, bias=False)
        self.w_down = nn.Linear(d_ffn,   d_model, bias=False)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # Gate signal × up-projection, then down-project
        gate   = F.silu(self.w_gate(x))  # SiLU = Swish activation
        up     = self.w_up(x)
        return self.w_down(gate * up)
Chapter 4

Pretraining

How models acquire world knowledge — learning to predict the next token across trillions of examples.

🎯 The Training Objective

LLMs are pretrained with Causal Language Modeling (CLM): given a sequence of tokens, predict the next token at each position. This is fully self-supervised — the data provides its own labels.

L = -Σₜ log P(token_t | token₁, token₂, ..., token_{t-1}) Minimize the negative log-likelihood of the correct next token across all positions in the training sequence.

📊 Scaling Laws

Kaplan et al. (2020) and Hoffmann et al. (2022, "Chinchilla") showed that model performance follows power laws in compute, data, and parameters. The Chinchilla result: for a given compute budget, you should roughly use equal scaling of parameters AND tokens.

ModelParametersTraining TokensTokens/Param
GPT-3175B300B1.7×
Chinchilla70B1.4T20× optimal
Llama-2 7B7B2T286×

Inference-Optimized Training: Llama-3 8B was deliberately over-trained on 15T tokens — far beyond compute-optimal. The rationale: a smaller model trained on more data is cheaper to serve at inference time, even if training cost is higher. This reflects production priorities.

🏗️ Training Infrastructure

Large model pretraining requires distributed training across hundreds or thousands of GPUs. Three types of parallelism are typically combined:

📦

Data Parallelism

  • Each GPU has full model copy
  • Different batch per GPU
  • AllReduce gradients after backward
  • FSDP: shards optimizer state
⚙️

Tensor Parallelism

  • Split weight matrices across GPUs
  • Each GPU holds a column/row shard
  • AllReduce within each layer
  • Megatron-LM style
🏗️

Pipeline Parallelism

  • Different layers on different GPUs
  • Micro-batching to fill pipeline
  • Reduces P2P communication
  • GPipe / 1F1B schedules

📉 Training Stability Techniques

1

Mixed Precision (BF16/FP16)

Train in BF16, keep FP32 master weights for optimizer. BF16 preferred for its wider dynamic range.

2

Gradient Clipping

Clip gradient norm to ~1.0 to prevent explosive updates that crash training.

3

Learning Rate Schedule

Linear warmup → cosine decay. Warmup stabilizes early training with random weights.

4

Flash Attention

IO-aware attention — never materializes full N×N attention matrix. O(N) memory instead of O(N²).

Pretraining produces the base model — a powerful next-token predictor with vast world knowledge, but not yet aligned to human preferences. The next steps (RLHF, instruction tuning) turn this raw capability into a helpful assistant. See Part 2.
