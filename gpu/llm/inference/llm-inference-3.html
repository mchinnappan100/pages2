Why Quantize?

Reducing numerical precision to fit more model into less memory — without catastrophic quality loss.

📐 The Memory Problem

A 70B-parameter model stored in BF16 (2 bytes/parameter) requires 140 GB of VRAM just for weights — more than fits on two A100 80GB GPUs. Quantization reduces this by representing weights in fewer bits.

Memory footprint per 70B parameters: FP32 (4 bytes): 280 GB ← training master weights BF16 (2 bytes): 140 GB ← standard inference INT8 (1 byte): 70 GB ← 2× reduction INT4 (0.5 byte): 35 GB ← 4× reduction ✓ fits on 2×A100 INT2 (0.25 byte): 17 GB ← 8× reduction (quality degraded) Rule of thumb for serving: INT4 = sweet spot (4× memory, ~1-3% quality drop)

🎯 The Inference Bottleneck: Memory Bandwidth

During token generation (decode phase), the model generates one token per step. At each step, every single weight must be loaded from GPU memory (HBM) into compute units. This makes decode memory-bandwidth-bound, not compute-bound — the GPU spends most of its time waiting for weights to arrive from memory, not computing.

Arithmetic Intensity = FLOPs / Bytes_accessed Llama-3 8B decode (batch=1): FLOPs ≈ 2 × 8B = 16 GFLOPs Bytes ≈ 8B × 2 = 16 GB (load all weights in BF16) Intensity ≈ 1 FLOP/Byte ← far below GPU's peak intensity (~200) → GPU is 200× underutilized during decode → Quantization helps: INT4 halves bytes → halves time
Prefill vs Decode: Prefill (processing the prompt) IS compute-bound — all tokens processed in parallel, high arithmetic intensity. Only decode is memory-bandwidth-bound. This asymmetry drives the architecture of modern inference systems like vLLM and TensorRT-LLM.

INT8, INT4 & FP8 Quantization

The algorithms that make precision reduction work without destroying model quality.

🔢 Absmax INT8 Quantization

The simplest form: find the maximum absolute value in a tensor, scale so it maps to 127, round to nearest integer. Dequantize by multiplying by the scale factor.

pythonAbsmax INT8 Quantization
import torch

def absmax_quantize(W: torch.Tensor):
    """Quantize weight matrix to INT8 using absmax scaling."""
    scale  = W.abs().max() / 127.0
    W_int8 = (W / scale).round().clamp(-128, 127).to(torch.int8)
    return W_int8, scale

def absmax_dequantize(W_int8: torch.Tensor, scale: torch.Tensor):
    return W_int8.to(torch.float32) * scale

# Zero-point quantization (handles asymmetric distributions)
def zeropoint_quantize(W: torch.Tensor):
    W_min, W_max = W.min(), W.max()
    scale   = (W_max - W_min) / 255.0
    zp      = (-128 - W_min / scale).round()
    W_int8  = (W / scale + zp).round().clamp(-128, 127).to(torch.int8)
    return W_int8, scale, zp

# Example: quantize a 4096×4096 weight matrix
W       = torch.randn(4096, 4096)
W8, sc  = absmax_quantize(W)
W_back  = absmax_dequantize(W8, sc)
err     = (W - W_back).abs().mean()
print(f"Mean quant error: {err:.5f}")  # typically ~0.001
print(f"Memory: {W.numel()*4/1e6:.1f}MB → {W8.numel()/1e6:.1f}MB")

🏋️ GPTQ: Post-Training INT4 Quantization

GPTQ (Frantar et al., 2022) uses second-order information (Hessian) to minimize quantization error layer by layer. It quantizes weights column-by-column, compensating for error in later columns using the remaining unquantized weights.

pythonGPTQ Quantization (using auto-gptq)
from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from transformers import AutoTokenizer

tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B")

quant_config = BaseQuantizeConfig(
    bits=4,             # INT4
    group_size=128,     # quantize in groups of 128 columns
    desc_act=False,     # activation order (True = better quality)
)

model = AutoGPTQForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B",
    quantize_config=quant_config
)

# Calibration data — small sample of representative text
calibration = [tokenizer("Machine learning is "), ...]
model.quantize(calibration)          # ~10-30 min for 8B model
model.save_quantized("./llama3-8b-gptq-int4")
# Result: 8B model → ~4.5 GB  (was ~16 GB in BF16)

🎯 AWQ: Activation-Aware Weight Quantization

AWQ (Lin et al., 2023) observes that only ~1% of weights are "salient" — corresponding to large activation magnitudes. By protecting these salient weights (scaling them up before quantization), AWQ achieves better quality than GPTQ at INT4.

pythonAWQ Quantization
from awq import AutoAWQForCausalLM

model = AutoAWQForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B",
    torch_dtype=torch.float16
)
quant_config = {
    "zero_point": True,
    "q_group_size": 128,
    "w_bit": 4,
    "version": "GEMM",      # fast inference kernel
}
model.quantize(tokenizer, quant_config=quant_config)
model.save_quantized("./llama3-8b-awq")

KV Cache: Cost & Mechanics

The essential caching trick that makes autoregressive generation tractable — and its hidden memory cost.

🔄 Why KV Cache Exists

In autoregressive generation, the model generates one token at a time. Without caching, generating token N would require recomputing the Key and Value matrices for all N-1 previous tokens — O(N²) total work. The KV cache stores these, reducing each step to O(N).

WITHOUT KV Cache (naïve): Step 1: compute K,V for [t₁] → generate t₂ Step 2: compute K,V for [t₁, t₂] → generate t₃ Step 3: compute K,V for [t₁,t₂,t₃] → generate t₄ Total work: O(N²) WITH KV Cache: Step 1: compute K,V for [t₁] → cache: {t₁:K₁,V₁} → t₂ Step 2: compute K,V for [t₂] → cache: {t₁,t₂: K,V} → t₃ Step 3: compute K,V for [t₃] → cache: {t₁,t₂,t₃: K,V} → t₄ Each step: O(1) new computation + O(N) cache lookup Total work: O(N) ← linear!

💰 KV Cache Memory Formula

KV_bytes = 2 (K and V) × n_layers (one KV pair per transformer layer) × n_kv_heads (GQA: fewer heads than query heads) × head_dim (= d_model / n_heads) × seq_len (grows with each generated token) × bytes_per_elem (2 for BF16, 1 for INT8) Example — Llama-3 8B (BF16): = 2 × 32 × 8 × 128 × 4096 × 2 bytes = 2 × 32 × 8 × 128 × 4096 × 2 = 536 MB per sequence Example — Llama-3 70B (BF16, 4096 seq): = 2 × 80 × 8 × 128 × 4096 × 2 = 1.34 GB per sequence
pythonKV Cache Size Calculator
def kv_cache_gb(n_layers, n_kv_heads, head_dim,
               seq_len, dtype_bytes=2, batch_size=1):
    """Calculate KV cache memory in GB."""
    kv = (2           # K and V
          * n_layers
          * n_kv_heads
          * head_dim
          * seq_len
          * dtype_bytes
          * batch_size)
    return kv / (1024**3)

# Llama-3 8B: 32 layers, 8 KV heads, 128 head_dim
print(f"Llama-3 8B  @ 4K  seq: {kv_cache_gb(32,  8, 128, 4096):.2f} GB")
print(f"Llama-3 8B  @ 128K seq: {kv_cache_gb(32,  8, 128, 131072):.2f} GB")
print(f"Llama-3 70B @ 4K  seq: {kv_cache_gb(80,  8, 128, 4096):.2f} GB")
print(f"Llama-3 70B @ 32K seq: {kv_cache_gb(80,  8, 128, 32768):.2f} GB")
# Output:
# Llama-3 8B  @ 4K  seq: 0.53 GB
# Llama-3 8B  @ 128K seq: 17.0 GB  ← KV alone fills an A100!
# Llama-3 70B @ 4K  seq: 1.34 GB
# Llama-3 70B @ 32K seq: 10.7 GB

📈 Batch Size Impact

Serving 100 concurrent users each with a 4K context window on Llama-3 70B requires: 100 × 1.34 GB = 134 GB for KV cache alone — more than the model weights themselves (140 GB). This is why KV cache management is the primary challenge in LLM serving.

KV Cache Optimization

PagedAttention, Flash Attention, and GQA — the techniques that make long-context inference economically viable.

📄 PagedAttention (vLLM)

Traditional KV cache allocates a contiguous block for each request's maximum sequence length upfront — wasting up to 60-80% of memory on padding. vLLM's PagedAttention, inspired by OS virtual memory, stores KV cache in non-contiguous pages (blocks of tokens), allocated on demand.

Traditional KV Cache (contiguous, pre-allocated): Request 1: [KV₁ KV₂ KV₃ KV₄ __ __ __ __] ← 50% wasted Request 2: [KV₁ KV₂ __ __ __ __ __ __] ← 75% wasted Request 3: [KV₁ KV₂ KV₃ KV₄ KV₅ KV₆ __ __] ← 25% wasted PagedAttention (block-based, on-demand): Physical blocks: [B0][B1][B2][B3][B4][B5][B6] Request 1 → B0, B2 (logical: 0,1) Request 2 → B1 (logical: 0) Request 3 → B3, B5, B6 (logical: 0,1,2) Memory waste: < 4% ← near-optimal!
pythonvLLM with PagedAttention
from vllm import LLM, SamplingParams

# vLLM automatically uses PagedAttention
llm = LLM(
    model="meta-llama/Meta-Llama-3-8B-Instruct",
    max_model_len=8192,
    gpu_memory_utilization=0.9,   # use 90% of GPU for KV cache
    dtype="bfloat16",
)
sampling = SamplingParams(temperature=0.7, top_p=0.9, max_tokens=512)

prompts = [
    "Explain the transformer architecture:",
    "Write a Python function to sort a list:",
]
outputs = llm.generate(prompts, sampling)
for out in outputs:
    print(out.outputs[0].text)

⚡ Flash Attention: IO-Aware Exact Attention

Standard attention materializes the full N×N attention matrix in HBM (slow, O(N²) memory). Flash Attention tiles the computation to fit in fast SRAM, never writing the full matrix to HBM. Same mathematical result, 2-4× faster, O(N) memory.

Standard Attention (N=4096): Q,K,V in HBM → load to SRAM Compute S = QKᵀ/√d → write N×N = 64M floats to HBM ← SLOW Compute P = softmax(S) → write 64M floats to HBM ← SLOW Compute O = PV → write to HBM Total HBM reads/writes: ~3 × N² × d Flash Attention (N=4096): Split Q,K,V into tiles that fit in SRAM (~96KB) For each tile pair: compute partial softmax in SRAM Accumulate into output — never write N×N matrix to HBM Total HBM reads/writes: ~N × d (linear!)
pythonFlash Attention 2 in PyTorch
import torch
import torch.nn.functional as F

# PyTorch 2.0+ uses Flash Attention automatically via scaled_dot_product_attention
def flash_attention_forward(Q, K, V, is_causal=True):
    # Automatically selects Flash Attention 2 when available
    with torch.backends.cuda.sdp_kernel(
        enable_flash=True,
        enable_math=False,    # disable non-Flash fallback
        enable_mem_efficient=False
    ):
        out = F.scaled_dot_product_attention(
            Q, K, V,
            attn_mask=None,
            dropout_p=0.0,
            is_causal=is_causal     # applies causal mask efficiently
        )
    return out

# Load model with Flash Attention 2 enabled
from transformers import AutoModelForCausalLM
model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-8B",
    attn_implementation="flash_attention_2",  # use FA2
    torch_dtype=torch.bfloat16,
    device_map="auto",
)

🗜️ KV Cache Quantization

The KV cache itself can be quantized (INT8 or even INT4) since small precision errors in cached attention states accumulate less harmfully than weight errors.

pythonKV Cache in INT8 with vLLM
from vllm import LLM

llm = LLM(
    model="meta-llama/Meta-Llama-3-70B-Instruct",
    kv_cache_dtype="fp8",          # FP8 KV cache (2× memory saving)
    quantization="fp8",            # weight quantization
    tensor_parallel_size=4,         # 4 GPUs for 70B model
    gpu_memory_utilization=0.92,
)
# FP8 KV cache: 1.34 GB → 0.67 GB per sequence for 70B @ 4K
# Enables ~2× more concurrent requests

🏎️ Continuous Batching

Traditional static batching waits for the entire batch to finish before starting new requests, causing GPU idling. Continuous batching (Orca/vLLM) inserts new requests as soon as slots free up, dramatically increasing throughput.

📄

PagedAttention

  • OS-inspired virtual memory
  • Non-contiguous KV pages
  • <4% memory waste
  • Prefix sharing across requests
⚡

Flash Attention 2

  • SRAM-tiled computation
  • O(N) memory (vs O(N²))
  • 2-4× faster than standard
  • Supports causal masking
🔀

GQA / MQA

  • Share K/V across Q heads
  • GQA: groups of 8 → 8× smaller KV
  • MQA: single K/V → max savings
  • Used by Llama-3, Mistral
🔄

Continuous Batching

  • No waiting for whole batch
  • New requests fill freed slots
  • 3-10× throughput improvement
  • Core feature of vLLM
Optimization Stack: The combination of (1) AWQ INT4 weights, (2) FP8 KV cache, (3) Flash Attention 2, (4) GQA, and (5) PagedAttention with continuous batching can serve Llama-3 70B at 5-10× higher throughput than a naïve BF16 implementation — at the same hardware cost. This stack is what vLLM and TensorRT-LLM implement out of the box.
