RLHF: Reinforcement Learning from Human Feedback

Bridging the gap between next-token prediction and genuinely helpful behaviour.

RLHF (Christiano et al., 2017; InstructGPT, 2022) is the technique that transformed GPT-3 into ChatGPT. It teaches models to be helpful, harmless, and honest by optimizing against human preferences rather than just token prediction likelihood.

🗺️ The Three-Stage Pipeline

1

Supervised Fine-Tuning (SFT)

Fine-tune the pretrained model on high-quality demonstration data: (prompt, ideal response) pairs written by human annotators. This teaches the model the format of helpful responses.

2

Reward Model Training

Collect human preference data: show two model responses to the same prompt, ask which is better. Train a reward model to predict these preferences as a scalar score.

3

RL Fine-Tuning with PPO

Use the reward model as a signal to fine-tune the SFT model via PPO. A KL-divergence penalty prevents the policy from drifting too far from the SFT model (reward hacking prevention).

Stage 1: SFT Dataset: {(prompt₁, response₁), (prompt₂, response₂), ...} Loss: standard cross-entropy on response tokens Output: SFT model (π_sft) Stage 2: Reward Model Dataset: {(prompt, response_win, response_lose)} Loss: -log σ(r(prompt, response_win) - r(prompt, response_lose)) Output: reward model r_θ Stage 3: PPO Fine-Tuning For each prompt x, generate response y ~ π_θ(y|x): reward = r_θ(x, y) - β · KL(π_θ(y|x) ‖ π_sft(y|x)) Maximize expected reward via PPO update Output: aligned model (π_RL)

🧮 Why KL Penalty Matters

Without the KL penalty, the policy would discover degenerate strategies — outputs that score high on the reward model but are garbage to humans ("reward hacking"). The penalty term keeps the RL-tuned model close to the well-behaved SFT baseline.

Total Objective = E_{x,y}[ r_θ(x,y) ] - β · KL( π_RL(·|x) ‖ π_SFT(·|x) ) β controls the tradeoff: β → 0: maximize reward regardless of divergence (reward hacking) β → ∞: never deviate from SFT (no alignment gain) Typical β: 0.01 – 0.1
Reward Hacking: A famous example — if the reward model prefers longer answers, the policy learns to be verbose without being more helpful. Human evaluation is essential to catch these failure modes. Constitutional AI (Anthropic) partially addresses this by using an AI-generated set of principles as an additional constraint.
Chapter 6

Reward Model Training

Teaching a model to predict human preferences — the critic that guides alignment.

🏗️ Architecture

The reward model is typically initialized from the SFT model with its language model head replaced by a single scalar output head. It takes a prompt-response pair and outputs a scalar score representing quality.

pythonReward Model (simplified)
import torch
import torch.nn as nn

class RewardModel(nn.Module):
    def __init__(self, base_model, d_model: int):
        super().__init__()
        self.base   = base_model          # frozen or partially frozen LLM
        self.head   = nn.Linear(d_model, 1, bias=False)  # scalar output

    def forward(self, input_ids, attention_mask):
        # Get last hidden state from base model
        out    = self.base(input_ids, attention_mask=attention_mask,
                           output_hidden_states=True)
        hidden = out.hidden_states[-1]          # (B, T, d_model)
        # Use the last non-padding token as the reward signal
        seq_lens = attention_mask.sum(dim=1) - 1  # (B,)
        last_h   = hidden[torch.arange(hidden.size(0)), seq_lens]
        reward   = self.head(last_h).squeeze(-1)   # (B,)
        return reward

def bradley_terry_loss(r_win: torch.Tensor, r_lose: torch.Tensor):
    # Preference loss: log-sigmoid of reward difference
    return -torch.log(torch.sigmoid(r_win - r_lose)).mean()

📊 Training Data Collection

Human annotators are shown pairs of model responses to the same prompt and asked to rank them. This produces a dataset of (prompt, chosen, rejected) triples. Key considerations:

  • Annotator agreement: preferences are noisy — only use pairs with high inter-annotator agreement
  • Diversity: cover a wide range of topics and difficulty levels
  • Calibration: avoid annotators who always prefer longer or more confident responses
  • Scale: OpenAI used ~33K comparisons for InstructGPT; Llama 2 used ~1M

PPO vs DPO: RL Algorithms for Alignment

Two approaches to the same goal — one online and complex, one offline and elegant.

🔄 PPO (Proximal Policy Optimization)

PPO is an online RL algorithm that generates responses at training time and uses the reward model to compute advantages. It requires running four models simultaneously (policy, reference, reward, value), making it memory-intensive and complex to tune.

pythonPPO Training Loop (conceptual)
from trl import PPOTrainer, PPOConfig

config = PPOConfig(
    model_name="sft_model",
    learning_rate=1.41e-5,
    batch_size=64,
    mini_batch_size=8,
    gradient_accumulation_steps=1,
    optimize_cuda_cache=True,
    kl_penalty="kl",      # KL from reference model
    init_kl_coef=0.2,
    adap_kl_ctrl=True,     # adaptively adjust KL coefficient
)
trainer = PPOTrainer(config, policy_model, ref_model, tokenizer,
                     dataset, reward_model)

for batch in trainer.dataloader:
    # Step 1: generate responses with current policy
    response_tensors = trainer.generate(batch["input_ids"],
                                        max_new_tokens=128)

    # Step 2: score with reward model
    rewards = [reward_model(q, r) for q, r in
               zip(batch["input_ids"], response_tensors)]

    # Step 3: PPO update
    stats = trainer.step(batch["input_ids"], response_tensors, rewards)

✨ DPO (Direct Preference Optimization)

DPO (Rafailov et al., 2023) bypasses the reward model entirely. It optimizes the policy directly on preference data, showing that the RLHF objective has a closed-form solution in terms of the policy and reference model. DPO is simpler, more stable, and competitive with PPO.

DPO Loss: L_DPO = -E[ log σ( β · log(π_θ(y_w|x) / π_ref(y_w|x)) - β · log(π_θ(y_l|x) / π_ref(y_l|x)) ) ] y_w = winning (preferred) response y_l = losing (rejected) response π_ref = frozen reference model (SFT checkpoint) β = temperature (typically 0.1)
pythonDPO Training with TRL
from trl import DPOTrainer, DPOConfig
from transformers import AutoModelForCausalLM, AutoTokenizer

model   = AutoModelForCausalLM.from_pretrained("sft_checkpoint")
ref_model = AutoModelForCausalLM.from_pretrained("sft_checkpoint")  # frozen
tokenizer = AutoTokenizer.from_pretrained("sft_checkpoint")

# Dataset format: {"prompt": ..., "chosen": ..., "rejected": ...}
config  = DPOConfig(
    beta=0.1,
    learning_rate=5e-7,
    num_train_epochs=1,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    bf16=True,
    max_length=1024,
)
trainer = DPOTrainer(model, ref_model, args=config,
                     train_dataset=pref_dataset, tokenizer=tokenizer)
trainer.train()

Chapter 8

Fine-Tuning: SFT & LoRA

Adapting large models to specific tasks without retraining billions of parameters from scratch.

📝 Supervised Fine-Tuning (SFT)

SFT is standard next-token prediction on a curated instruction-following dataset. The model learns to format responses properly, follow system prompts, and maintain conversation context. Loss is typically computed only on the assistant response tokens (not the prompt).

pythonSFT Training Loop
from transformers import AutoModelForCausalLM, AutoTokenizer, TrainingArguments
from trl import SFTTrainer
from datasets import load_dataset

model     = AutoModelForCausalLM.from_pretrained("meta-llama/Meta-Llama-3-8B",
                                                  torch_dtype=torch.bfloat16)
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Meta-Llama-3-8B")
dataset   = load_dataset("HuggingFaceH4/ultrachat_200k", split="train_sft")

args = TrainingArguments(
    output_dir="./sft-llama3-8b",
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,    # effective batch = 16
    num_train_epochs=1,
    learning_rate=2e-5,
    bf16=True,
    lr_scheduler_type="cosine",
    warmup_ratio=0.1,
    logging_steps=10,
    save_steps=500,
)
trainer = SFTTrainer(model=model, tokenizer=tokenizer,
                     args=args, train_dataset=dataset,
                     max_seq_length=4096)
trainer.train()

🔬 LoRA: Low-Rank Adaptation

LoRA (Hu et al., 2022) freezes the original model weights and injects trainable low-rank decomposition matrices into each attention layer. Instead of updating W (d×d), we train two small matrices A (d×r) and B (r×d) where r ≪ d.

Original forward: h = W·x LoRA forward: h = W·x + (B·A)·x · (α/r) W frozen, A and B trainable r = rank (e.g., 8, 16, 64) α = scaling factor Parameters saved: Full fine-tune: d² parameters LoRA (r=16): 2 × d × r (e.g., 4096×16×2 = 131K vs 16.7M)
pythonLoRA with PEFT
from peft import LoraConfig, get_peft_model, TaskType

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=16,                          # rank
    lora_alpha=32,                  # scaling = alpha/r = 2
    target_modules=["q_proj", "v_proj",
                    "k_proj", "o_proj"],
    lora_dropout=0.05,
    bias="none",
)

model = get_peft_model(base_model, lora_config)
model.print_trainable_parameters()
# trainable params: 3,407,872  |  all params: 6,742,609,920
# trainable%: 0.0506%  → 99.95% of weights frozen!

# After training, merge LoRA weights back into base model:
merged_model = model.merge_and_unload()
merged_model.save_pretrained("./lora-merged-llama3")

💡 QLoRA: Quantized LoRA

QLoRA (Dettmers et al., 2023) combines 4-bit quantization of the base model with LoRA adapters trained in 16-bit. This makes it possible to fine-tune a 70B model on a single 80GB A100 GPU.

pythonQLoRA Setup
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_use_double_quant=True,   # quantize quantization constants
    bnb_4bit_quant_type="nf4",         # NF4: optimal for normal distributions
    bnb_4bit_compute_dtype=torch.bfloat16
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Meta-Llama-3-70B",
    quantization_config=bnb_config,
    device_map="auto",               # auto-shard across GPUs
)
# Base model: 4-bit (frozen), LoRA adapters: BF16 (trainable)
model = get_peft_model(model, lora_config)
# 70B model now fits in ~40GB VRAM!

Full Fine-Tuning

  • All parameters updated
  • Best quality ceiling
  • Requires massive GPU RAM
  • Risk of catastrophic forgetting
🔬

LoRA

  • <1% params trainable
  • ~3% quality gap vs full
  • Fast, low-memory
  • Easy to share adapters
🗜️

QLoRA

  • 4-bit base + LoRA adapters
  • 70B on single A100 80GB
  • ~5% quality gap vs full
  • State-of-art efficiency
Practical Recommendation: For most teams, QLoRA + DPO is the production-ready alignment recipe. Train on preference data with DPO, use QLoRA for memory efficiency, and evaluate on MT-Bench or AlpacaEval to validate alignment quality. This stack can fine-tune Llama-3 70B on a single A100 in under 24 hours.
