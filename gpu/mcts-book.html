Monte Carlo
Tree Search

A beginner-friendly guide to one of the most powerful AI decision-making algorithms — the engine behind AlphaGo, game-playing AIs, and intelligent planning systems. No prior AI knowledge required.

What is Monte Carlo Tree Search?

Imagine you are playing chess and you want to figure out your best next move. You could try to analyse every possible future — but there are more chess positions than atoms in the universe. You can't check them all.

Instead, you might do what a clever player does: focus your thinking on the most promising lines of play, and randomly simulate a few games from each candidate position to estimate which one leads to winning most often. That intuition, made precise and algorithmic, is Monte Carlo Tree Search.

In One Sentence

MCTS is an algorithm that builds a search tree by intelligently selecting nodes to explore, randomly simulating outcomes from those nodes, and using the results to decide where to search next — all without needing a hand-crafted evaluation function.

Why MCTS Is Special

Most game-playing algorithms need an evaluation function — a formula that scores any board position. Writing a good evaluation function is hard and domain-specific. MCTS doesn't need one. It estimates the value of a position by playing out random games from it and seeing who wins. The more random games it plays from a position, the more accurate its estimate becomes.

The Four Pillars

STEP 1
Selection

Start at the root and navigate to the most promising unexplored node using a policy (UCB1).

STEP 2
Expansion

Add one (or more) new child nodes to the selected node, representing unexplored moves.

STEP 3
Simulation (Rollout)

Play out the game randomly from the new node until a terminal state (win/lose/draw).

STEP 4
Backpropagation

Update the win/visit counts of every node on the path from the new node back to the root.

CHAPTER ONE
Foundations

Game Trees and
the Search Problem

Before MCTS makes sense, you need to understand what a game tree is and why searching it naively is impossible for most real games.

The Search Tree Structure

A game tree represents all possible game states as a tree of nodes. Each node is a board state. Each edge is a move. The root is the current position. Leaves are terminal states (win, lose, draw).

Figure 1.1 — A game tree. Root = current state. Edges = moves. Nodes = states reached after those moves.

The Search Problem

The branching factor is the average number of legal moves at each state. The depth is how many moves until the game ends. The total number of leaf nodes is roughly:

Total states ≈ branching_factor ^ depth
GameBranching FactorAvg DepthApprox. States
Tic-Tac-Toe~59~362,880
Chess~35~80~10¹²³
Go (19×19)~250~150~10³⁶⁰

For Chess and Go, exhaustive search is impossible. We need smart sampling — which is exactly what MCTS provides.

Analogy

Think of MCTS as a smart explorer in a forest with millions of paths. Instead of checking every path, it keeps a rough mental map, spends more time exploring promising regions, and occasionally takes random walks to discover new areas.

CHAPTER TWO
Step 1

Selection

Selection is the navigation step — starting at the root, the algorithm descends the tree by choosing which child to visit at each node, guided by a policy that balances exploration and exploitation.

How Selection Works

Starting from the root, the algorithm repeatedly asks: "Which child should I visit next?"

The answer uses the UCT (Upper Confidence Bound applied to Trees) formula, which we'll explore in depth in Chapter 6. For now, the key idea is:

  • Nodes that have been visited many times and look good → visited more (exploitation)
  • Nodes that have been visited few times → also visited to reduce uncertainty (exploration)

Selection continues until we reach a node that has unexplored children (a node that can be expanded) or a terminal node.

Figure 2.1 — Selection navigates from root to a node with an unexplored child (shown as dashed circle). Numbers show wins/visits. Bold yellow path = selected route.

Node Statistics

Each node in the MCTS tree stores two values:

  • N(v) — the visit count: how many times this node has been visited
  • W(v) — the win count: how many wins were recorded through this node

The estimated win rate is simply W(v) / N(v). A node with 30 wins out of 60 visits has a 50% win rate.

CHAPTER THREE
Step 2

Expansion

Adding New Nodes

Once selection lands on a node with unexplored children, expansion creates one or more new child nodes representing the unexplored moves.

The simplest strategy: add one new child per iteration. Some variants add all children at once.

Figure 3.1 — Expansion: a previously unexplored child node (move3) is added to the tree. It starts with 0 wins and 0 visits.
Why Not Expand Everything at Once?

Expanding all children immediately wastes memory and computation on positions we might never need. Expanding one child per iteration lets the algorithm focus resources on the most promising regions.

CHAPTER FOUR
Step 3

Simulation (Rollout)

Playing Out Random Games

From the newly expanded node, the simulation (also called rollout or playout) plays a complete game to the end using a random policy — at each step, a legal move is chosen uniformly at random.

The result (win = +1, loss = 0, draw = 0.5) is recorded and passed back to the backpropagation step.

Figure 4.1 — Rollout: from the newly expanded node (A1c), random moves are played until the game ends. The rollout states are NOT added to the tree — only the final result is used.

Random vs Informed Rollout

Rollout PolicySpeedAccuracyNotes
Uniform randomVery fastLow per rolloutAccurate with many iterations
Domain heuristicSlowerHigher per rolloutBetter estimates with fewer iterations
Neural network policySlowVery highAlphaGo approach — see Chapter 10
The Monte Carlo Insight

Even though each individual random rollout is low-quality, the average result over many rollouts converges to a reliable estimate of the true position value. This is the statistical power of Monte Carlo methods — the "law of large numbers" in action.

CHAPTER FIVE
Step 4

Backpropagation

Updating the Tree

The rollout result propagates back up the tree — every node on the path from the new node to the root has its visit count and win count updated.

Figure 5.1 — Backpropagation: the rollout result (WIN = +1) flows from the new node back to the root. Every node on the path increments both its win count (W) and visit count (N).

The Update Rule

For each node v on the path from new_node to root:
N(v) = N(v) + 1
W(v) = W(v) + result
Important — Perspective Flip

In two-player games, the result must be flipped at alternating levels. A win for Player 1 at depth d is a loss for Player 2. When backpropagating, add the result for the player who just moved at that node, and add the complement (1 − result) for the opposing player's nodes. This ensures each node's win count reflects the perspective of the player choosing that move.

One Complete Iteration

The four steps (Select → Expand → Simulate → Backpropagate) form one iteration. MCTS runs thousands of these iterations within a given time budget, then picks the move at the root with the highest visit count (or win rate).

CHAPTER SIX
The Math

The UCB1 Formula &
Exploration vs Exploitation

The Core Tension: Explore vs Exploit

At every selection step, MCTS faces a dilemma:

  • Exploit — visit the node that currently looks best (highest win rate). Risk: might miss a better unexplored option.
  • Explore — visit a less-tried node to reduce uncertainty. Risk: wastes time on bad options.
Figure 6.1 — The exploration-exploitation trade-off: exploit what's known to be good vs explore the unknown.

The UCT Formula

UCT (Upper Confidence Bound applied to Trees) solves this dilemma by scoring each child node with a formula that combines both exploitation and exploration:

UCT(v) = W(v)/N(v) + C × √( ln(N(parent)) / N(v) )

W(v) = wins through node v
N(v) = visits to node v
N(parent) = visits to parent node
C = exploration constant (typically √2 ≈ 1.41)

Breaking Down the Formula

TermWhat It MeasuresEffect
W(v)/N(v)Estimated win rate (exploitation term)Prefer nodes with higher win rate
√(ln(N_p)/N(v))Uncertainty bonus (exploration term)Large when node is rarely visited; decreases as N(v) grows
CExploration constantControls the explore/exploit balance. Higher C = more exploration
Worked Example

Parent has N=100 visits. Two children: A has W=70, N=80; B has W=5, N=5. C=√2≈1.41.
UCT(A) = 70/80 + 1.41×√(ln(100)/80) = 0.875 + 1.41×0.239 = 0.875 + 0.337 = 1.21
UCT(B) = 5/5 + 1.41×√(ln(100)/5) = 1.0 + 1.41×0.959 = 1.0 + 1.35 = 2.35
Despite B having a perfect win rate but only 5 visits, its high uncertainty bonus (1.35) makes it worth exploring more. MCTS selects B next.

The Initial Position

It's X's turn on an empty board. MCTS must decide which of the 9 cells to play. The tree starts with just the root node (the empty board).

Figure 7.1 — Starting position: empty Tic-Tac-Toe board. Cells numbered 1–9.

Iteration 1: First Exploration

Select: Root has no visited children → select root for expansion.
Expand: Add child node for playing cell 5 (center — a reasonable first move).
Rollout: Play randomly from cell 5. Result: X wins!
Backpropagate: Root: 1/1. Node-5: 1/1.

Iteration 2: Second Exploration

Select: Root still has 8 unvisited children → expand another one.
Expand: Add child node for playing cell 1 (corner).
Rollout: Random play from cell 1. Result: X loses.
Backpropagate: Root: 1/2. Node-5: 1/1. Node-1: 0/1.

Iteration 3: Selection Begins

Select: UCT(node-5) = 1/1 + C×√(ln(2)/1) = 1.0 + 1.17 = 2.17. UCT(node-1) = 0 + C×√(ln(2)/1) = 1.17. UCT of 7 unvisited nodes = ∞ (never visited). Select one unvisited node (e.g., cell 9).
Expand: Add child for cell 9 (corner).
Rollout: Random play from cell 9. Result: Draw (0.5).
Backpropagate: Root: 1.5/3. Node-9: 0.5/1.

Figure 7.2 — Tree after 3 iterations. Cell 5 currently has the best win rate (100%). After thousands of iterations, the statistics converge and MCTS picks the best first move.

After thousands of iterations, MCTS would converge on playing the center (cell 5) as the best first move in Tic-Tac-Toe — which matches optimal play theory.

CHAPTER EIGHT
Implementation

Python Implementation

Node Class

import math
import random

class MCTSNode:
    """Represents one node in the MCTS tree."""

    def __init__(self, state, parent=None, move=None):
        self.state     = state          # Game state at this node
        self.parent    = parent         # Parent node (None for root)
        self.move      = move           # Move that led here from parent
        self.children  = []             # Expanded children
        self.wins      = 0.0           # W(v) — total result
        self.visits    = 0             # N(v) — visit count
        self.untried_moves = state.legal_moves()  # Not yet expanded

    def ucb1(self, c=1.41421):
        """Upper Confidence Bound score for this node."""
        if self.visits == 0:
            return float('inf')   # Unvisited nodes always explored first
        exploitation = self.wins / self.visits
        exploration  = c * math.sqrt(math.log(self.parent.visits) / self.visits)
        return exploitation + exploration

    def best_child(self, c=1.41421):
        """Return child with highest UCB1 score."""
        return max(self.children, key=lambda child: child.ucb1(c))

    def expand(self):
        """Add one unexplored child and return it."""
        move  = self.untried_moves.pop()         # Pick one unexplored move
        child = MCTSNode(
            state  = self.state.apply_move(move),
            parent = self,
            move   = move
        )
        self.children.append(child)
        return child

    def is_fully_expanded(self):
        return len(self.untried_moves) == 0

    def is_terminal(self):
        return self.state.is_terminal()

MCTS Main Loop

def mcts(root_state, n_iterations=1000):
    """Run MCTS for n_iterations and return the best move."""
    root = MCTSNode(root_state)

    for _ in range(n_iterations):

        # ── STEP 1: SELECTION ─────────────────────────────────────
        node = root
        while node.is_fully_expanded() and not node.is_terminal():
            node = node.best_child()    # Navigate by UCB1

        # ── STEP 2: EXPANSION ─────────────────────────────────────
        if not node.is_terminal() and not node.is_fully_expanded():
            node = node.expand()        # Add one new child

        # ── STEP 3: SIMULATION (ROLLOUT) ──────────────────────────
        result = rollout(node.state)    # Random game to terminal

        # ── STEP 4: BACKPROPAGATION ───────────────────────────────
        while node is not None:
            node.visits += 1
            node.wins   += result
            result       = 1 - result   # Flip for alternating players
            node         = node.parent

    # After all iterations, pick most-visited child of root
    best = max(root.children, key=lambda c: c.visits)
    return best.move


def rollout(state):
    """Play randomly from state until terminal. Return result."""
    s = state.clone()
    while not s.is_terminal():
        move = random.choice(s.legal_moves())
        s    = s.apply_move(move)
    return s.result()   # e.g., 1.0=win, 0.0=loss, 0.5=draw

Key Design Note

The State object needs four methods: legal_moves(), apply_move(m), is_terminal(), and result(). MCTS itself is completely game-agnostic — only the state object knows the rules. This makes MCTS extremely reusable across different games and planning problems.

CHAPTER NINE
Advanced Topics

Variations & Enhancements

Progressive Widening

Standard MCTS expands one child per iteration. Progressive Widening limits the number of children added as a function of visit count, useful when the branching factor is very large (e.g., continuous action spaces). New children are added only when the current node has been visited enough times.

RAVE (Rapid Action Value Estimation)

RAVE improves sample efficiency by sharing statistics across related nodes. If move m has been played well in any simulation starting from a node's subtree, its estimated value is shared with all nodes where move m is legal — providing faster convergence early in search.

UCT-RAVE(v) = (1−β) × Q(v) + β × Q̃(v)
β = blending weight (decreases as N(v) grows)
Q(v) = standard win rate
Q̃(v) = AMAF (all-moves-as-first) win rate estimate

Parallel MCTS

MCTS is naturally parallelisable. Three main strategies:

StrategyHow It WorksTradeoff
Leaf ParallelismMultiple threads run rollouts from the same leaf simultaneouslyEasy to implement; limited speedup
Root ParallelismMultiple independent trees from root; results merged at endNo synchronisation needed; some redundancy
Tree ParallelismMultiple threads share a single tree with virtual lossesMost efficient; requires careful locking

Transposition Tables

Different move sequences can reach the same game state. A transposition table (hash map from state → node) prevents the tree from maintaining duplicate nodes for equivalent positions, saving memory and sharing statistics across paths.

CHAPTER TEN
Modern AI

AlphaGo and Neural MCTS

The AlphaGo Breakthrough

Go was considered the hardest board game for AI — its branching factor (~250) made pure tree search hopeless. AlphaGo (DeepMind, 2016) combined MCTS with deep neural networks to beat world champion Lee Sedol, a historic milestone in AI.

Two Neural Networks Replace Random Rollouts

AlphaGo replaced the two weakest parts of vanilla MCTS:

Policy Network π(s,a)
Replaces Selection Policy

Instead of UCB1 alone, a CNN trained on human games assigns prior probabilities to each move. High-probability moves get explored more, focusing search on expert-like moves.

Value Network V(s)
Replaces Random Rollout

Instead of playing a full random game (which takes time and is low quality), a CNN directly estimates the win probability from any board position. Much faster and more accurate.

Figure 10.1 — AlphaGo replaces the random rollout with a Value Network and improves selection with a Policy Network, dramatically increasing search quality.

AlphaZero: Self-Play Training

AlphaZero (2017) went further: it trains both networks entirely through self-play — starting from random weights and playing millions of games against itself. No human game data is needed. This approach generalised to Chess, Shogi, and Go, reaching superhuman level in each game in a matter of hours.

The cycle: MCTS improves play quality → better play generates better training data → better training data improves the networks → better networks improve MCTS. A virtuous cycle.

MCTS in Modern LLMs

OpenAI's o1 and o3 models use MCTS-like tree search over reasoning chains at inference time. The model generates multiple "thinking steps" as tree branches, scores them with a value model (analogous to the value network in AlphaGo), and selects the best reasoning path. This is why these models use more compute at inference time and solve harder problems.

The Anytime Property

MCTS has an important practical advantage: it's an anytime algorithm. After any number of iterations, you can ask for the current best move and get a reasonable answer. Give it more time → better answer. This makes it ideal for real-time applications with variable time budgets (e.g., chess clocks, real-time games).
