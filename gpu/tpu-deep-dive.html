What Is a TPU?

Google's custom ASIC designed specifically for neural network matrix operations

A Tensor Processing Unit (TPU) is an application-specific integrated circuit (ASIC) designed by Google specifically for machine learning workloads. Unlike GPUs, which are general-purpose parallel processors adapted for ML, TPUs are built from the ground up with one purpose: performing the matrix multiplications that dominate neural network compute as efficiently as possible.

The first TPU was deployed internally at Google in 2015 and announced publicly in 2016. Google built it because they projected that if all voice searches used deep neural networks, they would need to double their data centre capacity. A custom chip designed for this exact workload was far more efficient than off-the-shelf GPUs.

ASIC vs GPU: the key trade-off

  • GPU: general-purpose parallel processor. Programmable with CUDA. Runs any parallel workload. Good flexibility but some hardware is wasted on features irrelevant to ML.
  • TPU: fixed-function ASIC. Only does matrix multiply and a small set of activation functions. Almost no hardware wasted. Very high efficiency for exactly the ML ops it supports.
Key Distinction

TPUs are domain-specific accelerators (DSAs) — hardware designed for a narrow domain (matrix math for ML). The trade-off: far more efficient than a GPU for supported ops, but cannot run arbitrary CUDA kernels. You must compile your model through XLA, which translates to TPU instruction sequences. If your model uses ops not supported by XLA, it may run on the CPU fallback path with poor performance.

Systolic Array — the Core Innovation

How a grid of simple multiplier-accumulator cells achieves extreme GEMM throughput

The heart of a TPU is a systolic array — a 2D grid of multiply-accumulator (MAC) cells. Data flows through the array like a wave: weights are preloaded into each cell, and input activations stream in from the left while partial sums accumulate downward. This achieves very high utilisation of all MAC units simultaneously.

The name "systolic" comes from the Greek word for heartbeat — data pulses rhythmically through the array, just as blood pulses through the heart.

Systolic Array for Matrix Multiply C = A × B (simplified 4×4 example; real TPU v4 uses 128×128 = 16,384 MACs) Weights preloaded (B matrix) — one weight per cell: col0 col1 col2 col3 row0 [b00] [b01] [b02] [b03] ← each cell holds one weight permanently row1 [b10] [b11] [b12] [b13] row2 [b20] [b21] [b22] [b23] row3 [b30] [b31] [b32] [b33] Activations stream in from the left (one row of A per cycle): Cycle 1: [a00, a01, a02, a03] → row 0 accumulates C[0,:] Cycle 2: [a10, a11, a12, a13] → row 1 accumulates C[1,:] ... Every cell: multiply input by stored weight, add to running sum, pass to right. All 16,384 MACs fire every clock cycle → maximum utilisation. No memory access during computation — weights cached in cells.
⚡

Why Systolic Arrays Are Efficient

Each MAC cell only needs to receive one value per cycle from its left and top neighbours, then send one value right and down. No shared bus, no contention, no memory bandwidth wasted on weights during computation.

  • Each weight is loaded once, reused across many input activations
  • No on-chip memory bandwidth needed for weight reads during matmul
  • Fully pipelined: new inputs every cycle, outputs every cycle
  • Power: ~4× more energy efficient than GPU for GEMM
📏

Limitations of Systolic Arrays

The fixed data-flow pattern that makes systolic arrays efficient also constrains what operations they can run fast. Irregular operations, dynamic shapes, and sparse computation are all harder on TPUs than on GPUs.

  • Fixed-size tiles: operations must fit 128×128 blocks
  • Padding overhead for non-multiples of 128
  • Sparse tensors: no hardware sparse support (wastes MACs on zeros)
  • Dynamic shapes: XLA recompiles, breaking the static computation graph

TPU Architecture

MXU, VPU, HBM, and the scalar/vector/matrix execution units

TPU v4 Core Architecture TPU v4 Chip ├── MXU (Matrix Multiply Unit) │ ├── Systolic Array: 128 × 128 = 16,384 MAC cells ← BF16 accumulate to FP32 │ └── Weight FIFO: preloads weights ahead of computation │ ├── VPU (Vector Processing Unit) │ ├── 128-wide SIMD vector lanes │ └── Handles: activation functions, LayerNorm, softmax, element-wise ops │ ├── Scalar Unit │ └── Control flow, address computation, loop management │ ├── On-chip SRAM (Unified Buffer) 16–32 MB — feeds MXU and VPU │ ├── HBM (High Bandwidth Memory) 32 GB @ 1.2 TB/s (v4) │ └── Inter-Chip Interconnect (ICI) high-speed links to adjacent chips in pod Total FLOPS (v4 BF16): ~275 TFLOPS per chip Peak systolic throughput: 16,384 MACs × 2 ops × clock freq

Matrix Multiply Unit (MXU)

The MXU contains the systolic array and is the primary compute engine. It accepts weights pre-loaded into a Weight FIFO and processes activation matrices streaming from the Unified Buffer. It outputs results in FP32 accumulation format. The MXU operates in BF16 (Brain Float 16) — the format Google pioneered and which is now standard across the industry.

Vector Processing Unit (VPU)

Not everything in a neural network is a matrix multiply. Activation functions (GELU, ReLU, sigmoid), normalisation (LayerNorm, RMSNorm), and element-wise operations are handled by the VPU — a wide SIMD unit that operates on vectors of 128 BF16/FP32 values simultaneously.

Memory & Interconnect

HBM, Unified Buffer, Inter-Chip Interconnect, and TPU Pods

Unified Buffer (On-chip SRAM)

The Unified Buffer is a large on-chip SRAM (~16–32 MB per chip) that sits between HBM and the MXU/VPU. It is the TPU equivalent of GPU shared memory, but much larger and centrally managed by the XLA compiler rather than the programmer. The compiler analyses the computation graph and schedules which tensors to prefetch into the Unified Buffer ahead of time, hiding HBM latency.

Inter-Chip Interconnect (ICI)

TPU chips are connected to each other via dedicated high-speed ICI links (up to 4 links per chip in a 2D torus topology for TPU v4). This is fundamentally different from GPU clusters which use InfiniBand through a network switch. TPU pods form a low-latency, high-bandwidth mesh — critical for tensor and model parallelism at the pod scale.

TPU v4 Pod — 4096 chips in a 3D torus Chip ←ICI→ Chip ←ICI→ Chip ←ICI→ Chip (X dimension) ↑ICI ↑ICI Chip Chip Chip Chip (Y dimension) ↑ICI ↑ICI Chip Chip Chip Chip ... (Z dimension wraps around) 4096 chips × 275 TFLOPS = ~1.1 PFLOPS (BF16) All-to-all bandwidth: ~2 PB/s within the pod No separate InfiniBand network needed — ICI is the interconnect vs GPU Cluster: GPU ←NVLink→ GPU (8 per node) ↑PCIe ↑PCIe InfiniBand switch ←→ InfiniBand switch Inter-node bandwidth limited to InfiniBand (~200–400 Gbps per GPU)

Gemini Training

Gemini Ultra was trained on TPU v5p pods — 8960 chips connected in a high-bandwidth 3D torus, with a combined ~4 EFLOPS of BF16 compute. The Chinchilla-optimal training required running for months on this infrastructure. The seamless scaling from a single chip to a 8960-chip pod — without needing InfiniBand — is a key advantage of the TPU ICI mesh design.

Programming TPUs with JAX & XLA

How code written in Python becomes TPU instructions

TPUs are programmed through JAX + XLA (Accelerated Linear Algebra). JAX provides NumPy-like Python APIs with automatic differentiation; XLA is the compiler that transforms JAX programs into efficient TPU (and GPU) instruction sequences.

Unlike CUDA (which requires explicit kernel writing), XLA operates at the level of computation graphs — you describe what to compute, and XLA decides how to map it to hardware optimally, including operator fusion, memory layout optimisation, and systolic array tile scheduling.

Python · JAX on TPU — key patterns
import jax
import jax.numpy as jnp

# Check available devices
print(jax.devices())           # [TpuDevice(id=0), TpuDevice(id=1), ...]
print(jax.device_count())      # 8 (for a v4-8 TPU slice)

# JIT compilation — XLA compiles and caches the first call
@jax.jit
def matmul_relu(A, B):
    return jnp.maximum(0, A @ B)   # XLA fuses matmul + relu into one kernel

A = jnp.ones((4096, 4096), dtype=jnp.bfloat16)
B = jnp.ones((4096, 4096), dtype=jnp.bfloat16)
C = matmul_relu(A, B)   # first call: JIT compile + run. Subsequent: fast.

# Automatic differentiation (autograd)
def loss_fn(W, x, y):
    pred = jax.nn.sigmoid(x @ W)
    return jnp.mean((pred - y) ** 2)

grad_fn = jax.grad(loss_fn)   # automatic gradient of loss_fn w.r.t. W
W = jnp.zeros((128, 1))
grads = grad_fn(W, x_data, y_data)   # ∂loss/∂W

# Multi-device: pmap for data parallelism across all TPU cores
@jax.pmap
def parallel_forward(x):
    return model(x)   # each device gets a shard of the batch

# Sharded arrays (shard_map) for tensor parallelism (JAX 0.4+)
from jax.experimental.shard_map import shard_map
from jax.sharding import Mesh, PartitionSpec

mesh = Mesh(jax.devices(), axis_names=('data',))
# Partition along batch dimension across all devices

XLA Compilation Pipeline

  • HLO (High-Level Operations) — JAX transforms your Python code to a graph of HLO operations (add, dot, reduce, etc.)
  • Optimisation passes — XLA applies operator fusion, constant folding, algebraic simplification, and layout assignment
  • Tiling & scheduling — XLA chooses tile sizes for the systolic array, schedules memory prefetches, and pipelines computation with data movement
  • LLO (Low-Level Operations) — Architecture-specific instruction selection for TPU or GPU

When to Use GPU vs TPU

Practical guidance for choosing the right accelerator

⚡

Choose GPU When

GPUs are the default choice for most ML practitioners due to the breadth of software support, availability, and flexibility.

  • Using PyTorch (dominant research framework)
  • Need HuggingFace, vLLM, or CUDA-native libraries
  • Prototyping — fast iteration with CUDA debuggers
  • Running diverse workloads beyond matrix multiply
  • Inference serving (vLLM, TensorRT, ONNX Runtime)
  • Not locked into Google Cloud
🧊

Choose TPU When

TPUs shine for large-scale, static-shape training jobs that run for weeks on Google Cloud, especially with JAX-native frameworks.

  • Training at pod scale (1000+ chips) on Google Cloud
  • Using JAX + Flax / MaxText natively
  • Fixed-shape training (no dynamic sequences)
  • Replicating Google's model training (PaLM, Gemini, T5)
  • Cost-sensitive long training runs (TPU v5e)
  • Google's own production ML infrastructure
Chapter Summary & Series Summary

GPU: general-purpose parallel processor; CUDA ecosystem; Tensor Cores for GEMM; NVLink + InfiniBand for multi-GPU. Best for flexibility and the broadest software ecosystem. TPU: purpose-built ASIC; systolic array for GEMM; XLA compiler; ICI pod topology. Best for large-scale static training on Google Cloud with JAX. In practice, the most important frontier model companies use both: Google trains on TPUs, most others (OpenAI, Anthropic, Meta) train on NVIDIA GPUs. Understanding both architectures is essential for any serious ML systems engineer.
