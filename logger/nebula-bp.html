01

🔭 Overview

Nebula Logger should be treated as an enterprise observability platform, not a replacement for System.debug(). It captures meaningful business and operational events while minimising noise and storage consumption.

Goal Capture meaningful business and operational events while minimising noise and storage consumption. Every log entry should answer: what happened, why it matters, and what action (if any) is required.
📊

Observability

  • Business-process tracing
  • Integration health monitoring
  • Performance baselines
  • Audit trails
⚙️

Where to Use It

  • Apex triggers & classes
  • Flows & Process Builders
  • LWC components
  • Integration services
🚫

When NOT to Use It

  • Temporary development traces
  • Verbose loop iterations
  • Pure debug noise
  • Already-handled info events
02

🎚️ Recommended Logging Levels

Set logging levels per environment through Nebula Logger's custom metadata. Misconfigured levels are the most common cause of runaway storage costs.

Environment Recommended Level Rationale
Production WARN / ERROR Minimise storage; capture actionable events only
UAT INFO Business-flow visibility during acceptance testing
QA DEBUG Detailed tracing for testers
Developer Sandbox FINE / FINEST Maximum verbosity during active development
⚠️ Warning Avoid DEBUG or lower globally in Production. Even when troubleshooting, enable verbose logging only for targeted users or transactions, and revert immediately.
03

🏗️ Structured Logging

Use scenarios, tags, and related records to create logs that are searchable, reportable, and meaningful to operations teams without reading source code. 

// Set the business scenario for every entry in this transaction
Logger.setScenario('OrderFulfillment');

Logger.error('Payment Failed')
    .addTag('Payments')
    .addTag('Stripe')
    .setRecord(orderRecord);

Logger.info('Order Created')
    .setScenario('OrderFulfillment')
    .addTag('Orders')
    .setRecord(newOrder);

Logger.saveLog();

Severity Tags

Tag Best Practices

  • Use a defined taxonomy — no free text
  • Apply at least one domain tag per entry
  • Apply severity tags to ERROR entries
  • Review tag list quarterly
04

🔐 Security Guidelines

❌ Never Log Passwords · OAuth tokens · Session IDs · Credit card data · SSNs · HIPAA PHI · Encryption keys · Any PII not strictly necessary for debugging.

✅ Good

// Log the event, not the secret
Logger.warn('Token validation failed for user');

Logger.error('Payment declined', ex)
    .addTag('Payments')
    .setRecord(orderRecord);

❌ Bad

// Never expose credentials in log messages
Logger.warn('Token=' + accessToken);

Logger.error('CC=' + cardNumber
    + ' CVV=' + cvv);

Additional Security Controls

Data Masking

  • Mask fields in your wrapper before passing to Nebula
  • Use Nebula's field-level masking metadata where available

Access Control

  • Restrict Log__c visibility to Ops & Support profiles
  • Do not expose log records in community or Experience portals

Compliance Review

  • Annual review of logged fields against GDPR / HIPAA requirements
  • Document data classes stored in logs for your DPA register
05

🗄️ Retention & Purging

Unbounded log growth is one of the most common causes of Salesforce data-storage overages. Define explicit retention windows per environment and automate purge jobs from day one.

Environment Retention Window Purge Mechanism
Production 30–60 Days Scheduled Apex job + Nebula Purge Plugin
UAT 30 Days Scheduled Apex job
Sandbox 90 Days Manual or scheduled Apex job
💡 Recommended Purge Setup Schedule LogBatchPurger to run nightly. Monitor Log__c record counts via a simple Flow-based alert when row count exceeds your threshold.

Storage Monitoring Checklist

  1. Create a dashboard tile for Log__c record count
  2. Set a Flow alert at 80% of your target retention ceiling
  3. Review purge-job logs weekly for the first month post-go-live
  4. Document your retention policy in your org's data dictionary
06

🔔 Alerting Strategy

Nebula Logger integrates with Platform Events to trigger real-time notifications. Use this pipeline for every P1/P2 ERROR entry.

ERROR log saved → Platform Event fired → Flow subscription → Slack / Teams / Email
🚨

Alert On

  • Unhandled exceptions
  • Integration failures
  • Authentication failures
  • Payment processing failures
  • Batch job failures
  • P1 / P2 tagged entries
🔕

Do Not Alert On

  • INFO or DEBUG messages
  • Minor / expected warnings
  • Retry attempts before final failure
  • Non-actionable system noise

Alert Quality Rules

  • Every alert must be actionable
  • Include record ID and scenario in the message
  • Route by tag to the right team channel
  • Review & tune alert thresholds monthly
07

📦 Logging Wrapper Pattern

Wrap Nebula Logger in a centralised class so all callers receive consistent tagging, masking, and saveLog() behaviour. This also isolates upgrade effort to a single file. 

public class AppLogger {

    /**
     * Log an error with the related SObject record.
     * All enterprise error logs are tagged and saved immediately.
     */
    public static void error(
        String message,
        Exception ex,
        SObject record
    ) {
        Logger.error(message, ex)
            .setRecord(record)
            .addTag('Enterprise')
            .addTag('P1');

        Logger.saveLog();
    }

    public static void warn(String message) {
        Logger.warn(message)
            .addTag('Enterprise');

        Logger.saveLog();
    }

    public static void info(String scenario, String message, SObject record) {
        Logger.setScenario(scenario);
        Logger.info(message)
            .setRecord(record)
            .addTag('Enterprise');

        Logger.saveLog();
    }
}
Benefits of the Wrapper Pattern Centralised masking · Consistent tagging · Easier framework upgrades · Single saveLog() point · Simplified mocking in unit tests.

Usage

// From any Apex class or trigger
try {
    processOrder(order);
    AppLogger.info('OrderFulfillment', 'Order processed', order);
} catch (Exception ex) {
    AppLogger.error('Order processing failed', ex, order);
}

📋 Enterprise Logging Standards

Category Production Standard Status
Default Log Level WARN Mandatory
Alert Threshold ERROR Mandatory
Retention Policy 30–60 Days Mandatory
Tagging At least one domain tag per entry Mandatory
Automated Purging Nightly scheduled job Mandatory
Wrapper Class All callers use AppLogger Recommended
Sensitive Data Zero PII / credentials in log messages Mandatory
✅ Key Takeaway Production logs should answer four questions:
  • What business process started?
  • What business process completed?
  • What failed — and why?
  • What action is required?
If a log entry cannot help answer one of these questions, it probably does not belong in Production.
