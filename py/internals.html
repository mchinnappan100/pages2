Basic Concepts

🏗️
01

Architecture of the Python Language

Python is a specification — a language grammar and semantics. CPython is the reference C implementation most people mean when they say "Python."

  • Language spec vs implementation (CPython, PyPy, Jython, IronPython, GraalPy)
  • Front end: tokenizer → parser → AST
  • Compiler: AST → code objects containing bytecode
  • Runtime: the eval loop executes bytecode against objects
See Example

A single line of Python passes through four distinct stages before anything actually runs:

source.py           "x = a + b"
  ↓ tokenizer
tokens               NAME('x') OP('=') NAME('a') OP('+') NAME('b')
  ↓ parser
AST                  Assign(target=x, value=BinOp(a, Add, b))
  ↓ compiler
code object          bytecode: LOAD_NAME a, LOAD_NAME b, BINARY_ADD, STORE_NAME x
  ↓ ceval.c
result                the interpreter's eval loop executes each instruction

Everything past the compiler step is just the eval loop walking a bytecode array — no more "parsing" happens at run time.

🐍
02

CPython

CPython is a C program that implements a stack-based bytecode interpreter, plus the standard library and C-API that everything else builds on.

  • Written in portable C, distributed as the reference `python` binary
  • Compiles source to .pyc bytecode cached under __pycache__
  • Ships the C-API (Python.h) used by every native extension
  • Uses reference counting + a cyclic garbage collector for memory
See Example

You can inspect the compiled bytecode CPython actually executes with the built-in dis module:

import dis
def add(a, b):
    return a + b

dis.dis(add)

#  2   0 LOAD_FAST     a
#      2 LOAD_FAST     b
#      4 BINARY_ADD
#      6 RETURN_VALUE

This bytecode is what the CPython eval loop (ceval.c) actually walks through, instruction by instruction, at runtime.

⚙️
03

Bytecode & the Eval Loop

The heart of CPython is a single giant C function — the eval loop — that repeatedly fetches, dispatches, and executes bytecode instructions.

  • Stack-based virtual machine — most ops push/pop a value stack
  • Each frame holds its own bytecode, locals, and stack
  • Since 3.11, the "specializing adaptive interpreter" caches faster paths per instruction
  • No JIT in stock CPython — every call re-walks bytecode (3.13 adds an experimental JIT)
See Example

A simplified sketch of the dispatch loop CPython runs internally:

for (;;) {
    opcode = *next_instr++;
    switch (opcode) {
        case LOAD_FAST:   PUSH(locals[oparg]); break;
        case BINARY_ADD:  {
            right = POP(); left = POP();
            PUSH(PyNumber_Add(left, right));
            break;
        }
        case RETURN_VALUE: return POP();
        /* ~160 more opcodes ... */
    }
}
📦
04

Objects & Reference Counting

In CPython, everything is a PyObject* — an int, a function, a class, a module — all sharing a common C header.

  • Every object header carries ob_refcnt and ob_type
  • Assignment increments a refcount; going out of scope decrements it
  • Refcount hitting zero frees the object immediately (deterministic-ish)
  • A separate generational GC handles reference cycles refcounting can't
See Example 
import sys
a = [1, 2, 3]
print(sys.getrefcount(a))   # 2 (a itself + the temp arg to getrefcount)

b = a                        # refcount +1
print(sys.getrefcount(a))    # 3

del b                        # refcount -1

Circular references (e.g. two objects pointing at each other) can never reach zero via refcounting alone — that's what the cyclic collector in the gc module cleans up periodically.

🧠
05

Memory Management

CPython manages small-object allocation itself via pymalloc, layered on top of the system allocator.

  • Objects < 512 bytes are served from pymalloc's pool/arena allocator
  • Common immutable objects (small ints -5..256, interned strings) are cached and reused
  • The generational GC has 3 generations — young objects are collected most often
  • PEP 703 (3.13+) introduces an optional build without reference-count locking (free-threaded / "no-GIL")

Concurrency & the GIL

🧵
06

Threads vs Processes in Python

Threads share one memory space and one interpreter; processes each get their own memory space and their own interpreter (and GIL).

  • threading — OS-level threads, but bound by one GIL per process
  • multiprocessing — separate processes, separate GILs, real parallel CPU work
  • Threads: great for I/O-bound waiting (network, disk); poor for CPU-bound math
  • Processes: true parallelism, but pay for pickling/IPC to move data around
See Example

Same CPU-bound workload, two approaches — only one actually parallelizes:

from threading import Thread
from multiprocessing import Process

def burn():
    x = 0
    for i in range(50_000_000):
        x += i

# Threads: contend for the same GIL, run ~sequentially on CPU work
threads = [Thread(target=burn) for _ in range(4)]

# Processes: 4 independent interpreters, 4 independent GILs
procs = [Process(target=burn) for _ in range(4)]

On an 8-core machine the process version finishes roughly 4x faster; the thread version finishes about the same as running burn() once, sometimes slower due to GIL handoff overhead.

🔒
07

The Global Interpreter Lock (GIL)

A single mutex inside CPython ensures only one thread executes Python bytecode at a time — even on a multi-core machine.

  • Exists mainly to keep CPython's refcounting simple and thread-safe
  • CPython periodically forces a thread switch (every ~5ms by default, or on bytecode-count intervals in older versions)
  • I/O and blocking C calls release the GIL — that's why threaded I/O still scales
  • PEP 703 lets you build a "free-threaded" CPython (3.13+) that removes the GIL, at the cost of single-thread speed and extension compatibility
See Example

The GIL only guards Python bytecode. Any C extension that does its own heavy lifting can explicitly release it:

/* inside a C extension function */
static PyObject* slow_native_call(PyObject *self, PyObject *args) {
    Py_BEGIN_ALLOW_THREADS
    /* pure C / C++ work here runs with the GIL released,
       so other Python threads can run concurrently   */
    do_expensive_c_computation();
    Py_END_ALLOW_THREADS
    Py_RETURN_NONE;
}

This is exactly why NumPy, PyTorch, and similar libraries can appear to "beat" the GIL — the actual number-crunching happens in C/C++/CUDA with the GIL dropped, and only re-acquires it to hand the result back as a Python object.

🧩
08

Multiprocessing & Shared Memory

Sidestepping the GIL entirely by running separate OS processes, each with its own interpreter and heap.

  • Data crosses process boundaries via pickling — has real serialization cost
  • multiprocessing.shared_memory avoids copies for large arrays
  • ProcessPoolExecutor gives a simple map/submit API over a worker pool
  • Fork vs spawn start methods change what state child processes inherit
🔁
09

asyncio & the Event Loop

A third concurrency model: single-threaded cooperative multitasking, ideal for many concurrent I/O-bound tasks.

  • Coroutines yield control at await points instead of using OS thread switches
  • Still runs on one thread, one GIL — CPU-bound code still blocks the whole loop
  • Combine with run_in_executor to offload blocking/CPU work to a thread or process pool
  • Great for thousands of concurrent sockets; bad for math-heavy loops

10

How NumPy, JAX & PyTorch Call Into C/C++

These libraries are thin Python wrappers around compiled C/C++/CUDA kernels — Python mostly orchestrates, the native code does the math.

  • NumPy's ndarray is a C struct: a raw data pointer, shape, strides, dtype
  • The buffer protocol (PEP 3118) lets libraries share that memory with zero copy
  • PyTorch's tensors dispatch through ATen (C++), which routes to CPU kernels or CUDA
  • JAX traces Python into XLA HLO, then compiles and runs that on CPU/GPU/TPU — Python barely runs at all during the hot loop
See Example

Conceptually, a + b on two NumPy arrays skips the Python eval loop almost entirely:

import numpy as np
a = np.arange(10_000_000, dtype=np.float64)
b = np.arange(10_000_000, dtype=np.float64)
c = a + b
# Python:  looks up __add__ on the ndarray type
# C:       ufunc dispatch picks a compiled loop for (float64, float64) -> float64
# C loop:  a tight, vectorized C for-loop writes directly into c's buffer
# Python:  wraps the resulting C buffer back into an ndarray object

Only the first and last steps touch the Python interpreter — the actual addition is a compiled C loop operating on contiguous memory, often using SIMD instructions.

11

Writing a C Extension

The most direct route: use Python.h and the CPython C-API to build a module that Python can import like any other.

  • Define PyObject* functions matching CPython's calling convention
  • Register them in a PyMethodDef table and a PyModuleDef
  • Compile to a shared library (.so/.pyd) that import loads directly
  • Tools like Cython or pybind11 generate this boilerplate for you
See Example

A minimal C extension exposing one function, fastmath.add(a, b):

#include <Python.h>

static PyObject* add(PyObject *self, PyObject *args) {
    long a, b;
    if (!PyArg_ParseTuple(args, "ll", &a, &b))
        return NULL;
    return PyLong_FromLong(a + b);
}

static PyMethodDef Methods[] = {
    {"add", add, METH_VARARGS, "Add two integers"},
    {NULL, NULL, 0, NULL}
};

static struct PyModuleDef fastmathmodule = {
    PyModuleDef_HEAD_INIT, "fastmath", NULL, -1, Methods
};

PyMODINIT_FUNC PyInit_fastmath(void) {
    return PyModule_Create(&fastmathmodule);
}
# build it with setuptools, then:
import fastmath
fastmath.add(2, 3)   # -> 5, executed entirely in compiled C

12

Calling Rust from Python (PyO3)

PyO3 + maturin let you write a Rust crate, compile it as a native module, and import it from Python with no C boilerplate.

  • #[pyfunction] / #[pymodule] macros generate the CPython glue for you
  • maturin develop builds and installs the module into your venv directly
  • Rust's memory safety avoids a whole class of native-extension segfault bugs
  • Release the GIL with py.allow_threads(|| ...) for CPU-heavy Rust code
See Example 
// src/lib.rs
use pyo3::prelude::*;

#[pyfunction]
fn add(a: i64, b: i64) -> i64 {
    a + b
}

#[pymodule]
fn fastmath_rs(_py: Python, m: &PyModule) -> PyResult<()> {
    m.add_function(wrap_pyfunction!(add, m)?)?;
    Ok(())
}
# Cargo.toml
[lib]
crate-type = ["cdylib"]
[dependencies]
pyo3 = { version = "0.22", features = ["extension-module"] }

# then:
$ maturin develop
>>> import fastmath_rs
>>> fastmath_rs.add(2, 3)
5

13

Calling Go from Python (cgo + ctypes/cffi)

Go can't plug into the CPython C-API directly, but it can export a plain C ABI via cgo — which Python then loads with ctypes or cffi.

  • Build the Go code as a C shared library: go build -buildmode=c-shared
  • //export FuncName comments mark which Go functions get a C symbol
  • Python loads the resulting .so with ctypes.CDLL(...)
  • Only C-compatible types cross the boundary — no Go slices/maps/strings directly
See Example 
// add.go
package main

import "C"

//export Add
func Add(a, b C.long) C.long {
    return a + b
}

func main() {}
$ go build -buildmode=c-shared -o libadd.so add.go
# call it from Python
import ctypes
lib = ctypes.CDLL("./libadd.so")
lib.Add.argtypes = [ctypes.c_long, ctypes.c_long]
lib.Add.restype = ctypes.c_long
lib.Add(2, 3)   # -> 5
🔀
14

ctypes vs cffi vs the C-API

Three different ways to cross the Python/native boundary, trading off ease of use against control and performance.

  • ctypes — stdlib, no compilation needed, but manual type declarations and slower calls
  • cffi — nicer C-declaration parsing, works in both "ABI" and faster "API" mode
  • Raw C-API (Python.h) — fastest, full control, but the most boilerplate and platform-specific build steps
  • pybind11 / nanobind — C++ wrapper libraries that make C-API-level extensions much less painful
04 Best Practices

What Great Python Engineers Do

✓ Profile before you parallelize

Confirm whether your bottleneck is I/O-bound or CPU-bound before reaching for threads, processes, or async — each solves a different problem.

python -m cProfile -s cumtime script.py

✓ Use processes for CPU-bound parallelism

Threads won't help pure-Python CPU-bound work because of the GIL. Reach for multiprocessing or push the hot loop into a native extension instead.

✓ Vectorize instead of looping

A Python-level for loop over an array pays the interpreter overhead per element. Let NumPy/PyTorch's C loops do the iteration instead.

# slow: Python loop, one bytecode dispatch per element
total = 0
for x in big_array:
    total += x * 2

# fast: one vectorized C call for the whole array
total = (big_array * 2).sum()

✓ Release the GIL in native code

If you write a C/C++/Rust extension, explicitly drop the GIL around any long-running, pure-native section so other Python threads keep progressing.

✓ Prefer PyO3/pybind11 over raw C-API

Unless you need absolute minimal overhead, a binding library saves you from manual refcounting bugs and gives you far better error messages.

✓ Watch out for the fork/copy-on-write trap

On Linux, multiprocessing's default fork start method inherits parent memory via copy-on-write — great for speed, but be careful with open file handles, threads, and locks held at fork time.

✓ Measure GIL contention directly

Don't guess — use sys.setswitchinterval() experiments or the new free-threaded build's tooling to see whether the GIL is actually your bottleneck.

✓ Keep the Python/native boundary thin

Cross into C/Rust/Go once with a batch of data, do all the heavy work natively, and cross back once — many small calls across the boundary add up fast.

✓ Pin native extension ABI compatibility

Extensions compiled against one CPython version's C-API aren't guaranteed compatible with another unless they use the stable ABI (Py_LIMITED_API).

✓ Know your build targets before choosing Rust vs Go vs C

Rust (via PyO3) integrates most natively with CPython's object model; Go must go through a plain C ABI; raw C gives maximum control with maximum responsibility.
