Overview

This guide covers two common Copado pipeline architectures — one with an integration sandbox (DevMerge) for teams that need a shared merge point before QA, and a simplified direct-to-QA pattern for smaller teams or projects. Both enforce the same quality principles.

🔒
Safety by Default
Every promotion stage is guarded by automated quality checks before any human review gate is reached. Nothing skips the pipeline.
🔗
Full Traceability
Branch naming tied to User Stories gives end-to-end audit visibility from JIRA ticket to production deployment.
🔄
Environment Consistency
Back promotion keeps all environments in sync after production deployments, eliminating configuration drift between orgs.
⚡
Shift-Left Quality
PMD static analysis and coverage gates catch issues at the first stage — before they propagate to UAT or production.

Pipeline Use Cases

Choose the architecture that fits your team's size, complexity, and governance requirements. Both patterns share the same quality gate philosophy.

When to use this pattern: Multiple teams or workstreams committing to the same org simultaneously. An integration sandbox (DevMerge) acts as a shared buffer, catching cross-team conflicts before QA is involved.

Deployment Flow

The following steps apply to both pipeline patterns. Stage 3 (Integration Sandbox) is skipped in Use Case B.

1
Feature Development
Check out from main or release. Name your branch after the User Story (e.g., feature/US-0042). Develop Apex, metadata, and configuration changes locally. Keep branches short-lived — ideally within a single sprint.
2
Pull Request & Code Review
Open a PR targeting the next stage branch. Automated quality gates (PMD + coverage) must pass before the PR can be reviewed. A Dev Lead approval is required. Failed checks return the work item to the developer — the PR cannot merge until all checks are green.
3
Integration Sandbox (Use Case A only)
Copado Smart Merge attempts automated conflict resolution. If conflicts are detected, the developer resolves them manually. A successful merge triggers an immediate deployment to DevMerge. This stage exists purely to protect QA from integration noise.
4
QA Validation
Quality gates re-run against the QA org. On success, QA engineers perform functional and regression testing. Only after QA sign-off is the User Story marked Ready for UAT. Defects raised here create new feature branches — they do not delay the original story in the queue.
5
UAT & Business Acceptance
Change is promoted to UAT. Quality gates run again. Business stakeholders perform acceptance testing. Defects raised here are scoped and fixed in a new branch — they do not block other User Stories progressing through UAT independently.
6
Production Deployment
After UAT sign-off and Change Management approval, the change is promoted to production. Schedule during low-traffic windows. Monitor the deployment in the Copado Deployment Dashboard in real time. Trigger back promotion immediately after success.

Environment Strategy

Each sandbox has a single clear responsibility. Avoid using sandboxes for purposes outside their lane — this is the most common source of pipeline instability.

EnvironmentTypeResponsibilityDeployment Trigger
Developer Org SOURCE Feature development, unit testing Developer push
DevMerge Sandbox ENV Cross-team integration testing (Use Case A) Approved PR merge
QA Sandbox ENV Functional & regression testing Automatic (post DevMerge or PR)
UAT Sandbox ENV Business acceptance testing Manual promotion
Production PROD Live environment Controlled, change-managed
Hotfix path: Hotfixes branch from production, bypass DevMerge and QA, and target UAT → Prod directly. Back-promote the fix to all lower environments immediately after production deployment.

Quality Gates

Quality gates are applied at every promotion stage — not just the first. A failure at any gate halts the pipeline and returns work to the developer.

🔍 PMD Static Analysis
Scans for anti-patterns, security vulnerabilities, and performance issues. Zero tolerance for critical and high-severity findings. Warnings are tracked but do not block.
Critical: 0
High: 0
Violations → Block
🧪 Apex Test Coverage
Coverage must exceed 90% across all Apex classes included in the deployment. Enforced automatically by Copado — failing coverage halts the pipeline before any human gate is reached.
Coverage ≥ 90%
Coverage < 90% → Block
👤 Human Review
A Dev Lead must approve the PR. Automated checks are a prerequisite — not a replacement — for human judgment. One approving review required; two recommended for production-impacting changes.
Dev Lead approved
Requested changes → Block
# Gate execution sequence (every promotion stage)
[1] PMD static analysis          → zero critical violations required
[2] Apex test execution            → all tests pass
[3] Coverage measurement           → ≥ 90% across deployment scope
[4] Dev Lead PR approval           → at least one approver
[5] Merge / Promote                → triggers next stage deployment

Merge Strategy

Copado handles the majority of merges automatically. Manual intervention is only required when the same component lines have been modified by two branches simultaneously.

⚡ Copado Smart Merge (preferred)
Analyses component-level diffs and resolves non-overlapping changes automatically. Handles the majority of merges where two branches have touched different metadata components. No developer action required.
🔧 Manual Merge (fallback)
Used when two branches have modified the same lines of the same component. Pull both branches locally, use a three-way diff tool to resolve, run tests locally, then push the resolved branch and re-trigger the Copado deployment.
Recurring merge conflicts are a signal, not just a problem to resolve. If the same component conflicts repeatedly, the team needs to review component ownership, branching strategy, or feature decomposition.

Back Promotion

After every production deployment, lower environments must be synced to reflect the production baseline. Skipping back promotion is the leading cause of "works in QA, fails in prod" issues.

# Back promotion sequence (run after every prod deployment)
Prod ──▶ UAT ──▶ QA ──▶ DevMerge ──▶ Dev Orgs
When to run
After every production release — regardless of size. After hotfixes that bypass the standard path. Before the start of a new sprint to ensure all dev orgs start from the latest baseline.
Conflict handling
Back promotion may introduce conflicts if in-flight development has diverged from production. Resolve using the same process as forward merges. Conflicts during back promotion are expected — they are not a sign of a broken process.

Best Practices

Branching
  • Keep feature branches within a single sprint
  • Prefix branch names with the User Story ID
  • Delete merged branches promptly
  • Never commit directly to main
Quality
  • Write tests alongside the feature, not after
  • Never mock tests to hit the 90% threshold
  • Treat PMD warnings as early signals
  • Review coverage reports in every PR
Deployments
  • Schedule production during low-traffic windows
  • Run back promotion immediately after prod
  • Monitor via Copado Deployment Dashboard
  • Never re-deploy without root-cause analysis
Hotfixes
  • Branch hotfixes from prod only
  • Back-promote the fix immediately after prod
  • Retrospect on every hotfix
  • Limit concurrent hotfixes to one
Team Habits
  • Review Copado dashboard in standups
  • Rotate deployment ownership weekly
  • Document all pipeline changes in the repo
  • Conduct quarterly pipeline health reviews
Governance
  • Require Change Management for all prod deploys
  • Audit PR approvers quarterly
  • Keep environment count minimal
  • Review Copado license utilisation monthly

Manual Pre & Post-Deployment Steps

Not everything in a Salesforce org can be captured in metadata and deployed through a pipeline. Certain platform features, data assets, and configuration entities fall outside the Metadata API's coverage and must be handled manually before or after each promotion. Failing to account for these is one of the most common causes of post-deployment incidents.

Golden rule: If it cannot be source-controlled and deployed via Copado, it must appear in a documented manual step — logged in your change ticket, with an owner and a verification check. "It was already there" is not an audit trail.

PRE-DEPLOY Steps Before the Pipeline Runs

These actions must be completed in the target environment before Copado begins deploying. If the pipeline runs first, these items may cause deployment failures or leave the org in an inconsistent state.

Asset / EntityWhy It's ManualRequired ActionApplies To
Named Credentials & External Credentials Secrets and auth tokens are environment-specific and not stored in metadata Create or update Named Credentials in target org; configure auth parameters manually All environments
Custom Settings (protected / hierarchy) Protected Custom Settings and hierarchy-level values are not deployable via Metadata API Set required values directly in Setup → Custom Settings before deployment QA, UAT, Prod
Platform Encryption (Shield) Encryption policies and tenant secrets cannot be deployed; they are org-specific Enable encryption policies and rotate tenant secrets in the target org ahead of deployment UAT, Prod
OmniStudio / Vlocity DataPacks OmniScript, FlexCards, and Integration Procedures are deployed via DataPacks, not Metadata API Deploy DataPacks using the OmniStudio CLI or DataPack tool before Copado runs; verify activation status All environments
Salesforce Connect / External Data Sources Connection credentials and endpoint URLs differ per environment Create or update External Data Sources with environment-specific endpoints and credentials QA, UAT, Prod
Permission Set Assignments (data-dependent) Assignments to specific users are data, not metadata Assign permission sets to integration users and service accounts in target org before deployment All environments
Data Migration / Seed Data Reference data (picklist masters, config records) must pre-exist for metadata to deploy cleanly Load required reference data via Data Loader or Copado Data Template before the deployment QA, UAT, Prod
Big Objects / Archiving Setup Big Object record-level configuration and archiving rules are not in metadata Configure archiving policies and verify Big Object field mappings in target org UAT, Prod

POST-DEPLOY Steps After the Pipeline Completes

These actions must be completed after Copado finishes deploying. Some activate features deployed by the pipeline; others clean up or verify the target state.

Asset / EntityWhy It's ManualRequired ActionApplies To
Flows & Process Builder — Activation Flows deploy in inactive state; active versions cannot be deployed over an existing active version automatically Deactivate the prior version and activate the new version in Setup → Flows after deployment All environments
Scheduled Jobs & Batch Apex Scheduled job instances are runtime data, not deployable configuration Re-schedule Batch Apex or Scheduled Apex classes via Anonymous Apex or Setup after deployment All environments
Email-to-Case / Web-to-Lead / Web-to-Case Routing addresses and endpoint URLs are org-specific and not in metadata Reconfigure routing addresses and update embedded HTML forms with new endpoint URLs UAT, Prod
Connected App OAuth Settings Client secrets, callback URLs, and IP allowlists are environment-specific Update OAuth callback URLs and rotate client secrets in target org's Connected Apps UAT, Prod
Einstein Features (Prediction Builder, Next Best Action) AI model training, activation, and strategy enablement are not deployable Re-enable Einstein features, retrain or repoint prediction models, activate strategies UAT, Prod
Knowledge Settings & Article Types Knowledge data categories, visibility settings, and article publication state are not in metadata Review Knowledge settings, publish new article versions, reconfigure data category visibility UAT, Prod
Community / Experience Cloud Publishing Community pages must be explicitly published after deployment to be visible to end users Publish the Experience Cloud site from Experience Builder after deployment All environments
Reporting Snapshots Snapshot schedules and target objects are org-specific runtime configuration Verify or re-create Reporting Snapshot schedules in the target org UAT, Prod
Delegated Authentication / SSO Certificates SSO certificates and identity provider endpoints differ per environment Upload or update identity provider certificates and verify SSO endpoints post-deployment UAT, Prod
Smoke Test & Sign-off Automated tests do not cover end-to-end business flows or manual-only features Run a defined smoke test script covering all affected user journeys; obtain formal sign-off before closing the change ticket UAT, Prod

📋 Where to Store It
Keep the registry as a versioned file in your Copado Git repository (e.g., docs/manual-steps.md). Link it from the Copado User Story template so every deployment automatically includes it.
👤 Assign Owners
Every manual step needs a named owner and a separate verifier — never the same person. Unowned steps are the ones that get skipped. Review ownership at sprint kickoff, not on the day of deployment.
🤖 Automate Where Possible
If a manual step recurs across every deployment, invest in automating it — Copado Functions, anonymous Apex scripts, or a Salesforce CLI script in your pipeline. Every eliminated manual step reduces deployment risk.

Troubleshooting

SymptomSeverityLikely CauseAction
Deployment stuck in Pending MEDIUM Copado job queue backed up Check Copado Job Monitor; re-queue if stalled >15 min
PMD fails on unchanged code LOW Baseline shift after tooling upgrade Run PMD locally against full org; fix all findings before re-deploying
Coverage drops unexpectedly HIGH New Apex added without tests Identify uncovered classes via Salesforce Setup → Apex Test Execution
Smart Merge loop HIGH Circular dependency between branches Resolve manually; review branching strategy with the team
Back promotion conflicts repeatedly MEDIUM Long-lived feature branches diverging Break large features into smaller incremental deliverables
"Works in QA, fails in Prod" HIGH Back promotion not run; data differences Verify back promotion was run; check for hard-coded IDs or record-dependent logic
Deployment succeeds but wrong metadata MEDIUM Wrong branch selected at promotion Audit Copado User Story metadata; enforce naming conventions via automation
