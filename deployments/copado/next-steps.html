Salesforce Deployment Strategy

Change in Deployment Architecture

As we transition away from a centralized integration sandbox (DevMerge), we must introduce stronger governance at the pull request level to protect QA from instability.

Before: With DevMerge (Shock Absorber)

DevMerge sandbox acted as a buffer, catching cross-team conflicts before QA.

After: Direct to QA (No Buffer)

Feature branches now promote directly to QA, increasing risk of integration issues.

⚠️ Problem

Without DevMerge, QA becomes the first point of integration — exposing it to:

  • Cross-team metadata conflicts
  • Deployment failures
  • Unstable builds

✅ Solution: Shift-Left Quality Control

QA protection must now be enforced at the Pull Request level by Team Leads.

Key Responsibilities

  • Review metadata changes thoroughly
  • Validate Apex code quality
  • Ensure deployments are successful in lower orgs

🔧 Required Tooling

VS Code Extension

Apex PMD Code Scanner

Static analysis directly inside IDE to catch issues early.

CLI Scanner

sf-code-scanner

Run automated scans in CI/CD pipelines or locally.

Immediate Test Class runs

🧪 Apex Test Results — Viewer

Run all test classes locally before PR submission to ensure code quality and deployment readiness.

📋 PR Approval Checklist

  • ✅ Metadata reviewed (profiles, permissions, objects)
  • ✅ Apex code scanned (PMD / sf-code-scanner)
  • ✅ All test classes executed locally
  • ✅ Deployment validated in lower org
  • ✅ Proof attached (logs/screenshots)

📎 Required Proof

Each PR must include:

  • Test execution results
  • Deployment success logs
  • Static scan reports
