Requirements & Setup

Q

What are the hardware and software requirements for using Platform Encryption?

A

None. The crypto functions run natively on the Salesforce platform. No custom code is required to encrypt or decrypt data.

‚úì Key Benefit

Platform Encryption is fully managed by Salesforce, requiring zero infrastructure on your end.

Q

Must I encrypt all of my data when using Platform Encryption?

A

No. Not all data is sensitive, so encryption isn't always required. Also, unnecessarily encrypting data can affect performance and functionality. However, if you choose, you can encrypt the entire transactional database by using Database Encryption.

üí° Best Practice

Encrypt only sensitive data like PII, financial information, and regulated data to optimize performance while maintaining security.

Q

When I enable Platform Encryption, how are my existing encrypted fields affected?

A

The Platform Encryption process doesn't affect fields encrypted using Classic Encryption.

Understanding the Difference:

  • ‚Ä¢ Classic Encryption: Masks data in UI and reports, but visible in API and SOQL
  • ‚Ä¢ Platform Encryption: Encrypts at rest, protects in all contexts including API and backups
  • ‚Ä¢ Both can coexist without interference

Performance & User Experience

Q

When I enable Platform Encryption, will my users experience performance delays?

A

Your users are unlikely to detect any difference in the performance of their day-to-day use of Salesforce when Shield Platform Encryption is enabled. Before enabling it in your production org, we recommend that you enable Shield Platform Encryption in a full copy sandbox and test it under real-world conditions.

‚ö†Ô∏è Recommendation

Always test in a full-copy sandbox with production data volumes and user load before enabling in production.

Technical Details

Q

What encryption algorithm is used with Platform Encryption?

A

For FLE and Database Encryption, Platform Encryption uses symmetric key encryption and a 256-bit Advanced Encryption Standard (AES) algorithm to encrypt data stored on the Salesforce platform. Data encryption and decryption occur on the application servers. Encryption is integrated into the Salesforce application so the application knows when data must be encrypted or decrypted. Whether you're accessing data through the user interface or the API, encryption and decryption are handled the same way.

Encryption Architecture:

Algorithm: AES-256

Type: Symmetric Key

Location: Application Servers

Scope: UI and API

Standard: FIPS 140-2

Key Length: 256 bits

Q

How long are the tenant secret, primary secret, and data encryption keys?

A

256 bits in length.

All encryption keys use the industry-standard 256-bit key length, providing maximum security strength for AES encryption.

Q

Where is my data encryption key stored?

A

The keys are stored only in memory and never persisted on disk.

üîí Security Feature

Keys exist only in volatile memory during encryption/decryption operations, providing an additional layer of security.

Q

How is my organization-specific key generated?

A

The data encryption keys are derived by a key derivation function (KDF) that combines a primary secret with an organization-specific tenant secret and a randomly generated 128-bit string.

Key Derivation Process:

    +
Tenant Secret (Organization-specific)
    +
Random 128-bit Salt
    ‚Üì

    ‚Üì
Data Encryption Key (256-bit)

I use Platform Encryption, and the Encrypted checkbox isn't visible when I create or edit an existing custom field. Why?

A

Only Email, Phone, Text, Text Area, Text Area (Long), Text Area (Rich), Date, Date/Time and URL custom field types are available for encryption.

Q

How finely can I control what data is encrypted with Platform Encryption?

A

For field data, you control which supported standard and custom fields to encrypt. For files and attachments, you control whether encryption is enabled in your organization.

Granularity Levels:

  • ‚Ä¢ Field Level: Individual field selection
  • ‚Ä¢ Files & Attachments: Organization-wide toggle
  • ‚Ä¢ Database Encryption: Entire database (when enabled)
Q

If I enable Platform Encryption, is the format for custom phone, email, and URL fields preserved?

A

Yes, formats for custom phone, email, and URL fields are preserved when they're encrypted.

‚úì Format Preservation

Phone numbers, email addresses, and URLs maintain their formatting and validation rules even when encrypted.

Q

Can I re-encrypt encrypted data?

A

Yes. You can review your encryption coverage statistics to find your active key coverage. Then if you want, you can synchronize the encryption of your data with the most recent tenant secret using the Background Encryption Service.

Re-encryption Use Cases:

  • ‚Ä¢ After key rotation to use new tenant secret
  • ‚Ä¢ Migrating from old to new encryption scheme
  • ‚Ä¢ Consolidating data under a single key
  • ‚Ä¢ Compliance requirements for key refresh

Key Management

Q

Can I access tenant secrets using the API?

A

Yes. For example, for FLE, you can use the API to define an automatic process to rotate the Platform Encryption key regularly. For detailed information, search for TenantSecret in the Object Reference for Salesforce and Lightning Platform.

Example: Query TenantSecret via API
-- SOQL Query for Tenant Secrets
SELECT Id, Source, Status, Description, 
       CreatedDate, LastModifiedDate
FROM TenantSecret
WHERE Status = 'Active'
ORDER BY CreatedDate DESC
Q

Do data encryption keys held in memory rotate automatically when Salesforce rotates the master secret?

A

No. While Salesforce rotates the master secret on a per-release basis, customers' data encryption keys aren't impacted. No new data encryption key is derived automatically.

Key Rotation Scope:

  • ‚Ä¢ Master Secret: Rotated automatically by Salesforce
  • ‚Ä¢ Tenant Secret: Customer-initiated rotation
  • ‚Ä¢ Data Encryption Keys: Derived on-demand, not persisted
Q

What happens to existing data if I rotate a tenant secret?

A

When you generate a new tenant secret, existing encrypted data remains encrypted and accessible as long as the old tenant secret isn't destroyed. (Because of the architecture of Database Encryption, its keys are never destroyed.) New data is encrypted by using the new tenant secret. There's no functional difference to the user.

Key Rotation Behavior:

Data State Encryption Key Used Accessible?
Existing data Old tenant secret ‚úì Yes
New data New tenant secret ‚úì Yes
After re-encryption New tenant secret ‚úì Yes
Q

Can I manage my keys outside of Salesforce?

A

Yes. You can store your key outside of Salesforce and have either the External Key Management service or the Cache-Only Key Service fetch it on demand from a key service that you control.

External Key Management Options:

Bring Your Own Key (BYOK)

Import your own key material into Salesforce

External Key Management (EKM)

Keys stored in external KMS (AWS KMS, Azure Key Vault)

Cache-Only Key Service

Keys fetched on-demand from your key service

Q

What is the limit for how many keys we can have?

A

You can have up to 50 active and archived FLE tenant secrets across all types of tenant secret. Database tenant secrets are not included in this limit. For example, you can have one active and 49 archived Fields and Files (Probabilistic) tenant secrets, and the same number of Analytics tenant secrets. This limit includes Salesforce-generated and customer-supplied key material.

‚ö†Ô∏è Key Limit

50 active + archived keys per secret type (excluding Database Encryption keys)

Q

What if I already have too many active and archived secrets?

A

If you run into the 50 limit, review your encryption coverage statistics to find your active key coverage. Choose one or more keys to destroy. Don't destroy any of them until you synchronize the data they encrypt with an active key.

‚ö†Ô∏è Critical Warning

Never destroy a key until all data encrypted with it has been re-encrypted with a new key. Data loss is permanent!

Q

Are keys I store outside of Salesforce part of the 50-key limit?

A

There is an across-the-board limit of 50 undestroyed keys. This includes keys managed by external services via EKM, BYOK, and the Cache-Only Key service.

Q

Can a Platform Encryption key be shared across more than one organization?

A

No. Encryption keys are specific to an organization and can't be shared with other organizations. Further, a key can be used for only 1 feature. For example, a key for FLE can't be used for Search Indexes.

Key Isolation:

  • ‚Ä¢ Each org has its own unique tenant secret
  • ‚Ä¢ Keys are feature-specific (FLE, Files, Search, etc.)
  • ‚Ä¢ No cross-org or cross-feature key sharing
Q

Where are encryption policies defined?

A

Your organization defines its own policies.

Organizations have full control over which fields, files, and data to encrypt based on their security, compliance, and business requirements.

Security & Hardware

Q

Are the Hardware Security Module (HSM) network appliances shared by multiple tenants?

A

Yes. Key material produced by an HSM is either a per-release secret or a per-tenant secret. Both are required to encrypt your data, so no two tenants have the same data encryption keys.

üîí Multi-Tenant Security

While HSMs are shared infrastructure, the combination of master secret + tenant secret ensures unique keys per organization.

Q

Do third-party vendors have access to the Hardware Security Modules (HSM)?

A

No. Salesforce controls access to the HSMs exclusively.

‚úì Exclusive Control

Salesforce maintains complete control over HSM access with no third-party vendor involvement.

Q

If I can see encrypted data, can Salesforce Support representatives also see the data?

A

Yes, if they have access to the object, record and field.

Access Control

Platform Encryption protects data at rest, but standard Salesforce permission controls still apply for data access. Support representatives with appropriate permissions can view decrypted data.

Storage, Licensing & Limits

Q

Does encrypting fields, files, and attachments with Platform Encryption count against my organization's storage limits?

A

No. Encryption and decryption do count against your organization's per-transaction Apex limits, but they aren't counted as separate transactions.

‚úì No Additional Storage Cost

Encrypted data doesn't consume additional storage beyond the normal data footprint.

Q

How many fields can I encrypt?

A

Salesforce doesn't impose a hard limit on the number of fields that you can encrypt, but there is one functional limitation. If you have Shield Platform Encryption enabled and you want to deploy packages, you can only deploy 80 fields at a time. If you configure 80 or more fields to encrypt, you must deploy the package in phases, with fewer than 80 fields in each phase.

‚ö†Ô∏è Deployment Limitation

Maximum 80 encrypted fields per deployment package. Plan phased deployments for larger implementations.

Q

Since turning on Database Encryption is a permanent action, what happens if I don't renew my subscription?

A

If your Salesforce Shield or Shield Platform Encryption license expires, you lose the ability to manage and rotate your encryption key material. Salesforce no longer guarantees Database Encryption, and it's harder to verify the encryption status of your data at any given time.

‚ö†Ô∏è Critical: License Expiration Impact

  • ‚Ä¢ Loss of key management capabilities
  • ‚Ä¢ No key rotation possible
  • ‚Ä¢ Database Encryption guarantee removed
  • ‚Ä¢ Limited encryption verification
  • ‚Ä¢ Data remains encrypted but unmanageable
