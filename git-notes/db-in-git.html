Git ≠ Database Storage

Storing database content in Git
is usually not a good idea

Git is fantastic for code. For live production data, binary dumps, large tables or frequently changing records — almost always the wrong tool.

1. Repository size explodes very quickly

Every pg_dump / mysqldump / CSV / JSON export → new commit.

Git stores near-full copies of binary/large files → history balloons to GBs very fast.

→ slow clones, huge disk usage, painful git gc, CI/CD timeouts, hosting limits exceeded

2. Git is terrible at versioning large binary blobs

Database dumps are usually not line-based text → delta compression is almost useless.

→ each version stored nearly in full → even Git LFS only moves the problem elsewhere (still slow fetches, no good history browsing).

3. Security & compliance disaster waiting to happen

Production data often contains:

  • Personal information (PII)
  • Passwords / API keys / tokens
  • Financial / health records

Once committed → present forever in history — even after git rm --cached + force push.

4. Databases already have much better tools

What databases give you

  • Point-in-time recovery
  • Incremental backups
  • WAL / binlogs / oplog
  • Replication & PITR
  • Fast & selective restore

What Git gives you (for data)

  • Slow full-history clones
  • No incremental restore
  • No transaction consistency
  • Terrible large-file performance
  • Accidental secret exposure

Very narrow cases where it can be acceptable

Tiny seed / lookup data

< 1–5 MB, rarely changes

Test fixtures / sample data

Small, static, non-sensitive

Dolt / versioned SQL database

Use tools made for this

For everything else → use proper backups, logical dumps with rotation, replication, or data versioning tools (DVC, lakeFS, Dolt, etc.)

Git is version control for code — not a backup or data storage system
