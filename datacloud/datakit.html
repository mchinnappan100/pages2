Data Kits: package, standardize, and activate customer data

Data Kits in Salesforce Data Cloud bundle data models, transformations, and activation mappings so teams can onboard, standardize, and operationalize customer datasets quickly — without reinventing the model for each use case.

  • Composable: Kits can be extended or combined.

What is a Data Kit?

A Data Kit is a packaged set of artifacts in Data Cloud that defines a domain-specific data model, ETL/transformations, derived attributes, identity resolution rules, and activation mappings. Think of it as a re-usable blueprint for a particular data domain (e.g., product catalog, orders, loyalty).

Core Components

  • Schema: Predefined entities and fields to ensure consistent representation.
  • Transformations / Recipes: ETL pipelines (dataflows) or SQL steps to normalize and enrich data.
  • Identity resolution: Match rules and identity graphs that link records to a single profile.
  • Enrichments: Computed attributes (LTV, segment tags, scoring).
  • Activations: Mappings to downstream destinations (Salesforce CDP, Journey Builder, advertising platforms).

Benefits

  1. Speed up onboarding of new data sources
  2. Reduce duplication of mapping & transformation logic
  3. Provide governance-friendly, repeatable data patterns
  4. Make activation to marketing or analytics frictionless

Getting started — practical steps

  1. Choose a kit: Pick the Data Kit that matches your domain (e.g., Orders Kit).
  2. Review schema: Validate fields and map your source fields to kit schema.
  3. Ingest sample data: Run a small ingestion to verify identity stitching and transforms.
  4. Tweak transforms: Add custom recipes or enrichments where business logic differs.
  5. Activate: Map kit attributes to destinations (Audiences, Journeys, Analytics).
Tip: Start with a test environment and small datasets. Data Kits speed iteration, but always validate identity rules on real samples.

Example: Data Kit manifest (illustrative)

Below is a simplified manifest-like snippet to show typical kit artifacts.


{
  "kitName": "Orders Kit",
  "schemas": [
    {"name": "order", "fields": ["orderId","customerId","amount","status","createdAt"]},
    {"name": "order_item", "fields": ["itemId","orderId","productId","qty","price"]}
  ],
  "transforms": [
    {"id":"normalize-order", "type":"sql", "script":"SELECT ... FROM raw_orders"}
  ],
  "identity": {"primaryKeys":["customerId"], "matchRules":["email","phone"]},
  "activations": [
    {"destination":"audiences","mapping":{"customerId":"profile.id","loyaltyTier":"profile.tags.loyalty"}}
  ]
}

Best practices

  • Align with business domains: Choose kits that reflect team ownership (e.g., commerce team uses Orders Kit).
  • Validate identity early: Identity resolution impacts everything — measure match rates and edge cases.
  • Keep transforms observable: Log intermediate outputs and monitor data quality metrics.
  • Version & document: Use kit versions and keep clear release notes for downstream consumers.
  • Secure PII: Apply encryption and governance controls to sensitive fields before activation.
