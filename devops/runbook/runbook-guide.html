Section 01

Core Principles

01
🎯
Clear and Unambiguous
Every step must state what, where, and how to validate
  • What to do
  • Where to do it
  • Expected result
  • Validation method
  • Rollback or escalation path if failure occurs
❌ "Check permissions"
✅ Navigate to Setup → Profiles → System Administrator → Verify 'Modify All Data' is enabled
👥
Written for Non-Domain Experts
Any authorized engineer can execute without deep context
  • Full navigation paths
  • Exact CLI commands
  • Explicit button names
  • Expected UI screenshots
  • Validation checkpoints
Assume executor may not know: Salesforce metadata architecture, Copado internals, Apex deployment behavior, or environment-specific dependencies. Use full paths and literal UI labels.
🖼️
Visual Guidance Required
Screenshots for every critical step
  • Navigation path screenshots
  • Required selections
  • Expected confirmation dialogs
  • Successful execution state
  • Validation result
Step: Disable Scheduled Apex Jobs — include screenshot showing: Setup → Apex Jobs, filter criteria, jobs to pause, and the expected paused state.
✅
Peer-Reviewed & Approved
No document executes without formal review
  • Steps are technically correct
  • Instructions are complete
  • Screenshots are current
  • Validation steps are testable
  • Rollback steps are accurate
🤝
Collaboratively Executed
Deployment is a team activity, never solo
  • Dev Team Leads — feature correctness
  • DevOps Team — pipeline execution
  • QA / Business Validation
  • Release Coordinator — commands timeline
🔁
Verifiable & Repeatable
Double verification for every executed step
  • Executor performs action
  • Verifier confirms outcome
  • Team signs off
  • Repeatable across all environments
DevOps aborts jobs → Dev Lead confirms jobs stopped → Both sign deployment sheet. Prevents silent execution mistakes.
Section 02

Standard Document Structure

02
📋
Deployment Header
Identity and ownership metadata
  • Deployment Name & ID / Copado Story / Release Number
  • Target Environment
  • Deployment Date & Time
  • Deployment Owner, Technical Lead, DevOps Lead
  • Business Owner, Rollback Owner
🎯
Deployment Objective
Why, what, and the business impact
  • What is being deployed
  • Why it is needed
  • Business impact
  • Risk level
Deploy Opportunity validation framework enhancement to support automated partner approval workflow.
☑️
Prerequisites Checklist
All required readiness checks before proceeding
  • Code merged to correct branch
  • Automated tests passed
  • Manual UAT signoff complete
  • Backup taken & org health verified
  • Deployment window approved & stakeholders notified
Checklist format: Item | Status | Verified By | Timestamp — every row must be signed before proceeding.
Section 03

Writing Pre-Deployment Steps

03
🔢
Step Numbering
Reference ID: PRE-01, PRE-02…
  • Unique identifier per step
  • Sequential, never reused
  • Referenced in signoff sheets
📝
Objective + Owner
What it achieves and who is responsible
  • Objective: e.g., disable batch jobs to prevent record locking
  • Owner: DevOps, Dev Lead, Salesforce Admin, QA, Release Manager
⚙️
Exact Execution Steps
Every click, navigate, and confirm spelled out
  • Login to target org
  • Navigate to exact Setup path
  • Locate named jobs/components
  • Perform explicit action (e.g., click Abort)
PRE-01 — Disable Scheduled Jobs: 1. Login to Production 2. Navigate to Setup → Scheduled Jobs 3. Locate Batch_OpportunityRefresh, Batch_RecalculateTerritory 4. Click Abort on each
🖼️
Screenshot + Validation
Visual proof and success criteria
  • Screenshot: correct menu, job list, post-action state
  • Validation: e.g., job status no longer shows "Queued" or "Processing"
✍️
Step Signoff
Executed By | Verified By | Time | Signature
  • Unsigned steps are unauditable
  • Both executor and verifier sign
  • Timestamp is mandatory
Section 04

Deployment Execution Steps

04
🚀
DEP-01: Promote Package via Copado
Highly explicit deployment instructions
  • Open Copado Pipeline
  • Select User Story Bundle
  • Verify stories included (e.g., US-1024, US-1040, US-1088)
  • Click Promote → Select Target: Production → Confirm
Expected Result: Deployment status = Success Validation: • Metadata count matches package manifest • No skipped components • Apex tests pass
❌
Failure Action
What to do if a deployment step fails
  • Capture error logs immediately
  • Notify release bridge team
  • Do NOT retry until reviewed
  • Escalate per rollback plan in header
Section 05

Writing Post-Deployment Steps

05
🧪
Functional Validation
Business-level verification
  • Create a test record in affected object
  • Walk through the deployed business process
  • Confirm expected record/related-list changes
Verify Opportunity stage automation: 1. Create Opportunity 2. Move to Proposal 3. Confirm approval request generated in related list
🔧
Technical Validation
System health checks
  • Apex Jobs resumed
  • Platform Events processing
  • Async jobs healthy
  • Integration endpoints responding
  • No new errors in debug logs
📡
Monitoring Window
Specify duration and what to watch
  • Monitor for at least 60 minutes post-deployment
  • Apex Exception Emails
  • Failed Flows & integration failures
  • Copado deployment logs
Section 06–08

Governance & Collaboration

06
🔍
Peer Review Requirement
No document executes without review
  • Steps are technically correct
  • Instructions are complete
  • Screenshots are current
  • Validation steps are testable
  • Rollback steps are accurate
📊
Approval Matrix
All roles must sign before deployment proceeds
  • Dev Lead
  • DevOps Lead
  • Salesforce Architect
  • QA Lead
  • Release Manager
🔄
Double Verification Rule
Every step independently confirmed
  • Executor performs action
  • Verifier confirms outcome independently
  • Both sign deployment sheet
  • Prevents silent execution mistakes
DevOps aborts jobs ↓ Dev Lead confirms jobs stopped ↓ Both sign deployment sheet
📦
Deployment Closure Checklist
Deployment closes only after all items pass
  • All post-deployment validations pass
  • Monitoring period complete
  • Stakeholder confirmation received
  • Team signoff recorded
  • Deployment artifacts archived (docs, screenshots, logs, Copado results)
Section 10

Common Mistakes to Avoid

10

✗ Avoid Assumptions

❌ "Pause jobs as needed"
✅ Explicitly list every job by name

✗ Avoid Missing Validation

❌ "Deploy package"
✅ Include expected success indicators

✗ Avoid Missing Screenshots

Visual guidance is mandatory for every critical step. Screenshots reduce interpretation errors significantly.

✗ Avoid Single-Person Execution

Critical deployments require collaborative verification. Deployment is a team activity, never a solo effort.

✗ Avoid Unsigned Steps

Unsigned steps are unauditable and considered incomplete. Every step requires Executed By and Verified By signatures.

✗ Avoid Vague Wording

❌ "Check permissions"
✅ Full navigation path to exact permission setting

⭐ The Golden Rule

If a reasonably technical person unfamiliar with the feature cannot execute your deployment document successfully, the document is not production-ready.

A great deployment document is: clear enough for anyone authorized to execute · visual enough to eliminate interpretation · reviewed enough to prevent mistakes · collaborative enough to ensure safety · verified enough to guarantee confidence.
