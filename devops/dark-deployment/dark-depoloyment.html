01Concept

What is Dark Deployment?

🌑

Deploy ≠ Release

Dark deployment means pushing new metadata and code all the way to production, but keeping the feature invisible or inactive until it is deliberately switched on. The deployment event and the release event become two separate decisions.

  • Code ships on the normal pipeline cadence
  • Feature stays "dark" — hidden from users
  • A separate, low-risk toggle controls visibility
  • Rollback becomes a config change, not a deployment
See Example
A team builds a new "Partner Portal Dashboard" Lightning page. It is deployed to Production on Tuesday with everyone else's sprint, but a Custom Permission gates the tab. No one sees it. On Thursday, Support turns on the permission for 5 pilot accounts to validate in real production data before a wider rollout.
🎯

Where It Fits in DevOps

Dark deployment is one technique in a family of progressive delivery strategies, alongside canary releases, blue-green deployments, and A/B rollouts — all aimed at separating "did it ship" from "is it live."

  • Reduces the blast radius of any single release
  • Lets QA and pilot users test in real Production data
  • Removes the Friday-deploy-freeze anti-pattern
  • Pairs naturally with Copado's promotion pipeline
See Example
Instead of holding a release branch for three weeks until Marketing is ready to announce a feature, the team merges and deploys continuously. The feature sits dark behind a Custom Metadata Type flag. Marketing flips the flag on launch day — no deployment required, no code freeze needed.
02Rationale

Why It Matters on the Salesforce Platform

Salesforce orgs are shared, multi-tenant, and often multi-team. A single production org may serve Sales, Service, and Partner users simultaneously, all riding on the same release train. Dark deployment gives release managers a way to keep the deployment cadence fast and frequent while still protecting each business unit from unfinished or unapproved work becoming visible.

The core benefit Smaller, more frequent deployments are inherently lower-risk than large infrequent ones — but only if "deployed" doesn't also mean "everyone sees it right now." Dark deployment breaks that coupling.
03Foundations

Toggle Mechanisms Native to Salesforce

🔑

Custom Permissions

The most common gate. Assign a Custom Permission to a Permission Set, keep it unassigned from everyone until go-live, then check it in Apex, Flow, or visibility rules.

if (FeatureManagement.checkPermission('Dark_Feature_Partner_Dashboard')) {
    // render the new dashboard component
}
🧩

Custom Metadata Types

Deployable, environment-aware toggles. Because Custom Metadata is deployed like code, a flag's default state travels with the release, and can still be flipped post-deploy without another deployment.

Feature_Toggle__mdt toggle =
    Feature_Toggle__mdt.getInstance('Partner_Dashboard');
if (toggle.Is_Active__c) {
    // proceed with dark feature logic
}
⚙️

Feature Management (Custom Permissions + Package Settings)

Salesforce's built-in FeatureManagement Apex class was designed for exactly this — gating functionality per-org or per-user in managed and unlocked packages.

  • Works with License Management for ISV-style rollout
  • Can drive Boolean, Date, or Integer feature parameters
  • Integrates cleanly with Permission Set Groups
👥

Permission Set Groups & Audience Rules

Layer visibility by audience instead of a single global switch — enabling a true canary rollout inside one org.

  • Pilot group gets the Permission Set Group first
  • Lightning App Builder component visibility rules
  • Flow entry conditions checking the permission
04Foundations

Architecture Patterns

Feature Flagging Canary Release Ring-Based Rollout Shadow / Parallel Run
🕯️

Ring-Based Rollout

Expand visibility in concentric rings: internal team → pilot customers → 10% of orgs → 100%. Each ring is a Permission Set Group assignment, not a deployment.

👻

Shadow / Parallel Run

New logic runs silently alongside the old path — writing to a staging object or logging comparison results — before it's ever trusted to drive the UI.

05Workflow

A Typical Pipeline Workflow

Build behind a flag

Developer wraps the new component or Apex path in a Custom Permission or Custom Metadata check from the very first commit — not retrofitted later.

Promote through the pipeline as normal

The User Story moves Dev → QA → UAT → Production on the regular Copado cadence. The flag ships everywhere, default OFF in Production.

Validate dark in Production

QA or a pilot group gets the permission assigned in Production itself — the safest possible test environment because it's the real one.

Progressive enablement

Widen the audience ring by ring, watching error rates and adoption at each step.

Flip on, then clean up

Once fully released, remove the toggle in a follow-up story so the codebase doesn't accumulate permanent flag debt.

06Applying It

Apex & LWC Patterns

☁️

Apex Guard Clause

public with sharing class PartnerDashboardController {
    @AuraEnabled(cacheable=true)
    public static List<Metric> getMetrics() {
        if (!FeatureManagement.checkPermission('Dark_Partner_Dashboard')) {
            throw new AuraHandledException('Feature not available.');
        }
        return MetricSelector.getForCurrentUser();
    }
}
🧵

LWC Visibility Check

import { LightningElement, wire } from 'lwc';
import hasDarkFeature from '@salesforce/customPermission/Dark_Partner_Dashboard';

export default class PartnerDashboard extends LightningElement {
    get isEnabled() {
        return hasDarkFeature;
    }
}
Tip Keep the flag check at the boundary — controller entry point or component render guard — rather than scattered through business logic. This keeps cleanup a one-line removal later.
07Applying It

Dark Deployment with Copado & CI/CD

🚀

Deployment Task for the Flag Flip

Create a lightweight Copado Deployment Task that only updates the Custom Metadata record's Is_Active__c field — no Apex, no metadata retrieve, just a data change that can be promoted independently of the feature code.

🔔

Decouple the Release Record

Attach the dark feature's User Stories to a Release, but hold the Milestone that represents "go live" separately — so Release Managers can promote code on schedule and trigger visibility on a completely different schedule.

See Example
Release: "Summer '26 — Wave 2" Milestone: Code Complete (deployed to Prod, flag OFF) ├── US-210 Partner Dashboard — Apex & LWC └── US-211 Partner Dashboard — Reporting objects Milestone: Go-Live (flag ON, no deployment) └── Deployment Task: Update Feature_Toggle__mdt.Is_Active__c = true
08Applying It

Risks & Guardrails

⚠️

Flag Debt

Toggles that never get removed accumulate as permanent branching logic and shared-state hazards. Track every flag with an owner and a removal date.

🧪

Untested Dark Code Paths

Dark code still runs in Production if the guard clause is missed anywhere — triggers, batch jobs, flows. Audit every entry point, not just the UI.

🔒

Sharing & Governor Limits

Dark features still consume SOQL, DML, and CPU governor limits even when hidden from the UI if they run in shared triggers or scheduled jobs. Isolate expensive dark logic behind async patterns.

📋

Data Model Drift

New fields and objects tied to a dark feature are visible to admins and integrations even when the UI is hidden. Coordinate field-level security alongside the toggle.

09Reference

Best Practices

✓ One Flag, One Owner

Every toggle has a single named owner responsible for enabling and eventually removing it.

✓ Default OFF in Every Environment

New flags should default to inactive everywhere, including UAT, so accidental exposure never reaches Production by inheritance.

✓ Gate at the Boundary

Check the flag once at the controller, trigger handler entry, or component root — not deep inside business logic.

✓ Track Flags in a Register

A simple Custom Metadata list view of all active toggles, their owners, and target removal sprint prevents silent flag sprawl.

✓ Remove the Flag After Full Rollout

Once a feature reaches 100% and is stable, ship a dedicated cleanup story that deletes the toggle and the dead branch.

✓ Pair with Quality Gates

Dark code still needs Apex test coverage and PMD scans before promotion — invisibility isn't an excuse for lower quality bars.
