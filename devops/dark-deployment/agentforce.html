Terminology Changes in Recent Agentforce Releases

Salesforce significantly restructured Agentforce terminology in the Summer '25 and Winter '26 releases as the platform shifted toward a native multi-agent architecture. The table below maps the old terms you may see in earlier docs to the current ones used in this guide and in the latest Setup UI.

Old TermNew TermWhat Changed
Topic SubAgent Topics evolved into first-class SubAgents — each with its own identity, scope, instructions, and action library. A SubAgent is now itself a mini-agent rather than a simple routing category.
Agent (top-level) Orchestrator Agent The parent agent that routes user messages is now explicitly called the Orchestrator. It delegates to SubAgents rather than simply selecting a Topic.
Topic Instructions SubAgent Instructions / Job To Be Done Instructions within a SubAgent are now framed as a "Job To Be Done" — a concise statement of purpose that Atlas uses for routing and constraint enforcement.
Topic Description (routing signal) SubAgent Scope The description field used by Atlas to decide whether to delegate to a SubAgent is now labelled Scope, making its role as a routing boundary clearer.
Action (within a Topic) Action (within a SubAgent) Actions themselves are unchanged — the same Flow, Apex, Prompt Template, and External action types remain. They are now registered to a SubAgent rather than a Topic.
Agent-to-Agent (multi-agent) Agent Network / SubAgent delegation The multi-agent pattern is now called an Agent Network. An Orchestrator delegates to SubAgents (which may themselves be full agents), forming a composable network of specialists.
Einstein Copilot Agentforce for Salesforce (internal agent) Einstein Copilot, the in-app Salesforce UI assistant, was rebranded as Agentforce for Salesforce to unify it under the Agentforce platform umbrella.
What this means for builders If you are reading Trailhead modules or Help docs from before Summer '25, mentally substitute "SubAgent" wherever you see "Topic" and "Orchestrator" wherever you see the top-level Agent. The underlying mechanics — action selection, routing via descriptions/scope, instruction-based guardrails — are the same. The new naming makes the multi-agent nature of every Agentforce deployment explicit.
Setup UI path change Agent Builder navigation in Winter '26: Setup → Agentforce → Agent Builder → [Select Agent] → SubAgents tab. The old "Topics" tab is replaced by "SubAgents". Each SubAgent card now shows its Scope, Job To Be Done, and Actions in a unified panel.
01Introduction

What is Agentforce?

🤖

Autonomy, Not Automation

Agentforce is Salesforce's platform for deploying autonomous AI agents — software that can reason, decide, and act on multi-step tasks without a human driving every step. Unlike traditional automation (flows, bots) that follows a fixed script, an Agentforce agent reads its instructions, evaluates the situation, selects the right action, and executes it.

  • Understands natural language intent, not just keywords
  • Chooses from a library of actions to fulfill a goal
  • Handles deviations and edge cases gracefully
  • Knows when to escalate to a human agent
See Example
A customer emails: "I want to change my shipping address and upgrade my subscription." A traditional bot routes to two separate scripted flows. An Agentforce Service Agent reads the intent, queries the CRM to confirm account ownership, updates the shipping address via a Flow action, upgrades the subscription via an Apex action — all in one conversation thread — then confirms and closes.
🧭

The Four Pillars

Every Agentforce agent is defined by four elements that together govern how it thinks and what it can do.

  • Role / Identity — who the agent is and what it's responsible for
  • Topics — the domains it's allowed to handle (e.g. "Order Management")
  • Instructions — natural-language rules per topic guiding its decisions
  • Actions — the concrete tools it can invoke (Flows, Apex, APIs)
See Example
Role: "You are Aria, a service agent for Acme Corp. You help customers with orders, returns, and billing. Always confirm identity before making account changes."

Topic: "Returns" → Instructions: "Offer a replacement if item is under 30 days old. Always issue a case number. Never issue a refund above $500 without manager approval."

Actions: Create_Return_Case (Flow), Process_Refund (Apex), Send_Confirmation_Email (Flow).
The paradigm shift With classical bots you design every path. With Agentforce you define goals, guardrails, and tools — the LLM reasons across them. You stop being a flow designer and start being an agent architect.
02Introduction

Core Architecture

Agentforce is not a standalone product — it is a layer woven through the Salesforce platform, drawing on the Einstein AI foundation, Data Cloud for context, and the full Salesforce metadata model for actions.

🧠

The Reasoning Loop

At runtime an agent runs a ReAct-style loop: it Reasons about what the user needs, selects an Action, observes the result, then decides to act again or respond.

  • User message → LLM reads instructions + conversation history
  • LLM selects a topic and an action to invoke
  • Action executes (Flow, Apex, API call)
  • Result is fed back to the LLM as context
  • LLM composes a response or picks the next action
🏗️

Platform Layers

Agentforce is built as a composition of Salesforce platform capabilities, not a separate cloud.

  • Einstein AI — large language model (Atlas Reasoning Engine)
  • Data Cloud — unified real-time customer data for grounding
  • Einstein Trust Layer — PII masking, toxicity filter, zero-retention
  • Agent Builder — low-code UI for assembling agents
  • Omni-Channel — routes conversations to agents and humans
🔗

Atlas Reasoning Engine

Salesforce's purpose-built orchestration layer on top of the LLM. Atlas handles topic classification, action selection, context window management, and response safety — abstracting away raw prompt engineering from builders.

  • Classifies which Topic applies to each user message
  • Selects the most relevant Action from the agent's toolset
  • Enforces instruction guardrails before executing
  • Manages conversation memory across turns
📦

Metadata Model

Agents, topics, instructions, and action references are all first-class Salesforce metadata — they live in your org as deployable, version-controlled records that travel through your DevOps pipeline.

  • Agent metadata type
  • AgentTopic and AgentTopicAction
  • GenAiPromptTemplate for prompt templates
  • Deployable via Salesforce CLI, Copado, and CI/CD
03Agent Anatomy

Agent Types

Service Agent SDR Agent Sales Coach Employee Agent Custom Agent
🎧

Service Agent

Replaces the traditional scripted chatbot for customer-facing service interactions. Handles case creation, order inquiries, returns, password resets, and appointment booking — and knows when to escalate to a live rep.

  • Deployed on Messaging channels (WhatsApp, SMS, Chat)
  • Reads Case, Order, and Contract records in real time
  • Performs CRM writes — creates cases, updates records
  • Transfers conversation context on human handoff
📈

Sales Development Representative (SDR) Agent

An outbound prospecting agent that researches leads, drafts personalized outreach, follows up autonomously on inbound inquiries, and qualifies pipeline before routing to an Account Executive.

  • Nurtures web-to-lead form submissions
  • Crafts email sequences grounded in company data
  • Schedules meetings via calendar integration
  • Updates Lead and Opportunity fields automatically
🏋️

Sales Coach Agent

An internal-facing agent that listens to (or reads transcripts of) sales calls and provides real-time coaching, objection-handling suggestions, and post-call summaries with next-best-action recommendations.

  • Surfaces relevant battle cards during live calls
  • Scores call quality against methodology criteria
  • Generates follow-up email drafts automatically
  • Logs notes and updates Opportunity stage
🧑‍💼

Employee-Facing Agent

Internal agents deployed on Experience Cloud, Slack, or the Salesforce app to help employees navigate HR policies, IT requests, onboarding tasks, and internal knowledge bases.

  • Answers policy questions against an HR Knowledge Base
  • Opens IT tickets and tracks resolution status
  • Guides new hires through onboarding checklists
  • Surfaces relevant internal documentation instantly
🛠️

Custom Agents

Built from scratch in Agent Builder. You define the identity, topics, instructions, and actions from the ground up — enabling fully domain-specific agents for industry verticals, back-office operations, or proprietary business processes.

  • Full control over scope and guardrails
  • Can compose Apex, Flow, Prompt Templates, and external APIs
  • Deployable across any Salesforce channel
  • Testable in a built-in conversation simulator
🌐

Agentforce for Experience Cloud

Embed agents directly into public or authenticated portal sites built on Experience Cloud — giving partners, customers, and community members a conversational interface into your CRM data and processes.

  • Respects Experience Cloud sharing and guest user permissions
  • Can authenticate portal users mid-conversation
  • Surfaced as a floating widget or embedded component
  • Supports multilingual responses via Einstein language models
04Agent Anatomy

Actions & Skills — The Agent's Toolbox

An agent is only as capable as the actions it can invoke. Actions are the bridge between the LLM's reasoning and real work inside (or outside) Salesforce.

⚡

Flow Actions

Any auto-launched Flow can be registered as an agent action. This gives low-code builders full authoring power — write a Flow, expose it as a callable tool, and the agent picks it up.

// In Agent Builder: Action Type = Flow
// Flow API Name: Create_Service_Case
// Input Variables: Subject, Description, AccountId
// Output Variables: CaseNumber, CaseId
See Example
A Service Agent handles a customer complaint. The agent invokes the Create_Service_Case Flow action, passing the summary it extracted from the conversation. The Flow creates the Case record, triggers an assignment rule, and returns the case number. The agent reads the output and tells the customer their case number.
☁️

Apex Actions

Invocable Apex methods annotated with @InvocableMethod are surfaced as actions. Use these for complex logic, callouts to external systems, or operations that exceed Flow's capabilities.

public class RefundProcessor {
    @InvocableMethod(
        label='Process Refund'
        description='Issue a refund for an order. Returns confirmation ID.'
    )
    public static List<Result> processRefund(
            List<Request> requests) {
        // callout to payment gateway
        // update Order status
        // return confirmation
    }
}
📝

Prompt Template Actions

Actions backed by a GenAiPromptTemplate — the agent passes dynamic context into a prompt template and gets a generated text artifact back (draft email, summary, recommendation).

  • Generate a personalised follow-up email from opportunity data
  • Summarise a long case thread into three bullet points
  • Draft an order confirmation from structured order fields
  • Produce a meeting agenda from calendar context
🔌

External & MuleSoft Actions

Call any REST API via a named credential. MuleSoft integration surfaces as a first-class action type with automatic schema generation from the API spec.

  • Fetch real-time inventory levels from an ERP
  • Post a Slack message when a case is escalated
  • Retrieve customer CSAT scores from a third-party survey tool
  • Trigger a logistics provider to reroute a shipment
🔍

Knowledge Retrieval Actions

Built-in actions that query Salesforce Knowledge Articles or a Data Cloud vector index — the foundation of RAG (Retrieval-Augmented Generation) inside Agentforce.

  • Semantic search over the Knowledge Base
  • Returns ranked articles with relevance scores
  • Feeds retrieved text into the LLM as grounded context
  • Reduces hallucination by anchoring answers to real content
📊

Standard CRM Actions

Out-of-the-box actions that let agents read and write core Salesforce objects without any custom code.

  • Query any sObject with field-level security enforced
  • Create and update records (Case, Lead, Opportunity)
  • Log an activity (Task, Event) against any record
  • Send emails via org email services
Action design principle Write actions that do one thing clearly. The LLM selects actions by name and description — a well-named, single-purpose action (Get_Order_Status) is far more reliably invoked than a broad multi-purpose one (Handle_Order_Stuff). Treat action descriptions as contracts, not comments.
05Agent Anatomy

Prompt Builder & Instructions

Prompt Builder is the declarative tool for crafting the instructions and prompt templates that govern how agents (and standalone Einstein features) think and respond.

📋

System Prompt & Role Instructions

The top-level agent instructions define its identity, tone, scope, and absolute guardrails — the equivalent of a system prompt in a direct LLM API call, but managed as deployable Salesforce metadata.

  • Set the agent's persona and communication style
  • Define hard limits ("never promise a refund without approval")
  • Specify languages and accessibility requirements
  • Control data disclosure rules
🗂️

Topic Instructions

Each Topic carries its own instruction block — the specific rules the agent follows when handling that domain. Topic instructions override or supplement the system-level instructions for their scope.

/* Topic: Billing Disputes */
You help customers understand and dispute charges.
- Always retrieve the account ledger before responding.
- For disputes under $50, you may issue a courtesy credit
  without escalation.
- For disputes above $50, create a Billing Case and inform
  the customer a specialist will follow up within 24 hours.
- Never discuss competitor pricing.
✏️

Prompt Templates with Dynamic Grounding

Prompt Builder supports merge fields that pull live CRM and Data Cloud data into the prompt at execution time — combining the structure of a template with real-time factual grounding.

// GenAI Prompt Template: Draft_Renewal_Email
You are a Salesforce account manager.
Write a renewal email for {!Account.Name}.
Their current contract expires {!Contract.EndDate}.
Their primary contact is {!Contact.FirstName} {!Contact.LastName}.
Last NPS score: {!Account.NPS_Score__c}.
Tone: professional, concise, 150 words max.
🔗

Prompt Chaining

Complex reasoning tasks can be split across multiple prompt templates that pipe into each other — each step refining or extending the previous output before the final response is delivered.

  • Step 1: Extract structured intent from raw user message
  • Step 2: Retrieve relevant knowledge articles
  • Step 3: Generate grounded response using extracted intent + articles
  • Step 4: Safety/tone check before delivery
Instruction precision matters Vague instructions produce inconsistent agent behaviour. Use concrete, testable rules: "If the order is older than 30 days, do X" is far better than "Use your judgement on older orders." Think of instructions as unit-testable logic, not prose guidance.
06Platform Integration

Data Cloud & Grounding

Grounding is the practice of injecting verified, up-to-date facts into the LLM's context window so responses are anchored to reality rather than to training data. Data Cloud is Agentforce's grounding engine.

🌊

Why Grounding is Critical

An ungrounded LLM answers from its training cut-off — it knows nothing about your customers, products, or current inventory. Grounding replaces generic answers with customer-specific, real-time truth.

  • Customer name, account status, and entitlements — live from CRM
  • Product availability and pricing — live from ERP via Data Cloud
  • Knowledge Base articles — latest approved versions only
  • Interaction history — last 10 cases, emails, chat transcripts
🗄️

Unified Data Model (UDM)

Data Cloud harmonises data from any source — Salesforce, external data lakes, web analytics, third-party SaaS — into a canonical Unified Data Model. Agentforce agents query the UDM, not raw source systems.

  • Unified Individual, Account, and Engagement objects
  • Real-time identity resolution across sources
  • Calculated Insights for pre-aggregated ML features
  • Data Streams ingest from S3, Snowflake, MuleSoft, and more
🧲

Retrieval-Augmented Generation (RAG)

For unstructured content (knowledge articles, PDFs, policy docs), Data Cloud builds a vector index. When an agent needs to answer a factual question, it performs a semantic similarity search, retrieves the top-matching passages, and feeds them into the LLM as context.

  • Documents are chunked, embedded, and indexed in Data Cloud
  • Semantic search runs at query time, not training time
  • Retrieved passages are cited in the agent's response
  • Index updates automatically when source content changes
⚡

Real-Time Streaming Data

Data Cloud can ingest streaming events — click-streams, IoT telemetry, payment webhooks — and make them available to agents within seconds, enabling truly real-time context.

  • Customer just browsed the pricing page → agent knows to offer a demo
  • Payment failed 10 minutes ago → agent proactively addresses billing
  • Device telemetry triggered an alert → agent pre-fills a service case
See Example
A Data Cloud Calculated Insight computes "Churn Risk Score" in real time as customer engagement data flows in. An Agentforce Service Agent, when greeting a high-risk customer, retrieves this score, sees it is above threshold, and proactively offers a retention discount — without the rep knowing to ask.

Deploying Agents — Channels, Handoffs & Lifecycle

📱

Channels

Agents are deployed via Omni-Channel and can serve customers across every Salesforce-supported messaging surface from a single agent definition.

  • Embedded web chat widget (Experience Cloud or any website)
  • WhatsApp Business via Messaging for In-App and Web
  • SMS (via Salesforce Messaging)
  • Email-to-Case with agent auto-reply
  • Slack (Employee-facing agents)
  • Custom channels via REST Messaging API
🤝

Human Handoff

When the agent reaches its limits — complexity, policy exception, emotional escalation — it transfers to an Omni-Channel queue, carrying the full conversation transcript and a structured handoff note it writes itself.

  • Agent generates a plain-English summary for the receiving rep
  • All turns are visible in the Service Console transcript
  • Omni-Channel routing rules determine which queue or rep receives
  • Human rep can resume in the same conversation thread
See Example
Handoff note auto-generated by the agent:

"Customer is Lisa Chen, Account #58201. She is requesting a refund of $340 for Order #10928, placed 45 days ago (outside the standard 30-day window). She is frustrated — this is her third contact on this issue. I have pulled her order history and the previous two case numbers (CS-4421, CS-4509) for context. Recommend supervisor review for policy exception."

Security, Trust & the Einstein Trust Layer

Every LLM call Agentforce makes passes through the Einstein Trust Layer — Salesforce's security envelope that ensures enterprise data never leaves the platform's trust boundary and every AI interaction is auditable.

🔒

Zero Data Retention

By contractual commitment, Salesforce (and its LLM partners) do not retain prompts or completions for training. Data sent to the LLM in a grounded prompt is ephemeral — it is used to generate the response, then discarded.

  • No customer data stored outside the org's trust boundary
  • Contractually enforced with all LLM providers
  • Verified at the network layer, not just policy
🕵️

PII & Sensitive Data Masking

Before any prompt is dispatched to the LLM, the Trust Layer detects and masks PII — credit card numbers, social security numbers, health identifiers, and any custom-defined sensitive patterns.

// Trust Layer replaces before sending to LLM:
// "Customer SSN is 123-45-6789"
// → "Customer SSN is [MASKED_SSN]"
// LLM response with [MASKED_SSN] is then un-masked
// before returning to the application layer.
🛡️

Toxicity Detection & Prompt Injection Defense

Responses are screened for harmful, discriminatory, or off-topic content before delivery. Prompt injection attempts — where a user's message tries to override the agent's instructions — are detected and blocked.

  • Configurable toxicity threshold per channel
  • Blocks jailbreak and instruction-override attempts
  • Logs blocked attempts in the Audit Trail
  • Triggers graceful fallback response when blocked
📜

Audit Trail & Compliance

Every agent interaction — prompts sent, actions invoked, responses generated — is logged in the Salesforce Audit Trail and the Einstein Generative AI Audit object for compliance and debugging.

  • GenAiAudit sObject stores full interaction records
  • Queryable via SOQL for compliance reporting
  • Retention policies configurable via Data Management
  • Supports data residency requirements (EU, US)
🔑

Agent Permissions & Org Security

Agents run under a named Salesforce User (the "Agent User") whose Profile, Permission Sets, and Sharing Rules control exactly what data and operations the agent can access — no more, no less.

  • Least-privilege principle: agent user has only required permissions
  • Field-Level Security respected on all CRM queries
  • Record sharing and OWD enforced as for any user
  • Named credentials for all external system callouts
🌐

Model Selection & Bring-Your-Own-LLM

Agentforce ships with Salesforce-hosted models but supports connecting your own (Azure OpenAI, AWS Bedrock, Google Vertex AI) via the Model Provider framework — keeping data within your contracted cloud boundary.

  • Model Provider metadata type registers external LLM endpoints
  • Named credentials secure API keys for external models
  • Trust Layer still enforces PII masking even on BYO models
  • Model benchmarks available in the Einstein AI Centre
Trust as a feature, not a constraint The Einstein Trust Layer is what makes Agentforce viable in regulated industries — financial services, healthcare, government. The zero-retention commitment and field-level security enforcement turn "we can't use AI with customer data" into a solved problem.
09Developer Guide

Building Custom Agents with Apex & Flow

Custom agents are where the Agentforce platform meets real-world complexity. Here is the end-to-end developer workflow for building a production-grade agent.

🔧

Invocable Apex — Best Practices

Apex actions are the most powerful tool in an agent's arsenal. Write them with the LLM caller in mind: clear input/output schemas, good descriptions, and no side effects the agent can't observe.

public class GetOrderStatus {
    public class Request {
        @InvocableVariable(required=true
            description='Salesforce Order record Id')
        public Id orderId;
    }
    public class Result {
        @InvocableVariable(description='Current order status')
        public String status;
        @InvocableVariable(description='Estimated delivery date')
        public Date estimatedDelivery;
        @InvocableVariable(description='Carrier tracking URL')
        public String trackingUrl;
    }
    @InvocableMethod(
        label='Get Order Status'
        description='Returns real-time shipping status, estimated delivery '
                  + 'date, and tracking URL for a given order.'
        category='Order Management'
    )
    public static List<Result> execute(List<Request> reqs) {
        List<Result> results = new List<Result>();
        for (Request r : reqs) {
            Order o = [SELECT Status, Estimated_Delivery__c,
                              Tracking_URL__c
                       FROM Order WHERE Id = :r.orderId LIMIT 1];
            Result res = new Result();
            res.status = o.Status;
            res.estimatedDelivery = o.Estimated_Delivery__c;
            res.trackingUrl = o.Tracking_URL__c;
            results.add(res);
        }
        return results;
    }
}
🌊

Auto-launched Flow Action — Pattern

For record operations that don't require Apex, an auto-launched Flow is the right choice. Define clear input/output variables and document their purpose — the agent reads them.

  • Flow API Name: Agent_Create_Return_Request
  • Input: OrderId, ReturnReason, CustomerNote
  • Output: ReturnCaseNumber, DropOffLocations
  • Description: "Creates a return request case for an order and returns the case number and nearest drop-off locations."
See Example
In Agent Builder → Actions → Add Action → Type: Flow → Select Agent_Create_Return_Request. The agent builder auto-generates the parameter mapping. In the Topic Instruction, add: "When a customer wants to return an item, use the Create Return Request action. Always confirm the reason before invoking it."
🧪

Agent Testing Framework

Agentforce ships a dedicated AgentTestSuite metadata type that records expected conversation flows as replayable automated tests — runnable in CI/CD pipelines before each promotion.

// AgentTestSuite: Service_Agent_Core_Flows
{
  "testCases": [
    {
      "name": "Order Status Happy Path",
      "userUtterances": [

      "expectedActionsInvoked": ["Get_Order_Status"],

    },
    {
      "name": "Escalation Trigger",
      "userUtterances": [
        "This is ridiculous, I want to speak to a manager NOW"
      ],

    }
  ]
}
📐

Topic & Instruction Design Patterns

Well-structured topics make agents more reliable. Follow these design conventions to minimise topic misclassification and action selection errors.

  • One topic per business domain (Billing, Orders, Returns, Technical Support)
  • Topics should not overlap in scope — ambiguity causes mis-routing
  • Write instructions as numbered rules, not paragraphs
  • End every topic with an explicit escalation condition
  • Include "what NOT to do" rules for common misuse patterns
Test each topic in isolation Use the Agent Simulator with utterances designed to stress-test the boundary between adjacent topics — e.g., "I need to cancel and refund my order" tests whether the agent classifies under Orders, Cancellations, or Billing.

Define the agent identity

Write the system-level role, persona, and hard guardrails in Agent Builder's Identity section.

Create topics and instructions

Enumerate the domains the agent handles; write precise, numbered instructions per topic.

Build and register actions

Create Apex @InvocableMethod or auto-launched Flows; register them to the relevant topics in Agent Builder.

Test in the Simulator

Run through all happy-path and edge-case scenarios; observe the reasoning chain and action selection log.

Write AgentTestSuite cases

Convert manual test scenarios into automated AgentTestSuite records for CI/CD regression coverage.

Promote and activate

Deploy via Copado or SFDX to Production; assign to a channel; activate and begin monitoring.

Monitoring, Analytics & Iteration

Deploying an agent is the beginning, not the end. A production Agentforce agent requires the same operational discipline as any other production system — dashboards, alerts, feedback loops, and a continuous improvement cadence.

📊

Agentforce Analytics Dashboard

The out-of-the-box Agentforce analytics dashboard surfaces the metrics that matter most for evaluating agent quality and coverage.

  • Containment Rate — % of conversations fully resolved without human handoff
  • Escalation Rate — % transferred to human agents
  • Topic Distribution — volume breakdown by topic classification
  • Action Success Rate — % of action invocations that completed without error
  • CSAT — post-conversation satisfaction scores where collected
💬

Conversation Review & Feedback

Every agent conversation is stored and reviewable. Use the Conversation Review tool in Service Cloud to annotate problematic turns, flag incorrect action selections, and generate training examples for instruction refinement.

  • Filter by escalated, abandoned, or low-CSAT conversations
  • View the full reasoning chain and actions invoked per turn
  • Tag conversations with quality labels
  • Export tagged samples to refine topic instructions
🔔

Operational Alerts

Set up Salesforce Flow-based alerts to notify the owning team when key agent health metrics breach thresholds.

  • Escalation rate exceeds 40% → alert the agent owner
  • Action error rate > 5% → alert the developer on-call
  • Average handling time up 20% week-over-week → review topic instructions
  • Any blocked prompt injection attempt → alert the security team
🔁

The Continuous Improvement Loop

The most successful Agentforce deployments treat the agent as a living system — with a weekly iteration cadence, not a set-and-forget launch.

Collect

Pull the week's low-CSAT and escalated conversations from the Conversation Review tool.

Diagnose

Classify failures: wrong topic? wrong action? missing action? bad instruction rule?

Refine

Update instructions, add missing actions, or tune topic scope to address identified failure modes.

Test

Re-run the AgentTestSuite and simulate the specific failing scenarios in the Simulator.

Deploy

Promote the updated agent metadata through the pipeline to Production.
