PILLAR 01 RunAllTests — Regular Cadence
Test Coverage Health

Run All Test Classes on Wednesdays and Fridays — Without Exception

Running Run All Tests on a fixed mid-week cadence is not optional housekeeping. It is your earliest warning system for coverage rot, newly broken assertions, and inter-class dependencies that were silently introduced since the last deployment.

Recommended Weekly Schedule
📋
Mon
Sprint work begins
🔨
Tue
Feature dev continues
✅
Wed
Run All Tests — mid-week health check
🔨
Thu
Fix failures from Wed run
✅
Fri
Run All Tests — pre-weekend gate
💡
Why Wednesday AND Friday? Wednesday catches issues early enough to fix them before the week ends. Friday confirms the fix is stable and the codebase is clean before the weekend — when no one is around to respond to a broken deployment.
📉
Why Test Coverage Quietly Degrades

New Apex classes added without corresponding test classes silently lower overall org coverage. A class at 90% stays in memory — but ten new untested utility classes will drag org coverage below 75% without a single existing test failing.

Trigger refactors, handler splits, and framework extractions all move covered lines between files. Coverage numbers shift invisibly until the next Run All Tests reveals the truth.

⚠️
Silent failure mode Copado will block a Production promotion if Apex coverage drops below 75%. Discovering this during a release window — not on a Wednesday — is a preventable crisis.
🔗
Catching Hidden Inter-Class Dependencies

Running tests in isolation (per user story) hides failures that only surface when all classes execute together. A shared utility refactored in one story can break six test classes from unrelated stories — discoverable only by running everything.

The Wednesday run catches these cross-story collisions in the middle of the sprint, while the team has context and capacity to investigate. The Friday run confirms the fix.

✅
Rule of thumb If a test fails in Run All Tests but passes in isolation, you have a shared state problem — a static variable, a hardcoded Id, or a missing Test.startTest() boundary. These must be fixed, not suppressed.
📋
The Weekly Test Run Checklist
✓
Run All Tests from Setup → Apex Test Execution, not just local tests
✓
Record overall org coverage percentage — track trend week-over-week
✓
Flag any class below 80% coverage for immediate remediation
✓
Investigate all failures — never mark a failing test "known issue" without a ticket
✓
Review time-to-complete — a run that grows >20 min indicates test bloat to address
✓
Post results to the team Slack channel for shared visibility
✓
Block Friday promotions to UAT if any test is failing

PILLAR 02 Design Documents & Change Control for Shared Code
Governance for Cross-Track Code

Shared Apex Classes and Metadata Are Org-Wide Infrastructure — Treat Them That Way

Any Apex class, trigger handler, utility, or metadata record that is consumed by more than one track (Sales Cloud, Service Cloud, Communities, Integrations…) carries the blast radius of infrastructure. Changes need a design document and formal change control — not just a user story and a PR.

📄
Why a Design Document Is Non-Negotiable

When a shared utility like AccountUtils, a trigger framework base class, or a custom metadata type is changed without documented intent, every consuming team is flying blind. A design doc answers the three questions that prevent incidents: What is changing, why is it changing, and what else does this touch?

→
Problem statement — what gap or defect the change addresses
→
Consumer inventory — every class, flow, and package that calls this code
→
Interface contract — method signatures, return types, expected exceptions
→
Backward compatibility — what breaks if an existing caller is not updated
→
Test plan — which existing tests must pass, and what new tests are added
🔁
Change Control — The Gate That Protects Every Track

A Copado user story is sufficient for a change that lives in a single team's track. It is not sufficient for shared infrastructure. Cross-track changes require a formal change control process involving all affected track leads before any code is written.

🔵
Change Control Workflow Design doc drafted → reviewed by all track leads → approved in CAB or equivalent → implementation in a dedicated branch → full Run All Tests before merge → monitored deployment window.
⚠️
Without change control Track A refactors a shared trigger handler to fix their bug. Track B's tests silently start failing. Track C's integration breaks in UAT three sprints later. No one has the context to diagnose it because the change was never documented.
🗂️
What Counts as "Shared" Metadata

Not all metadata is equal. The following categories require design doc + change control because a mistake in any one of them propagates to every track simultaneously.

📌
Trigger frameworks and base handler classes
📌
Shared utility classes (DateUtils, SObjectUtils, etc.)
📌
Custom Metadata Types consumed across multiple features
📌
Permission Sets and Permission Set Groups used org-wide
📌
Validation rules on high-volume objects (Account, Contact, Opportunity)
📌
Sharing rules and OWD settings
📌
Named Credentials and External Services
🛡️
The Living Design Document

A design document written once and never updated becomes misleading faster than no document at all. For shared infrastructure, the doc must be a living artifact — versioned alongside the code in the same Git repository, updated with every change control cycle.

📁
Recommended structure Store design docs in /docs/shared/ in your repo. Each shared component gets its own markdown file. Changes to the component require a corresponding update to the doc in the same PR — reviewed by all track leads before merge is permitted.
PILLAR 03 System.runAs in Every Test Class
Permission-Aware Testing

Tests That Don't Use runAs Are Testing the Admin — Not the User

By default, Apex tests execute as a system administrator with all permissions enabled. This means a test can pass perfectly in your sandbox, then fail in Production because a real user doesn't have the field-level security, record access, or CRUD permission your code assumed. System.runAs() is the only way to test reality.

🔐
What runAs Validates That Default Tests Cannot

Without System.runAs(), Apex tests bypass all of Salesforce's security enforcement: sharing rules, field-level security, object-level CRUD, and record visibility. Any code that queries, updates, or renders data through a user's security context is untested if runAs is absent.

✓
Field-Level Security — can this user actually read/write the fields being accessed?
✓
Object CRUD — can this profile create, edit, or delete this SObject?
✓
Sharing rules — does this user see only the records they should?
✓
Permission Set features — is the feature correctly gated behind the PS?
✓
Role hierarchy visibility — does a manager see subordinate records as expected?
💻
The Correct runAs Pattern

Always create the test user with explicit profile and permission set assignment inside the test. Never rely on existing org users — org data is not test data.

@IsTest
private class OpportunityServiceTest {

  @TestSetup
  static void makeData() {
    // Create a user with the exact profile under test
    User salesRep = TestDataFactory.createUser(
        'Sales Rep', 'Standard User'
    );
    insert salesRep;
  }

  @IsTest
  static void testCloneOpportunity_asRep() {
    User salesRep = [
        SELECT Id FROM User
        WHERE LastName = 'TestRep' LIMIT 1
    ];

    System.runAs(salesRep) {
      // All assertions here run as the sales rep
      // FLS, CRUD, sharing rules all enforced
      Test.startTest();
        OpportunityService.cloneWithProducts(oppId);
      Test.stopTest();

      System.assertEquals(
          1,
          [SELECT COUNT() FROM Opportunity
           WHERE Name LIKE '%Clone%']
      );
    }
  }
}
⚖️
Profiles and Permission Sets to Always Test

Every significant code path should have a runAs test for the minimum-privilege user that would realistically encounter it. Testing only as Admin is testing a scenario that is almost never the real production case.

User TypeWhy Test It
Standard UserMost common profile — catches CRUD and FLS gaps
Read-OnlyVerifies no write operations leak through
Community / PortalHighly restricted — sharing and visibility critical
Integration UserConfirms API-only profile can reach needed objects
🚫
Common runAs Anti-Patterns
❌
Querying existing org users SELECT Id FROM User WHERE Name = 'John Smith' — org users change, get deactivated, and their profiles drift. Tests that depend on them are fragile and non-portable.
❌
Using System Admin for all runAs calls Admin bypasses every security check. Using Admin in runAs tests gives you the false comfort of coverage without testing real user scenarios.
✓
Always create users in @TestSetup Create users with precise profile and permission set assignments in @TestSetup via your Test Data Factory. This is the only repeatable, portable pattern.
PILLAR 04 Test Data Factory — One Factory for the Entire Org
Shared Test Infrastructure

Duplicated Test Data Is the Root of Fragile Tests — Centralize Everything

When every test class creates its own records with hardcoded values, the org becomes impossible to maintain. A single required field added to Account breaks 40 test classes overnight. A Test Data Factory means that change happens in exactly one place.

🧱
The Core Factory Pattern

A single TestDataFactory class with static builder methods is the standard. Methods should accept overrides so callers can customize only what they care about, while the factory handles all required fields and relationships.

@IsTest
public class TestDataFactory {

  // Account — all required fields set by default
  public static Account createAccount(String name) {
    return new Account(
      Name        = name,
      BillingCity = 'Boston',
      Phone       = '555-0100',
      Type        = 'Customer'
    );
  }

  // User — profile + permission set in one call
  public static User createUser(
    String lastName, String profileName
  ) {
    Profile p = [
      SELECT Id FROM Profile
      WHERE  Name = :profileName LIMIT 1
    ];
    return new User(
      LastName          = lastName,
      FirstName         = 'Test',
      Alias             = lastName.left(5),
      Email             = lastName + '@test.com',
      Username          = lastName + '@test.org.sf',
      ProfileId         = p.Id,
      TimeZoneSidKey    = 'America/New_York',
      LocaleSidKey      = 'en_US',
      EmailEncodingKey  = 'UTF-8',
      LanguageLocaleKey = 'en_US'
    );
  }

  // Opportunity with all mandatory fields
  public static Opportunity createOpportunity(
    Id accountId, String name
  ) {
    return new Opportunity(
      AccountId = accountId,
      Name      = name,
      StageName = 'Prospecting',
      CloseDate = Date.today().addDays(30)
    );
  }
}
💡
Why the Factory Pays for Itself Immediately

The value of a Test Data Factory compounds with every test class written. The break-even is the second test class that uses it. By the time you have ten test classes, each new required field or validation rule change costs one edit instead of ten.

🏭
The compounding return Org adds a new required field to Account. Without a factory: 40 test classes need individual edits — guaranteed broken tests discovered on the next Run All Tests. With a factory: 1 line changes in TestDataFactory.createAccount() — all 40 classes pass immediately.
⚡
Speed benefit Factory methods with doInsert=false patterns allow bulk DML in @TestSetup, dramatically reducing test execution time — a major factor in keeping Run All Tests fast enough to run twice weekly.
📐
Factory Design Principles
1
No hardcoded IDs anywhere — ever. Record Ids change between environments. Always query by name or external key.
2
Sensible defaults, explicit overrides — factory methods fill every required field; callers only pass what they need to vary.
3
doInsert parameter pattern — return the unsaved SObject for bulk caller inserts; avoids DML-in-a-loop inside the factory.
4
Builders for complex graphs — Account → Contact → Opportunity chains should have a single createOpportunityWithAccount() method rather than repeated boilerplate in every test.
5
Versioned alongside the code — the factory is production code that happens to carry the @IsTest annotation. It must be code-reviewed and kept current.
🚫
Anti-Patterns That Break Teams
❌
seeAllData=true Never use @IsTest(seeAllData=true). It makes tests depend on org data that differs between environments, causing tests to pass in Dev and fail silently in UAT.
❌
Inline record creation in every test method Repeated new Account(Name='Test'...) inside individual test methods. Each one is a future maintenance burden and a future broken test waiting to happen.
❌
Multiple competing factories Three different teams each building their own TestUtil class. They diverge, conflict, and create duplicate maintenance overhead. There must be one factory for the entire org — governed like shared infrastructure.
