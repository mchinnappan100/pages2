What is Copado?

Copado is a native Salesforce DevOps platform that manages the full software delivery lifecycle within Salesforce orgs.

Native Salesforce app built on the platform
End-to-end CI/CD for Salesforce metadata
Manages deployments across multiple orgs
Integrates with Git version control

A retail company has 3 Salesforce orgs: Dev, UAT, and Production. They install Copado on their Production org and connect all three orgs to a single pipeline. Developers now push Apex and Flows through a single tracked path instead of manually exporting change sets.

Result: full audit trail, no more "who deployed what" debates.

Copado Functions

Extend pipeline logic with serverless Functions — custom scripts that run at any pipeline stage.

Writing Functions in Python, Node, Bash
Attaching Functions to Quality Gates
Passing parameters and secrets securely
Chaining Functions with Results

A custom Python Function validates that all Jira tickets linked to User Stories have status "Approved" before Production deployment:

def run(context): stories = context.get("userStories") for story in stories: jira_id = story.get("jiraKey") status = jira_api.get_status(jira_id) if status != "Approved": return {"success": False, "message": f"{jira_id} not approved" } return {"success": True}

This Function is attached as a Quality Gate on the Production stage. If any ticket isn't approved, the promotion is blocked automatically.

Best Practices

What Great Teams Do

04

✓ One User Story per Feature

Keep User Stories small and focused. One story should map to one logical change. This minimizes conflicts and speeds up code review.

❌ Bad: US-001 "Q4 Salesforce Improvements" (50 metadata items) ✅ Good: US-045 "Add Renewal Date field to Contract" US-046 "Auto-populate Account Owner on Case"

✓ Always Retrieve Before You Commit

Run a fresh Retrieve Changes before committing metadata to ensure your User Story reflects the latest org state and avoids stale metadata issues.

Workflow: 1. Make changes in Dev sandbox 2. Click "Retrieve Changes" in Copado 3. Review diff — confirm only your changes appear 4. Commit to Git branch ⚠️ Skipping step 2 can capture unrelated changes

✓ Protect Production with Gates

Configure Quality Gates on the production pipeline stage — requiring green Apex tests, static analysis, and manual approval before any deployment proceeds.

Production Stage Gates (in order): Gate 1: PMD Scan → 0 Critical violations Gate 2: Apex Tests → ≥ 80% coverage Gate 3: Robotic Tests → all E2E tests green Gate 4: Manual Approval → Release Manager sign-off

✓ Never Deploy on Fridays

Schedule production deployments mid-week. This gives teams full business days to respond to any post-deployment issues without weekend emergency support.

Recommended deployment window: ✅ Tuesday 10:00–14:00 local time ✅ Wednesday 10:00–14:00 local time ⚠️ Thursday — only if hotfix is critical ❌ Friday afternoons — avoid entirely ❌ Day before a holiday

✓ Use Environment Variables for Config

Never hardcode org-specific values (URLs, IDs, credentials) in metadata. Use Copado's Environment Variables to manage these differences cleanly.

Environment Variable: PARTNER_API_URL Dev org: https://api-sandbox.partner.com QA org: https://api-qa.partner.com Prod org: https://api.partner.com In Apex: String url = System.Label.PARTNER_API_URL;

✓ Tag Releases with Milestones

Create Copado Release records and Milestones to group related User Stories. This makes it easy to rollback a complete feature set if needed.

Release: "Spring '25 — Wave 1" Milestone: Core Platform (due Mar 1) ├── US-101 Custom Lead Scoring ├── US-102 Opportunity Kanban View Milestone: Integrations (due Mar 8) ├── US-110 DocuSign Envelope Trigger └── US-112 Jira Auto-Sync

✓ Maintain a Healthy Git Repo

Periodically clean stale feature branches and run Git snapshots. A cluttered repository leads to slow merges and confusing conflict scenarios.

Monthly Git hygiene checklist: □ Delete merged feature branches older than 30 days □ Run a full Git Snapshot on Production □ Verify integration branches are up-to-date with main □ Review and close abandoned User Stories

✓ Test in Scratch Orgs (2GP)

For package-based development, spin up scratch orgs for isolated testing. This ensures that all dependencies are explicit and reproducible.

sfdx force:org:create -f config/project-scratch-def.json \ -a FinanceCore-Test \ --durationdays 7 # Install package dependencies first: sfdx force:package:install \ --package 04t5g000001XkLm \ --targetusername FinanceCore-Test

✓ Document Deployment Steps

Use Deployment Task descriptions and checklist items to capture manual steps (data loads, permission updates). These become your runbook automatically.

Deployment Task: "Post-Deploy Data Load" □ Export Account records from QA (report: Acct_Backup) □ Run DataLoader import on Production □ Verify record count matches QA (expected: 1,240) □ Notify #salesforce-ops channel when complete

✓ Enforce Code Coverage Minimums

Set a minimum Apex code coverage threshold (75%+) as a Quality Gate. Never let coverage degrade — it's a leading indicator of technical debt.

Quality Gate config: Metric: Apex Code Coverage Operator: Greater than or equal to Value: 80 Trend alert: If coverage drops >5% week-over-week, send alert to #dev-leads Slack channel

✓ Review Copado's Audit Logs

Regularly review deployment history and audit logs, especially in regulated environments. Copado provides this natively — use it for change management evidence.

Monthly compliance report pull: Filter: Deployment History → Last 30 days Export fields: • Deployed By (user) • Deployment Date/Time • Target Environment • Metadata Components changed • Approval chain (who signed off)

✓ Train Teams on Git Basics

Copado abstracts Git but developers still benefit from understanding branches, commits, and merges. Invest in Git literacy to prevent avoidable incidents.

Recommended Git training checklist for new Copado devs: □ What is a branch and why it matters □ How to read a diff / understand a conflict marker □ git log, git status — reading history □ What "rebase vs merge" means in plain English □ How to recover from a bad commit (git revert)
