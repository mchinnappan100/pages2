Parallel Pipeline Patterns

🚀
Option 1 — Multiple Pipelines Recommended
Best for teams with dedicated environments per release train
  • Create a separate Copado pipeline per release train
  • Each train gets dedicated Git branches and lower environments
  • Independent promotion flow — no shared gates
  • All trains converge into the shared Production branch
  • Supports hotfix pipelines and process-specific pipelines natively
Release Train A
  DEV-A ──► QA-A ──► UAT-A ──► PROD
                                 ▲
Release Train B                  │
  DEV-B ──► QA-B ──► UAT-B ──►──┤
                                 │
Hotfix Train                     │
  HOTFIX ───────────────────────►┘

Typical program breakdown:
  Main Release   Sprint 12, Sprint 13
  Feature Prog   Loyalty Project
  Hotfix         PROD Fixes

Each uses its own pipeline and release manager.
🍃
Option 2 — Multiple Release Branches
Useful when sharing QA / UAT environments across trains
  • Single set of environments, multiple release branches in Git
  • July release stabilises while August development continues
  • Hotfixes on hotfix/* can bypass all release trains
  • Similar to GitFlow, implemented within Copado branching
  • User Stories promote: feature → release branch → master
Branch strategy:

  master          ──► Production
  release/2026.07 ──► July release
  release/2026.08 ──► August release
  hotfix/*        ──► Emergency fixes
  feature/*       ──► User stories

Promotion path for a user story:
  feature/US-101
    └──► release/2026.07
          └──► master (Production deploy)
📦
Option 3 — Release Bundles
Ideal for weekly, bi-weekly, or monthly cadence releases
  • Group user stories into named release bundles
  • Stories promote independently through QA and UAT
  • Bundle deploys together to Production as a unit
  • Minimises metadata collisions across concurrent stories
  • Supports major, minor, and hotfix release stream classification
Release 2026-07
  ├── US-101  ✅ UAT passed
  ├── US-102  ✅ UAT passed
  └── US-103  ✅ UAT passed
  → Deploy bundle together to PROD

Release 2026-08
  ├── US-201  🔄 In QA
  └── US-202  🔄 In QA
  → Still progressing independently

Benefit: US-101 through 103 don't block
each other during testing — only join
at the Production deployment gate.
🔥
Option 4 — Feature Release + Hotfix Train
Very common Salesforce setup — parallel lanes to Production
  • Hotfix pipeline runs alongside the main release train
  • Emergency fixes go directly to Production without waiting
  • Back-promote hotfixes to active release branches after deploy
  • Keep release train frozen while hotfixes proceed
  • Single Production org serves both lanes safely
                     ┌──► Hotfix Pipeline ──────────► PROD
                     │                                  ▲
DEV ──► QA ──► UAT ──┤                                  │
                     │                                  │
                     └──► Monthly Release ─────────────►┘

Rules:
  1. Hotfixes bypass the release train entirely
  2. After hotfix deploys: back-promote to release branch
  3. Freeze release train metadata during active hotfix
  4. Resume release train once hotfix is merged back

Git Branch & Release Train Layout
Three concurrent trains running against a common master
  • master — Production-ready truth
  • release/current — active sprint deploying to PROD
  • release/next — next sprint in development
  • hotfix/* — emergency fixes, bypass release trains
  • feature/* — individual user story branches
Git branches:
  master           ← Production source of truth
  release/current  ← Current Release train
  release/next     ← Next Release train
  hotfix/*         ← Emergency fixes
  feature/*        ← User story work

Concurrent trains:
  Current Release  → Team A deploying now
  Next Release     → Team B building ahead
  Hotfixes         → Ops deploying anytime

All three run simultaneously without blocking.
Critical Controls

What to Enforce When Running Parallel Trains

⚠ 1 — Enable Overlap Detection

Prevent two user stories in different trains from modifying the same metadata component simultaneously. Copado's overlap awareness flags competing changes before they reach a merge conflict at Production.

High-risk components to monitor: Account.object Opportunity.object Profile PermissionSet Copado overlap detection surfaces these conflicts during promotion — not at deploy.

↑ 2 — Frequent Back Promotions

After every Production deployment, back-promote changes down the pipeline to keep all trains in sync with PROD. This prevents branches from drifting apart and accumulating merge debt.

Back-promote after each PROD deploy: PROD ↓ back-promote UAT ↓ back-promote QA ↓ back-promote DEV branches Prevents pipeline drift.

🔒 3 — Separate Hotfix Pipeline

Never route emergency fixes through a frozen release train. A dedicated hotfix pipeline gives Ops a clear, unblocked path to Production at any time without disturbing in-flight releases.

❌ Wrong: Hotfix queued behind Sprint 13 release → 2-week delay for a production outage fix ✅ Right: Hotfix Pipeline → PROD (same day) Then back-promote hotfix to release branches

👥 4 — Assign Release Managers

Each train must have a dedicated release manager with clear ownership. Without ownership boundaries, cross-train decisions stall and merge windows get missed.

Train A → Release Manager A Train B → Release Manager B Hotfixes → Production Support Lead Each RM owns: - Promotion approvals - Merge conflict resolution - Deployment scheduling

Pattern Comparison

Option Pattern Best For Key Requirement Complexity
Option 1 Multiple Pipelines Teams with dedicated lower environments per train Separate sandbox orgs for each train Medium — org count increases
Option 2 Multiple Release Branches Teams sharing QA / UAT environments Disciplined branch naming and merge strategy Low — fewer orgs, more Git discipline
Option 3 Release Bundles Cadence-driven releases (weekly / bi-weekly) Stories must be independently promotable Low — native Copado bundle feature
Option 4 Feature + Hotfix Train Any org needing emergency fix capability Dedicated hotfix pipeline and back-promote process Low–Medium — common Salesforce pattern
Recommended Multiple Pipelines + GitFlow Branches Large multi-team enterprise programs Release managers per train, overlap detection on Medium — highest control and visibility
