01

Foundational Principles

The non-negotiable rules every high-performing Copado team lives by. Get these right first before optimizing anything else.

Source of truth

Git is the single source of truth

Every metadata change, merge, and conflict resolution must be tracked in Git. All promotions go through version control — no exceptions and no side-door changes.

Automation

Quality gates are non-negotiable

Gates enforce standards automatically. No human can bypass a failed PMD scan or insufficient Apex coverage without an explicit override and a documented justification.

Traceability

Every change links to a User Story

If a metadata change cannot be traced to a User Story, it should not exist in your pipeline. This is the foundation of a full audit trail and enables clean, surgical rollback.

Size

Small, focused User Stories win

One story per logical change. Atomic User Stories minimize merge conflicts, speed up code review, isolate failures, and make rollback precise rather than catastrophic.

Compliance

Audit trail is automatic — use it

Every deployment action (who, what, when, with whose approval) is logged natively in Salesforce. No separate infrastructure is needed for compliance reporting.

Separation

Separation of duties, always

The developer who writes the code must never be the same person who approves it for Production. Configure Copado Approvals to enforce this — not just a policy document.

Protect Production with layered gates

Quality Gate configuration

Configure Quality Gates in order on the production stage. Each gate builds on the last — no gate should be bypassed even for "urgent" fixes. Urgent fixes still need all gates.

Production Stage Gates (enforce in order): Gate 1: PMD Scan → 0 Critical violations Gate 2: Apex Tests → ≥ 80% coverage Gate 3: Robotic E2E → all suites green Gate 4: CAB Approval → Release Manager sign-off Override policy: document justification, senior approval
⏰

Never deploy on Fridays

Deployment scheduling policy

Schedule production deployments mid-week. This gives teams full business days to respond to any post-deployment issues without weekend emergency support burnout.

✓ Tuesday 10:00–14:00 local time ✓ Wednesday 10:00–14:00 local time ⚠ Thursday only if hotfix is truly critical ✗ Friday afternoon — avoid entirely ✗ Holiday eve — avoid entirely Exception: P1 incidents follow the hotfix runbook
⚙️

Use Environment Variables for config

No hardcoded org-specific values

Never hardcode org-specific values — URLs, IDs, credentials — inside metadata. Use Copado's Environment Variables so the same User Story deploys cleanly across all orgs.

Environment Variable: PARTNER_API_URL Dev org : https://api-sandbox.partner.com QA org : https://api-qa.partner.com Prod org : https://api.partner.com // In Apex: String url = System.Label.PARTNER_API_URL;

Copado Release records

Create Copado Release records and Milestones to group related User Stories. This makes it easy to rollback a complete feature set and communicate scope to stakeholders.

03

Git Practices

A clean, well-maintained Git repository is the backbone of a healthy Copado implementation. These practices prevent 80% of common merge disasters.

🌿

Always retrieve before you commit

Prevent stale metadata issues

Run a fresh Retrieve Changes before committing metadata. This ensures your User Story reflects the latest org state and avoids capturing unrelated changes made by colleagues.

  • 1Make changes in Dev sandbox
  • 2Click "Retrieve Changes" in Copado
  • 3Review diff — confirm only your changes appear
  • 4Commit to Git branch
⚠

Skipping the retrieve step is the single most common cause of "phantom changes" appearing in deployments.

🔀

Maintain a healthy Git repository

Monthly hygiene checklist

Periodically clean stale feature branches and run fresh Git snapshots. A cluttered repository leads to slow merges, confusing conflicts, and stale baselines.

  • □Delete merged feature branches older than 30 days
  • □Run a full Git Snapshot on Production
  • □Verify integration branches are up-to-date with main
  • □Review and close abandoned User Stories
  • □Confirm branch naming conventions are followed
🌳

Trunk-based branching model

Short-lived feature branches

Copado implements a trunk-based model. Feature branches live only as long as the User Story they represent. Long-lived branches accumulate drift and produce painful merges.

main ← Production baseline └── integration/dev ← Dev environment branch │ └── feature/US-042 ← Lives ≤ 1 sprint │ └── feature/US-043 └── integration/qa ← QA environment branch └── integration/uat ← UAT environment branch
🏷️

Meaningful commit messages

Structured commit convention

Enforce a commit message convention across the team. A clear commit history is the fastest path to understanding what changed and when — especially at 2am during an incident.

✓ feat(US-042): Add Renewal Date field to Contract ✓ fix(US-047): Correct validation rule on Opportunity ✓ chore(US-050): Remove deprecated workflow rule ✗ "fixed stuff" ✗ "WIP" ✗ "asdf"
04

Testing & Code Quality

Automated quality enforcement that catches issues in Dev and QA — not at 3pm on release day. Prevention is always cheaper than firefighting.

📊

Enforce code coverage minimums

Quality Gate configuration

Set a minimum Apex code coverage threshold (80%+) as a Quality Gate. Never let coverage degrade — it is a leading indicator of technical debt and future deployment failures.

Quality Gate config: Metric : Apex Code Coverage Operator: Greater than or equal to Value : 80 On Fail : Block promotion, notify developer Trend alert: if coverage drops >5% week-over-week → alert #dev-leads Slack channel
🔍

Zero tolerance for critical PMD violations

Static code analysis

Critical PMD violations always block promotion. Medium violations generate a warning. Configure this per pipeline stage — QA can be lenient, Production must be strict.

PMD scan result on AccountService.cls: [CRITICAL] Line 42: SOQL inside for loop Rule: AvoidSoqlInLoops [MEDIUM] Line 78: Empty catch block Rule: EmptyCatchBlock Gate result: ❌ BLOCKED — 1 Critical violation Fix: move SOQL outside loop, re-promote
🤖

Automate E2E tests with Robotic Testing

Copado Robotic Testing

Record Robotic test suites for critical user journeys and run them automatically on every promotion to UAT. E2E tests are the safety net static analysis cannot provide.

Test Suite: Opportunity E2E Step 1: Navigate to /lightning/o/Opportunity/new Step 2: Assert modal = "New Opportunity" Step 3: Fill Name = "Test Deal Q1" Step 4: Select Stage = "Prospecting" Step 5: Click Save Step 6: Assert record created successfully On failure: block promotion, link to User Story
🧪

Test in Scratch Orgs (2GP)

Package-based development

For package-based development, spin up scratch orgs for isolated testing. This ensures all dependencies are explicit and reproducible — no hidden org state can mask bugs.

# Create a fresh scratch org sfdx force:org:create \ -f config/project-scratch-def.json \ -a FinanceCore-Test --durationdays 7 # Install package dependencies first sfdx force:package:install \ --package 04t5g000001XkLm \ --targetusername FinanceCore-Test
05

Governance & Compliance

Enterprise-grade controls for regulated industries — and a good idea for everyone else too. Change management that is auditable by default.

📋

Document deployment steps as runbooks

Deployment Task descriptions

Use Deployment Task descriptions and checklist items to capture every manual step — data loads, permission updates, record sharing changes. These become your runbook automatically.

Deployment Task: "Post-Deploy Data Load" □ Export Account records from QA □ Run DataLoader import on Production □ Verify record count matches QA (1,240) □ Notify #salesforce-ops channel
🔎

Review audit logs regularly

Compliance reporting

Review deployment history and audit logs regularly — especially in regulated environments. Copado provides this natively. Use it for evidence, post-incident review, and quarterly audits.

Monthly compliance report pull: Filter : Deployment History → Last 30 days Export fields: • Deployed By (user) • Deployment Date/Time • Target Environment • Metadata Components changed • Approval chain (who signed off)
🔐

Configure roles and permissions correctly

Access control model

Copado ships with pre-built permission sets that map to DevOps roles. Map them deliberately to real people — do not give everyone Copado Admin just for convenience.

Sarah (Admin) → Configure pipelines, manage Git James (Dev) → Create stories, commit, promote Dev/QA Lisa (Rel Mgr) → Promote to UAT and Production Bob (Read Only)→ View history and logs only Org credentials: OAuth only — no plain-text passwords
🌐

Multi-org all-or-nothing releases

Release orchestration

When releasing across multiple org pipelines simultaneously, use Copado Release records to create an all-or-nothing gate. Partial releases that break cross-org integrations are the worst outcome.

Release: Spring '25 — Wave 1 Pipeline A: Sales Cloud NA → UAT ✅ Pipeline B: Sales Cloud EU → UAT ✅ Pipeline C: Service Cloud → UAT ✅ Release Gate: All 3 pipelines at UAT? → Yes: proceed to coordinated prod deploy
06

Team Culture & Collaboration

The best pipelines fail when teams do not support them. These practices build the human side of DevOps maturity.

🎓

Train teams on Git fundamentals

Copado abstracts Git — but not enough

Copado abstracts Git workflows but developers still benefit greatly from understanding branches, commits, and merges. Invest in Git literacy to prevent avoidable incidents.

  • □What is a branch and why it matters
  • □How to read a diff and understand a conflict marker
  • □git log, git status — reading history
  • □What "rebase vs merge" means in plain English
  • □How to recover from a bad commit with git revert
🔔

Wire Slack and Jira notifications

Deployment awareness for everyone

Connect Copado events to Slack and Jira so the entire team has real-time visibility into pipeline state. Eliminates the "did it deploy yet?" message thread entirely.

🚀 Copado Deployment — Production Pipeline : Sales Cloud Release Stories : US-105, US-112, US-118 Status : ✅ Success Duration : 4m 22s By : Lisa Chen (Release Manager) Jira: linked tickets auto-transition to "Done"
📅

Run a weekly pipeline health review

15-minute standing meeting

Hold a short weekly review of pipeline metrics — deployment frequency, lead time, failure rate, and mean time to restore. What gets measured gets managed.

Deployment frequency

Are teams shipping small changes often, or batching large risky releases?

Gate failure rate

Which Quality Gates fire most often? That is where to invest in developer enablement.

Mean time to restore

When production incidents occur, how quickly can the team detect and roll back?

🤝

Shift-left on conflict prevention

Proactive coordination beats reactive resolution

Most merge conflicts are preventable. Developers working on overlapping components should communicate proactively — before writing the first line of code.

  • ✓Check open User Stories in Copado before starting work
  • ✓Flag shared metadata components in daily standups
  • ✓Merge integration branches into feature branches early and often
  • ✗Assume nobody else is touching the same component

Advanced Patterns

Patterns for teams who have mastered the basics and are ready to push their DevOps practice to the next level.

⚡

Extend pipelines with Copado Functions

Custom serverless logic

Any custom logic — Jira validation, security checks, data migration scripts — becomes a first-class pipeline citizen via Copado Functions with full result reporting back to Copado.

def run(context): stories = context.get("userStories") for story in stories: status = jira_api.get_status(story["jiraKey"]) if status != "Approved": return {"success": False} return {"success": True}
🤖

Leverage Copado AI for risk prediction

AI-assisted delivery decisions

Copado AI analyzes deployment history and metadata patterns to predict risk scores per deployment. Use these scores to prioritize testing effort and make informed go/no-go decisions.

Story: "Allow Sales reps to clone Opportunities" AI-Generated Test Cases: TC-01: Clone Opp without Products TC-02: Clone Opp with 5 Products TC-03: Clone Opp — verify stage resets TC-04: Clone as non-owner — sharing rules Risk: MEDIUM — Opp trigger touched by 3 active stories
🔗

Integrate via Copado REST API

Trigger pipelines from external systems

Use the Copado REST API to trigger promotions from GitHub Actions, Jenkins, or any external system. This enables full CI/CD automation without being limited to Salesforce-native tools.

POST /services/apexrest/copado/v1/promote Authorization: Bearer {session_token} { "userStoryId": "a0B5g000001XkLm", "targetEnvironment": "QA", "runTests": true, "notifySlack": true }
📦

Adopt 2nd-Gen Packaging (2GP)

Module-based deployments

Shift from org-based deployments to modular, versioned Salesforce 2nd-generation packages. Dependencies become explicit, versions are reproducible, and rollback is a version number change.

{ "packageDirectories": [ { "package": "FinanceCore", "versionNumber": "2.4.0.NEXT" }, { "package": "FinanceReports", "dependencies": [{ "package": "FinanceCore@2.4.0.1" }]}] }

Define what qualifies as a hotfix

Severity classification first

Not every Production issue is a hotfix. Establish clear P1/P2 criteria — and stick to them. Hotfix pipelines bypass normal safeguards, so invoking them for non-critical issues erodes both quality and trust in the process.

P1 — Hotfix required immediately: • Production down for all users • Data loss or corruption actively occurring • Critical integration broken (payments, auth) • Regulatory/compliance breach P2 — Hotfix may be warranted: • Core feature broken for majority of users • Workaround exists but is unacceptable at scale P3/P4 — Normal pipeline, next release: • Cosmetic issues, minor defects • Feature not working as intended (not broken)
🌿

Create a dedicated hotfix branch

Branch off main — never off integration

Always branch your hotfix off main — the Production baseline — not off integration/dev. Branching off dev risks pulling unreleased changes into Production alongside your fix.

# Branch off the Production baseline main └── hotfix/HF-2025-001-payment-timeout # Fix scoped to the issue only # No refactoring, no unrelated changes # NOT this: integration/dev └── hotfix/... ← risks pulling unreleased work to Prod
⚡

Streamlined — not zero — quality gates

Speed with a minimum safety floor

A hotfix pipeline can skip E2E Robotic tests and manual approval layers — but it must never skip Apex tests on the classes directly touched by the fix. A broken hotfix is worse than the original incident.

Hotfix gate configuration (minimal floor): ✓ Keep: PMD scan on changed files only ✓ Keep: Apex tests on directly affected classes ✓ Keep: Release Manager approval (expedited) ✓ Keep: Deployment to a Hotfix sandbox first ⚡ Streamline: Skip full E2E Robotic suite ⚡ Streamline: Skip UAT sign-off (time-boxed) ✗ Never skip: Apex tests on touched classes ✗ Never skip: Deployment validation step
👥

Assemble the right team immediately

Hotfix war room protocol

Speed in a hotfix comes from having the right people in the room instantly — not from skipping process. Define roles before an incident happens so nobody wastes time figuring out who does what at 2am.

  • ✓Incident Commander — owns the war room, makes go/no-go calls
  • ✓Developer — writes the fix, scoped to the issue only
  • ✓Release Manager — owns the Copado promotion and approval
  • ✓QA / Verifier — validates fix in hotfix sandbox
  • ✓Comms owner — updates stakeholders throughout
↩

Back-merge within 24 hours

The step teams most often forget

After the hotfix deploys, merge the hotfix branch back into main AND into all active integration branches. Skipping this creates a permanent divergence — your next regular release will overwrite the hotfix.

# After Production deploy — same day: 1. Merge hotfix → main git merge hotfix/HF-2025-001 → main 2. Merge hotfix → integration/dev git merge hotfix/HF-2025-001 → integration/dev 3. Merge hotfix → integration/qa git merge hotfix/HF-2025-001 → integration/qa ⚠ Missing step 2 or 3: Next regular release overwrites the hotfix
⚠

The back-merge is the step most teams skip when they're exhausted after an incident. Assign it to a named person before the war room disbands.

📝

Post-incident review within 48 hours

Blameless, forward-looking, documented

Every P1 hotfix must be followed by a blameless post-incident review. The goal is not to assign fault — it is to understand the system failure and make it impossible to repeat.

  • □Timeline of events (detection → resolution)
  • □Root cause identified (not just symptoms)
  • □Contributing factors — process, tooling, communication
  • □Action items with owners and due dates
  • □Detection gap: why did monitoring not catch this earlier?
  • □Prevention: what pipeline or test change prevents recurrence?
🔄

Consider rollback before hotfix

Rollback is often faster than a fix

Before spending time writing a fix, always ask: can we rollback the most recent deployment instead? Rolling back is faster, lower risk, and buys time to develop a proper fix without production pressure.

Decision tree at incident declaration: Was a deployment made in the last 2 hrs? Yes → Can we rollback that deployment? Yes → ROLLBACK FIRST, fix in normal pipeline No → Proceed with hotfix process No → Is there a known Salesforce platform issue? Yes → Open Salesforce support case, monitor trust.salesforce.com No → Proceed with hotfix process
📋

Keep the hotfix narrowly scoped

Fix the issue — nothing else

A hotfix must contain only the minimum change needed to resolve the incident. No refactoring. No "while we're in here" improvements. Every extra line is extra risk in an already high-pressure deployment.

✓ Good hotfix scope: • 1 Apex class modified (null check added) • 1 validation rule disabled temporarily • 1 permission set assignment corrected ✗ Scope creep to avoid: • "We also cleaned up the trigger while there" • "We refactored the method for readability" • "We added the feature that was blocked on this" Rule: if the diff is >200 lines, it's not a hotfix
