Sprint Rollover Breaks the Jira–Copado Sync

When a closed sprint rolls unfinished stories into the next one, Jira's Sprint field keeps both values. Copado's Sprint lookup can only ever hold one — and that mismatch is what silently breaks your sync.

01 Symptom

What's actually happening

The reported chain of events, in order — this is the exact failure sequence teams run into after a sprint close.

🔁

Sprint closes with open stories

Any Story still unresolved when a sprint is completed automatically rolls into the next sprint.
🧾

Sprint field keeps both values

Jira's native Sprint field doesn't overwrite — it appends. The story now shows the previous (closed) sprint and the new one.
🚫

Copado sync breaks

Copado's Sprint field is a single-value lookup. A multi-value Jira field has nowhere to go — the sync fails or maps incorrectly.
🔒

Can't clean it up in Jira

You can't manually strip the old sprint value — Jira won't let you edit the Sprint field on a closed sprint.
02 Why it happens

This is Jira's design, not a bug

Confirmed directly by Atlassian support — the multi-value behavior is intentional, and Copado's connector simply wasn't built to receive it.

🏛️

Atlassian's official explanation: the Sprint field retains the completed sprint because Jira is preserving the history of where the issue was originally planned to finish — so historic sprint reports stay accurate. It's expected behavior by design, triggered specifically by the "Move open issues to <Sprint>" option used when completing an active sprint.

See what Jira stores internally

A Story planned for Sprint 12, unresolved when Sprint 12 closes and rolled to Sprint 13, ends up with:

// Jira Sprint field (cf[10020]) — array, not scalar
Sprint: [ "Sprint 12 (closed)", "Sprint 13 (active)" ]

// Copado's mapped field — Sprint__c (Lookup, single value)
Sprint__c: null  // ← sync has nothing valid to write
03 Sequence

The break, step by step

Where in the pipeline the single-value assumption fails.

1

Scrum Master completes Sprint 12

"Move open issues to Sprint 13" is selected during Complete Sprint.

2

Jira auto-rolls unresolved stories

Sprint field is updated by the system user "Anonymous" to hold both sprint IDs.

3

Copado's Jira connector polls the issue

It reads the Sprint field expecting one value, receives an array instead.

✕

Sync fails or writes the wrong sprint

Depending on connector version, the record errors out, silently keeps a stale sprint, or drops the field entirely.

04 Fix paths

Four ways to resolve it

Ranked roughly from "fixes root cause" to "escalate to vendor." You don't need all four — pick based on how much you can change process vs. tooling.

✅

Move stories before sprint close

Fix
Prevents the dual-value from ever being written. Reassign unfinished stories manually before hitting "Complete Sprint" instead of using the auto-rollover option.
See the 3 ways to do this

Any of these move an issue to a new sprint without retaining the old value:

Method 1 — Backlog view → right-click issue → "Send to" → target sprint
Method 2 — Backlog view → drag-and-drop issue into next sprint
Method 3 — Open issue → click Sprint field → set new value directly
⚙️

Add a "Current Sprint" automation field

Workaround
Doesn't touch process. A Jira Automation rule mirrors only the latest sprint into a dedicated field, which Copado maps to instead of the native Sprint field.
See the automation rule

Create a custom field (e.g. Current_Sprint__c equivalent, a Jira text field), then:

Trigger:  Sprint field changed on issue
Action:   Edit issue field → Current Sprint
Value:    {{sprint.last.name}}  // always resolves to newest sprint only

Then in Copado: map the Jira→Salesforce sync to
                this custom field, not the native Sprint field.
🧯

Turn off sprint auto-complete

Workaround
If sprints are configured to auto-start/auto-complete, that's what's silently triggering the rollover in the first place — under the "Anonymous" user, with no chance to intervene.
What to check

On the sprint's settings, look for both of these being enabled together:

☑ Start and complete sprint automatically
☑ After sprint completes → Move open issues to: New Sprint

Disabling auto-complete gives your Scrum Master the chance to manually move issues first (Solution 1) before the sprint closes.

🎟️

Escalate to Copado support

Vendor
The single-value constraint lives in Copado's connector, not Jira. Other customers have hit this exact issue — worth asking if a "take latest sprint only" mapping mode exists or is planned.
What to reference in the ticket

Point support to the field mismatch directly:

Jira Sprint field:   customfield_10020 (array, multi-value)
Copado field:        Sprint__c (Lookup, single-value)
Ask for:             latest-sprint-only mapping option,
                      or graceful fallback instead of sync failure
