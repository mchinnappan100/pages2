How It Works

ℹ️ Copado connects to Jira via the Copado Integration User (a dedicated Salesforce automation user). When a Jira issue is linked to a User Story, Copado's webhook or Apex automation writes the Jira Key into the Jira_Key__c field on the User_Story__c object.
🔄

The Integration Pipeline

Jira issues and Copado User Stories are joined through a three-step synchronisation pathway managed entirely by the Copado Automation User — no developer intervention required.

  • A Jira issue (e.g. SF2-7856) is created in your Jira project
  • A Copado User Story (e.g. US-0001380) is created — manually or auto-generated
  • The Copado Automation User writes SF2-7856 into the Jira_Key__c field on US-0001380
  • From that point, status changes in Jira propagate back to the User Story and vice versa

Who is the "Copado Automation User"?

This is a dedicated Salesforce system user configured during Copado setup. It acts as the service account for all programmatic changes made by Copado's backend — including writing Jira Keys.

  • Username typically follows pattern: copado.automation@yourorg.com
  • Assigned the Copado_User permission set + Modify All Data on User Story objects
  • All field writes appear in Salesforce audit history as this user — giving a clear automated trail
  • Not a human user — logins are service-token based, not password-based

// Jira Webhook Payload (POST to Copado endpoint)
{
  "webhookEvent": "jira:issue_updated",
  "issue": {
    "key":     "SF2-7856",
    "fields": {
      "summary":  "Add Renewal Date to Contract",
      "status":   { "name": "In Progress" },
      "assignee": { "emailAddress": "j.smith@acme.com" },
      "sprint":   { "name": "Sprint 24" }
    }
  }
}

// Copado Apex Handler (simplified)
public class CopadoJiraKeyHandler {
  public static void upsertUserStory(JiraPayload payload) {
    User_Story__c us = [
      SELECT Id, Jira_Key__c, Status__c
      FROM User_Story__c
      WHERE Jira_Key__c = :payload.issueKey
      LIMIT 1
    ];
    if (us == null) {
      us = new User_Story__c();
      us.Jira_Key__c = payload.issueKey; // "SF2-7856"
    }
    us.Name      = payload.summary;
    us.Status__c = payload.status;
    upsert us; // Run as Copado Automation User
  }
}

Automation Flow — Step by Step

🎯 The Real Scenario

A Jira ticket SF2-7856 ("Add Renewal Date to Contract") is created by a BA in Jira. Within moments, Copado's automation user creates and populates Copado User Story US-0001380 with the Jira Key — so developers can immediately commit Salesforce metadata against it.

📍

Step-by-Step: SF2-7856 → US-0001380

Follow exactly what happens from Jira ticket creation to fully-linked Copado User Story, driven by the automation user.

  1. Jira ticket created: Business Analyst creates SF2-7856 in Jira project "SF2 — Sales Cloud". Summary: "Add Renewal Date to Contract". Priority: High. Sprint: Sprint 24.
  2. Webhook fires: Jira sends a jira:issue_created webhook to Copado's configured endpoint in Salesforce.
  3. Copado handler runs: Copado's Apex CopadoJiraKeyHandler receives the payload, running as the Copado Automation User.
  4. User Story created: The handler creates a new User_Story__c record. Auto-name assigned: US-0001380.
  5. Jira Key stamped: Jira_Key__c is set to SF2-7856 by the automation user. Field history log records the change with timestamp.
  6. Developer assigned: Jira assignee j.smith@acme.com is matched to Salesforce user James Smith and set on Developer__c.
  7. Git branch auto-created: Copado creates a feature branch feature/US-0001380 in the connected Git repo.
  8. Developer starts work: James sees US-0001380 in his Copado queue, commits Salesforce metadata (custom field, page layout) to the feature branch.
  9. Promotion triggers status sync: When James promotes US-0001380 to QA, Copado's automation user calls the Jira API and transitions SF2-7856 to "QA Review".
  10. Deployment to Production: After final promotion, Jira ticket SF2-7856 is automatically closed as "Done".

Troubleshooting

⚠️ If Jira_Key__c is blank on a User Story that should be linked, start by checking the Copado Debug Log and confirming the automation user has field-level security access to Jira_Key__c.
🔎

Common Issues & Fixes

The most frequent reasons a Jira Key doesn't appear on a User Story — and how to resolve each one.

Issue 1: Jira_Key__c is blank after Jira ticket is created
─────────────────────────────────────────────────────────────────
SYMPTOM: US-0001380 created but Jira_Key__c = (blank)
CHECK 1: Jira Webhook → confirm URL points to correct Copado endpoint
          Expected: https://<your-org>.salesforce.com/services/apexrest/copado/jira
CHECK 2: Copado Debug Logs → look for "JiraKeyHandler: issueKey is null"
CHECK 3: Field-Level Security → ensure Copado Automation User profile
          has READ + EDIT access to Jira_Key__c
CHECK 4: Jira Project Mapping → confirm "SF2" is mapped to this pipeline
          Copado Setup → Jira Settings → Project Mappings

Issue 2: Jira Key written but status not syncing back
─────────────────────────────────────────────────────────────────
SYMPTOM: SF2-7856 still shows "In Progress" after US deployed to Prod
CHECK 1: Jira Workflow → does "Done" transition exist in SF2 project?
CHECK 2: Copado Status Mapping → verify "Deployed" → "Done" mapping
          Copado Setup → Jira Settings → Status Mapping
CHECK 3: Jira API Token → confirm automation user's Jira API token
          has not expired (tokens expire every 90 days by default)

Issue 3: Duplicate User Stories created
─────────────────────────────────────────────────────────────────
SYMPTOM: SF2-7856 resulted in US-0001380 AND US-0001381
CAUSE:   Webhook fired twice (Jira retry on timeout)
FIX:     Enable idempotency check in Copado Jira handler:
          check if Jira_Key__c already exists before insert

Best Practices

✓ Always Use a Dedicated Automation User

Never reuse a human developer's account as the Copado automation user. A dedicated service account makes field history readable — every Jira Key write is clearly attributed to automation, not a person.

✅ Good:  copado.automation@acme.com  (service account)
❌ Bad:   j.smith@acme.com            (developer account used for automation)

✓ One Jira Ticket = One User Story (where possible)

Resist bundling multiple Jira tickets into one User Story. Each story should map to exactly one Jira Key for clean traceability, rollback capability, and conflict-free Git branching.

✅ Clean traceability:
  SF2-7856 → US-0001380  (1:1 mapping, one feature branch)

❌ Messy:
  SF2-7856 + SF2-7857 → US-0001380  (shared story, harder to rollback)

✓ Validate the Jira_Key__c Format in Apex

Add a validation rule or Apex trigger to enforce that Jira_Key__c matches your project key pattern. This prevents typos that break the bi-directional sync.

// Salesforce Validation Rule on User_Story__c
// Error Message: "Jira Key must match pattern SF2-NNNNN"
AND(
  NOT(ISBLANK(Jira_Key__c)),
  NOT(REGEX(Jira_Key__c, "SF2-[0-9]{1,6}"))
)

✓ Monitor the Automation User's API Call Usage

The automation user makes Jira API calls on every status change. In busy sprints this can approach Salesforce API limits. Set up a daily alert on API usage for the automation user profile.

-- Query to check today's API usage by automation user --
SELECT UserId, Username, ApiCalls
FROM   ApiCallMetrics
WHERE  Username = 'copado.automation@acme.com'
AND    Date = TODAY
-- Alert if ApiCalls > 80% of daily limit

Frequently Asked Questions

❓

Can I manually type a Jira Key onto a User Story?

Yes. While the automation user writes the key programmatically, any Copado Admin can also manually type a Jira Key into Jira_Key__c. The two-way sync will activate as soon as a valid key is present in the field.

❓

What if the Jira project key changes?

If your Jira project is renamed (e.g. SF2 → SF3), you must update the Project Key Mapping in Copado Setup → Jira Settings → Project Mappings. Existing Jira_Key__c values on old User Stories are not auto-updated — those historical links remain as-is.

❓

Can one User Story link to multiple Jira tickets?

The standard Jira_Key__c field holds one key. For multi-ticket scenarios, use Copado's Jira Links related list (a child object) which supports multiple Jira keys per User Story — though two-way status sync only applies to the primary key in Jira_Key__c.
