Copado DevOps Governance

Critical DO NOTs for Branch Integrity Across Dev → QA → UAT → Main

Golden Rule

The branch is the source of truth. If metadata changes directly inside an org, the pipeline becomes unreliable and drift propagates downstream.

1. Do NOT Modify Metadata Directly in QA / UAT / Production

Developers should never update flows, validation rules, custom metadata, permissions, layouts, Apex, or config directly in shared environments.

Direct org changes create branch drift. Promotions later overwrite, miss, or conflict with actual org state.

2. Do NOT Grant Broad Deployment Access

Developers should not have unrestricted edit/deploy access to QA, UAT, or Mainline orgs.

Access bypasses Copado promotion controls and eliminates deployment traceability.

3. Do NOT Fix Bugs Directly in Higher Environments

Hotfixes made directly in QA/UAT “just to unblock testing” guarantee future merge conflicts and lost changes.

4. Do NOT Skip Promotion Gates

Never manually mark user stories complete or bypass Ready-to-Promote validation.

5. Do NOT Bundle Unvalidated Stories

Bundling unrelated stories masks hidden dependencies and causes metadata collisions downstream.

6. Do NOT Allow Environment Drift to Persist

Any detected drift must trigger immediate reconciliation: retrieve → commit → promote cleanly.

7. Do NOT Ignore Manual Steps Documentation

Missing flow activation steps, feature toggles, or post-deployment actions cause production instability despite successful deployments.

What Good Looks Like

  • All changes originate from feature branches
  • Every promotion is traceable to a Copado user story
  • Environment access is role-restricted
  • Drift detection runs regularly
  • Post-deployment manual steps are mandatory and reviewed
  • QA/UAT remain immutable validation mirrors
