Section 01
Phase Zero
Release Planning & Scope
📋
Define the Release Scope in Copado Before Any Work Begins
CriticalCopado

Every release must have a named Copado Release record with a defined scope before the sprint begins. Scope creep during a release is the single largest driver of deployment failures. Additions after UAT has started are strictly prohibited.

  • Create a Copado Release record before sprint kickoff
  • Link all User Stories to the Release at planning time
  • Set a release freeze date — no new stories after this date
  • Classify each story: Feature, Bug Fix, Config Change, or Data Migration
  • Assign a Release Manager as the single accountable owner
# Copado Release record — sprint kickoff: Release Name: REL-2025-Q2-Sprint-14 Target Env: Production Release Type: Standard Sprint Release Freeze Date: 2025-05-08 (no new Stories after this) Deploy Window: 2025-05-15 21:00–22:00 IST Release Manager: Naveen Kumar # Stories scoped at planning: US-1101 Feature — Opportunity scoring automation ✅ In scope US-1102 Bug Fix — Case SLA timer calculation ✅ In scope US-1103 Config — Profile permission adjustment ✅ In scope US-1104 Feature — Einstein Recommendation widget ❌ Moved to next sprint ❌ Never add stories after the freeze date ❌ Never add stories after UAT has begun
⚠️
Classify Release Risk Before Scheduling the Deployment Window
High Priority

Risk classification drives the depth of pre-deployment review, the CAB approval path, and the monitoring duration. Never schedule a deployment window without a documented risk assessment on the Copado Release record.

  • Low — config changes, permission sets, report/dashboard updates
  • Medium — Apex code, Flow changes, process automations, integration callouts
  • High — schema changes, data migrations, critical path automation, multi-org changes
  • High-risk releases require synchronous CAB review, not async email approval
  • Risk level determines post-deploy monitoring duration (1h / 4h / 24h)
🌿
Branch Strategy — One Story, One Branch, One Release
CriticalCopado Git

Every Copado User Story gets its own feature branch auto-created by Copado from the correct source. Never manually create branches or branch from another feature branch. All merging is orchestrated through Copado's deployment pipeline.

# Branch naming — Copado auto-creates: feature/US-1101-opportunity-scoring feature/US-1102-case-sla-timer-fix feature/US-1103-profile-permission-update # Source branch rules: main (Production) ← hotfixes and P1 defects only develop ← standard sprint features # After Production deployment — back-merge: develop ← merge feature/* (Copado handles automatically) release/sprint-14 ← merge develop ❌ Never branch from another feature/* branch ❌ Never manually push to develop or main
Section 02
Phase One → Four
Pipeline Execution
🔒
Gate-Controlled Promotion — No Manual Environment Pushes
CriticalCopado

Each environment gate requires a story status check. Copado Pipeline prevents promotion until the gate condition is satisfied — this is by design and must never be bypassed.

  • DEV → QA: Story status = "Committed"
  • QA → UAT: Story status = "QA Complete"
  • UAT → Prod: Release status = "UAT Complete" + CAB approved
  • No direct sandbox pushes via VS Code or Workbench
📦
Explicit Metadata — No Wildcards in Deployments
Critical

Every Copado deployment must use an explicit, minimal component list in the package manifest. Wildcard or full-org deployments have caused Production incidents at every org that attempted them.

  • Commit only the components changed for this story
  • Review the Copado diff before every commit
  • Run a metadata comparison against the target before deploying
  • Never include unrelated cleanup in the same branch
⚡
Use Copado Quick Deploy for Production — Validate + Deploy Separation
High PriorityCopado

Run a full validation deployment (including Apex tests) against Production during off-peak hours. Then Quick Deploy during the maintenance window. This reduces the actual maintenance window to under 5 minutes.

# Copado Validate → Quick Deploy workflow: Step 1 — Validate (run during off-peak hours, T-1 day) Copado Release → Deploy to Production (Validate Only) Run Tests: Run Specified Tests → OpportunityScoringTriggerTest → CaseSLACalculatorTest Result: PASS ✅ (valid for 10 days) Step 2 — Quick Deploy (during maintenance window) Copado Release → Quick Deploy Estimated time: ~3 minutes No test re-run required (Salesforce platform optimisation) ✅ Users experience <5 min impact window ❌ Never run full deploy + tests during peak hours
📸
Org Snapshots Before Every Promotion Step
High PriorityCopado

Take a Copado Org Snapshot of the target environment before every promotion. This creates a baseline for comparison, enables metadata drift detection, and serves as the rollback reference if a deployment must be reversed.

  • Snapshot before QA promotion — catches environment drift since last deploy
  • Snapshot before UAT promotion — validates QA deployment was clean
  • Snapshot before Production deployment — mandatory rollback baseline
  • Store snapshots named with date and release ID for traceability
Note: Copado Snapshot takes a point-in-time metadata inventory of the org. Compare the pre-deploy snapshot to post-deploy to verify only the expected components changed.
Section 03
Emergency Release
Hotfix Management
🔥
What Qualifies as a Hotfix — Strict Definition Required
CriticalHotfix

A hotfix is an emergency production fix for a Severity-1 or Severity-2 defect causing active business impact. Hotfixes bypass the standard sprint release cycle but must still flow through a controlled, documented process. "Urgency" is not sufficient justification — the impact criteria must be met.

Hotfix Criteria (ALL must apply): Active Production defect · User-facing or data-integrity impact · Cannot wait for the next sprint release · Formally approved by Release Manager + Business Owner
  • Sev-1: Production is down or data is corrupted — immediate action
  • Sev-2: Critical business process blocked for a group of users — same-day fix
  • Sev-3 and below: Schedule for the next sprint defect release, not a hotfix
  • The Release Manager makes the final call — developers do not self-declare hotfixes
# Hotfix trigger evaluation: Issue reported: Case trigger throws Apex exception in Prod Affected users: All Service Agents (250 users) Business impact: No new cases can be created — SLA breach risk ✅ Qualifies as Sev-1 Hotfix Approved by: Naveen Kumar (Release Manager) Business sign: Sarah Mitchell (Service Ops VP) Hotfix ticket: HF-2025-0042 # Does NOT qualify as hotfix: Issue: Dashboard label is misaligned for one user ❌ Sev-3 — defer to next sprint release
🌿
Hotfix Branch Strategy — Always Branch from main (Production)
CriticalCopado GitHotfix

Hotfix branches must always originate from main — the branch that mirrors Production. Branching from develop risks pulling in unreleased sprint changes, which could introduce regressions in the hotfix deployment.

# Hotfix branch — Copado creates from main: hotfix/HF-2025-0042-case-trigger-apex-exception # Source: main (mirrors Production exactly) ❌ Never branch hotfix from develop ❌ Never branch hotfix from release/sprint-* branches # After Production deployment — back-merge: git checkout develop git merge --no-ff hotfix/HF-2025-0042-case-trigger-... git push origin develop git checkout release/sprint-14 # active sprint branch git merge --no-ff hotfix/HF-2025-0042-case-trigger-... git push origin release/sprint-14 ⚠️ Back-merge must complete within 1 hour of Production deploy ⚠️ Failure to back-merge causes the same bug to reappear in next sprint release
📝
Create a Dedicated Copado User Story and Release for Every Hotfix
CriticalCopadoHotfix

Every hotfix must be tracked as a Copado User Story linked to an emergency Release record. This maintains audit trail, traceability from defect to commit to deployment, and enables Copado's gate controls even on the accelerated hotfix path.

# Copado User Story for hotfix: Story Title: [HOTFIX] HF-2025-0042 — Case trigger Apex exception Story Type: Hotfix Severity: Sev-1 Linked Defect: DEF-2231 Branch: hotfix/HF-2025-0042-case-trigger-apex-exception Release: HF-REL-2025-0042 # Emergency Release record: Release Name: HF-REL-2025-0042 Type: Emergency / Hotfix Risk Level: High (mandatory for all hotfixes) Deploy Window: 2025-05-09 18:00–18:30 IST CAB Approval: Emergency CAB (async + verbal bridge) ✅ All changes tracked in Copado — full audit trail ❌ No "quick push" via Workbench or VS Code to Production
🧪
Hotfix Testing — Accelerated but Never Skipped
CriticalHotfix

The hotfix path is accelerated, not untested. The testing scope narrows — but reproducing the defect, verifying the fix, running specified Apex tests, and obtaining QA and business sign-off are all mandatory steps that cannot be skipped.

  • Reproduce the defect in a hotfix sandbox (cloned from Prod snapshot)
  • Apply and verify the fix in the hotfix sandbox
  • Run specified Apex tests — all tests touching changed classes must pass
  • QA Lead sign-off: independent verification (not the developer)
  • Business Owner verbal + written confirmation before Production deploy
  • Full regression deferred — run within 24 hours post-deploy
# Hotfix QA Sign-Off — HF-2025-0042: Tester: Priya Ramaswamy (QA Lead) Sandbox: HOTFIX-SB01 (refreshed from Prod snapshot) Time elapsed: 2h 15m from defect report to QA pass Defect reproduced first: YES ✅ Fix verified (no exception): PASS ✅ Apex tests — specified run: 18/18 PASS ✅ Coverage: 82% ✅ Business Owner confirmation: Sarah Mitchell ✅ Story Status: QA Complete → Ready for Emergency Prod Deploy ⚠️ Full regression suite to be run post-deploy (within 24h)
🔙
Hotfix Rollback Plan — Written Before the Deployment Window Opens
CriticalHotfix

Even emergency deployments require a written rollback plan. In a high-pressure Sev-1 situation, improvising a rollback leads to extended outages. The rollback plan must be attached to the Copado Release record before the deployment window opens.

# Rollback Plan — HF-REL-2025-0042: Trigger: Post-deploy defect found within 60 minutes If deploy fails mid-flight: → Salesforce atomic deploy auto-rolls back ✅ → No manual action required If post-deploy defect discovered: Step 1: Release Manager initiates rollback in Copado Copado → Rollback Deployment → Confirm Step 2: Verify original defect re-appears (confirms rollback) Step 3: Notify stakeholders via Slack #hotfix-bridge Step 4: Log incident — DO NOT re-deploy without RCA Rollback Owner: Naveen Kumar (Release Manager) Estimated time: ~8 minutes Rollback tested: Yes — HOTFIX-SB01 ✅ ❌ "We'll figure it out" is not a rollback plan ❌ Rollback taking >15 min → change is too large for hotfix
📊
Post-Hotfix Review — Root Cause Required Within 48 Hours
High PriorityHotfix

Every hotfix must generate a post-incident review within 48 hours. The purpose is to identify root cause, document preventive measures, and ensure the defect class does not recur. Teams with more than 2 hotfixes per month must review their testing process.

  • Root cause — what broke and why it was not caught in QA
  • Fix summary — what was changed and why this approach was selected
  • Process gap — what testing or review step was missing
  • Prevention action — what checklist or automation now prevents recurrence
  • KPI: target ≤ 2 hotfixes per team per quarter
Hotfix KPI Thresholds: ≤ 2 hotfixes/team/quarter is healthy · 3–5/quarter signals a testing gap · 6+/quarter requires a release process review with leadership
Section 04
Planned Fix Release
Defect Fix Promotion
🔗
One Story Per Defect — No Bundling
CriticalCopado

Every defect fix must be its own Copado User Story with its own branch, its own traceability chain, and its own deployment record. Bundling defeats the ability to selectively roll back a single fix.

  • One Story per defect — always
  • Multiple defect Stories may share a single Release
  • Never combine a defect fix with a feature in the same Story
🚫
No Direct Changes in UAT — Absolute Rule
Critical

If a defect is found in UAT, it must be fixed back in QA and re-promoted through the full cycle. UAT is a read-only validation environment — never make direct metadata changes to the UAT sandbox.

  • UAT defect → fix in QA → QA re-validates → re-promote to UAT
  • No VS Code / Workbench patches applied to UAT
  • One violation breaks the entire audit trail
✅
Independent Sign-Off at Every Gate — Never Self-Approved
Critical

QA sign-off must be done by someone other than the developer. UAT sign-off must be done by the business owner, not QA. The fix author must not approve their own work at any stage. Separation of duties is non-negotiable.

# Sign-off Responsibility Matrix: Gate Sign-Off By Must NOT be ──────────────────────────────────────────────────── QA Gate QA Engineer / QA Lead The developer UAT Gate Business Owner / BizOps The QA team Prod Approval Release Manager The developer CAB Approval IT Change Manager Any dev team member # Copado User Story Status lifecycle: In Development → In QA → QA Complete → In UAT → UAT Complete → Deployed to Production ✅ ✅ QA Complete status unlocks UAT promotion in Copado ✅ UAT Complete + CAB Approved unlocks Prod promotion
🔄
Post-Deploy Back-Merge Within 2 Hours — No Exceptions
High PriorityCopado

After every Production deployment, immediately merge the fix branch back to develop and all active sprint branches. Failure to do this is the #1 cause of regressions in the next sprint release — the same bug re-appears because the fix was never merged forward.

# Post-deploy back-merge (within 2 hours): git checkout develop git merge --no-ff feature/DEF-2201-case-assignment-fix git push origin develop git checkout release/sprint-14 git merge --no-ff feature/DEF-2201-case-assignment-fix git push origin release/sprint-14 # Update records: Defect DEF-2201: Status = "Closed — Fixed in Prod" Fixed in Release: REL-2025-05-W2-Defects ✅ Track back-merge as a Copado Release checklist item ❌ Back-merge skipped → same defect recurs next sprint release

Change Advisory Board (CAB) Process
Critical

All Production deployments require an approved change ticket. Standard change templates reduce CAB overhead for routine sprint releases without bypassing governance. Every deployment — standard or hotfix — must have an approved change record before the deployment window opens.

# Change Ticket Template — Copado Sprint Release: Change Type: Standard (pre-approved template) Template: CHGT-STD-004 "Copado Sprint Release" Risk Level: Low / Medium / High (select one) Affected System: Salesforce Production Business Impact: {describe user-facing impact} Rollback Plan: {link to rollback doc in Copado Release} Test Evidence: {link to QA + UAT sign-off records} # CAB Review path by risk: Low risk → Async (email approval, 24h notice) Medium risk → Async (email approval, 48h notice) High risk → Synchronous CAB meeting required # Hotfix / Emergency change: Emergency CAB bridge call (15 min) 3 of 5 approvers minimum for Sev-1 Full CAB retroactive approval within 24h
📡
Post-Deploy Monitoring — Duration by Risk Level
High Priority

Assign a named on-call engineer to actively monitor Production after every deployment. Silent failures — Apex exceptions, integration errors, report anomalies — often surface hours after go-live. Monitoring duration is determined by release risk level.

  • Low risk release — 1-hour active monitoring window
  • Medium risk release — 4-hour active monitoring window
  • High risk release or hotfix — 24-hour active monitoring window
  • Review Apex Exception Logs, Failed Flows, integration error rates
  • Close with "All clear" note added to the Copado Release record
Section 06
Summary
What Great Teams Do

✓ Never Skip a Gate

QA → UAT → Prod is not a suggestion. Skipping UAT "because it's a small fix" is the #1 cause of Production regressions. Every release, regardless of size, goes through the full cycle.

✓ All Changes Flow Through Copado

No VS Code, Workbench, or Data Loader pushes to shared environments. Every metadata change must be committed to Copado and deployed through the tracked pipeline — always.

✓ Define Hotfix Criteria — Don't Self-Declare

Only the Release Manager approves a hotfix classification. Developers feeling pressure to "fix it now" is not a hotfix trigger. The severity criteria must be met.

✓ Write Rollback Plan Before Deploy

The rollback plan must be written and reviewed before the deployment window opens — for standard releases and hotfixes alike. A plan taking more than 15 minutes means the change is too large.

✓ Use Quick Deploy

Separate validation (with test runs, off-peak) from the actual go-live. Quick Deploy minimises the maintenance window to under 5 minutes while keeping governance intact.

✓ Back-Merge Within 2 Hours

Set a team rule: fix branches — including hotfixes — must be merged back to develop and all active sprint branches within 2 hours of Production deployment.

✓ Hotfix Sandbox = Prod Clone

Hotfix sandboxes must be refreshed from a Production snapshot taken immediately before the fix. Testing a hotfix in a stale sandbox produces false positives.

✓ Automate the Checklists

Encode pre-deployment and hotfix checklists as Copado checklist templates on the Release record. Humans skip manual checklists under pressure — templates make compliance the default.

✓ Post-Incident Review for Every Hotfix

Every hotfix generates a 15-minute structured review: root cause, fix summary, and what process change prevents recurrence. Log findings in the team wiki within 48 hours.

✓ Track Release KPIs Monthly

More than 3 unplanned defect fix releases or more than 2 hotfixes per quarter signals a systemic quality gap. Review in the sprint retrospective.

Target KPIs: ≤ 3 defect releases/team/month ≤ 2 hotfixes/team/quarter ≥ 95% first-time deploy success ≤ 5 days QA → UAT → Prod cycle

✓ Explicit Metadata — No Wildcards

Every Copado deployment must use a precise, minimal component list. Wildcard or full-org deployments have caused Production incidents at every org that tried them.

✓ Independent Sign-Off at Every Gate

QA sign-off must not be the developer. UAT sign-off must not be QA. The fix author must not approve their own work. Separation of duties at every gate — always.

⭐ The Golden Rule

If a reasonably technical person unfamiliar with the feature cannot execute your release or hotfix plan successfully, the plan is not production-ready.

A great release process is: clear enough for anyone authorised to execute · governed enough to satisfy CAB · fast enough to handle emergencies safely · verified enough to guarantee confidence.

Section 07
Anti-Patterns
Common Mistakes to Avoid

✗ Self-Declaring a Hotfix

❌ "This is urgent, I'll just hotfix it"
✅ Submit to Release Manager for Sev classification

✗ Branching Hotfix from develop

❌ git checkout -b hotfix/... develop
✅ Always branch hotfix from main (Production)

✗ Skipping UAT for "Small" Fixes

❌ "It's a one-line change, UAT is overkill"
✅ Every fix through full QA → UAT → Prod cycle

✗ No Rollback Plan

❌ "We'll figure it out if something breaks"
✅ Written rollback plan before deployment window opens

✗ Direct Changes in UAT

❌ "Quick fix" applied directly in UAT sandbox
✅ Fix in QA → re-promote through full cycle

✗ Missing Back-Merge

❌ Hotfix/fix merged to Prod only
✅ Back-merge to develop + sprint branches within 2h

✗ Wildcard Deployments

❌ Deploy * (full org metadata)
✅ Explicit, minimal package manifest every time

✗ Scope Creep After Freeze

❌ Adding stories to a Release after UAT starts
✅ Hard freeze date — additions go to next release

✗ No Post-Hotfix Review

❌ Fix deployed, move on, forget root cause
✅ 15-min RCA within 48h, findings in team wiki
