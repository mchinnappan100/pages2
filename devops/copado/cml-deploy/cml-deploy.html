Section 01 Setup

Why CML Deploy?

🔬
What is a CML Blob?
Salesforce Revenue Cloud · ExpressionSetDefinitionVersion

Constraint Model Language (CML) source code is stored as a binary blob inside ExpressionSetDefinitionVersion.ConstraintModel. Standard sf project retrieve/deploy and Copado's normal metadata promotion do not capture it.

⚠️
Standard Copado promotions skip CML. Even if everything else deploys correctly, the CML blob stays behind in the source org. You need a dedicated step.

The sf-cml-deploy tool handles the REST API calls required to fetch and push these blobs between authenticated orgs.

🏗️
Object Model
How CML is stored in Salesforce
Object hierarchy 
ExpressionSetDefinition          ← container (name, type, context)
  └── ExpressionSetDefinitionVersion   ← version (number, status)
        └── ConstraintModel            ← binary blob (the CML source)
  • Fetch — GET the blob via /services/data/v66.0/sobjects/ExpressionSetDefinitionVersion/{id}/ConstraintModel
  • Deploy — PATCH the version record with a base64-encoded CML string
  • HTTP 204 = success (no body returned by Salesforce on a successful blob PATCH)

Section 02 Recommended

Copado Function

⭐
Best approach. Copado Functions are parameterized, logged in the Copado UI, and reusable across multiple pipeline stages — no duplicating scripts per environment.
📋
Step 1 — Create the Function Record
Copado → Functions → New
FieldValue
NameCML Deploy Copy
API NameCML_Deploy_Copy
TypeCustom
ParametersSOURCE_ORG_SFDX_URL, TARGET_ORG_SFDX_URL, MODEL_NAME
📝
Step 2 — The Function Script
Paste this into the Function's Script field
bash 
#!/bin/bash
set -e

# Install tool if not already present on the runner
if ! command -v cml-deploy &> /dev/null; then
  npm install -g sf-cml-deploy
fi

# Authenticate source org using Copado-injected SFDX URL
echo "$SOURCE_ORG_SFDX_URL" | sf org login sfdx-url \
  --stdin --alias source-org --no-prompt

# Authenticate target org
echo "$TARGET_ORG_SFDX_URL" | sf org login sfdx-url \
  --stdin --alias target-org --no-prompt

# Copy the CML blob directly org-to-org
cml-deploy copy \
  -s source-org \
  -t target-org \
  -m "$MODEL_NAME"

echo "✅ CML copy complete: $MODEL_NAME"
🔗
Step 3 — Map Parameters
Function Parameters tab → map env vars from Org Credential records
ParameterSource in CopadoNotes
SOURCE_ORG_SFDX_URLSource Org Credential → Auth URLInjected at runtime by Copado
TARGET_ORG_SFDX_URLTarget Org Credential → Auth URLInjected at runtime by Copado
MODEL_NAMEPipeline Variable or hardcodede.g. RevenueCloudModel
JWT authentication (non-interactive CI) 
sf org login jwt \
  --client-id   "$CONSUMER_KEY" \
  --jwt-key-file server.key \
  --username    "$ORG_USERNAME" \
  --alias       source-org

Use when your Copado pipeline runner operates headlessly (no browser popup). Requires a Connected App with the certificate pre-configured in both orgs.

🚀
Step 4 — Add to Pipeline
Pipeline → Edit → Add Step → Function
  1. Go to your Copado Pipeline record and click Edit
  2. On the target stage (e.g. UAT), click Add Step
  3. Choose Function as the step type
  4. Select CML_Deploy_Copy from the lookup
  5. Set the execution order: after the standard metadata deployment step
  6. Save and run a test promotion to verify the Function executes
Section 03 Simple

Deployment Task (Script)

ℹ️
Good for one-off deployments. Simpler than a Function but less reusable — you'll duplicate the script across pipeline stages if needed in multiple places.
📋
Create the Deployment Task
Deployment → Deployment Tasks → New
FieldValue
Task NameCML Deploy Copy
TypeManual Task / Script
Run OrderAfter Metadata Deployment
StageYour target pipeline stage
📝
Deployment Task Script
Attach as a Before/After Step script on your Deployment record
bash 
#!/bin/bash
set -e

npm install -g sf-cml-deploy

# Authenticate from SFDX URL files on the runner
sf org login sfdx-url \
  --sfdx-url-file /path/to/source.sfdxurl \
  --alias source-org

sf org login sfdx-url \
  --sfdx-url-file /path/to/target.sfdxurl \
  --alias target-org

# Copy the CML blob
cml-deploy copy \
  -s source-org \
  -t target-org \
  -m "YourModelName"
⚠️
Hardcoded model name. Unlike the Function approach, the model name is embedded in the script. Update it per deployment task when promoting different models.
Section 04 Advanced

Quality Gate (Blocking)

⚠️
Use this when CML copy failure must stop the release. If the CML blob fails to transfer, the Quality Gate will block the promotion — no partial releases.
🚧
Create the Quality Gate Rule
Quality Gates → New Rule
FieldValue
ExtensionShell Script
StageAfter Promotion
EnvironmentYour pipeline stage (e.g. UAT)
Block on Failure✅ true
Quality Gate shell script 
cml-deploy copy \
  -s "$COPADO_SOURCE_ORG_ALIAS" \
  -t "$COPADO_TARGET_ORG_ALIAS" \
  -m "$CML_MODEL_NAME"
  • Copado marks the Quality Gate as Failed
  • The promotion is blocked — target org does not receive a broken state
  • Copado logs the stderr output from cml-deploy in the gate results
  • Developers are notified via Copado's notification rules (Slack, email, etc.)
  • Fix the root cause (auth, model name, connectivity) then re-trigger promotion
Section 05 Scenarios

Deploy Scenarios

🔄
Scenario A — Model Already Exists in Target Org
Typical promotion: sandbox → UAT → Production

The ExpressionSetDefinition record already exists in the target org. You only need to copy the updated CML blob.

One command — direct org-to-org copy 
cml-deploy copy \
  -s <sourceOrgAlias> \
  -t <targetOrgAlias> \
  -m <ModelName>
✅
This is the most common Copado scenario. Add this as a Copado Function step after each standard metadata promotion.

If you track the CML in your git repo (recommended), deploy from file instead:

cml-deploy deploy \
  -t <targetOrgAlias> \
  -m <ModelName> \
  ./force-app/main/default/cml/<ModelName>.cml
🆕
Scenario B — Model Does Not Exist in Target Org
First-time deploy of a new model

You must first deploy the ExpressionSetDefinition XML as Inactive, then upload the CML blob, then activate.

StepActionTool
1Deploy definition XML (Inactive status)Copado standard promotion or sf project deploy start
2Upload CML blobcml-deploy deploy
3Activate the versionSetup UI or redeploy XML with Active status
Step 1 — deploy definition XML via SF CLI 
sf project deploy start \
  --source-dir force-app/main/default/expressionSetDefinition/<ModelName>.expressionSetDefinition-meta.xml \
  --target-org  <targetOrgAlias> \
  --ignore-conflicts
Step 2 — upload the CML blob 
cml-deploy deploy \
  -t <targetOrgAlias> \
  -m <ModelName> \
  ./force-app/main/default/cml/<ModelName>.cml
Step 3 — activate (Setup UI or redeploy with Active status in XML) 
# In the XML: change <status>Inactive</status> → <status>Active</status>
sf project deploy start \
  --source-dir force-app/main/default/expressionSetDefinition/<ModelName>.expressionSetDefinition-meta.xml \
  --target-org  <targetOrgAlias>
💾
Fetching CML Into Your Git Repo
Run before Copado commit — track .cml files alongside your metadata

Add this as a pre-commit script or developer workflow step to keep CML in version control alongside your standard metadata.

Fetch and commit 
# Fetch from dev org
cml-deploy fetch \
  -s <devOrgAlias> \
  -m <ModelName> \
  -o force-app/main/default/cml/<ModelName>.cml

# Commit alongside your metadata
git add force-app/main/default/cml/<ModelName>.cml
git commit -m "chore: update CML blob for <ModelName>"
