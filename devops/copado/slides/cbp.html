Branch Strategy

Git Branching Model for Copado

Every delivery type maps to a specific branch pattern. Consistency prevents merge chaos and enables automation.

main / production

Protected. Only accepts merges via approved Copado promotions. No direct commits. Mirrors production org state exactly.

release/YYYYMMDD

Time-boxed release trains. Created per sprint end. QA-validated user stories are promoted onto this branch before UAT.

feature/US-XXXX

One Copado User Story = one branch. Auto-created by Copado on User Story activation. Merged via back-promotion only.

hotfix/HF-XXXX

Branched directly from main. Short-lived — merged back to main AND all integration branches within 24 hours. Never from feature branches.

Hotfix Protocol

Hotfix: Fast & Safe Production Fixes

Hotfixes bypass the sprint cycle but must never bypass quality. Every minute saved in process must be recovered with discipline.

01

Declare a P1 incident & get approval

Hotfixes require explicit sign-off from Release Manager before a branch is created. Document root cause in incident ticket.

02

Branch from main, not feature

Create hotfix/HF-XXXX from main only. Keeps the fix isolated from in-flight sprint work.

03

Expedited CI — never skipped

Run targeted Apex test classes (not full suite) to pass the ≥85% gate. PMD critical rules still enforced. Time budget: 30 min max.

04

Merge back to main AND all active branches

After production deploy, immediately back-promote the hotfix to release and integration branches. Prevents regression at next sprint release.

slide 05 / 07
Promotion Gates

Promotion Quality Gates

Copado promotions are checkpoints, not just transports. Define gates per environment and enforce them as pipeline steps.

Dev → QA

CI build passes · Apex test coverage ≥85% · PMD: no critical violations · peer review approved

QA → UAT

QA sign-off in Jira · No open P1/P2 defects · Regression suite green · Test class added for every fix

UAT → Prod

Business owner sign-off · Change Advisory Board (CAB) approved · Deployment window confirmed · Rollback plan documented

Hotfix → Prod

P1 incident ticket linked · RM approval · Targeted tests pass · Post-deploy verification plan ready

Release Management

Release Planning & Cadence

Structured release trains reduce risk, improve predictability, and give stakeholders clear commit windows.

Bi-weekly release train (recommended)

Sprint end triggers a release branch cut. All QA-passed User Stories are cherry-picked onto the release branch. No new features after cut.

Deployment windows: off-peak only

Schedule Copado promotions to production on weekday evenings (6–10 PM local) or Sunday mornings. Never during business hours or month-end close.

Rollback plan is mandatory

Every release must have a documented rollback: Copado Snapshot + destructive changes XML + data backup checklist ready before deploy starts.

Stakeholder comms at every gate

Automated Slack/email notifications on promotion start, success, and failure. Release notes auto-generated from linked User Story titles in Copado.

slide 07 / 07
Summary

Key Takeaways

Discipline at the process level makes speed at the delivery level possible. These practices compound over time.

One story, one branch

Enforce strict US-to-branch mapping in Copado. Avoid umbrella branches that bundle unrelated changes.

Gates can't be skipped

CI, PMD, and test coverage gates apply even to hotfixes. Speed comes from pre-approval, not gate removal.

Hotfixes are exceptional

Require explicit P1 declaration and RM sign-off. Back-promote within 24 hours or you accumulate drift.

Ship on a cadence

Predictable bi-weekly trains build stakeholder trust and reduce last-minute pressure on the pipeline.

Document everything

Rollback plan, CAB approval, and release notes are non-negotiable. Copado's audit trail is your compliance record.

Measure & improve

Track lead time, change failure rate, and MTTR in Copado Analytics. Use them in sprint retros.
