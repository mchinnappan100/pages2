Copado Production Deployment Strategies

Executive Summary

This white paper analyzes two common approaches for deploying Salesforce metadata to Production using Copado: Bundle-based deployments and Promotion-based deployments. It highlights architectural trade-offs, risks, and best practices based on real-world DevOps patterns.

Approach 1: Bundle-Based Deployment

In this approach, bundles are created by grouping User Stories across sprints. Copado retrieves metadata directly from an environment such as UAT and builds a deployment package. Suitable if sprints are well-defined and have minimal cross-dependencies.

Example:

  • Bundle 1_3 → Sprints 1 to 3
  • Bundle 4_7 → Sprints 4 to 7
graph LR S1[Sprint 1] --> B1[Bundle 1_3] S2[Sprint 2] --> B1 S3[Sprint 3] --> B1 S4[Sprint 4] --> B2[Bundle 4_7] S5[Sprint 5] --> B2 S6[Sprint 6] --> B2 S7[Sprint 7] --> B2 B1 --> UAT B2 --> UAT UAT --> PROD[Production Deployment]

Key Issue: Metadata Drift

This approach introduces a critical risk due to how Copado retrieves metadata: it always pulls the latest version from the source org (e.g., UAT).

Problem Scenario:

  • Apex class Constants.cls is modified in Sprint 2 (Bundle 1_3)
  • It is modified again in Sprint 6 (Bundle 4_7)
  • UAT now contains only the latest version (Sprint 6)

When deploying Bundle 4_7, dependencies may exist on metadata expected from earlier bundles. However, since Bundle 1_3 is not yet deployed, required components (e.g., fields, objects) may be missing, leading to deployment failures.

Challenges

  • Loss of historical metadata versions
  • Hidden dependencies across bundles
  • Deployment order sensitivity
  • High failure rate during validation

Approach 2: Promotion-Based Deployment

In this approach, a promotion (e.g., P11036) is created containing a defined set of user stories. Metadata is sourced from version control (Git), not from a mutable org like UAT. This ensures that the exact state of metadata is preserved and can be reliably deployed to Production.

graph TB DEV[Development] --> GIT[Git Repository] GIT --> MAIN[Main Branch - Prod State] S1[Stories Sprint 1-3] --> P1[Promotion P11036] P1 --> GIT MAIN --> MERGE[Merge & Conflict Resolution] MERGE --> VALIDATE[Validate Deployment] VALIDATE --> PROD[Production]

Key Concept: Source of Truth

The Git repository serves as the single source of truth, ensuring consistent and versioned metadata. Promotions align deployments with a specific commit state rather than a fluctuating environment.

Merge Conflict Management

Since Production is represented by the main branch, all promotions must be merged into it. This requires resolving merge conflicts when multiple stories modify the same metadata.

Advantages

  • Version-controlled deployments
  • Predictable and reproducible builds
  • Better dependency management
  • Reduced risk of missing components

Comparison

Criteria Bundle-Based Promotion-Based
Source of Metadata UAT (latest state) Git (versioned)
Version Control Limited Strong
Risk of Drift High Low
Merge Conflicts Hidden Explicit
Deployment Reliability Lower Higher

Conclusion

While bundle-based deployments may appear simpler, they introduce significant risks due to reliance on mutable environments like UAT. Promotion-based deployments, anchored in version control, provide a more robust, scalable, and reliable strategy for enterprise Salesforce DevOps.

Recommendation:

Adopt promotion-based deployments with strict Git governance, enforce merge validation, and ensure proper dependency tracking for production releases.
