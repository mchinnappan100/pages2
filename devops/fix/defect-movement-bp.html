Phase One
QA Sandbox — Fix Verification
🔗
Link the Fix to a Copado User Story
Critical Copado

Every defect fix must be tracked as a Copado User Story linked to the originating defect record. Never deploy uncommitted, free-floating changes — traceability from defect to commit to deployment is non-negotiable.

See Example 
# Copado User Story setup for a defect fix:
  Story Title:  [DEFECT] DEF-2201 — Case assignment rule fires twice
  Story Type:   Bug Fix
  Linked to:    Defect Record DEF-2201 (Status: In QA)
  Sprint:       Fix Sprint 12
  Environment:  QA Sandbox

# Git branch auto-created by Copado:
  feature/DEF-2201-case-assignment-double-fire

  ✅  All commits reference DEF-2201 in message
  ✅  package.xml is explicit — no wildcards
  ❌  Do NOT commit unrelated cleanup in this branch

Why it matters: Copado's deployment pipeline gates progression on Story status. A fix with no linked Story cannot be promoted — by design.

🧪
Reproduce the Defect in QA Before Fixing
Critical QA Discipline

Confirm the defect is reproducible in the QA sandbox using the same user profile, data, and steps as the original reporter. Do not begin the fix if it cannot be reproduced — escalate immediately.

See Example 
Reproduction Checklist — DEF-2201:
  □  Log in as: Service Agent profile (not Sys Admin)
  □  Create a Case with Origin = "Web"
  □  Assign to Queue: Tier-1 Support
  □  Observe: assignment rule fires → email sent twice
  □  Screenshot both email notifications
  □  Attach screenshots to defect record in Copado

  Result:
    Reproduced ✅  — proceed with fix
    NOT reproduced ❌ — STOP, escalate to Dev Lead

Reminder: A fix deployed for a defect that wasn't reproduced has a high chance of leaving the original bug intact while introducing new issues.

🔁
Regression Testing in QA
High Priority Risk Reduction

Test the full surface area adjacent to your fix — not just the defect scenario. Defect fixes commonly break edge cases in nearby logic that the developer wasn't aware of.

See Example 
# DEF-2201 (Case assignment double-fire) regression matrix:
  Scenario                               Result
  ──────────────────────────────────────────────
  Web case → Queue assignment            PASS ✅
  Phone case → Agent assignment          PASS ✅
  Escalation rule trigger (Sev1)         PASS ✅
  Re-assignment after status change      PASS ✅
  Bulk case import (50 records)          PASS ✅
  API-created cases (integration)        PASS ✅
  Case assignment via Omni-Channel       PASS ✅

  ✅  Test matrix created BEFORE fix, not after.
  ❌  Post-fix test authoring introduces confirmation bias.
✅
QA Sign-Off & Copado Story Status Update
Critical Governance

The QA engineer who did NOT write the fix must perform independent sign-off. Separation of duties is required. Once passed, update the Copado User Story status to "QA Complete" to unlock promotion to UAT.

See Example 
QA Sign-Off Template — DEF-2201:
  Tester:        Priya Ramaswamy (QA Engineer)
  Date / Time:   2025-04-10  11:15 IST
  Sandbox:       QA-SB02  (refreshed 2025-04-09)

  Defect scenario (original):    PASS ✅
  Regression — adjacent flows:   6/6  PASS ✅
  Apex test run:                 214 passed, 0 failed ✅
  Coverage:                      79%  (was 76%) ✅

  Copado Story Status:  QA Complete → Ready for UAT
  Sign-off by:          Priya Ramaswamy
  Approved by:          Naveen Kumar (QA Lead)

Governance note: This sign-off is stored permanently on the Copado User Story as a Salesforce attachment — your audit trail for change management.

📦
Copado Deployment: QA Commit & Snapshot
Copado High Priority

Commit all changed metadata to the Copado User Story, take an Org Snapshot, and run a Deployment from the branch to the QA sandbox. Do not manually push files — all changes must flow through Copado's tracked pipeline.

See Example 
# Copado deployment steps in QA:
  1. Commit changes via Copado Git Commit UI
     Components:
       ✅  AutoResponseRule — CaseAssignment
       ✅  ApexClass — CaseAssignmentTrigger
       ✅  ApexClass — CaseAssignmentTriggerTest

  2. Run Copado Deployment to QA
     Source Branch:  feature/DEF-2201-...
     Destination:    QA Sandbox
     Run Tests:      Run Specified Tests
                     → CaseAssignmentTriggerTest

  3. Validate in Copado deployment log
     Deployment Status: Succeeded ✅
     Test Results:      12/12 passed ✅

  ❌  Never use VS Code / Workbench to push to shared sandboxes
02
Phase Two
UAT Sandbox — Business Validation
🚦
Promote via Copado Release Only After QA Gate
Critical Copado

UAT promotion is gated on Copado User Story status "QA Complete." Never manually migrate changes to UAT outside of Copado Releases. The Release Manager must review the Release contents before triggering promotion.

See Example 
# Copado Release promotion checklist:
  □  All Stories in Release: Status = "QA Complete"
  □  No Stories with Status = "In Development" or "In QA"
  □  Release name: REL-2025-04-W2-Defects
  □  Release Manager review: APPROVED ✅

# Stories included in this Release:
  ✅  DEF-2201 — Case assignment double-fire
  ✅  DEF-2187 — Opportunity PDF missing line items
  ✅  DEF-2198 — Contact merge duplicate phone

  Promotion Type:   Metadata Deployment
  Destination:      UAT Sandbox
  Deployment Mode:  Validate Before Deploy

  ❌  Do NOT add Stories to a Release after UAT has started
👔
Business Owner UAT Execution
Governance High Priority

UAT must be executed by the business owner or product owner who reported or owns the defect — not by the development team. Testers must use real business data patterns, not synthetic test data.

See Example 
UAT Test Plan — DEF-2201:
  Tester:    Sarah Mitchell (Service Ops Manager)
  Dates:     2025-04-12 → 2025-04-13
  Sandbox:   UAT-SB01  (promoted from QA: 2025-04-11)

  Test Scenarios (business perspective):
  □  Create a web case — confirm single email notification
  □  Process 10 real-world case types (Tier-1 queue)
  □  Verify email format matches template standard
  □  Confirm Omni-Channel routing still works post-fix
  □  Check Case Assignment Report accuracy

  UAT Result:   PASSED ✅
  Sign-off by:  Sarah Mitchell  |  2025-04-13  16:00 IST
🚫
No Code Changes in UAT — Absolute Rule
Critical Non-Negotiable

If a defect is found in UAT, it must be fixed back in QA and re-promoted through the full cycle. UAT is a read-only validation environment — never make direct metadata changes to the UAT sandbox.

See Example 
If a defect is found during UAT:

  ✅  Correct workflow:
      1. Log defect found in UAT on the Copado Story
      2. Developer fixes in QA sandbox
      3. QA re-validates in QA sandbox
      4. Story re-promoted to UAT via new Release
      5. Business Owner re-tests in UAT

  ❌  Forbidden shortcuts:
      • "Quick fix" directly in UAT sandbox metadata
      • VS Code / Workbench patch applied to UAT
      • Skipping QA re-test ("it's just a tiny change")

  One violation invalidates the entire change record
  and breaks Copado's audit trail.
📋
UAT Defect Triage & Go/No-Go Decision
Governance Copado

At the conclusion of UAT, hold a formal triage session. Classify any remaining issues and make an explicit Go/No-Go decision before the Release Manager marks the Release as UAT Complete in Copado.

See Example 
UAT Triage Outcome — REL-2025-04-W2-Defects:
  Stories tested:  3
  Stories passed:  3  ✅

  New issues found during UAT:
    UAT-001: Minor UI label truncation in modal
             Severity: Low → Defer to next sprint
             New Story created: DEF-2204

  Go/No-Go Decision:    ✅  GO — cleared for Production
  Decision by:          Naveen Kumar (Release Manager)
                        + Sarah Mitchell (Service Ops)

  Copado Release Status: UAT Complete → Ready for Prod

Phase Three
Production Deployment
📋
Pre-Deployment Checklist (Written, Not Verbal)
Critical Zero Shortcuts

Never begin a Production deployment without completing a documented pre-flight checklist. Verbal confirmations are forgotten under pressure. Every item must be checked and signed in the Copado Release record.

See Example 
Production Pre-Deployment Checklist:
  □  Change ticket approved by Change Advisory Board (CAB)
  □  UAT sign-off documents attached to Copado Release
  □  Rollback plan written and reviewed by Release Manager
  □  Deployment window confirmed (avoid peak hours)
  □  Stakeholders notified via email + Slack
  □  On-call engineer assigned for post-deploy monitoring
  □  Copado Quick Deploy validation run: PASSED ✅
  □  Apex tests: Run Specified Tests list finalised
  □  Backup snapshot of Production components exported
  □  War-room Slack channel opened: #deploy-REL-2025-04-W2

  Release Manager sign-off: ____________  Date: ________
⚡
Use Copado Quick Deploy for Zero Downtime
Copado Best Practice

Use Copado's Validate + Quick Deploy flow: run a full validation deployment (including tests) against Production during off-peak hours, then Quick Deploy during the maintenance window. Reduces actual deployment time to under 5 minutes.

See Example 
# Copado Validate → Quick Deploy workflow:

  Step 1 — Validate (run during off-peak hours)
    Copado Release → Deploy to Production (Validate Only)
    Run Tests:  Run Specified Tests
                → CaseAssignmentTriggerTest
                → OpportunityPDFControllerTest
                → ContactMergeUtilTest
    Result: PASS ✅  (valid for 10 days)

  Step 2 — Quick Deploy (during maintenance window)
    Copado Release → Quick Deploy
    Estimated time: ~3 minutes
    No test re-run required (Salesforce platform optimisation)

  ✅  Users experience <5 min impact window
  ❌  Never run full deploy + tests during peak hours
🔙
Document the Rollback Plan Before Deploying
Critical Risk Management

Every deployment must have a written rollback plan with step-by-step instructions, a named rollback owner, and an estimated execution time. "We'll figure it out if something goes wrong" is not a rollback plan.

See Example 
Rollback Plan — REL-2025-04-W2-Defects:

  Trigger condition: Post-deploy defect within 1 hour

  If deployment fails mid-flight:
    → Salesforce atomic deploy auto-rolls back ✅
    → No manual action required

  If post-deploy defect discovered:
    Step 1: Release Manager initiates rollback in Copado
            Copado → Rollback Deployment → Confirm
    Step 2: Verify original defect re-appears (confirms rollback)
    Step 3: Notify stakeholders: "Rollback complete"
    Step 4: Log incident — root-cause before re-fix

  Rollback owner:      Naveen Kumar (Release Manager)
  Estimated time:      ~10 minutes
  Rollback tested in:  UAT sandbox ✅

  Rule: If rollback takes >15 min, the fix is too
  complex for one deployment — decompose it.
🕐
Deployment Window Rules
High Priority Timing

Schedule Production deployments to minimise user impact. Even critical defect fixes benefit from a low-traffic window. Document the window in the Change ticket.

See Example 
Deployment window guidelines:
  ✅  Preferred:   Off-peak (early morning, after 6 PM local)
  ✅  Acceptable:  Any time for Sev1 data-integrity risk

  ⚠️   Avoid:
       • Monday morning (peak login volume)
       • Month-end / quarter-end close period
       • During active scheduled batch jobs
       • During Salesforce Trust maintenance windows

  ❌  Never deploy:
       • Friday after 3 PM local time
       • During live customer events or demos
       • Major holidays or on-call blackout periods

  Change ticket field: "Deployment Window"
  Required format:  2025-04-14  22:00–22:30 IST
📊
Post-Deploy Monitoring — 24-Hour Watch
High Priority Monitoring

Assign an on-call engineer to actively monitor Production for 24 hours after any defect fix deployment. Silent failures — Apex exceptions, integration errors, report anomalies — often surface hours after deployment.

See Example 
Post-Deploy Monitoring Checklist (0–24h):
  □  Review Apex Exception Log every 2 hours
  □  Check Copado Deployment log for post-deploy warnings
  □  Monitor Slack #case-ops for user-reported issues
  □  Run "Case Assignment Report" — compare to baseline
  □  Verify integrations: no new error spikes (CTI / ERP)
  □  Confirm scheduled Apex jobs resumed normally

  Hour 0–1:    Naveen Kumar (Release Manager) on watch
  Hours 2–8:   Arjun Patel (On-Call Developer)
  Hours 9–24:  Standard NOC alerting

  Close-out: "All clear" note added to Copado Release
             Defect record updated to "Deployed to Prod"

Governance & Copado Controls

Branch Strategy for Defect Fixes
Copado Git Hygiene

Defect fix branches follow a standard naming convention and are always created by Copado from the correct source branch. Never manually create fix branches or branch from a feature/sprint branch.

See Example 
# Branch naming convention (Copado auto-creates):
  feature/{DEF-number}/{short-description}

  Examples:
    feature/DEF-2201-case-assignment-double-fire
    feature/DEF-2187-opp-pdf-missing-lines
    feature/DEF-2198-contact-merge-dupe-phone

# Branch source rules:
  main (Production)   ← source for critical / P1 defects
  release/sprint-12   ← source for sprint-cycle defects

# After Production deployment — back-merge:
  Copado handles main ← merge automatically in Release flow

  Developer also merges to develop + active sprint branch:
    git checkout develop
    git merge --no-ff feature/DEF-2201-...

  ❌  Never branch a defect fix from a feature branch
🔒
Environment Permission Controls
Governance Security

Restrict sandbox and Production access using Copado's Deployment Flow permissions. Only named roles may promote to UAT or Production — no developer should have unilateral Production deploy rights.

See Example 
Copado Role Matrix:
  Role                    QA    UAT   Prod
  ─────────────────────────────────────────
  Developer               ✅    ❌     ❌
  QA Engineer             ✅    RO    ❌
  QA Lead                 ✅    ✅     ❌
  Release Manager         ✅    ✅     ✅
  Emergency Change Mgr    ✅    ✅     ✅  (+ CAB approval)

  ✅ = Deploy rights   RO = Read-only   ❌ = No access

Copado Pipeline Gating:
  QA → UAT:    Story Status = "QA Complete" required
  UAT → Prod:  Release Status = "UAT Complete" required
               + Change ticket = Approved
               + Release Manager confirmation
📝
Change Advisory Board (CAB) Process
Governance Required

All Production deployments require an approved change ticket. Standard change templates reduce CAB overhead for routine defect fixes without bypassing governance.

See Example 
Change Ticket Template — Copado Defect Fix Release:
  Change Type:      Standard (pre-approved template)
  Template:         CHGT-STD-004 "Copado Defect Fix Release"
  Risk Level:       Low / Medium / High  (select one)
  Affected System:  Salesforce Production
  Business Impact:  {describe user-facing impact}
  Rollback Plan:    {link to rollback doc in Copado Release}
  Test Evidence:    {link to QA + UAT sign-off records}

  CAB Review:
    Low risk  → Async (email approval)
    High risk → Synchronous CAB meeting

  Required approvals:
    □  Release Manager
    □  IT Change Manager
    □  Business Owner  (High risk changes only)
🔄
Post-Deploy Back-Merge & Story Closure
High Priority Copado

After successful Production deployment, immediately merge the fix branch back to the development integration branch and any active sprint branches. Failure to do this causes regressions in the next sprint release.

See Example 
# Post-deploy back-merge (within 2 hours of deploy):

  git checkout develop
  git merge --no-ff feature/DEF-2201-case-assignment-...
  git push origin develop

  git checkout release/sprint-13
  git merge --no-ff feature/DEF-2201-case-assignment-...
  git push origin release/sprint-13

# Copado Story lifecycle:
  In Development → In QA → QA Complete →
  In UAT → UAT Complete → Deployed to Production ✅

# Close the defect record:
  Defect DEF-2201:  Status = "Closed — Fixed in Prod"
  Resolution date:  2025-04-14
  Fixed in Release: REL-2025-04-W2-Defects

Summary
What Great Teams Do
✓ One Story Per Defect

Never bundle multiple defect fixes into one Copado User Story. Each defect gets its own Story, its own branch, and its own traceability chain.

❌  DEF-2201 + DEF-2187 → one Story
✅  DEF-2201 → Story 1
✅  DEF-2187 → Story 2  (same Release is fine)
✓ Never Skip a Gate

QA → UAT → Prod is not a suggestion. Skipping UAT "because it's a small fix" is the #1 cause of Production regressions. Every defect, regardless of size, goes through the full cycle.

✓ Independent Sign-Off at Every Gate

QA sign-off must not be done by the developer. UAT sign-off must not be done by QA. The fix author must not approve their own work at any stage.

✓ Explicit Metadata — No Wildcards

Every Copado deployment must use an explicit, minimal component list. Wildcard or full-org deploys during defect releases have caused Production incidents at every org that tried them.

✓ All Changes Flow Through Copado

No VS Code, Workbench, or Data Loader pushes to shared environments. Every metadata change must be committed to Copado and deployed through the tracked pipeline — always.

✓ Write Rollback Plan Before Deploy

The rollback plan must be written and reviewed before the deployment window opens. A plan taking more than 15 minutes to execute means the change is too large for a single deployment.

✓ Use Quick Deploy

Separate validation (with test runs, off-peak) from the actual go-live. Quick Deploy minimises the maintenance window to under 5 minutes while keeping governance intact.

✓ Back-Merge Within 2 Hours

Set a team rule: fix branches must be merged back to develop and all active sprint branches within 2 hours of Production deployment. Track this as a Copado Release checklist item.

✓ Post-Deploy 24-Hour Watch

Assign a named on-call engineer to monitor Apex exceptions, integration error rates, and user-reported issues for 24 hours after every Production defect fix deployment.

✓ Run a Post-Incident Review

Every defect fix that reached Production should generate a 15-minute review: root cause, fix summary, and what process change prevents recurrence. Log findings in your team wiki.

✓ Track Defect Fix Rate as a KPI

More than 3 unplanned defect fix releases per month signals a systemic quality or testing gap. Track this on your Copado Release dashboard.

Target KPIs:
  ≤ 3 defect releases / team / month
  ≥ 95% first-time success rate
  ≤ 5 days QA → UAT → Prod cycle time
✓ Automate the Checklists

Encode pre-deployment checklists as Copado checklist templates on the Release record. Humans skip manual checklists under pressure — templates make compliance the default path.
