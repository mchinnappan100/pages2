What is OWD?

Foundations of Salesforce record access

🔐

Org-Wide Defaults Defined

OWD sets the baseline access level for all records across the entire organization — the floor that all other sharing layers build upon.

  • Public Read/Write — everyone can read and edit any record
  • Public Read Only — everyone can read, but only owners can edit
  • Private — only the record owner and those above in hierarchy can access
  • Controlled by Parent — access follows the parent record's OWD
Object: Opportunity Current OWD: Public Read Only What this means: • Any Salesforce user can VIEW all Opportunities in the org • Only the Opportunity owner (or those above in the Role Hierarchy) can EDIT • Sharing Rules can be layered on top to grant additional access When Private is chosen instead: • Users only see Opportunities they own or are explicitly shared with • The Role Hierarchy still allows managers to see their reports' records • Manual sharing / Apex sharing / Sharing Rules must be used to open up access beyond ownership
🏗️

Access Extension Layers

OWD is the baseline — these mechanisms extend access beyond it to meet business needs.

  • Role Hierarchy — managers automatically see reports' records
  • Sharing Rules — criteria or ownership-based access grants
  • Teams — Account/Opportunity/Case team members get access
  • Manual Sharing — record-by-record sharing with specific users
  • Apex Managed Sharing — programmatic sharing via Apex code
  • Permission Sets / Profiles — object-level CRUD permissions
OWD: Case = Private Access Extension Stack: ───────────────────────────────── Layer 1 (Role Hierarchy): Support Manager → can see all team reps' Cases Layer 2 (Sharing Rule): "Cases owned by West Region users shared with West Region Manager role" → Criteria-based, Public Read Only Layer 3 (Apex Managed Sharing): Custom logic: "Share any Case tagged VIP with the Executive Team group" → Reason: CaseExecShare Layer 4 (Manual Share): Sarah (VP) manually granted Read/Write on Case #10542 Result: A Private OWD with precise, auditable access grants

Why OWD Changes Are High Risk

Every OWD change is an enterprise security architecture decision

⚡

Sharing Recalculation Impact

Changing OWD often triggers a massive sharing table rebuild across potentially millions of records — affecting performance, availability, and data access in real time.

  • Sharing table rebuilds across all impacted objects
  • Record access propagation delays during recalculation
  • Queue backlogs that block other async processes
  • Integration failures due to visibility changes mid-flight
  • Apex timeout issues if triggers assume record visibility
  • Automation execution delays (Flows, Process Builder)
  • Unexpected visibility loss or privilege escalation
Scenario: Changing Account OWD from Public Read Only → Private Org profile: • 2.4 million Account records • 680 Sharing Rules on Account • 12 Apex Triggers referencing Account visibility • 3 active integrations pulling Account data hourly What happens at the moment of change: [T+0:00] OWD saved in Setup → recalculation job queued [T+0:05] Async recalculation begins — sharing table locked [T+0:15] Integration job fails — records now return 0 results [T+1:45] Apex trigger fires on new Account — SOQL returns empty (governor limit hit) [T+3:20] Flow automation fails — attempts to relate child records, no parent visible [T+6:00] Recalculation completes — normal access restored 6 hours of degraded access and integration failures → Treated as a P1 incident rather than a planned change
🏢

Complex Enterprise Risk Factors

The risk compounds in organizations with high data volumes, complex automations, and multiple integrated systems.

  • Large data volumes (millions of records per object)
  • Multiple integrations assuming specific visibility contracts
  • Territory models and complex Role Hierarchies
  • Custom sharing logic via Apex triggers
  • Heavy automation: Flows, Apex, and Process Builder
  • Cross-object dependencies and lookup relationships

Key Roles & Responsibilities

No single team should own an OWD change end-to-end

🏛️

COE Team — Center of Excellence

Owns governance and approval. The COE ensures OWD changes are business-driven, not convenience-driven.

  • Validate business justification for the change
  • Review security impact and compliance posture
  • Ensure enterprise standards compliance
  • Approve architectural risk acceptance
  • Coordinate change windows across teams
  • Review rollback readiness before execution
  • Provide final signoff for production execution
COE Review Checklist for Account OWD Change: ✅ Business justification documented? → "Regulatory requirement: field sales reps must not see competitor accounts" ✅ Security impact reviewed? → 1,200 users lose direct read access; compensating shares added via SR-045 ✅ Recalculation window agreed? → Saturday 10pm–4am maintenance window ✅ Rollback plan tested in sandbox? → Yes — reverting to Public Read Only takes ~45 min in full-copy sandbox ✅ All impacted integration owners notified? → ERP team (Mark), BI team (Sandra), Marketing Cloud (Jesse) → COE APPROVED — Change Window: Sat Nov 18, 10:00 PM PST
📐

Solution Architects

Own technical design and impact analysis. No OWD change should proceed without architectural signoff.

  • Assess data access model implications across objects
  • Analyze role hierarchy impact on existing access grants
  • Review sharing rule dependencies and conflicts
  • Evaluate performance risks given data volume
  • Identify automation and integration breakpoints
  • Design compensating sharing mechanisms
  • Produce a tested rollback strategy
Architect Impact Analysis — Case OWD: Public → Private Q: What users lose access? → 340 Community users currently see all Cases (no sharing rules cover them) → Action: Create Sharing Rule SR-012 "Share Cases with Community User group" Q: What users gain unintended access? → No new unintended grants detected in analysis Q: What automations depend on current visibility? → CaseEscalationTrigger.cls assumes running user can query parent Account → Risk: SOQL returns 0 rows if Account OWD is also Private → Action: Run with system context or add WITH SECURITY_ENFORCED guard Q: Expected recalculation duration? → Sandbox: 38 min for 120K records → Production estimate: ~3.5 hours for 1.1M Q: Sharing explosion risk? → Moderate — 42 sharing rules on Case; monitor queue depth during execution
🧪

QA Team

Validates business continuity and security correctness — both positive access and negative deny cases.

  • Execute visibility regression tests across user personas
  • Validate CRUD and record sharing behavior
  • Test all user persona access (not just happy paths)
  • Validate automation execution with new access model
  • Validate reports and dashboards still load correctly
  • Validate integrations succeed with changed visibility
QA Test Matrix — Opportunity OWD Change POSITIVE CASES (user SHOULD see record): Persona: Sales Rep (record owner) → ✅ Can read, edit own Opp Persona: Sales Manager (above in hierarchy)→ ✅ Can read, edit team's Opps Persona: Sales Ops (sharing rule member) → ✅ Read-only access confirmed Persona: Exec (manual share granted) → ✅ Full access confirmed NEGATIVE CASES (user MUST NOT see record): Persona: Support Rep (different BU) → ✅ No access — correct Persona: Marketing User → ✅ No access — correct Persona: Community Portal User → ✅ No access — correct Persona: Partner Community User → ⚠️ Unexpected read access via SR-007 → BUG FILED: SR-007 criteria too broad, must be scoped to Account owner BU Security testing is not optional — negative cases revealed a real gap.
🚀

DevOps Team

Ensures safe deployment execution. OWD changes must never be deployed casually through automated pipelines without orchestration controls.

  • Validate metadata promotion path and change set contents
  • Coordinate deployment window with all stakeholders
  • Verify environment consistency before deployment
  • Monitor deployment execution in real time
  • Track sharing recalculation progress
  • Coordinate rollback if criteria are triggered
  • Capture deployment evidence and post-deploy artifacts
DevOps Deployment Runbook — OWD Change Execution Pre-deployment (T-60 min): □ Confirm all stakeholders on bridge call □ Freeze other deployments in pipeline □ Take org snapshot / backup current sharing config □ Confirm rollback package is staged and validated Execution (T+0): □ Apply OWD change via Setup (manual or metadata deploy) □ Note exact timestamp — log to incident record □ Monitor: Salesforce Sharing Recalculation job queue □ Monitor: Async queue depth every 10 minutes Post-change (T+30 min): □ QA smoke tests → access verification □ Integration health check → API calls returning expected data □ Automation health → no stuck Flows or Apex failures □ Confirm recalculation job complete Rollback trigger: if recalculation not complete within 8 hours OR any P1 business impact detected → execute rollback package

Required Change Planning Process

Six phases — no phase is optional

1

Phase 1 — Impact Analysis

  • Document current and proposed OWD state for all impacted objects
  • Identify related child objects and cross-object dependencies
  • Map sharing rule and Apex sharing dependencies
  • Assess role hierarchy implications and data volume estimates
  • Estimate recalculation duration based on sandbox rehearsal
2

Phase 2 — Sandbox Simulation

  • Deploy in a full-copy or representative sandbox first
  • Measure sharing recalculation time and performance impact
  • Validate user access behavior, batch/flow health, and dashboards
  • Capture all evidence for cross-team review
3

Phase 3 — Cross-Team Review

  • Solution Architect signoff on design and impact
  • Security reviewer sign-off on exposure risk
  • COE approval of business justification and risk acceptance
  • QA signoff confirming test coverage
  • DevOps execution readiness confirmation
4

Phase 4 — Production Scheduling

  • Schedule during a low-traffic maintenance window
  • Avoid peak hours, concurrent large deployments, data migrations
  • Communicate change start time, recalculation window, and escalation contacts
  • Prepare and distribute rollback criteria to all team leads
5

Phase 5 — Controlled Deployment Execution

  • DevOps executes the change with all teams on standby
  • Architect validates behavior against expected access model
  • QA performs immediate smoke validation
  • COE monitors governance compliance in real time
6

Phase 6 — Post-Deployment Verification

  • Validate object accessibility and reports load correctly
  • Confirm automation execution and integration success
  • Verify sharing recalculation completion with no async backlog
  • Document final signoff from all four teams

Dos & Don'ts

Non-negotiable standards for every OWD change

✓ Dos

✓
Perform Full Access Impact MappingIdentify every affected persona and user group before any change proceeds.
✓
Test with Realistic Data VolumesSmall sandboxes may hide recalculation risks entirely. Use full-copy or representative sandboxes.
✓
Validate All Automation PathsFlows and Apex triggers often assume record visibility — validate every execution path.
✓
Have a Tested Rollback PlanRollback must be tested and packaged in advance, not discussed verbally during an incident.
✓
Monitor Sharing Recalculation JobsTrack queue depth, completion time, and failure states in real time during the change window.
✓
Require Written ApprovalsArchitectural changes to security models need formal, documented signoff from all teams.

✗ Don'ts

✗
Change OWD Directly in ProductionNever perform a "quick fix" to security settings directly in your production org.
✗
Skip Sandbox ValidationThis is the single most common and most damaging anti-pattern. Never skip it.
✗
Assume Sharing Rules Compensate CorrectlySharing rules must be explicitly validated — never assumed to cover the new access gaps.
✗
Bundle OWD Changes with Feature ReleasesIsolation improves troubleshooting and rollback scope dramatically.
✗
Ignore Sharing Recalculation TimeLarge orgs can require hours to recalculate — never schedule without a duration estimate.
✗
Let Developers Execute Without GovernanceOWD changes are architecture decisions — they require multi-team governance, not dev discretion.

Common Anti-Patterns

Patterns to recognize, refuse, and remediate

Anti-Pattern 1 — Convenience-Based Public Access
Setting OWD to Public Read/Write to avoid building proper sharing architecture. Introduces uncontrolled data exposure, becomes permanent, and breaks compliance posture over time.
✅ Correct approach: Fix the sharing architecture properly — invest in role hierarchy, sharing rules, and Apex sharing rather than using a broad OWD as a shortcut.
Anti-Pattern 2 — Production-First Testing
Changing OWD in production before sandbox rehearsal. Sharing recalculation can cascade unpredictably, and failures are extremely difficult to reverse under production load.
✅ Correct approach: Full sandbox rehearsal first — measure duration, capture evidence, confirm behavior before any production execution.
Anti-Pattern 3 — Missing Negative Security Tests
Testing only confirms intended access (positive cases) without validating that users who should NOT have access are genuinely denied. This misses real security gaps.
✅ Correct approach: Explicit deny-case validation for every persona that must be restricted. Negative tests are mandatory, not optional.
Anti-Pattern 4 — No Rollback Automation
Rollback planning is done only verbally in standup. When a P1 issue occurs, teams discover the rollback has never been tested and requires manual steps under pressure.
✅ Correct approach: A documented, tested rollback deployment package — validated in sandbox — is a hard requirement before production execution.
Anti-Pattern 5 — Ignoring Downstream Systems
Integrations, data warehouses, and external apps often depend on specific visibility assumptions. An OWD change can silently break data pipelines and reports.
✅ Correct approach: Include integration owners in the planning and validation process. Integration health must be verified before post-deployment signoff.

Best Practices

What high-performing teams consistently do

✓ Treat OWD as Architecture, Not Config

OWD changes warrant the same rigor as data model redesigns, authentication changes, or infrastructure modifications.

Comparable rigor level: ✓ Data model redesign ✓ Auth model changes ✓ Integration security ✓ Prod infrastructure

✓ Estimate Recalculation Before Scheduling

Run a rehearsal in a representative sandbox and extrapolate duration to production data volumes.

Sandbox: 120K records → 38 min Scale factor: 1.1M prod records Estimated prod duration: ~3.5 hrs Schedule buffer: 6 hr window

✓ Bundle Compensating Sharing Rules

Deploy compensating Sharing Rules in the same change set to prevent access gaps from occurring between the OWD change and rule activation.

Change Set Contents: [1] OWD Metadata (Security Settings) [2] SharingRules: Account [3] SharingRules: Opportunity → Deploy atomically, not sequentially

✓ Communicate Proactively to Business

Users will notice access changes. Proactive communication prevents a wave of support tickets and escalations.

Notify: change start time Notify: expected duration Notify: user-visible impact Notify: escalation contact Notify: completion confirmation

✓ Test Integration Health Post-Change

Run integration health checks immediately after the change window to confirm external systems still receive expected data.

Health Check Targets: □ ERP data sync (Account) □ BI/Analytics pipeline (Opp) □ Marketing platform (Lead) □ Partner portal (Case)

✓ Maintain a Governance Artifact

Every OWD change should produce a complete governance artifact for audit trails and future reference.

Required artifacts: □ Architectural review doc □ Sandbox execution evidence □ QA regression evidence □ Rollback package (tested) □ COE approval record □ Post-deploy signoff

Recommended Governance Standard

Required artifacts — if any are missing, deployment must not proceed

Final Principle

OWD changes are not simple configuration updates. They are enterprise security architecture changes. Treat them with the same rigor as foundational infrastructure decisions.

  • Data model redesign
  • Authentication model changes
  • Integration security changes
  • Production infrastructure modifications

A disciplined, collaborative, and well-orchestrated OWD deployment protects both business continuity and organizational trust.
