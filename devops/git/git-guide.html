Level 01

Basic Concepts

01

Git is a distributed version control system. Understanding its core model — the three-tree architecture, commits, branches, and remotes — is the foundation of everything else.

🌳
Git Three-Tree Architecture
How files move between the Working Directory, Staging Area (Index), and Repository
🗂️

Repository Setup

Initializing a local repository or cloning an existing one — the starting point for every Git project.

git init — create a new repo in current folder
git clone — copy a remote repo locally
.gitignore — exclude files from tracking
git config — set name, email, editor
git init my-project cd my-project git config user.name "Jane Dev" git config user.email "jane@acme.com" # Clone from GitHub git clone https://github.com/org/repo.git # .gitignore for Salesforce DX node_modules/ .sfdx/ *.dup
📸

Staging & Committing

Moving changes from your working directory through the staging area into a permanent commit in the repository.

git status — see what's changed
git add <file> or git add . — stage changes
git commit -m "message" — create a commit
git log --oneline — view history
git status # On branch feature/US-0042 # Changes not staged for commit: # modified: force-app/main/default/classes/AccountService.cls git add force-app/main/default/classes/AccountService.cls git commit -m "feat: add discount calculation to AccountService" git log --oneline -5 # a1b2c3d feat: add discount calculation # 9f8e7d6 fix: correct null check in trigger
🌿

Branches

Branches let you develop features in isolation. They're cheap pointers to commits — creating one takes microseconds.

git branch — list all branches
git checkout -b <name> — create & switch
git switch <branch> — switch branch (modern)
git branch -d <name> — delete merged branch
# Create and switch in one step git checkout -b feature/US-0042 # or using newer syntax: git switch -c feature/US-0042 # List all branches (local + remote) git branch -a # Delete after merging git branch -d feature/US-0042 # Force delete unmerged branch (careful!) git branch -D feature/abandoned
🔗

Remotes & Syncing

Connecting your local repository to a remote (GitHub, GitLab, Bitbucket) and keeping them in sync.

git remote add origin <url>
git push origin <branch> — push to remote
git fetch — download without merging
git pull — fetch + merge
# First-time push of new branch git push -u origin feature/US-0042 # -u sets upstream tracking # Pull latest from main git fetch origin git merge origin/main # Pull = fetch + merge in one command git pull origin main # See all remotes git remote -v
🔍

Viewing History & Diffs

Understanding what changed, when, and by whom using Git's powerful history inspection tools.

git log --oneline --graph --all
git diff — unstaged changes
git diff --staged — staged changes
git show <commit> — inspect a commit
# Visual branch graph git log --oneline --graph --all --decorate # Diff between two branches git diff main..feature/US-0042 # Who changed this line? (blame) git blame force-app/main/default/classes/AccountService.cls # Diff between two commits git diff a1b2c3d 9f8e7d6
↩️

Undoing Changes

Safely reverting changes at different stages — from unstaged edits to committed history.

git restore <file> — discard working dir change
git restore --staged — unstage a file
git revert <commit> — safe undo with new commit
git reset --soft HEAD~1 — undo last commit, keep staged
# Discard unstaged file changes git restore AccountService.cls # Unstage (but keep local changes) git restore --staged AccountService.cls # Safe undo — creates a new "revert" commit # Use this on shared branches! git revert a1b2c3d # Undo last commit (keep changes staged) git reset --soft HEAD~1 # ⚠️ Nuclear option — discard ALL local changes git reset --hard HEAD

⚠️ Use git reset --hard with extreme caution — it permanently discards uncommitted work. Prefer git revert on shared branches.

Level 02

Intermediate Topics

02

Once you're comfortable with commits and branches, the next level is mastering merges, rebasing, stashing, tagging, and working effectively in a team context.

🔀

Merging Strategies

Different ways to combine branches — each with trade-offs for history clarity and conflict risk.

Fast-forward merge — linear, no merge commit
3-way merge — creates a merge commit
--squash — combine commits into one
--no-ff — force merge commit even if FF possible
# Standard merge (creates merge commit) git checkout main git merge feature/US-0042 # Squash — all feature commits become one git merge --squash feature/US-0042 git commit -m "feat: US-0042 Discount field" # No fast-forward — always create merge commit git merge --no-ff feature/US-0042 # Abort a merge mid-conflict git merge --abort
🔁

Rebasing

Rebase replays your commits on top of another branch — giving you a cleaner, linear history without merge commits.

git rebase <branch> — replay onto another branch
git rebase --interactive — reorder, squash, edit
git rebase --continue / --abort
Golden rule: never rebase shared/public branches
# Rebase feature onto latest main git checkout feature/US-0042 git rebase main # Interactive rebase — clean up last 3 commits git rebase -i HEAD~3 # Opens editor: # pick a1b2c3d add field # squash 9f8e7d6 fix typo ← combines into above # reword c3d4e5f update layout ← edit message # After resolving conflict during rebase: git add . git rebase --continue # Bail out entirely git rebase --abort

⚠️ Never rebase branches that others are working on — it rewrites commit hashes and causes chaos for collaborators.

⚔️

Conflict Resolution

Merge conflicts happen when two branches change the same lines. Git marks them with conflict markers that you must resolve manually.

Identify conflicted files with git status
Understand <<<< HEAD / ==== / >>>> markers
Resolve, stage, then commit
Use a merge tool: git mergetool
# What a conflict looks like in a file: <<<<<<< HEAD (your branch) public class AccountService { public static Decimal getDiscount(Id aid) { ======= public class AccountService { public static Decimal getDiscountRate(Id aid) { >>>>>>> feature/US-0043 (incoming) # Steps to resolve: # 1. Edit the file — keep what you want # 2. Remove all conflict markers <<< === >>> # 3. Stage the resolved file git add force-app/main/default/classes/AccountService.cls # 4. Finish the merge git commit
📦

Stashing

Temporarily shelve changes so you can switch context without committing incomplete work.

git stash — save current changes
git stash pop — restore last stash
git stash list — view all stashes
git stash apply stash@{n} — apply specific stash
# You're mid-feature and need to fix a bug NOW git stash push -m "WIP: discount field changes" # Switch to hotfix git checkout -b hotfix/null-check # ... fix bug, commit, merge ... # Come back and restore your work git checkout feature/US-0042 git stash pop # View all stashes git stash list # stash@{0}: WIP: discount field changes # stash@{1}: WIP: layout changes # Drop a stash you don't need git stash drop stash@{1}
🏷️

Tags & Releases

Tags mark specific commits as important milestones — typically used for release versions.

Lightweight tags — just a name pointer
Annotated tags — include message & metadata
git push origin --tags — push tags to remote
git tag -d — delete a local tag
# Create annotated tag for a release git tag -a v2.4.0 -m "Spring '25 Release Wave 1" # Push tag to remote git push origin v2.4.0 # Push all tags at once git push origin --tags # List all tags git tag -l "v2.*" # Checkout a specific tag (detached HEAD) git checkout v2.4.0 # Delete a tag (local + remote) git tag -d v2.3.9 git push origin --delete v2.3.9
🍒

Cherry-Picking

Apply a specific commit from one branch to another without merging the entire branch.

git cherry-pick <commit-hash>
Cherry-pick a range of commits
--no-commit — stage without committing
Common use: backporting a hotfix
# Hotfix was committed on main: a1b2c3d # You need it on the release/v2.3 branch too git checkout release/v2.3 git cherry-pick a1b2c3d # Cherry-pick a range (exclusive..inclusive) git cherry-pick a1b2c3d..f9e8d7c # Apply without committing (inspect first) git cherry-pick --no-commit a1b2c3d git diff --staged # review before committing git commit

Cherry-pick copies a commit — the original commit and the new one will have different hashes even if the content is identical.

Advanced Topics

03

Power-user Git: understanding the object model, reflog recovery, hooks, submodules, bisect debugging, and writing clean commit history at scale.

🔬

Git Object Model

Git stores everything as content-addressed objects: blobs (files), trees (directories), commits, and tags. Understanding this unlocks everything.

Blobs — file content (SHA-1 hash)
Trees — directory structure
Commits — point to tree + parent(s)
Refs — branches/tags are just pointers to commits
# Inspect any git object by its SHA git cat-file -t a1b2c3d # → "commit" git cat-file -p a1b2c3d # → commit contents # See the tree of HEAD git ls-tree HEAD # See the tree of a specific path git ls-tree HEAD -- force-app/ # Count all objects in the repo git count-objects -v # Where does a branch pointer live? cat .git/refs/heads/main # → a1b2c3d4e5f6... (just a SHA)
🛡️

Reflog — Recovery Superpower

The reflog is Git's safety net — it records every movement of HEAD, letting you recover "lost" commits after resets or bad rebases.

git reflog — see full HEAD movement history
Recover after accidental git reset --hard
Reflog entries expire after 90 days (default)
git checkout HEAD@{n} — go to any past state
# You accidentally ran: git reset --hard HEAD~3 # Panic! 3 commits gone. # Step 1: Find the lost commit in reflog git reflog # HEAD@{0}: reset: moving to HEAD~3 # HEAD@{1}: commit: feat: add discount UI # HEAD@{2}: commit: feat: add discount logic # HEAD@{3}: commit: fix: null check # Step 2: Restore to before the reset git reset --hard HEAD@{3} # All 3 commits are back! # Or create a new branch from the lost state git checkout -b recovery HEAD@{1}
🪝

Git Hooks

Scripts that run automatically at key Git lifecycle events — enforce commit message formats, run tests, or block bad pushes.

pre-commit — run linting before commit
commit-msg — enforce message format
pre-push — run tests before pushing
Hooks live in .git/hooks/ — not version-controlled
#!/bin/sh # .git/hooks/commit-msg # Enforce: feat|fix|chore|docs: message MSG=$(cat "$1") PATTERN="^(feat|fix|chore|docs|refactor|test)(\(.+\))?: .+" if ! echo "$MSG" | grep -qE "$PATTERN"; then echo "❌ Bad commit message!" echo " Use: feat: description" echo " Got: $MSG" exit 1 fi # Make executable: chmod +x .git/hooks/commit-msg

Use Husky (npm) to version-control hooks and share them across your team — hooks in .git/hooks/ aren't pushed to remote.

🐛

Git Bisect — Bug Hunting

Binary search through commit history to find exactly which commit introduced a bug — down to one commit in log₂(N) steps.

git bisect start — begin the hunt
git bisect bad — mark current as broken
git bisect good <commit> — mark known good
git bisect run <script> — fully automate
# Bug exists now, was fine 2 weeks ago git bisect start git bisect bad HEAD # current is broken git bisect good v2.3.0 # this tag was fine # Git checks out middle commit # Test → working → mark good git bisect good # Git checks out another middle commit # Test → broken → mark bad git bisect bad # ... repeat until Git says: # a1b2c3d is the first bad commit # Automate with a test script git bisect run npm test -- --grep "discount" # Always reset when done! git bisect reset
📦

Submodules & Worktrees

Submodules embed one Git repo inside another. Worktrees let you check out multiple branches simultaneously without stashing.

git submodule add — embed a repo as dependency
git submodule update --init --recursive
git worktree add <path> <branch>
git worktree list — see all active worktrees
# Add a shared library as submodule git submodule add https://github.com/org/shared-lib.git libs/shared # Clone a repo that has submodules git clone --recurse-submodules https://github.com/org/main-repo.git # Update all submodules to latest git submodule update --remote --merge # Worktrees: work on hotfix while feature is in progress git worktree add ../hotfix-work hotfix/null-check cd ../hotfix-work # separate folder, same repo # Fix, commit — then return to main workspace git worktree remove ../hotfix-work
✨

Advanced Log & Search

Git's log and search capabilities go far beyond git log — find any change across all history with surgical precision.

git log -S "string" — pickaxe: find when code appeared
git log --follow — track file renames
git grep — search working tree content
git shortlog — summarize commits by author
# Find all commits that added/removed "getDiscount" git log -S "getDiscount" --oneline # Search commit messages git log --grep="US-0042" --oneline # Track a file through renames git log --follow --oneline -- force-app/.../AccountService.cls # Find string across all files (like grep) git grep "getDiscount" HEAD # Rich log format git log --pretty=format:"%h %an %ar %s" --all # Who contributed most commits? git shortlog -sn --all
Diff Tools

Semantic Diff Tools

04

Standard git diff is line-based — it can't understand XML structure or JSON keys. Semantic diff tools understand the meaning of file formats, surfacing structural changes rather than raw text differences.

⚡

FileDiff Pro — Semantic File Comparison

A modern browser-based diff tool that goes beyond line-by-line comparison. It supports Text mode, XML Semantic diff (element-level structural comparison), and JSON Semantic diff (deep key-by-key structural comparison) — with syntax highlighting, drag & drop, and inline editing.

Try it live: mchinnappan100.github.io → FileDiff Pro →

🔤 Text Diff Mode

Line-by-line comparison — same as git diff but in a visual side-by-side UI. Best for Apex classes, JavaScript, and plain text files.

Original: Modified: public static void public static Decimal process(Id aid) { getDiscount(Id aid) { Account a = [ Account a = [ SELECT Name SELECT Name, FROM Account Discount__c WHERE Id=:aid]; FROM Account } WHERE Id=:aid]; return a.Discount__c; }

Changed lines highlighted in red (removed) and green (added).

🧩 XML Semantic Diff

Understands XML structure — compares elements and attributes, not raw text lines. Ideal for Salesforce metadata (page layouts, profiles, permission sets, flows).

Original XML: <Profile> <userPermissions> <enabled>false</enabled> <name>EditReadonlyFields</name> </userPermissions> </Profile> Modified XML: <Profile> <userPermissions> <enabled>true</enabled> ← changed <name>EditReadonlyFields</name> </userPermissions> </Profile> Semantic result: element /Profile/userPermissions /enabled value changed: false → true

🗝️ JSON Semantic Diff

Key-by-key structural comparison — understands JSON objects and arrays, not just text. Perfect for package.json, sfdx-project.json, and metadata JSON files.

# sfdx-project.json — Original: { "packageDirectories": [{ "path": "force-app", "versionNumber": "2.3.0.NEXT" }] } # Modified: { "packageDirectories": [{ "path": "force-app", "versionNumber": "2.4.0.NEXT" ← changed }], "namespace": "myorg" ← added } # Semantic output: CHANGED: packageDirectories[0].versionNumber 2.3.0.NEXT → 2.4.0.NEXT ADDED: namespace = "myorg"

🔍 When to Use Semantic Diff in Salesforce

Semantic diff is essential for comparing Salesforce metadata files where the same content can appear in different element orders.

1Permission Set XML — detect added/removed permissions without false positives from reordering
2Page Layout XML — compare field positions and sections structurally
3Flow metadata XML — spot changed node types without noise from whitespace
4sfdx-project.json — verify package version bumps between branches
5Profile XML — audit permission changes during a security review
🆚

git diff for Salesforce Metadata

Using git diff effectively on Salesforce DX project structures to compare metadata between branches or commits.

Compare metadata between two branches
Filter diff by metadata type
Generate a list of changed files for deployment
Diff a specific User Story branch vs main
# What changed on this feature branch vs main? git diff main..feature/US-0042 --name-only # Only Apex class changes git diff main..feature/US-0042 -- \ "force-app/main/default/classes/**" # Only layout changes git diff main..feature/US-0042 -- \ "force-app/main/default/layouts/**" # Generate deployment manifest from diff git diff main..feature/US-0042 \ --name-only | grep "\.cls$"
🧰

Configuring a Diff Tool

Set up an external diff tool (VS Code, Beyond Compare, or a semantic tool) as your default for git difftool.

git difftool — open diffs in external tool
git mergetool — use GUI for conflict resolution
Configure VS Code as default diff tool
Custom difftool for XML/JSON files
# Set VS Code as diff and merge tool git config --global diff.tool vscode git config --global difftool.vscode.cmd \ 'code --wait --diff $LOCAL $REMOTE' git config --global merge.tool vscode git config --global mergetool.vscode.cmd \ 'code --wait $MERGED' # Use it git difftool main..feature/US-0042 # For XML files — custom semantic diff command # pointing at FileDiff Pro or xmldiff CLI: git config diff.xml.command "xmldiff" echo "*.xml diff=xml" >> .gitattributes

Best Practices

Git Best Practices

06

Habits that separate good Git users from great ones — applicable whether you're working solo or on a team of 50 Salesforce developers.

✓ Write Meaningful Commit Messages

Follow Conventional Commits: type(scope): short description. This enables automated changelogs and makes history scannable.

feat(account): add discount field to page layout fix(trigger): prevent null pointer on empty contact list docs(readme): update installation steps for DX refactor(service): extract getDiscount into utility class

✓ Never Force-Push to Shared Branches

Force-pushing rewrites history and will break every team member's local clone. Use --force-with-lease as a safer alternative if you must.

❌ git push --force origin main ✅ git push --force-with-lease origin main # Fails if remote has commits you haven't seen

✓ Keep Branches Short-Lived

Feature branches that live longer than a sprint accumulate conflicts and become hard to merge. Aim for branches that can be reviewed in under 400 lines of diff.

❌ feature/q1-overhaul (3 months, 80 files) ✅ feature/US-0042-discount-field (3 days, 4 files) ✅ feature/US-0043-approval-flow (2 days, 6 files)

✓ Use --force-with-lease Not --force

If you need to rewrite history on a branch only you're using, --force-with-lease refuses if the remote has changed since your last fetch — preventing accidental overwrites.

git rebase -i HEAD~3 # clean up commits git push --force-with-lease origin feature/US-0042

✓ Always Pull Before You Push

Fetch the latest changes before pushing your work. This minimizes divergence and makes merges simpler for everyone.

git fetch origin git rebase origin/main # or git merge origin/main git push origin feature/US-0042

✓ Use .gitattributes for Salesforce XML

Tell Git how to handle Salesforce metadata files — ensures consistent line endings and enables custom diff drivers for XML.

# .gitattributes *.cls text eol=lf *.xml text eol=lf diff=xml *.json text eol=lf *.md text eol=lf *.zip binary

✓ Use git add -p for Precise Staging

Stage changes interactively, hunk by hunk. This lets you make clean, focused commits even when your working directory has multiple unrelated changes.

git add -p AccountService.cls # Git asks for each hunk: # (y) stage this hunk # (n) skip this hunk # (s) split into smaller hunks # (e) manually edit the hunk

✓ Tag Every Production Release

Create annotated tags for every production deployment. If you ever need to compare "what changed between two releases," tags make this trivial.

git tag -a v2.4.0 -m "Spring '25 Wave 1" git push origin v2.4.0 # Compare two releases instantly git diff v2.3.0..v2.4.0 --stat
