Basic Concepts

01

GitLab CI/CD is powered by a single .gitlab-ci.yml file at the root of your repository. Understanding pipelines, stages, jobs, runners, and artifacts is the foundation for everything else.

🏗️

📄

Pipeline File Structure

Everything lives in a single .gitlab-ci.yml at the repo root. You define stages, global config, and jobs — all in one file (or split across includes).

Filename: .gitlab-ci.yml (fixed name)
stages: — ordered list of pipeline phases
Job names are user-defined keys
default: block sets global job defaults
# .gitlab-ci.yml — complete minimal pipeline stages: - build - test - deploy default: image: node:20-alpine before_script: - npm ci --cache .npm build-app: stage: build script: - npm run build artifacts: paths: [dist/] run-tests: stage: test script: - npm test deploy-production: stage: deploy script: - ./deploy.sh when: manual

GitLab runs stages sequentially but all jobs within a stage in parallel. The default: block applies to every job, reducing repetition.

⚡

Triggers & Pipeline Sources

GitLab pipelines can be triggered by many sources. You control per-trigger behaviour with rules: or the older only:/except: syntax.

Push to a branch
Merge request creation or update
Schedule (cron) — Settings → CI/CD → Schedules
API trigger / pipeline trigger token
# Modern rules: syntax (preferred) test-job: stage: test script: npm test rules: # Run on MR and push to main - if: '$CI_PIPELINE_SOURCE == "merge_request_event"' - if: '$CI_COMMIT_BRANCH == "main"' nightly-audit: stage: test script: npm audit rules: - if: '$CI_PIPELINE_SOURCE == "schedule"'

CI_PIPELINE_SOURCE is one of GitLab's many predefined CI/CD variables. Others: CI_COMMIT_BRANCH, CI_MERGE_REQUEST_IID, CI_COMMIT_TAG.

🐳

Docker Executor & Images

GitLab Runner's Docker executor runs each job in a fresh container. You set the image globally or per-job, with optional service containers (databases, etc.).

image: — Docker image for the job
services: — sidecar containers (Postgres, Redis)
Global default: image: applies to all jobs
Per-job image overrides the global default
default: image: python:3.12-slim unit-tests: stage: test script: - pip install -r requirements.txt - pytest tests/unit/ integration-tests: stage: test services: - name: postgres:15 alias: db - redis:7 variables: DATABASE_URL: postgresql://postgres@db/testdb REDIS_URL: redis://redis:6379 script: - pytest tests/integration/
📦

Artifacts

Artifacts are files generated by a job, stored in GitLab, and passed to downstream jobs. Unlike GitHub Actions, artifacts in the same pipeline are automatically available to later stages without explicit download steps.

artifacts.paths: — list of files/dirs to keep
artifacts.reports: — structured reports (JUnit, coverage)
artifacts.expire_in: — auto-delete after time
Auto-downloaded by subsequent stage jobs
build: stage: build script: npm run build artifacts: paths: - dist/ expire_in: 1 week test: stage: test script: npm test -- --reporter junit artifacts: when: always # upload even on failure reports: junit: test-results.xml coverage_report: coverage_format: cobertura path: coverage/cobertura.xml

JUnit report artifacts show test results inline in merge requests. Coverage reports show diff coverage annotations on changed lines.

🔐

Variables & Secrets

GitLab CI/CD variables can be set at the project, group, or instance level. Masked variables are hidden in logs; protected variables only run on protected branches/tags.

Settings → CI/CD → Variables — stored encrypted
Predefined variables: $CI_COMMIT_SHA, $CI_JOB_TOKEN
variables: block — define inline job vars
Masked: hidden in logs · Protected: main/tags only
# Global variables for all jobs variables: DOCKER_DRIVER: overlay2 APP_ENV: production deploy: stage: deploy variables: # Job-level override DEPLOY_REGION: us-east-1 script: - echo "Deploying $CI_PROJECT_NAME" - echo "Commit: $CI_COMMIT_SHORT_SHA" - echo "Branch: $CI_COMMIT_REF_NAME" - echo "Actor: $GITLAB_USER_NAME" # $AWS_SECRET_KEY comes from project settings - aws deploy --region $DEPLOY_REGION
🏷️

Runners & Tags

GitLab Runners execute jobs. You assign tags to runners and jobs to control which runner picks up which job — enabling specialised environments.

Shared runners — provided by GitLab.com
Group runners — shared across group projects
Project runners — registered to one project
tags: — match job to a runner with that tag
build-on-shared: stage: build script: npm run build # No tags → picks up shared runner deploy-prod: stage: deploy tags: - production - linux # Only a runner with BOTH tags handles this script: - ./deploy.sh ios-build: stage: build tags: - macos - xcode15 script: - xcodebuild -scheme MyApp

Rules — Conditional Job Execution

rules: is the modern, flexible way to control when a job runs. Each rule is evaluated in order and the first match wins.

if: — evaluate CI/CD variable expression
changes: — run only if specific files changed
exists: — run if a file exists in the repo
when: never — explicitly skip a job
frontend-tests: stage: test script: npm test rules: # Run on MR only if frontend files changed - if: '$CI_PIPELINE_SOURCE == "merge_request_event"' changes: - src/frontend/**/* - package.json # Always run on main push - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH' # Skip on tags - if: '$CI_COMMIT_TAG' when: never
⚡

Caching

Cache persists files between pipeline runs — typically dependency folders like node_modules or .venv. Unlike artifacts, caches are not guaranteed to exist.

cache.key: — unique identifier for the cache
cache.paths: — which directories to cache
cache.policy: pull — download only (read-only)
cache.key.files: — cache key based on file hash
default: cache: key: files: - package-lock.json # cache key = lockfile hash paths: - node_modules/ - .npm/ install: stage: build script: npm ci cache: policy: push # only this job writes cache test: stage: test script: npm test cache: policy: pull # test only reads cache, never writes
🏛️

Environments & Deployments

GitLab tracks deployments to named environments with full history, rollback, and stop actions. Merge requests show environment status and live previews.

environment.name: — named deployment target
environment.url: — link shown in MR
environment.on_stop: — job to tear down env
Deploy boards show per-env deployment status
deploy-review: stage: deploy script: ./deploy-preview.sh $CI_COMMIT_REF_SLUG environment: name: review/$CI_COMMIT_REF_SLUG url: https://$CI_COMMIT_REF_SLUG.preview.myapp.com on_stop: stop-review rules: - if: '$CI_PIPELINE_SOURCE == "merge_request_event"' stop-review: stage: deploy script: ./destroy-preview.sh $CI_COMMIT_REF_SLUG environment: name: review/$CI_COMMIT_REF_SLUG action: stop when: manual

Review apps automatically spin up a preview environment per MR, visible in the MR UI with a direct link — and can be stopped manually when done.

📂

Includes & Job Templates

Split your pipeline across multiple files using include:. Use YAML anchors or extends: to share job configurations without duplication.

include: local: — another file in same repo
include: project: — file from another GitLab project
include: template: — GitLab built-in CI templates
extends: — inherit from a job template
include: # GitLab built-in SAST template - template: Security/SAST.gitlab-ci.yml # Shared pipeline from another project - project: myorg/shared-pipelines ref: main file: /templates/deploy.yml # Local file in same repo - local: .gitlab/ci/test-jobs.yml # Reusable template (hidden with dot prefix) .deploy-template: image: alpine before_script: - apk add curl when: manual deploy-staging: extends: .deploy-template script: ./deploy.sh staging deploy-prod: extends: .deploy-template script: ./deploy.sh production

Jobs prefixed with a dot (.name) are hidden — they don't run directly but can be inherited via extends:.

🔢

Parallel Matrix Builds

Run the same job multiple times with different variable combinations — testing across multiple Node versions, OS types, or environments simultaneously.

parallel.matrix: — define variable combinations
Each combination spawns a separate job instance
Variables accessible via $VARIABLE_NAME
Max 200 parallel jobs per matrix
test-matrix: stage: test image: node:$NODE_VERSION parallel: matrix: - NODE_VERSION: ['18', '20', '22'] OS: [ubuntu, alpine] script: - echo "Testing Node $NODE_VERSION on $OS" - npm test # Creates 6 parallel jobs: # Node 18/ubuntu, 18/alpine, # Node 20/ubuntu, 20/alpine, # Node 22/ubuntu, 22/alpine
🔗

Job Dependencies & needs:

needs: creates a DAG (Directed Acyclic Graph) — jobs start as soon as their dependencies finish, regardless of stage boundaries, enabling true parallel fan-out pipelines.

needs: — start job when listed jobs complete
Breaks stage boundaries — runs out of stage order
needs: [] — start immediately, no dependencies
Download specific job's artifacts via needs
build-frontend: stage: build script: npm run build:frontend build-backend: stage: build script: npm run build:backend # test-fe starts AS SOON AS build-frontend done # doesn't wait for build-backend test-frontend: stage: test needs: [build-frontend] script: npm run test:frontend test-backend: stage: test needs: - job: build-backend artifacts: true # download artifacts too script: npm run test:backend

Dynamic Child Pipelines

A parent pipeline can dynamically generate and trigger a child pipeline YAML at runtime — enabling data-driven, monorepo-aware pipelines where the structure itself is computed, not hardcoded.

Parent job generates a .yml file as an artifact
trigger: include: artifact: spawns child pipeline
Child pipeline runs independently with its own jobs
Ideal for monorepos: build only changed packages
# Parent: .gitlab-ci.yml stages: [generate, child] generate-child-pipeline: stage: generate script: - | CHANGED=$(git diff --name-only HEAD~1 HEAD \ | grep "^packages/" | cut -d/ -f2 | sort -u) echo "stages: [build, test]" > child.yml for pkg in $CHANGED; do echo "${pkg}-build:" >> child.yml echo " stage: build" >> child.yml echo " script: cd packages/$pkg && npm build" >> child.yml done artifacts: paths: [child.yml] trigger-child: stage: child trigger: include: - artifact: child.yml job: generate-child-pipeline strategy: depend # parent waits for child
🌐

Multi-Project Pipelines

Trigger pipelines in other GitLab projects from within your pipeline — enabling cross-service orchestration, microservice deployments, and end-to-end integration testing.

trigger: project: — trigger another project's pipeline
trigger: branch: — target branch in downstream project
strategy: depend — wait for downstream to finish
Pass variables to downstream via variables:
deploy-microservice: stage: deploy trigger: project: myorg/payment-service branch: main strategy: depend # wait for it to complete variables: DEPLOY_ENV: production TRIGGER_SOURCE: $CI_PROJECT_NAME integration-tests: stage: test needs: [deploy-microservice] trigger: project: myorg/integration-test-suite strategy: depend
🔒

GitLab Security Scanning

GitLab Ultimate/Gold includes built-in security scanning templates for SAST, DAST, dependency scanning, secret detection, and container scanning — all integrated into MR pipelines.

SAST — static application security testing
DAST — dynamic testing against a live URL
Dependency scanning — CVE check on packages
Secret detection — scan for leaked credentials
include: # GitLab managed security scan templates - template: Security/SAST.gitlab-ci.yml - template: Security/Dependency-Scanning.gitlab-ci.yml - template: Security/Secret-Detection.gitlab-ci.yml - template: Security/Container-Scanning.gitlab-ci.yml # Override scanner variables variables: SAST_EXCLUDED_PATHS: spec,test,tests,tmp DS_MAX_DEPTH: 5 # DAST — test running app dast: variables: DAST_WEBSITE: https://staging.myapp.com DAST_FULL_SCAN_ENABLED: "true"

Security scan results appear in merge requests as a "Security" section, showing new vulnerabilities introduced by the MR compared to the target branch.

Kubernetes & GitLab Agent

GitLab's Kubernetes Agent (agentk) enables pull-based GitOps deployments without exposing your cluster to the internet, replacing the deprecated certificate-based integration.

Agent installed in cluster via Helm
Pull-based GitOps — cluster polls GitLab
CI_JOB_TOKEN authenticates pipeline to agent
No need to expose kube API to internet
deploy-k8s: stage: deploy image: bitnami/kubectl:latest environment: name: production kubernetes: namespace: production script: # Use GitLab Agent proxy for kubectl - kubectl config use-context $CI_PROJECT_PATH:my-agent - kubectl set image deployment/app app=$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA - kubectl rollout status deployment/app rules: - if: '$CI_COMMIT_BRANCH == "main"'

Compliance Frameworks

GitLab Ultimate allows group-level Compliance Frameworks that enforce required pipeline jobs on all projects in a group — for SOX, HIPAA, and similar compliance requirements.

Group-level pipeline configuration enforced on all projects
Required jobs cannot be overridden by project pipelines
Audit events for pipeline compliance
Push rules, merge approval policies enforced centrally

A compliance pipeline (set at group level) is prepended to every project's pipeline:

# Group compliance pipeline # .gitlab/ci/compliance.yml in compliance project stages: - compliance-check - audit-log security-scan-required: stage: compliance-check allow_failure: false # must pass script: - echo "Running mandatory security scan" - run-sast-scan.sh audit-deployment: stage: audit-log script: - post-to-audit-log.sh \ --project $CI_PROJECT_NAME \ --sha $CI_COMMIT_SHA \ --user $GITLAB_USER_NAME

Salesforce CI/CD with GitLab

Automate Salesforce deployments using GitLab CI/CD — JWT auth, Salesforce CLI, validation on MRs, and full deploy on merge to main.

JWT Connected App auth — no password, no MFA
Validate-only on merge requests, deploy on main merge
Run Apex tests and enforce coverage threshold
Store SF_JWT_KEY and SF_CLIENT_ID as masked variables
stages: [validate, deploy] default: image: node:20-alpine before_script: - npm install -g @salesforce/cli - echo "$SF_JWT_KEY" > server.key - sf org login jwt --client-id $SF_CLIENT_ID --jwt-key-file server.key --username $SF_USERNAME --alias target-org validate-on-mr: stage: validate script: - sf project deploy validate --source-dir force-app --test-level RunLocalTests --target-org target-org rules: - if: '$CI_PIPELINE_SOURCE == "merge_request_event"' deploy-to-org: stage: deploy script: - sf project deploy start --source-dir force-app --test-level RunLocalTests --target-org target-org environment: name: production-salesforce rules: - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH' when: manual # require explicit trigger

Best Practices

Best Practices

05

Habits that keep GitLab CI/CD pipelines fast, secure, maintainable, and cost-efficient — whether you're managing 5 projects or 500.

✓ Use rules: Instead of only:/except:

The modern rules: syntax is more flexible, readable, and avoids the "YAML merge" complexity of the deprecated only:/except:. Migrate all jobs to rules:.

❌ only: - main - merge_requests ✅ rules: - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH' - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'

✓ Use needs: for Faster Pipelines (DAG)

Don't wait for an entire stage to finish before starting the next. Use needs: to start jobs as soon as their direct dependencies complete — dramatically reducing total pipeline time.

test-fe: stage: test needs: [build-fe] # starts immediately when build-fe done # doesn't wait for build-be to finish

✓ Set interruptible: true on CI Jobs

Mark jobs that can be safely cancelled when a newer pipeline starts. This is GitLab's equivalent of GitHub's cancel-in-progress — prevents redundant CI runs on rapid pushes.

test: stage: test interruptible: true # cancel if newer pipeline starts script: npm test deploy: stage: deploy interruptible: false # never interrupt a deploy!

✓ Always Set artifacts: when: always on Test Jobs

Without when: always, test report artifacts are not uploaded when tests fail. But that's exactly when you need them most for debugging.

test: script: pytest --junit-xml=results.xml artifacts: when: always # ← critical! reports: junit: results.xml expire_in: 1 week

✓ Use cache: policy: pull/push Correctly

Only the job that installs dependencies should write the cache (push). All other jobs should read it (pull). This prevents cache corruption and speeds up all jobs.

install: cache: policy: push # only this job writes test: cache: policy: pull # all other jobs read only lint: cache: policy: pull

✓ Pin Docker Image Tags

Using image: node:latest means your pipeline can break silently when a new Node version is released. Always pin to a specific version or SHA digest.

❌ image: node:latest ❌ image: python:3 ✅ image: node:20.11-alpine3.19 ✅ image: python:3.12.2-slim # Even safer — SHA digest ✅ image: node@sha256:abc123...

✓ Use Protected + Masked Variables for Secrets

Always mark secrets as both Protected (only runs on protected branches/tags) and Masked (hidden in logs). Never put secrets in variables: in the YAML file.

❌ variables: AWS_SECRET: "my-secret-key" # visible in YAML! ✅ # Store in Settings → CI/CD → Variables # Mark as: ✅ Protected ✅ Masked # Then use as: $AWS_SECRET in scripts

✓ Extract Shared Config with includes

Don't duplicate job templates across projects. Store shared pipeline logic in a central GitLab project and include it — one change updates every project that uses it.

include: - project: myorg/shared-pipelines ref: main file: - /templates/node-jobs.yml - /templates/deploy-jobs.yml - /templates/security-jobs.yml

✓ Use GitLab's Built-in Security Templates

Don't write your own SAST/dependency scanning scripts. GitLab maintains official templates that integrate with MR security reports — include them with one line.

include: - template: Security/SAST.gitlab-ci.yml - template: Security/Dependency-Scanning.gitlab-ci.yml - template: Security/Secret-Detection.gitlab-ci.yml

✓ Validate .gitlab-ci.yml Before Pushing

Use GitLab's built-in CI Lint tool (CI/CD → Pipelines → Editor) or the API to validate your YAML before pushing. Syntax errors cause the entire pipeline to fail to start.

# Validate via GitLab API curl --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \ --form "content=$(cat .gitlab-ci.yml)" \ "https://gitlab.com/api/v4/ci/lint" # Or use the GitLab CI/CD Editor in the UI # → validates YAML + shows pipeline graph preview

✓ Use Review Apps for MR Preview Environments

Automatically deploy a preview of every MR to a dynamic URL — reviewers can click the environment link directly in the MR to test before approving.

deploy-review: environment: name: review/$CI_COMMIT_REF_SLUG url: https://$CI_COMMIT_REF_SLUG.preview.myapp.com on_stop: stop-review rules: - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'

✓ Set Appropriate expire_in on Artifacts

Artifacts consume storage. Set realistic expiry times — test reports can expire in days, release binaries may need months. Don't default to "never".

build: artifacts: paths: [dist/] expire_in: 1 week # CI artifacts: short release: artifacts: paths: [release.zip] expire_in: 6 months # Release artifacts: longer
