01
Phase One

Sandbox Setup & Preparation

🏗️

What Is a Hotfix Sandbox?

A dedicated, short-lived sandbox environment used exclusively for emergency fixes that must bypass the normal release pipeline.

Partial Copy preferred Short-lived Emergency only
▶ See Example 
Sandbox Type Recommendations:
  Critical bug fix (config/flow): Developer Pro
  Apex / trigger fix:             Partial Copy
  Data-related hotfix:            Full Copy (rare)

Keep it locked to a single purpose:
  ✅  "Fix payment validation bug - Sprint 24"
  ❌  "General sandbox - use for anything"

Result: Clear scope prevents accidental scope creep and keeps refresh time low for the next hotfix.

🔄

Refresh from Production

Always refresh the hotfix sandbox directly from Production immediately before starting work. Never reuse a stale sandbox.

Mandatory step Pre-work checklist
▶ See Example 
Pre-hotfix refresh checklist:
  □ Request sandbox refresh → source: Production
  □ Wait for "Refresh Complete" email
  □ Confirm sandbox metadata version matches Prod
  □ Verify connected apps & named credentials
  □ Disable outbound integrations (see §Integrations)
  □ Log start time + refreshed-by in the incident ticket

Why it matters: A sandbox refreshed 2 weeks ago may be missing recent Prod changes, causing a false-passing fix that breaks on deploy.

🔌

Disable External Integrations

Immediately after refresh, disable or mock all outbound integrations to prevent the sandbox from firing real API calls, emails, or triggers.

Safety critical Data protection
▶ See Example 
# Items to disable / reconfigure in hotfix sandbox:
  ✅ Email deliverability → set to "System Email Only"
  ✅ Named Credentials → point to sandbox endpoints
  ✅ Custom Settings (API URLs) → swap to test URLs
  ✅ Process Builder / Flow notifications → deactivate
  ✅ Scheduled Jobs → suspend in Setup > Scheduled Jobs
  ✅ Platform Events outbound → disable subscribers

Risk avoided: Prevents customer-facing emails, payment calls, or ERP writes from firing against real systems during testing.

👥

Access Control

Restrict hotfix sandbox access to only the people directly involved. A hotfix is not a playground — every extra user increases risk.

Governance Audit trail
▶ See Example 
Recommended access roster:
  Dev Lead (fix author)       → System Administrator
  QA Lead (tester)            → Minimum required profile
  Release Manager (approver)  → Read Only + Deploy
  Incident Manager            → Read Only audit

  ❌ Do NOT grant general developer access
  ❌ Do NOT share login credentials
  ❌ Do NOT let multiple devs modify simultaneously

Tip: Use a dedicated hotfix user login for the sandbox so all changes appear under one traceable identity.

02
Phase Two

Development Process

🎯

Scope the Fix Ruthlessly

A hotfix must address one specific, verified production defect. Reject any scope additions — those go into the next sprint.

Non-negotiable Discipline
▶ See Example 
Incident: INC-4821 — Opportunity close date not saving
  when Probability = 100%

✅ In scope:
   • Fix validation rule on Opportunity object
   • Update associated test class

❌ Out of scope (defer to sprint):
   • "While we're here, also clean up the layout"
   • "Let's also fix the currency display bug"
   • "Can we add the new Status field too?"

Rule: Every out-of-scope item gets logged in the incident ticket with "Deferred — Sprint XX" and nothing more.

📝

Document Every Change

Log each metadata component modified, the reason, and the before/after state in your incident ticket before making the change.

Traceability Rollback ready
▶ See Example 
Hotfix Change Log — INC-4821
  Component: ValidationRule — Opportunity.CloseDate_Required
  Before: ISPICKVAL(StageName, "Closed Won") &&
          ISBLANK(CloseDate)
  After:  ISPICKVAL(StageName, "Closed Won") &&
          ISBLANK(CloseDate) &&
          Probability != 100

  Component: ApexClass — OpportunityTriggerTest
  Change: Added test method testClosedWonHighProbability()
  Coverage before: 81% → after: 88%

Benefit: If the fix needs rolling back at 2 AM, any team member can execute it from the log without hunting through metadata.

🌿

Branch Naming Convention

Use a consistent, predictable branch naming pattern so hotfixes are immediately identifiable in your Git history and never confused with feature work.

Git hygiene Best practice
▶ See Example 
Recommended branch pattern:
  hotfix/{INC-number}/{short-description}

Examples:
  hotfix/INC-4821/opp-closedate-validation
  hotfix/INC-4850/case-email-routing-null-ptr
  hotfix/INC-4901/payment-timeout-increase

  main ← Production baseline
  └── hotfix/INC-4821/opp-closedate-validation
        ↳ Merge PR back to main AND to develop

Do NOT branch from develop or feature branches.

Critical: Hotfix branches must always branch from main (Production baseline), never from a development integration branch.

⚡

Minimize Metadata Footprint

Only retrieve and deploy the exact components that change. Packaging unrelated components risks deploying unintended changes to Production.

Safety Precision
▶ See Example 
# package.xml — keep it surgical:
<?xml version="1.0" encoding="UTF-8"?>
<Package xmlns="...salesforce.com/ant/migrationTool">
  <types>
    <members>Opportunity.CloseDate_Required</members>
    <name>ValidationRule</name>
  </types>
  <types>
    <members>OpportunityTriggerTest</members>
    <name>ApexClass</name>
  </types>
  <version>59.0</version>
</Package>

❌ Never use *  (wildcard) in a hotfix package.xml

Risk: Wildcard deploys have caused unintended overrides of Production configs during hotfixes. Always list components explicitly.

03
Phase Three

Testing & Validation

🧪

Reproduce Before You Fix

Confirm you can reproduce the exact Production defect in the hotfix sandbox before writing a single line of code. If you can't reproduce it, you can't verify the fix.

Mandatory QA discipline
▶ See Example 
Reproduction checklist:
  □ Copy exact steps from incident report
  □ Use same user profile as affected user (not Sys Admin)
  □ Use representative data (clone prod record if needed)
  □ Screenshot/record the failure
  □ Note any error messages verbatim
  □ Confirm: bug reproduced ✅ / not reproduced ❌

If not reproduced:
  → Escalate immediately
  → Do NOT guess a fix and deploy it

Reminder: A fix deployed without confirming reproduction has a high chance of introducing a new bug while leaving the original in place.

✅

Apex Test Requirements

Run the full local test suite even for config-only changes. Maintain ≥75% coverage and ensure all directly related test classes pass with new assertions.

Coverage gate Salesforce required
▶ See Example 
Hotfix test execution order:
  1. Run: Run Specified Tests → OpportunityTriggerTest
     Confirm: new test method passes
     Confirm: existing methods still pass (no regression)

  2. Run: Run All Local Tests
     Confirm: org-wide coverage ≥ 75%
     Confirm: 0 test failures

  3. Document results in incident ticket:
     Tests run: 142  Passed: 142  Failed: 0
     Coverage: 83%  (was 81% before fix)

Note: Salesforce will block deployment to Production if coverage drops below 75%. Don't discover this at deploy time.

🔁

Regression Testing

Test the workflows adjacent to your fix. Hotfixes commonly break edge cases in nearby code paths that weren't on the developer's radar.

High priority Risk reduction
▶ See Example 
For INC-4821 (Opportunity validation fix), test:
  Adjacent workflows to verify no regression:
  □ Create Opp → Stage: Prospecting (normal path)
  □ Create Opp → Stage: Closed Won, Prob 100% (fixed path)
  □ Create Opp → Stage: Closed Won, Prob 90% (edge case)
  □ Opp → Edit CloseDate after creation
  □ Bulk update via Data Loader (10 records)
  □ Trigger via API (integration test if feasible)
  □ Lightning + Classic UI (if Classic still in use)

Pattern: Use a regression test matrix — list scenarios before fixing, not after. Confirmation bias causes missed regressions when writing tests post-fix.

👁️

Independent QA Sign-Off

The person who writes the fix must NOT be the same person who signs off on QA. Fresh eyes catch issues the author is blind to.

Governance Separation of duties
▶ See Example 
QA Sign-Off Template (attach to incident ticket):

  Reviewer: Jane Smith (QA Lead)
  Date / Time: 2025-03-12  14:32 UTC
  Sandbox: HotfixSB01  (refreshed 2025-03-12 09:00)

  Scenarios tested: 7 / 7 passed  ✅
  Regression scenarios: 6 / 6 passed  ✅
  Apex test run: 142 passed, 0 failed  ✅
  Coverage: 83%  ✅

  Sign-off: APPROVED for Production deployment
  Approved by: Jane Smith  |  Release Manager: Tom Nguyen

Governance note: This sign-off document is your change management artifact. Store it with the incident ticket permanently.

04
Phase Four

Production Deployment

📋

Pre-Deployment Checklist

Never begin a hotfix Production deployment without completing a written pre-flight checklist. Verbal checks get forgotten under pressure.

Required Zero shortcuts
▶ See Example 
Hotfix Pre-Deployment Checklist:
  □ Incident ticket approved by Change Manager
  □ QA sign-off document attached to ticket
  □ Rollback plan documented (see rollback card)
  □ Deployment window confirmed (avoid peak hours)
  □ Stakeholders notified (brief maintenance window)
  □ On-call engineer assigned for post-deploy watch
  □ package.xml reviewed — no wildcard, no extras
  □ Apex test strategy confirmed (Run Specified Tests)
  □ Backup of affected Production components exported
  □ War-room channel opened (Slack #hotfix-INC-4821)
🔙

Always Have a Rollback Plan

Before deploying, document exactly how to undo every component change. Test the rollback in the sandbox if time permits.

Critical Risk management
▶ See Example 
Rollback Plan — INC-4821:

  If deployment fails mid-flight:
    → Salesforce will auto-rollback (atomic deploy)

  If post-deploy defect discovered within 1 hour:
    Step 1: Re-activate old ValidationRule from backup
            (stored in /backups/INC-4821/pre-fix.xml)
    Step 2: Deploy backup via VS Code / Workbench
    Step 3: Verify defect re-appears (confirm rollback)
    Step 4: Notify stakeholders: "Rollback complete"

  Estimated rollback time: ~8 minutes
  Rollback owner: Dev Lead + Release Manager

Rule: If your rollback plan takes more than 15 minutes to execute, your fix is too complex for a hotfix — decompose it.

🕐

Deployment Timing Rules

Schedule hotfix deployments to minimize user impact. Even emergencies benefit from a 30-minute window of low traffic.

Timing User impact
▶ See Example 
Deployment window guidelines:
  ✅ Preferred: Off-peak (early morning, lunch, after 6 PM)
  ✅ Acceptable: Any time if Sev1 data corruption risk

  ⚠️  Avoid:
     • Monday morning (peak login volume)
     • End of quarter / month close
     • During active batch jobs (check Scheduled Jobs)
     • During Salesforce maintenance windows

  ❌ Never deploy hotfixes:
     • Friday after 3 PM (no weekend coverage)
     • Major holidays
     • During live customer demos or events
📤

Back-Merge to Development

After a successful Production deployment, immediately merge the hotfix branch back to your development and release branches. Missing this step causes regression on the next sprint release.

Easy to forget Git hygiene
▶ See Example 
# After Production deploy — immediately:
git checkout main
git pull origin main

# Merge hotfix to develop:
git checkout develop
git merge --no-ff hotfix/INC-4821/opp-closedate-validation
git push origin develop

# Merge hotfix to any active release branch:
git checkout release/spring-25
git merge --no-ff hotfix/INC-4821/opp-closedate-validation
git push origin release/spring-25

# Delete the hotfix branch:
git branch -d hotfix/INC-4821/opp-closedate-validation
git push origin --delete hotfix/INC-4821/opp-closedate-validation

Automation tip: Add a CI rule that flags any unmerged hotfix branch older than 2 hours post-deploy as a blocking alert.

✓
Summary

What Great Teams Do

✓ One Hotfix, One Bug

Never bundle multiple fixes into one hotfix deployment. Each incident gets its own sandbox refresh, branch, and deployment. Bundling is the #1 cause of hotfix failures.

❌ HF-032 "Fix payment + fix case routing + tweak layout"
✅ HF-032 "Fix payment timeout — INC-4891"
✅ HF-033 "Fix case routing null ptr — INC-4893"

✓ Refresh From Production Every Time

A hotfix sandbox must be a fresh copy of Production for every incident. A sandbox refreshed 3 days ago may be missing critical Production state.

✓ Disable Integrations Immediately Post-Refresh

The very first action after a refresh is disabling all outbound integrations, email deliverability, and scheduled jobs. Build this into a post-refresh runbook.

Post-refresh automation (Flow or script):
  • Set Email Deliverability → System Email Only
  • Suspend all Scheduled Jobs
  • Set Named Credentials → sandbox endpoints

✓ Never Deploy Without Independent QA

Time pressure is the enemy of quality. Even a 20-minute independent QA review catches the class of issues that cause rollbacks. It's always faster than a post-deploy rollback.

✓ Use Explicit package.xml — No Wildcards

Every hotfix deployment must use an explicit, minimal package.xml listing only affected components. Wildcard (*) deploys in a hotfix context have caused Production incidents at every org that has tried them.

✓ Document Rollback Before You Deploy

Your rollback plan must be written and reviewed before the deployment window opens. "We'll figure it out if something goes wrong" is not a rollback plan.

✓ Back-Merge Within 2 Hours of Prod Deploy

Set a team rule: hotfix branches must be merged back to develop and any active release branches within 2 hours of Production deployment. Track this in your incident ticket.

✓ Retire the Sandbox After Use

Mark the sandbox as "Available for refresh" in your team's sandbox tracker immediately after deployment. Don't let hotfix sandboxes linger and get accidentally reused out of sequence.

Sandbox lifecycle:
  Refresh → Work → Test → Deploy → Retire → (next hotfix) → Refresh

✓ Conduct a Post-Incident Review

Every hotfix should generate a short (15-min) post-incident review: what caused the bug, what the fix was, and what process change prevents recurrence. Log findings in your team wiki.

✓ Monitor Production for 24 Hours Post-Deploy

Assign an on-call engineer to actively monitor error logs, Apex exceptions, and user reports for 24 hours after any hotfix deployment. Silent failures can emerge hours later.

Post-deploy monitoring checklist (first 24h):
  □ Review Apex Exception Logs every 2 hours
  □ Monitor Debug Log for affected user profile
  □ Check relevant Salesforce reports for anomalies
  □ Watch incident channel for user reports

✓ Automate the Checklist

Encode your hotfix pre-deployment checklist as a ticket template in Jira/ServiceNow. Humans skip manual checklists under pressure; templates make compliance the path of least resistance.

✓ Track Hotfix Frequency as a KPI

More than 2 hotfixes per month per team signals a systemic quality problem. Track hotfix rate as a dashboard KPI and use it to prioritize test coverage investment.

Target KPI:
  ≤ 2 hotfixes / team / month
  ≥ 95% hotfix-first-time-success rate
  ≤ 4 hours average hotfix cycle time
