Git Branching Strategy for Salesforce Releases

A clear guide to how Copado manages code across environments — from developer sandboxes to production — using Git branches, promotions, and automated deployments.

Foundations

Key Concepts at a Glance

Before diving into the branching flow, here are the essential building blocks you need to know.

📁

Files & Components

Every Salesforce change — an Apex class, a flow, a page layout — is stored as a file in your Git repository in source format.

📌

Git Commit

A snapshot of a change. When a developer saves work to Git, a commit is recorded with a description of what changed and when.

🌿

Git Branch

A separate line of development. Branches allow multiple people to work in isolation without interfering with each other's code.

🔀

Git Merge

The act of combining changes from one branch into another, keeping environments in sync as work progresses through the pipeline.

🏛️

Git Repository

The central store of all file versions, commits, and branches. It can be remote (cloud-hosted) or local (on a developer's machine).

✅

Pull Request

A peer-review step before merging. Another team member reviews the code changes and approves them before they're combined.

Branch Architecture

The Five Types of Branches

Copado uses a structured set of branches that mirror your Salesforce environments. Each branch has a specific role in the release lifecycle.

Branch Hierarchy & Environments
Each environment in Salesforce maps to a branch in Git. Branches are created from their parent — a Developer branch comes from Integration, not from Master.
How Deployments Work

The Promotion Flow

A promotion is Copado's automated mechanism to move user story changes from one environment to the next. Here's what happens under the hood.

Dev → UAT Promotion
When a developer's work is ready, Copado creates a temporary promotion branch and safely merges the changes before deploying.
UAT → Production Promotion
At release time, the release manager selects which user stories to include. Copado creates a promotion branch from Master and merges only the approved feature branches.
End-to-End Process

From Development to Production

Follow a user story as it travels through the full Copado release pipeline, step by step.

👩‍💻 Developer Workflow

1

Create a Feature Branch

Copado automatically creates a feature branch tied to the user story, based off the developer's sandbox branch.

2

Commit Changes

The developer makes changes in their Salesforce sandbox. Copado retrieves the metadata and commits it to the feature branch.

3

Pull Request Review

A pull request is raised against the destination branch. A team member reviews and approves the changes.

4

Merge Feature Branch

Once approved, the feature branch is merged — first into a temporary promotion branch, then into the destination environment branch.

📦 Release Manager Workflow

1

Create a Promotion

The release manager selects one or more approved user stories and initiates a promotion in Copado.

2

Promotion Branch Created

Copado creates a promotion branch forked from the destination environment's branch (e.g., UAT or Master).

3

Deploy & Validate

The promotion branch is deployed to the target org. Automated tests run. If deployment fails, the affected user story can simply be removed and retried.

4

Auto-Merge on Success

On a successful deployment, Copado automatically merges the promotion branch into the destination branch, keeping Git in sync.

Special Scenarios

Back Promotion

After deploying to a higher environment (e.g., Production), a back promotion ensures that lower-environment branches (e.g., Integration, Dev) contain all the latest changes — keeping every developer in sync.

Back Promotion Flow
Changes that reach Production are propagated back down to lower branches so developers always work on top of the latest code.
Why Copado

Benefits of This Approach

Copado automates the most error-prone parts of Git branching so your team can focus on delivering value.

🔒 Isolated Development

Each user story has its own feature branch. Developers work in complete isolation, so one person's changes can never accidentally break another's work.

⚡ Automation

Copado automatically creates feature and promotion branches, handles merges, and follows a consistent naming convention — eliminating tedious manual Git work.

🔍 Full Traceability

Every change is linked to a user story. You can always see exactly what was deployed, when, and by whom — all the way back through Git history.

🛡️ Safe Rollback

If a deployment fails, simply remove the affected user story from the promotion and retry. No messy reverting of commits needed.

✂️ Selective Releases

The release manager can choose exactly which user stories go to production in a given release — even if other work is ready. Deselect and re-run in seconds.

🔄 Continuous Sync

Back promotions ensure every developer's branch stays up to date with what's been released, preventing divergence and merge conflicts down the line.
