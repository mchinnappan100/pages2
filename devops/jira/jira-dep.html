Level 01
Basic Concepts
Understanding the foundation of issue linking and dependency tracking in JIRA.
🔗
What is a Dependency?
A formal relationship between two user stories indicating that one must precede or inform the other.
  • Blocking: Story A must complete before Story B starts
  • Relates to: general non-blocking association
  • Clones: derived or duplicated story link
  • Epic links and parent-child hierarchies
▶ See Example 
US-101: "Set up Auth0 authentication service"
  ↓ is blocked by
US-102: "Provision cloud infrastructure"

Until US-102 is Done, US-101 cannot begin.
JIRA will display the blocker badge on US-101's board card.
Result: the team sees a red 🚫 blocker indicator on US-101 and knows to prioritize infrastructure work first.
🧩
Creating an Issue Link
How to add a dependency relationship between two user stories using JIRA's built-in link dialog.
  • Open the issue → click "Link" button
  • Choose the relationship type from the dropdown
  • Search for and select the target issue
  • Add an optional comment explaining the link
▶ See Example 
Steps to link US-201 to US-200:

1. Open US-201 (Payment Gateway Integration)
2. Click [Link] → "This issue..."
3. Select: "is blocked by"
4. Search: US-200 "User Account Service"
5. Comment: "Needs user token from US-200"
6. Click [Save]
JIRA automatically creates the reverse link on US-200 showing "is blocking US-201".

How JIRA surfaces dependency information on Scrum and Kanban boards to help teams spot blockers.
  • Blocker icon appears on cards with open blockers
  • Backlog view shows dependency counts per story
  • Board settings → enable "Show blockers"
  • Use swimlanes by epic to group related work
▶ See Example 
Board card US-201 displays:

Epic & Parent-Child Hierarchy
Representing structural dependency through JIRA's native hierarchy of Epics, Stories, and Sub-tasks.
  • Epic → Story → Sub-task hierarchy
  • Stories inherit the Epic's timeline context
  • Sub-tasks are hard-dependencies of their parent
  • Epic Link field for cross-board linking
▶ See Example 
Epic: EP-10 "User Onboarding Flow"
  └── Story US-301: "Registration Form"
       ├── Sub-task: Design mockup
       └── Sub-task: API integration
  └── Story US-302: "Email Verification"
       ⬆ is blocked by US-301
  └── Story US-303: "Welcome Dashboard"
       ⬆ is blocked by US-302
The entire chain is visible under one Epic — making sprint planning and release forecasting straightforward.

Level 02
JIRA Link Types In Depth
Every native and custom link type — when to use each and how they appear to the team.
🚫
Blocks / Is Blocked By
Hard sequential dependency — work cannot start or finish until the blocker resolves.
🔁
Clones / Is Cloned By
Derived copy of a story — useful for multi-team or environment-specific duplicates.
🔗
Relates To
Soft non-blocking association — shared context, shared component, or related feature area.
✂️
Splits Into / Split From
Marks a story that was too large and broken into multiple child stories during refinement.
🐛
Caused By / Causes
Links a bug or defect back to the story whose implementation introduced the regression.
🔩
Custom Links
Admin-defined link types for domain-specific relationships (e.g., "Tested By", "Depends On").
🚫
Blocks / Is Blocked By
The most critical link type — represents a hard dependency that prevents progress.
  • Surfaced prominently on the board card
  • Prevents a story from being moved to "In Progress"
  • Flagged in sprint review reports
  • Use for infrastructure, API, and shared service deps
▶ See Example 
US-401: "Build Search Results Page"
  is blocked by US-400: "Implement Search API"

US-400 in Sprint 14, US-401 planned Sprint 15.
JIRA Advanced Roadmap shows a red dependency
arrow between the two story bars on the timeline.

When US-400 → Done:
  JIRA clears the blocker badge on US-401
  Automation fires Slack alert to assignee
🔗
Relates To
Non-blocking soft link — valuable for documenting shared concerns and keeping context connected.
  • No board badge or warning indicator
  • Visible in the "Issue Links" section of the issue
  • Useful for shared UI components or data models
  • Cross-team awareness without hard gating
▶ See Example 
US-501: "Redesign Navigation Menu"
  relates to US-502: "Update Mobile Header"
  relates to US-503: "Accessibility Audit"

All three can proceed in parallel.
Team knows to communicate changes to
shared CSS variables used by all three.
🔩
Custom Link Types
Create domain-specific link types in JIRA admin to capture relationships meaningful to your team.
  • Admin → Issue Linking → Add Link Type
  • Define outward and inward description
  • Examples: "Tested By", "Deployed With", "Risk For"
  • Filterable in JIRA Query Language (JQL)
▶ See Example 
Admin → Issue Linking → Add Link Type:

Name:     Depends On
Outward:  "depends on"
Inward:   "is a dependency of"

Usage:
US-601 "Checkout Flow" depends on US-600
US-600 "is a dependency of" US-601

JQL query:
issueFunction in linkedIssuesOf("US-600", "depends on")
✂️
Splits Into / Split From
Documents story decomposition during refinement — essential for traceability back to original requirements.
  • Original large story links to each split child
  • Original story is typically closed as Won't Do
  • Children inherit parent's epic and labels
  • Stakeholders can trace from requirement to delivery
▶ See Example 
US-700: "User Profile Management" (13 pts)
  → Too large for one sprint. Split during refinement:

  splits into US-701: "View Profile"    (3 pts)
  splits into US-702: "Edit Profile"    (5 pts)
  splits into US-703: "Delete Account"  (5 pts)

US-700 status: Won't Do (replaced by children)

Automation Rules for Deps
Use JIRA Automation to enforce dependency rules, auto-transition issues, and notify teams programmatically.
  • Trigger: "when issue transitions to Done"
  • Condition: "has linked issues of type blocks"
  • Action: update linked issue status or label
  • Action: post Slack/Teams message with context
▶ See Example 
Automation Rule: "Auto-unblock dependents"

TRIGGER: Issue transitioned → Done
CONDITION: Issue has link type "is blocking"
ACTION: For each linked issue...
  • Remove label "blocked"
  • Add label "unblocked"
  • Send Slack: "{{issue.key}} unblocked
    {{linkedIssue.key}} — ready to start!"
  • Log comment on blocked issue
This saves teams from manually hunting down downstream stories every time a blocker resolves.
🏢
Cross-Project Dependencies
When stories in different JIRA projects — or even different boards — have dependencies on each other.
  • JIRA links work across projects natively
  • Prefix issue keys with project code
  • Advanced Roadmaps spans multiple boards
  • Use "Team" custom field for ownership clarity
▶ See Example 
Project MOBILE — Story:
MOB-55: "Display live inventory count"
  is blocked by API-22: "Add /inventory endpoint"

Project API — Story:
API-22: "Add /inventory endpoint"
  is blocking MOB-55, WEB-108

API team sees full downstream impact.
Mobile and Web teams track their blocker
on their own project boards.

JQL for Dependency Queries
Use JIRA Query Language to build filters and dashboards that surface dependency health across the project.
  • Find all blocked stories in active sprints
  • Find all stories blocking others
  • Identify stale blockers older than N days
  • Export results to CSV for stakeholder reports
▶ See Example 
-- All blocked stories in current sprint --
sprint in openSprints()
  AND issueFunction in linkedIssuesOf(
    "project = MYPROJ", "is blocked by"
  )

-- All stories I'm blocking --
issueFunction in linkedIssuesOf(
  "assignee = currentUser()", "is blocking"
)

-- Blockers unresolved > 5 days --
issueLinkType = "blocks"
  AND updated <= -5d
  AND resolution = Unresolved

JIRA REST API for Links
Programmatically create, read, and delete issue links to integrate JIRA with CI/CD pipelines and custom tooling.
  • POST /rest/api/3/issueLink to create a link
  • GET /rest/api/3/issue/{key} — see fields.issuelinks
  • DELETE /rest/api/3/issueLink/{id}
  • Useful for GitHub PR merge → auto-link JIRA stories
▶ See Example 
POST /rest/api/3/issueLink
Authorization: Bearer {token}

{
  "type": {
    "name": "Blocks"
  },
  "inwardIssue": {
    "key": "US-201"
  },
  "outwardIssue": {
    "key": "US-200"
  },
  "comment": {
    "body": "Created by CI pipeline"
  }
}

Best Practices
What Great Teams Do
Proven patterns for healthy, manageable dependency chains in JIRA.
✓ Keep Dependency Chains Short
Aim for no more than 3 levels of sequential blocking per sprint. Long chains mean a single delay ripples through the entire release. 
❌ Bad:  US-1 → US-2 → US-3 → US-4 → US-5 (5 deep)
✅ Good: US-1 → US-2 → US-3
         US-4 (independent)
         US-5 (independent)
✓ Resolve Blockers in the Same Sprint
Never plan a blocked story in Sprint N if its blocker is planned for Sprint N+1. Blockers and their dependents must be co-planned or the blocker de-risked first. 
❌ Bad:  Sprint 14: US-201 (blocked by US-200)
         Sprint 15: US-200 (the blocker!)
✅ Good: Sprint 14: US-200, US-201 (same sprint)
✓ Use Labels to Flag Dependency Risk
Add a "cross-team-dep" or "external-dep" label to any story that depends on another team or external API. Filter these in the backlog for proactive risk management. 
Labels: cross-team-dep, external-dep, blocked
JQL: label = cross-team-dep
        AND sprint in openSprints()
        AND statusCategory != Done
✓ Review Dependency Health in Standups
Add a standing agenda item: "Are any active stories blocked?" Use a saved JIRA filter on the shared screen. Blockers identified late cost exponentially more to resolve. 
Daily standup dashboard filter:
sprint in openSprints()
  AND label = blocked
  AND assignee in membersOf("team-alpha")
✓ Link Before You Estimate
During refinement, add all known dependencies before story pointing. A story's estimate changes if it depends on uncertain upstream work — dependencies inform risk, and risk affects sizing. 
Refinement checklist:
□ Acceptance criteria defined
□ Dependencies linked in JIRA
□ Blockers identified & assigned
□ Story pointed after deps are known
✓ Don't Overuse "Relates To"
"Relates To" should mean shared technical context, not "we talked about this once." Over-linking creates noise and makes it hard to spot real blockers amid a wall of soft links. 
❌ Bad:  US-501 relates to US-502, US-503,
         US-504, US-505 ... (10 links)
✅ Good: US-501 relates to US-502 (shared component)
         All others: no link needed
✓ Use Advanced Roadmaps for PI Planning
During Program Increment (PI) planning, use Advanced Roadmaps to lay out the full 10-sprint dependency graph. Export a PDF for stakeholders. Identify critical-path stories early. 
PI Planning outputs in JIRA:
□ All Stories linked with dependencies
□ Advanced Roadmap timeline exported
□ Critical path stories labeled "PI-critical"
□ Risk register linked to blocker stories
✓ Clean Up Stale Links at Sprint End
At sprint retrospective, audit all "blocks" links. If the blocker resolved but the link wasn't removed, clean it up. Stale links distort dependency reports and dashboard counts. 
Stale link JQL audit:
issueLinkType = "blocks"
  AND status = Done
  AND updated <= -14d

→ Delete links on Done issues monthly
✓ Visualize Before Each Release
Before a release cut, generate a dependency graph (Advanced Roadmaps or a plugin) and walk it with the tech lead. A five-minute graph review prevents multi-day release failures. 
Pre-release dependency checklist:
□ No open blockers on release stories
□ All "depends on" links point to Done issues
□ Cross-team deps confirmed with owners
□ Dependency graph reviewed & approved
✓ Automate Blocker Notifications
Set up JIRA Automation rules so that when a story is blocked, the assignee and their team lead receive an immediate notification — without waiting for standup. 
Automation: "Alert on new blocker"
TRIGGER: Link added (type: blocks)
ACTION: Slack DM to assignee of blocked issue
  "🚫 {{issue.key}} is now blocked by
   {{linkedIssue.key}} — heads up!"
